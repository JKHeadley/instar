## What Changed

The CLAUDE.md template now documents which file actually controls whether a built-in job is enabled — and existing agents receive it via migration.

The `enabled:` line in `.instar/jobs/instar/<slug>.md` looks authoritative and isn't: that markdown is regenerated from the shipped template on every update, so edits there revert silently. The durable setting is `enabled` in `.instar/jobs/schedule/<slug>.json`, which the installer preserves and the loader reads.

## What to Tell Your User

Nothing to do. If you ever asked me to turn a scheduled job on and it seemed to work but wasn't running later, this is why — I was editing a file that gets overwritten on every update. The instructions I read at the start of every session now say which file actually holds the setting.

## Summary of New Capabilities

- Documentation only. No runtime change.

## Evidence

**Observed 2026-07-23 on a live machine:** `enabled: true` written into `.instar/jobs/instar/benchmark-divergence-analysis.md`; ~20 minutes later, after an auto-update, the same line read `enabled: false` again.

**Root cause, confirmed in source rather than inferred:**

- `InstallBuiltinJobs.ts` regenerates the `.md` from the shipped template each run, and reads `existing.enabled` / `existing.disabledAtBodyHash` from `jobs/schedule/<slug>.json` to PRESERVE them.
- `AgentMdJobLoader.ts` gates on the manifest: `if (!manifest.enabled) return`.

So the manifest JSON is both the durable store and the read authority; the markdown's `enabled:` line is only the seed used when no manifest exists yet.

**Cost:** roughly an hour, and a false blocker reported to the operator — having failed at the visible file, I concluded no durable path existed at all and escalated a decision that didn't need making.
