## What Changed

On a personal multi-machine setup with no shared git channel (a `LocalLeaseStore` mesh — the common 2-machine case), each machine knew only the peer's flaky Cloudflare tunnel address and never the peer's fast ropes (Tailscale / LAN). The fast-rope set was meant to cross via a git `commitAndPush` registry sync that a git-less setup doesn't run, so peer `endpoints` never propagated. The lease layer was then forced onto the one flaky Cloudflare rope; a Cloudflare latency spike made the real lease-holder briefly believe it had LOST the lease (`holdsLease=false`), the AutonomousLivenessReconciler saw `blocked-not-owner` and refused to revive a reaped session, and an autonomous session could die silently overnight.

This change propagates each machine's advertised mesh endpoints over the existing **authenticated, signed lease RPC body** (`/api/lease`, `/api/lease/pull`) — the same way `lastKnownUrl` already crosses at pairing — so a git-less mesh learns peer fast ropes structurally. A new `PeerEndpointRecorder` is the single chokepoint that records a peer's endpoints into the local registry (meshTransport-gated, absence-is-a-no-op, idempotent, advisory-only), and a new `MeshEndpointValidator` validates the incoming set per-kind before storage using the SAME host predicates the dial-time resolver uses. The lease/transfer/deliver paths then automatically prefer the fast rope, so a Cloudflare hiccup stops flipping the lease.

## What to Tell Your User

If you run me on more than one machine without a shared git repo (just two personal machines paired over a tunnel), my two machines now actually share their fast network addresses with each other. Before this, they only knew each other's slow/flaky tunnel address, which is what made my "who's in charge" badge flip incorrectly during a network hiccup — and that flip is what let an overnight autonomous session die silently. This makes the fast connections (Tailscale / local wifi) actually get used between machines, so a tunnel blip stops being able to knock a session offline.

## Summary of New Capabilities

- Git-less personal meshes now propagate peer fast-rope endpoints (Tailscale/LAN) over the signed lease RPC channel — multi-transport mesh comms is no longer inert on the exact setup it was designed for.
- Purely additive and `multiMachine.meshTransport`-gated; worst case is unchanged (cloudflare-only) behavior.

## Evidence

- 48 tests green: unit (`MeshEndpointValidator` 16, `PeerEndpointRecorder` 9, `HttpLeaseTransport-mesh` 15), integration (`mesh-endpoint-propagation` 5 — `/api/lease` records endpoints), e2e wiring (`mesh-endpoint-propagation-wiring` 3 — the resolver receives propagated endpoints).
- Real-hardware root cause: laptop registry's `machines[<mini-id>]` had `lastKnownUrl` (cloudflare) and an EMPTY `endpoints` field, while the Mini self-reported `meshEndpoints:['tailscale','lan']` via its own `/health` — proof the peer set never crossed.
