## What Changed

When an agent runs across more than one machine, the machines now talk to each other over **multiple independent connections** — Tailscale, the local network (LAN), and the Cloudflare tunnel — with automatic health-based failover, instead of a single fragile rope. This fixes the root cause of the lease "flap": a single flaky Cloudflare tunnel could make a machine look unreachable, which made the awake-election thrash (observed live: epoch 12748→12753 in 11 minutes).

- **Layers 0-2 — multi-rope hedged failover (ENABLED by default, strictly additive).** Each machine auto-advertises every address it's reachable at (its Tailscale `100.64/10` IP, its primary LAN IPv4, its Cloudflare hostname). A new `PeerEndpointResolver` validates each advertised rope by kind (a loopback / link-local / metadata / wrong-subnet address is dropped, never dialed), caps how many a peer can advertise, and orders them best-first with latency-and-liveness-aware health (a dead rope sinks and is probed on exponential backoff; a slow rope loses its "preferred" status). The lease transport then **hedges**: it tries the best rope first and, if it doesn't answer within 1.5s, fires the rest in parallel — the first rope that genuinely confirms wins. A peer is declared unreachable only when ALL its ropes fail. With no Tailscale and no LAN this is byte-for-byte the old single-rope behavior.
- **Freshness-bound accept-ack (correctness hardening).** A rope counts as "reached" only when the responding machine returns a signed handshake proving it (a) IS the expected peer, (b) folded *our current* lease, and (c) is answering *this* request (an echoed challenge nonce). A bare 200, a stranger on the same wifi, a black-hole proxy, or a replayed handshake is a FAILED rope — never a false confirmation. Request and ack signatures are domain-separated (`mesh-req-v1` / `mesh-ack-v1`) so neither can be replayed as the other.
- **Server bind (live-verify finding).** A multi-machine agent's server now also listens on its Tailscale/LAN interfaces (not just localhost) so peers can reach it on the advertised ropes — found during live-verify before any failover code shipped. This is strictly LESS exposure than the always-on public Cloudflare tunnel it already runs (every route keeps its existing machine-auth / PIN / Bearer). A single-machine agent keeps its localhost bind (gated on `multiMachine.enabled`).
- **Solo-captain hold (⚗️ EXPERIMENTAL, dark by default).** When enabled, a preferred *stationary* captain whose sole peer is provably gone (aged out of the heartbeat window — not merely unreachable for a moment) HOLDS its lease at the same epoch instead of self-suspending and thrashing. Safe-by-construction: it never advances the epoch or takes over a peer's lease, the existing cryptographic fence stays the authority, and a merely-unreachable peer still triggers the conservative self-suspend. Must be live-verified on a real pair (physically sever the peer) before its flag is ever turned on.

Observability: `GET /health → multiMachine.syncStatus.meshEndpoints` lists the rope KINDS this machine advertises (raw IPs stay off this surface). Config lives under `multiMachine.meshTransport`.

## What to Tell Your User

If you run one agent across more than one machine, the machines now have several independent ways to reach each other (Tailscale, local wifi, and the Cloudflare tunnel) and automatically use whichever is healthy — so one flaky connection is no longer a problem and the "who's in charge" election stays steady. This is on automatically and is a no-op on single-machine setups. Installing Tailscale on each machine (and signing into the same account) gives the strongest, most travel-friendly connection — the agent auto-detects and uses it. One further behavior (a stationary machine holding the lease alone when the other is genuinely gone) ships turned OFF and is opt-in after verification on a real two-machine setup.

## Summary of New Capabilities

- Multi-rope mesh transport (Tailscale / LAN / Cloudflare) with hedged health-based failover — ON by default, single-machine no-op.
- Freshness-bound, domain-separated accept-ack — a rope is "reached" only on a verified peer handshake (closes the black-hole / LAN-collision / replay vectors).
- `multiMachine.meshTransport` config (FLAT knobs: `enabled`, `hedgeDelayMs`, per-kind priorities, `tailscaleEnabled`, `lanSubnetGate`, `unhealthyAfterFailures`, `endpointEvictionMs`, `maxProbeBackoffMs`, `bindHost`).
- `leaseSelfHeal.soloCaptainHold` (dark) — a preferred stationary captain holds the lease when its peer is provably gone.
- `GET /health → multiMachine.syncStatus.meshEndpoints` visibility; a multi-machine server binds its Tailscale/LAN interfaces.

## ELI16

When an agent runs on two machines (say a Mac Mini that stays home and a laptop that travels), the machines constantly check in with each other to agree on which one is "in charge." Until now they had exactly ONE way to reach each other — a Cloudflare tunnel, a kind of public internet relay — and that tunnel kept dropping for a few seconds at a time. Every drop made the home machine panic that it had lost the laptop, so it nervously re-grabbed the "in charge" badge over and over. This change gives the machines SEVERAL ropes between them — Tailscale (a private, travel-friendly network), the local wifi when they're in the same place, and the Cloudflare tunnel as a last resort — and they automatically use whichever rope is working. Losing one rope becomes a non-event, and the panicking stops. To make sure a stranger on the same coffee-shop wifi can never be mistaken for the laptop, each machine proves its identity with a signed, replay-proof handshake every time. The sturdy parts are on by default because they only ADD ways to connect; the one part that changes a real decision (letting the home machine keep the badge alone when the laptop is genuinely gone) ships turned OFF until it's proven on a real pair of machines.
