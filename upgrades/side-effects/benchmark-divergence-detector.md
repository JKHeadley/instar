## Summary of the change

Implements the converged + approved spec `docs/specs/benchmark-divergence-detector.md` (4 rounds, codex + gemini cross-model): the benchmark-feedback loop's Increment A. The quality meter (#1458) produces GROUND TRUTH (real production grades per decision-point); INSTAR-Bench produces PREDICTIONS (per-(task, model) pass-rates); nothing compared them, and grade data aged out before anything learned from it. This adds: (1) a `decision_quality_rollup_by_model` parallel table maintained in the SAME transaction as the meter's existing rollup (dual recompute at the annotate chokepoint + reconcile with `GROUP BY model`; `decided_total` from raw DECISION rows, never outcomes; freeze-don't-zero beyond the raw retention floor — now threaded from the operator's real `decisionRetentionDays`); (2) `BenchmarkDivergenceAnalyzer` — a serving-lease-holder-only periodic pass that pool-collects every machine's matured-window aggregates (R2), compares real grade-rates to the mirrored benchmark predictions, and idempotently upserts observe-only findings (every row `advisory: true`) with cause-enumerated verdicts + chronic-streak honesty; (3) a per-machine watermark that is PURE loss-accounting (R1 — a day aging out un-analyzed increments a visible `unanalyzedLoss` counter, never silently); (4) three Bearer routes (503 dark) with a content-free envelope and an FD9-hardened `?scope=pool` merge; (5) config `benchmarkDivergence.*` (ships dark + dryRun), a fleet-dark cadence job template, idempotent migrations + CLAUDE.md template awareness. The static prompt-template extraction (FD6) touched `MessagingToneGate.ts` and `ExternalHogClassifierPrompt.ts` — verified byte-identical runtime prompt output, zero verdict-path change.

## Decision-point inventory

- **Divergence verdict per (decision-point × model)** — add — deterministic ladder over counted aggregates (Wilson intervals, enumerated verdicts `{divergent-worse, divergent-better, aligned, partial, insufficient-evidence, precondition-failed, no-benched-baseline}`); **observe-only** — every finding `advisory: true`; nothing routes, gates, or reconfigures on it (the spec's §Decision-points classifies it `invariant`; the feedback edge into routing remains the explicitly-out-of-scope later increment).
- **Chronic-streak escalation flag** — add — a deterministic counter (per FD8's enumerated semantics), a flag on the finding row only; no notification authority, no action.
- **MessagingToneGate** — pass-through — FD6 prompt-template extraction only; runtime prompt output byte-compared identical; the verdict path untouched (all 12 tone-gate suites + the provenance-enrollment suite pass unmodified).
- **Meter rollup recompute** — modify (mechanism) — the same transaction now maintains a second derived table; recompute-from-raw semantics preserved (supersede-correct, replay-idempotent — tested against a sentinel-superseded wrong grade).

## 1. Over-block

**No block/allow surface — over-block not applicable.** The detector is observability-only end to end: findings are advisory rows behind a dark flag; the routes are reads (plus one lease-gated trigger); nothing gates any message, session, build, or routing decision. The only refusals are structural: a non-holder `POST /analyze` gets a 409 naming the holder (correct-by-design, not an over-block), and dark mode 503s the routes.

## 2. Under-block

Not a blocking feature, but the honesty analogue — "what divergence does it MISS?" — is stated and bounded: (a) the mirror ships ABSENT on this machine (the bench harness physically lives on the benching machine), so until the first operational pull every enrolled pair reports `precondition-failed / stale-mirror` — an operational failure loudly visible (chronic after 3 cycles), never misreported as `no-benched-baseline` or silently skipped; the refresh runbook is registered as ACT-615. (b) A day that ages past by-model retention having never been analyzed increments the `unanalyzedLoss` counter per machine (R1) — data loss is counted, never silent. (c) Holder churn forfeits watermarks with a first-seen re-seed; the churn-gap undercount is accepted and named in the spec.

## 3. Level-of-abstraction fit

Correct layer: the by_model table rides the meter's existing SQLite substrate and its recompute/reconcile machinery (no second source of truth, no event-log/warehouse dependency — the spec argues this trade explicitly); the analyzer reuses the mesh's existing lease check, `PoolPollCache`, and peer-URL guard rather than new coordination; the job rides the standard fleet-dark job-template pattern; migrations ride `PostUpdateMigrator`. Nothing parallel-to an existing mechanism.

## 4. Signal vs authority compliance

**Compliant.** The detector produces signals (advisory findings) that feed the OPERATOR's judgment via the ranked re-evaluation questions — it holds zero authority. The spec's §3.5-equivalent invariant holds: no graded outcome or divergence verdict feeds back into any decision input (model routing, door selection, prompt, floor) — INSTAR-Bench remains the routing authority; the feedback edge is the explicitly-deferred later increment with its own spec `<!-- tracked: ACT-1195 -->` (per the spec's §Non-scope). The lease-gate 409 and dark-503 are structural validators, not judgment.

## 5. Interactions

- **The meter:** the dual recompute lives inside the meter's existing transaction — byte-identical meter behavior with the detector dark, dryRun, or live (three-state store-dump comparison test). The reconcile rebuilds by_model only for days whose raw truth exists (freeze-don't-zero), so the meter's aggressive raw retention cannot silently zero not-yet-analyzed history.
- **The tone gate:** FD6 extraction only; byte-compared prompt output; all 12 suites green. Any behavioral drift would have failed the provenance-enrollment integration test as well.
- **Mesh:** the analyzer runs on the serving-lease holder only (both entry points gated, lease re-checked before the findings upsert to bound a mid-pass flap); offline peers degrade to `partial` coverage honestly; a missing peer never stalls any watermark or retention (verdict-only coupling — tested with all peers offline).
- **No self-action loop:** the cadence tick is the standard job scheduler (fleet-dark manifest); per-pass bounded fan-out under one deadline; ±jitter (delay-only, documented) prevents systematic peer exclusion; the P19 posture is bounded retention knobs + idempotent upserts, no self-triggered retry loops.
- **Prune timer:** by_model + watermark prunes ride the existing 6h prune timer — no new timer.

## 6. External surfaces

Three new Bearer-gated routes (503 dark). The GET envelope is **content-free by test**: it never joins/inlines raw decision context (correlation-id pointers only; a test asserts no payload field), and every finding key is ⊆ a frozen envelope allowlist. The `?scope=pool` merge is FD9-hardened: peer inputs type-clamped (ids/dates/enums/counts), volume-ceilinged with `pool.suspect` classification, and **peer free-text never crosses the pool** — questions are dropped and regenerated locally from the verdict enum; failed/suspect rows carry normalized enums + clamped machine ids only (hostile-peer-text-never-survives is asserted at both the clamp and live-HTTP levels). All queries parameterized. No Telegram/notification surface — findings are read-surface only.

## 7. Multi-machine posture (Cross-Machine Coherence)

Per the spec's converged §Multi-machine posture: aggregates are **collected-on-analyze from every machine** (R2) by the lease holder via the declared peer route (bounded fan-out, `PoolPollCache`-fronted pool reads); the per-machine watermark is honestly **holder-local loss-accounting** (a lease move forfeits it; re-scan ≤ the window is duplicate-safe by idempotency — tested); findings live on the holder and merge pool-wide through FD9 with `(analysisWindow.toDay DESC, lastSeenAt DESC)` ordering so a stale-holder catch-up can never shadow a fresher window. Per-finding `coverage.byMachine` + `dominantMachineShare` make partial collection visible, never silent.

## 8. Rollback cost

**Cheap.** Ships dark (`benchmarkDivergence.enabled` absent ⇒ off; routes 503) AND dryRun-first (a pass computes but writes no detector-owned durable rows — verified by store-dump test). Disabling restores byte-identical behavior except the by_model table maintenance, which is a derived cache: dropping the flag stops the analyzer; the table itself is rebuilt-from-raw at any time (recompute-from-raw semantics), so no data migration or repair on rollback. A full revert is a plain git revert; the migrations are existence-checked/idempotent, and the fleet job manifest ships `enabled:false` so no agent runs anything until deliberately enabled.

**Non-blocking observations** (none rises to a concern; recording for a follow-up hardening pass):
1. `clampPeerFinding` admits `benchedPromptHash` as ANY string sliced to 128 chars and `mirrorCapturedAt` as any ≤40-char string (core :545–547) — length-clamped per the spec's residual-free-text clause, but not hex-/date-clamped the way the mirror loader clamps the same fields (registry :216). An authenticated pool peer can float ≤128 arbitrary chars through the pool merge in those two fields. Tight clamps (`^[0-9a-f]{64}$`, `Date.parse`) would close it cheaply.
2. Peer `retentionEdgeDay` is regex-only (`FD9_DAY_RE`, core :375) — no future-day rejection, so a hostile/buggy authenticated peer advertising e.g. `9999-12-31` can blank the intersected analysis window and degrade every finding for that pass. Observe-only surface, authenticated-peer threat, self-heals next pass without that peer — bounded impact.
3. A jitter-delayed cadence pass (delay up to ~2.4h at the 24h default) re-checks the lease before the findings upsert but performs the FD7 watermark accrual/prune at pass start without a fresh lease check. Watermark rows are machine-local pure loss accounting keyed on THIS machine's id, so a lease flap during the delay cannot corrupt another machine's state — consistent with FD8's re-check-once contract.
4. The migration strips any `benchmarkDivergence.enabled === false` on EVERY run, so an operator's later deliberate `enabled:false` opt-out on a dev agent would be silently removed at the next update. This follows established precedent (dashboard.liveInsights, cartographer, conformanceAudit strips) and `dryRun` is preserved, but the pattern-wide tension with "an explicit value always wins" (types.ts) is worth a registry-level look someday — it is not this PR's invention.
