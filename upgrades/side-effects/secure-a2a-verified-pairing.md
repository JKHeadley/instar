## Increment 1 — SAS derivation foundation (LANDED)
**Files:** `src/threadline/ThreadlineCrypto.ts` (added `deriveSasBits`/`deriveSAS`/`deriveSasFingerprint`/`derivePairingId`/`loadSasWordlist`/`SAS_WORDLIST_SHA256`), `src/threadline/data/sas-wordlist-en.json` (vendored BIP-39 English, 2048 words, sha256 187db04a…), `tests/unit/threadline-sas-derivation.test.ts` (10 tests), `package.json` (ship `src/threadline/data`).

### Phase 1 — Principle check (signal vs authority)
**Does this increment involve a decision point (gate/block/filter/constrain)?** NO. This increment is pure cryptographic derivation — HKDF/SHA-256 over a handshake shared secret producing deterministic values (a 6-word SAS, an 8-byte fingerprint, a 16-byte pairingId). It holds no blocking authority, makes no allow/deny decision, and filters nothing. The blocking authority of the feature (the credential-share gate, the operator-PIN verify) arrives in later increments and will get its own principle check + the Phase-5 second-pass review (it touches "gate"/trust-levels → high-risk). The foundation only produces values those later authorities will consume.

### Phase 4 — Side-effects (this increment)
1. **Over-block:** N/A — no blocking logic in this increment.
2. **Under-block:** N/A — no blocking logic; correctness is the concern, covered by determinism/order-independence/MITM-sensitivity tests.
3. **Level-of-abstraction fit:** Correct layer — these primitives belong in `ThreadlineCrypto` alongside the existing `sign`/`verify`/`ecdh`/`deriveRelayToken`, reusing the same `node:crypto` HKDF the relay token already uses. Domain-separated `info` strings (`threadline-sas-v1`, `threadline-sas-fp-v1`, `threadline-pairing-id-v1`) keep SAS/fingerprint/pairingId/relay-token outputs independent.
4. **Signal vs authority compliance:** Compliant — no authority added here. Pure functions.
5. **Interactions:** None destructive. New functions are additive; no existing symbol changed. The wordlist load is memoized and fails CLOSED on tamper (length≠2048 or sha256 mismatch → throw), so a corrupted wordlist can never silently produce a wrong SAS.
6. **External surfaces:** None yet — nothing is exposed over a route/MCP/message in this increment. The SAS is never transmitted by design; only the fingerprint is loggable.
7. **Multi-machine posture:** The SAS/pairingId are derived from the machine-local handshake shared secret → **machine-local BY DESIGN** (per spec §3.8). This increment introduces no replicated/proxied state. The replicated verified-IDENTITY result is a later increment (dark, `multiMachine.stateSync.threadlinePairing`).
8. **Rollback cost:** Trivial — additive pure functions + a data file behind no flag; nothing calls them yet. Reverting the commit fully removes them with zero migration/state impact.

## Increment 2 — pairing lifecycle + mutual-verified trust source (LANDED)
**Files:** `src/threadline/AgentTrustManager.ts` (trust source `mutual-verified` + op `credential-share` + `markMutualVerified` single-writer + pairing fields on the profile + `recordPendingVerification`/`markVerificationFailed`/`recordPeerAck`/`isCredentialShareAllowed*`), `src/threadline/PairingPendingStore.ts` (machine-local 0600 SAS-pending store), `tests/unit/threadline-pairing-trust.test.ts` (27 tests).

### Phase 1 — Principle check (signal vs authority)
**Decision point?** YES — `isCredentialShareAllowedForProfile` IS an allow/deny decision (it decides whether the `credential-share` operation is permitted for a peer). It is **compliant** with signal-vs-authority: the authority keys on a STRUCTURAL fact (trustSource==='mutual-verified' AND pairingState==='mutual-verified' AND level>=trusted), NOT on brittle/heuristic logic or message content. The source itself is un-self-grantable (only `markMutualVerified` writes it; the generic setter rejects it; unknown sources degrade to un-verified). This is exactly the "blocking authority on a structural fact, not a brittle check" pattern the principle endorses. NOTE: this increment adds the decision LOGIC; the ENFORCEMENT (refusing a send/ingest) is wired in Increment 3 at the relay-send chokepoint — so the feature-level **Phase-5 second-pass review runs before the PR merges**, once enforcement is load-bearing.

### Phase 4 — Side-effects (this increment)
1. **Over-block:** `credential-share` is denied for every peer that is not mutual-verified+trusted — by design; that is the feature. No legitimate non-credential operation is affected (credential-share is a NEW op, absent from all DEFAULT_ALLOWED_OPS).
2. **Under-block:** the decision logic cannot be bypassed via the generic setter (rejects the source) or an unknown source (degrades un-verified); belt-and-suspenders also requires pairingState==='mutual-verified' so a level-only auto-downgrade can't leave a stale grant.
3. **Level-of-abstraction fit:** pairing state lives ON the trust profile (single file, single atomic writer) — no cross-file torn state; SAS words (sensitive, machine-local) live in a SEPARATE 0600 pending store, never mixed into the replicated profile.
4. **Signal vs authority:** compliant (see Phase 1).
5. **Interactions:** purely additive to AgentTrustManager; existing trust suites (AgentTrustManager-fingerprint 23, TrustElevationSource 25, ThreadlineMCPServer-trust 18, +others = 319 touched) all green — no regression. `checkPermission`/`getAllowedOperationsByFingerprint` route credential-share solely through the new gate.
6. **External surfaces:** none yet — no route/MCP/message exposes this in increment 2 (those are increments 3–4).
7. **Multi-machine posture:** pairing state on the profile + the machine-local pending store are **machine-local** in this increment; the replicated verified-identity result is Increment 5 (dark). The pending SAS store is machine-local BY DESIGN (bound to the machine-local shared secret, spec §3.8).
8. **Rollback cost:** additive; nothing enforces yet, all behind no flag. Revert removes it cleanly; the new optional profile fields are forward-compat (unknown source degrades un-verified) so a downgrade-rollback cannot silently grant credential-share.

## Increment 3 — credential-share gate (enforcement) + pair-verify (LANDED)
**Files:** `src/threadline/CredentialShareGate.ts` (new — `assertCanShareCredential`, `evaluateOutboundCredentialShare`, fail-closed), `src/threadline/PairVerifyReceipt.ts` (new — receipt schema+sig+pairingId+sasFingerprint validation, sets `peerAcked` only), `src/threadline/InboundMessageGate.ts` (inbound credential gate @ :267 keyed on sender trust source; pair-verify @ :234 processed before the trust gate but fully validated), `src/server/routes.ts` (outbound chokepoint `refuseCredentialShareIfBlocked` @ :21901 enforced @ :22320 local + :22571 relay), `src/core/types.ts` (`ThreadlineVerifiedPairingConfig`), `src/threadline/ThreadlineBootstrap.ts` + `src/commands/server.ts` (wire ownFingerprint + live config reader), `src/threadline/client/ThreadlineClient.ts` (`hasEncryptedSendPath`), `tests/unit/threadline-credential-gate.test.ts` (24 tests).

### Phase 1 — Principle check (signal vs authority)
**Decision point?** YES — this is the feature's blocking authority (refuse a credential send/ingest). **Compliant:** the refusal keys on a STRUCTURAL identity fact (recipient/sender `mutual-verified` trust source via `isCredentialShareAllowedByFingerprint`) and the encrypted-path availability — NOT on brittle heuristics or message content (FD5). The `kind`/`credentialShare` hint is the caller's declaration of INTENT to send a credential (a fast-path), and the inbound `kind` fast-path is explicitly courtesy, NOT the security boundary (the security decision is sender-trust-source). Fail-CLOSED (FD9). This is "blocking authority on a structural fact," the endorsed pattern.

### Phase 4 — Side-effects (this increment)
1. **Over-block:** A genuine credential to a not-yet-verified peer is refused — by design (the feature's purpose). Non-credential traffic is untouched (a test asserts a normal message passes flag-on AND flag-off). Risk: a caller who legitimately wants to send a credential to an unverified peer is blocked until they pair — intended, and the refusal reason names the fix.
2. **Under-block:** Honest scope (spec §3.5) — this gates the SANCTIONED credential path (`kind:'credential-share'`/the ingestion point), NOT free-text exfiltration (that's the ExternalOperationGate/DLP family). A secret pasted into an ordinary message is not caught structurally here — documented, not silently assumed covered.
3. **Level-of-abstraction fit:** outbound at the single relay-send funnel (the chokepoint every outbound A2A message passes), inbound at the ingestion decision in InboundMessageGate — the right two layers. `assertCanShareCredential` is the agent-facing READ; the GUARANTEE is at the funnel (Structure>Willpower).
4. **Signal vs authority:** compliant (see Phase 1) — structural authority, fail-closed, flag-gated.
5. **Interactions:** Flag OFF ⇒ complete pass-through, byte-identical legacy behavior (credential-share still hits the legacy `insufficient_trust` path exactly as before — tested). When enabled, the credential gate is authoritative for the credential op so it isn't double-blocked. pair-verify is processed before the trust gate but is still subject to schema+Ed25519-sig+pairingId+sasFingerprint+pending-state+size+replay(seenMessageIds)+rate checks (control-plane-exempt ≠ security-exempt). No double-fire: pair-verify and credential-share are distinct `content.type`s.
6. **External surfaces:** the relay-send route now returns HTTP 403 `{refused, reason}` for a blocked credential send — a NEW response only on the credential path with the flag on. No change to non-credential sends. `pair-verify` is a new inbound control message kind.
7. **Multi-machine posture:** the gate reads the LOCAL trust profile's mutual-verified state (machine-local in increments ≤3); Increment 5 adds the dark replication of the verified-identity result so a pairing verified on machine A is honored on B. Until then the gate is correctly machine-local (a credential send is refused on a machine that hasn't verified — fail-closed, never falsely-allowed). The pair-verify receipt binds to the machine-local pending record's `peerIdentityPub`.
8. **Rollback cost:** flag-gated (`threadline.verifiedPairing.enabled` default false) ⇒ default install sees ZERO behavior change; rollback = flip flag off or revert. No persisted-state migration risk (the new config keys default safe; unknown trust source already degrades un-verified from inc 2).

**FD10 dryRun split (verified):** OUTBOUND refusal is LIVE whenever `enabled` (a leak gate has no allow-by-default soak); `dryRun` governs ONLY inbound observability (logs the verdict it WOULD apply). Tested.

### Phase 5 — Second-pass review
Independent reviewer audited the CODE (not the artifact's claims) across 7 failure modes: (1) fail-open — none (every error/unknown-peer/missing-dep path refuses, both send sites :22320+:22571 gated, FD9); (2) dryRun leak — none (outbound never reads dryRun; dryRun is inbound-observability-only, FD10); (3) plaintext leak — refused (`hasEncryptedSendPath` mirrors `sendAuto`'s key check, refuses before sendAuto); (4) pair-verify — fully validated (schema+self-pair+bound-peerIdentityPub Ed25519 sig+pairingId+sasFingerprint+pending-state+replay via seenMessageIds) before any write; sets `peerAcked` only, cannot self-flip mutual-verified (markMutualVerified is sole writer, requires operatorConfirm, unreachable from receipt path); (5) flag-off byte-identical pass-through (credential-share/pair-verify fall to legacy `insufficient_trust`); (6) authority keyed on structural trust fact, not content; (7) no material bypass/shadowing (honest free-text-exfil scope documented + delegated to DLP family). **Verdict: "Concur with the review."**

## Increment 4 — pairing routes (operator-PIN verify) + MCP tool (LANDED)
**Files:** `src/server/routes.ts` (`GET /threadline/pairing`, `GET /threadline/pairing/:peerFp`, `POST /threadline/pairing/:peerFp/verify`), `src/threadline/ThreadlineEndpoints.ts` (`mutualVerifiedCount` on `/threadline/health`), `src/threadline/ThreadlineMCPServer.ts` (`threadline_pair` tool), `tests/integration/threadline-pairing-routes.test.ts` (16), `tests/integration/threadline/pair-mcp-tool.test.ts` (5), + 3 stale tool-count (8→9) fixes.

### Phase 1 — Principle check
**Decision point?** The verify route is the operator-authority that flips a pairing to `mutual-verified`. **Compliant** — it adds NO new gate logic; it WIRES two already-audited authorities: (a) the exact existing `checkMandatePin` operator-PIN auth (routes.ts:7359, the same closure `/mandate/issue` + 7 other operator-only routes use — sha256+timingSafeEqual+per-IP cap) and (b) the inc-2 single-writer `markMutualVerified` (whose un-self-grantable property is inc-2 tested). FD7 (verify = operator PIN, not bearer) is satisfied by reusing the audited PIN gate verbatim.

### Phase 4 — Side-effects (this increment)
1. **Over-block:** none new — reads are bearer-gated (like all routes); the only block is "verify without the PIN → 403" (the FD7 requirement). 2. **Under-block:** the MCP `threadline_pair verify/deny` deliberately CANNOT flip state (MCP can't satisfy the PIN) — it returns `requiresOperatorPin` + the dashboard path, so the PIN gate can't be bypassed via MCP. 3. **Abstraction fit:** routes live in routes.ts (where the trust manager + checkMandatePin are); `mutualVerifiedCount` threaded into the relay router's `/health` via a callback (that router has no trust-manager access). 4. **Signal vs authority:** reuses audited authorities; no brittle logic added. 5. **Interactions:** SAS words on the detail GET are gated by the same sha256+timingSafeEqual PIN compare (non-throwing: absent/wrong PIN omits the words, doesn't 403 the bearer-gated read — §3.9); 16 integration tests prove BOTH sides of the PIN boundary (verify-without-PIN→403+NO flip; verify-with-PIN→flips; match:false→verification-failed). 6. **External surfaces:** 3 new routes + 1 new MCP tool, all 503 when the flag is off; `/health` gains `mutualVerifiedCount` (omitted when no callback → legacy shape preserved). 7. **Multi-machine:** `mutualVerifiedCount` is per-machine in this increment; the pool-scope read is inc 5. 8. **Rollback:** flag-gated; routes 503 when off; revert is clean.

### Phase 5 — Second-pass review
**not-required for inc 4** — and here's the honest rationale (not a skip): inc 4 introduces NO new decision LOGIC. Its authority surface is entirely (a) the already-independently-second-passed credential gate is untouched, (b) the operator-PIN flip reuses the EXACT `checkMandatePin` closure already audited + used by `/mandate/issue`, and (c) `markMutualVerified`'s single-writer/un-self-grantable property was inc-2 unit-tested. The risk an independent reviewer would probe ("can verify flip without the PIN?") is covered by 16 integration tests asserting verify-without-PIN→403+no-flip and verify-wrong-PIN→403+no-flip, which pass. No new brittle authority, so the Phase-5 trigger (new high-risk gate logic) does not fire on this increment.

## Increment 5 — multi-machine replication of the verified-identity result (LANDED, hard-dark)

### Phase 1 — Principle check
**Decision point?** It extends the credential authority across machines. **Compliant** — the inherited record is honored ONLY by pinning `peerIdentityPub` (a mismatch downgrades to pending, never auto-honors); inherited `identity-verified` satisfies only the IDENTITY half; the CHANNEL half (own live encrypted path) is still independently required at the outbound funnel. ### Phase 4 — Side-effects
1–4: **Secret-exclusion is structural** — the replicated record's input type has no SAS/secret/token field; the serializer projects an explicit closed 5-field set (no spread); the receive-side allowlist is the same closed set (smuggled fields dropped, non-`mutual-verified`/non-ISO/non-hex records rejected). 5. **Interactions:** flag-off ⇒ emitter not injected ⇒ strict no-op; tombstone (verification-failed/revoke) un-verifies pool-wide and refuses to resurrect a local verification-failed. 6. **External surfaces:** `?scope=pool` merged read (mirrors `jobsPoolMerge`); a new dark replicated journal kind. 7. **Multi-machine posture:** THIS is the increment that gives the verified-identity result the replicated posture (SAS/secret/token stay machine-local); identity-key pinning prevents a substituted-key peer from inheriting. 8. **Rollback:** hard-dark (`enabled:false` default) ⇒ single-machine + un-enabled agents see ZERO change; the credential-gate's inherited branch is inert until replication is enabled (no records land).

Server emitter wiring confirmed at server.ts:12619 (gated on the enabled flag). **Verdict: "Concur with the review."**

**Honest residual (consistent with all WS2 stores):** the replication APPLY path (a peer's record auto-landing into a peer-origin journal stream and driving `inheritReplicatedVerification`) rides the SAME generic journal-apply rollout stage as relationships/learnings/etc. (own-origin at the union reader today); the consumer/key-pinning logic + the emit/tombstone send-side are fully built + unit-tested.
