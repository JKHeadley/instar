## 1. Over-block

The hook could over-block if it fired falsely on legitimate research-heavy sessions.

**Mitigations:**
- Threshold is 5 consecutive read-only calls. Plenty of room for normal "read three related files then act" workflows.
- The hook NEVER blocks. It returns `decision: "approve"` with an `additionalContext` checklist. The tool call proceeds; only the next prompt sees the nudge.
- Any action tool (Edit, Write, Bash, NotebookEdit) resets the counter. Research-then-act flows trigger at most once per stretch.
- Sessions tracked independently — a slow research session in one tmux pane does not pollute another.

## 2. Under-block

The hook could under-block if read-only loops continued unnoticed.

**Mitigations:**
- 5 is the spec's recommended threshold (from gsd-executor's prompt, battle-tested on thousands of real sessions).
- TaskCreate / TaskUpdate / TaskList are not classified as either read-only or action, so they don't reset the counter falsely. Pure planning still trips the guard.

## 3. Level-of-abstraction fit

- Lives in `.instar/hooks/instar/` alongside `deferral-detector.js`, `external-communication-guard.js`, etc. — same layer, same idiom.
- Hooks into PostToolUse with empty matcher (all tools) — same pattern as `hook-event-reporter.js`.
- Generated by `PostUpdateMigrator.getAnalysisParalysisGuardHook()` — same pattern as every other built-in safety hook.
- State file at `.instar/state/analysis-paralysis-state.json` — same convention as other small ephemeral state files.

## 4. Signal-vs-authority compliance

Per `feedback_signal_vs_authority`: brittle/low-context filters detect signals; only higher-level intelligent gates with full context block.

**Assessment:** Compliant. The hook is a pattern-match counter — pure low-context filter. Its output is a checklist injected as `additionalContext`, never a block. The agent itself (full-context authority) decides whether to act on the nudge.

## 5. Cross-feature interactions

- **deferral-detector.js** — runs on PreToolUse for Bash (communication commands). No overlap.
- **hook-event-reporter.js** — runs on PostToolUse for all tools, posts to `/hooks/events`. The new guard also runs PostToolUse all-tools but writes locally to `.instar/state/`. Two parallel hooks; no shared mutable state.
- **claim-intercept.js / response-review.js** — Stop / Response hooks. Different lifecycle.
- **skill-usage-telemetry.sh** — PostToolUse matching `Skill` only. No overlap.

No conflicts identified.

## 6. Rollback cost

Single hook file generated from a single method on `PostUpdateMigrator`. Single entry in settings template. Single entry in manifest. Single entry in the known-builtin-hooks list. Single test file. No DB migrations. No config changes.

**Net rollback cost: trivial.** Revert the commit; the hook stops generating; existing installed copies become inert (their state file is harmless). Agents re-`upgrade` and the file disappears from their settings.

## 7. Migration parity

- Hook script: lives in `.instar/hooks/instar/` (built-in). Per the Migration Parity Standard, built-in hooks are ALWAYS overwritten on every `installBuiltinHooks()` call — no install-if-missing. Existing agents get the new hook on their next `instar upgrade`.
- settings-template.json: a new entry in `PostToolUse`. The settings migrator merges new entries by hook command — existing agents will receive this new entry on `migrateSettings()` run.
- builtin-manifest.json: a new entry. The manifest is generated fresh, no migration needed.
- known-builtin-hooks list (PostUpdateMigrator.ts:1549): a new entry. This list governs the orphan-cleanup pass, so adding the new hook here prevents it from being moved to `custom/`.

All migration parity boxes checked. Existing agents get full coverage on update.
