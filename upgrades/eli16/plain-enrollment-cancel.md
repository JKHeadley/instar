# Plain enrollment cancellation — ELI16

Starting a subscription account login creates two related things: a durable “pending login” record and a small tmux process waiting for the browser sign-in to finish. Until now the ordinary enrollment API could start, refresh, and complete that login, but it could not cancel it. If the sign-in became stuck, the record and waiting process remained until the normal 15-minute expiry.

This change adds one cancellation endpoint for ordinary subscription enrollment. It first validates the login identifier, then looks up the durable record, and only then derives the tmux process name. That ordering matters because malformed or invented identifiers must never reach process-control code. A pending or expired login is marked abandoned first, making it unavailable to the active enrollment flow, and its tmux process is then killed on a best-effort basis. If process cleanup fails, the durable state is still correct and the next enrollment start already cleans stale panes.

Completed and already-abandoned records are treated as calm, idempotent reads: the endpoint returns success information but does not mutate the store or kill anything. Cancellation also refuses with a retryable conflict while completion for the same login is actively running. Both the existing follow-me endpoint and this new ordinary endpoint call the same cancellation core, preventing their safety ordering from drifting apart. The dashboard button belongs to a separately tracked dashboard change; this increment completes the API lifecycle and agent-awareness surfaces.
