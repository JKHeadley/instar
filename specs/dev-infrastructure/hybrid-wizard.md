# Hybrid wizard — state machine + per-framework intelligence

## Problem statement

First end-to-end test of `npx instar` → "Codex CLI" runtime prompt
against v1.2.11 confirmed two earlier fixes (PR #289 bareword prompt,
PR #299 model-pin) were working: Codex spawned with the right model,
received the wizard skill prompt, and read the skill file. But the
user experience was still broken — Codex parsed the wizard's JSON
context, internally resolved an entry point, and **executed the
entire setup non-interactively** before showing the user any
conversational walkthrough. The skill's behavioral instructions
("speak conversationally", "display this and wait for input") were
treated as a task description by Codex's training rather than a
behavioral contract.

A v1.2.12 attempt (PR #300) routed the wizard through Claude
unconditionally regardless of host framework. Justin rejected that
fix: it would block Codex-only users on a host without Claude
installed. He requested instead a hybrid that provides structure AND
intelligence, with specialized wizard paths per framework.

## Proposed design

A two-layer wizard architecture.

### Layer 1: deterministic state machine (`src/commands/setup-wizard/state-machine.ts`)

The state machine is the structural backbone. Each state is one of:

- `narrative-then-prompt`: an LLM-generated warm intro paragraph
  followed by a structural prompt printed verbatim by instar. The
  user's answer is matched against the state's choice resolver, and
  the transition is deterministic.
- `action`: a side effect (init, user add, server start, autostart,
  Telegram setup, GitHub backup, greeting). Side effects call
  existing instar CLI subcommands or hand off to an agentic
  framework session. LLM does NOT drive side effects.
- `terminal`: the wizard ends. Farewell text mentions that all
  settings (name, personality, autonomy, messaging) can be changed
  later by chatting the agent — no terminal needed.

State machine for the fresh project install (the only entry point
ported in v1.2.12):

```
welcome → agent-name → agent-role → user-name → autonomy
       → [action: init]
       → [action: add-user]
       → messaging (choice)
         ├─ telegram → [action: setup-telegram-agentic]
         ├─ whatsapp → [action: setup-whatsapp-agentic]
         ├─ slack    → [action: setup-slack-agentic]
         └─ skip
       → [action: start-server]
       → [action: install-autostart]
       → [action: send-greeting]
       → complete
```

Restore-flow, multi-user/multi-machine variants, and the WhatsApp/
Slack agentic actions stay on the existing SKILL.md path for now;
they ship as follow-ups in subsequent releases.

### Layer 2: per-framework driver

Two driver implementations consume the state machine:

- **Claude path**: setup.ts continues to spawn Claude with the
  existing `/setup-wizard` slash-command and the full SKILL.md
  context. Claude reliably follows the skill; no reason to break
  what works. This path stays unchanged.

- **Codex path** (`src/commands/setup-wizard/codex-driver.ts`):
  instar drives the state machine turn-by-turn. For each
  `narrative-then-prompt` state, the driver spawns `codex exec -s
  read-only -m gpt-5.3-codex --ephemeral` with a tightly-constrained
  prompt asking Codex to generate ONE warm 2-3 sentence paragraph.
  Codex cannot use tools (read-only sandbox + ephemeral mode +
  prompt explicitly says "no tools"). The structural prompt text is
  printed by instar from the state machine, NOT generated by Codex
  — so Codex cannot reword the question, skip steps, or decide to
  execute the setup. instar reads the user's answer with `readline`
  and advances the state.

- **Action states**: side effects call existing CLI commands via
  `execFileSync('npx', [...])`. Telegram setup is the lone exception
  — it hands off to a full agentic Codex session (with browser
  automation, sandbox bypassed) because driving Playwright + the
  BotFather conversation IS execution, which is Codex's strength.

### Dispatch in setup.ts

```ts
if (framework === 'codex-cli') {
  const { runCodexWizard } = await import('./setup-wizard/codex-driver.js');
  await runCodexWizard({ codexPath, projectDir, instarRoot });
  return;
}
// Claude path: existing /setup-wizard slash-command spawn (unchanged).
```

### Why the per-turn approach beats one-big-session for Codex

- **Bounded scope per call**: each codex exec generates ~30 tokens
  of narrative text. There's no way to "drift" into execution
  because there's nothing to execute and the prompt asks for a
  single paragraph.
- **Read-only sandbox**: even if Codex tried to invoke bash, it
  could not write anything.
- **Ephemeral mode**: no session persistence between turns; each
  turn is independent.
- **Suppressible**: if a narrative call fails or times out, the
  driver prints a deterministic fallback paragraph and continues.
  The wizard is robust to LLM unavailability.

### "Settings can be changed later" UX promise

The farewell text and the autonomy step's narrative both explicitly
tell the user that anything set up during the wizard (name,
personality, autonomy, messaging) can be changed later by just
chatting the agent. The agent is expected to update its own config
in response — that's already an existing instar capability (the
`/dispatch` action set and the agent's own `AGENT.md` self-editing
loop). The wizard just surfaces the affordance.

## Decision points touched

- Adds two new operator-intent SIGNALS (the state machine ID and
  the per-turn narrative prompt). Neither is a routing or
  authorization decision — both are conversation-flow signals.
- AUTHORITY for "what model is acceptable on subscription auth"
  remains with `src/providers/adapters/openai-codex/models.ts`. The
  driver imports `WIZARD_CODEX_MODEL` from a shared constants file.
- No new privilege. The codex narrative spawns use `-s read-only`
  (strictly less than the existing wizard spawn's
  `--dangerously-bypass-approvals-and-sandbox`). The Telegram-
  agentic phase keeps the existing sandbox-bypass posture for
  consistency with how the existing Claude path drives Playwright.

## Open questions

None for the fresh-project-install scope. Two follow-ups noted:

1. Port restore-flow, multi-user, multi-machine entry points into
   the state machine.
2. Add agentic Codex sessions for WhatsApp and Slack setup (today
   v1.2.12 emits a clear "configure later" pointer for those).

## Out of scope

- Restore-flow and multi-user-machine variants. The fresh install
  is the failure mode the v1.2.11 instar-codey log surfaced; that's
  what v1.2.12 fixes.
- A standalone "Codex Telegram-setup" skill file. The driver inlines
  a tight Telegram-only prompt that references the existing wizard
  SKILL.md for detail. Good enough until a real Telegram setup test
  surfaces a refinement.
- Cross-framework state-machine drivers (e.g., a single driver that
  uses whichever framework is available for narrative). The split
  is intentional: Claude's wizard is end-to-end via SKILL.md,
  Codex's is per-turn via state machine. Forcing them through a
  shared driver would compromise the strengths of each.
