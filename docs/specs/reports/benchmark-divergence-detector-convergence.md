## ELI10 Overview

We have a benchmark that predicts how good each AI model should be at each kind of job, and (as of today) a quality meter that records how the agent's real automated decisions actually turned out. This spec adds the piece that compares the two: a small, read-only "divergence detector" that lines up each job's REAL success rate against the benchmark's PREDICTED rate, model by model, and records a finding when they disagree by more than a fair, sample-size-aware margin.

Each finding sorts the disagreement before anyone acts on it: first "can we even trust the benchmark here?" (if it tested a stale prompt, or the benchmark data is old or missing, the finding says THAT — it never blames the model); then, for real underperformance, the operator's three ranked questions (enough context? right prompt? does the benchmark represent reality?); and for suspicious OVER-performance, a different first question — "is the grading too lenient?" — so a gamed score never reads as "promote this model."

It only ever observes. Every finding is stamped advisory; nothing anywhere may auto-act on one. It ships dark on the fleet and in a write-nothing rehearsal mode on the development agent, and the operator's two requirements are structural: no grade data is deleted before its lesson is captured (the lesson is folded into a permanent summary at grading time), and analysis gathers every machine's data — an offline machine makes the result "partial," never a wrong conclusion.

## Original vs Converged

**The retention redesign (round 1, the big one).** The original draft made the quality meter's data-cleanup wait for this detector's progress. Five reviewers independently showed the two default states were both hazardous: detector off (the fleet default) meant the meter's data could never be cleaned up — unbounded growth; rehearsal mode risked marking data "analyzed" while writing nothing. The converged design removes the coupling entirely: per-day, per-model grade tallies live in a new parallel summary table maintained by the meter's own grading machinery, and the detector reads only that. The meter's cleanup is byte-identical whatever the detector does.

**Reading the foundation's real code (round 2).** Reviewers grounded the design against the merged meter implementation and reshaped three load-bearing pieces: the meter's existing summary table has a fixed primary key that SQLite cannot alter — so the per-model counts became a NEW parallel table instead of a risky rebuild of a live one; a "closed" day's grades keep being corrected for up to ~30 days (late evidence, re-grades, a 6-hourly reconcile) — so "analyze each day exactly once" became "re-read a rolling window every pass," meaning late corrections always update the verdict; and the "too much ungraded data to judge" honesty gate turned out to have no data source at all — fixed by counting ALL recorded decisions, not just graded ones. Round 2 also hardened the benchmark side (its predictions carry counts, not just rates, so a tiny test battery can't manufacture a false alarm) and closed a code-execution hazard (the mirror file can name a source path, but the live prompt is now resolved only through a compile-time registry — never by importing whatever path a data file names).

**Tightening to zero (rounds 3–4).** Round 3 verified every round-2 fix landed genuinely and yielded ten one-line tightenings (examples: fully-ungraded days must still weigh in the honesty gate; a machine-switch must not fabricate months of false "data lost" warnings; peer machines' free-text is dropped and regenerated locally so a compromised machine has no text channel into the reader). Round 4 verified all ten and found nothing new; both external models returned only restatements of trade-offs the spec already documents.
