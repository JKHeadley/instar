# Token-Audit Completeness

## Problem statement

Operator directive (Justin, topic 22726, 2026-06-10): **full token auditability — per feature AND per model — is a requirement for any feature, and a precondition before the cartographer freshness sweep (codex-routed) is ever enabled.**

The FeatureMetricsLedger + `/metrics/features` system (docs/specs/llm-feature-metrics-spec.md) already records per-call feature, tokens, latency, model, and framework through the single funnel (`CircuitBreakingIntelligenceProvider.evaluate`). Gaps that keep it short of the directive:

1. **Codex calls are token-blind.** `CodexCliIntelligenceProvider` runs `codex exec` with plain output, which carries no usage block (`src/core/CodexCliIntelligenceProvider.ts:98-103`). Every codex-routed call records `tokensIn/tokensOut = null`. The cartographer freshness sweep routes to codex — the exact feature the operator wants audited would report zero tokens.
2. **Failed calls drop already-burned tokens.** The funnel's error path (`CircuitBreakingIntelligenceProvider.ts:204-210`) records model/framework but not the `usage` variable sitting in scope. A flaky or timeout-prone feature systematically under-reports its true cost — the inversion of auditability.
3. **No per-model token split.** `FeatureRollup` lists distinct `models[]` per feature but aggregates tokens across them.
4. **Attribution is willpower, not structure.** `IntelligenceOptions.attribution` is optional; untagged calls bucket under `unlabeled`. Nothing fails the build when a new LLM-calling feature omits its tag. Real unattributed callsites exist today (e.g. WarrantsReplyGate, CoherenceGate, AutoDispatcher-class gates).
5. **No tripwire when usage parsing rots.** A future codex CLI shape change would silently null tokens again — the exact regression this spec exists to fix, with no alarm.

## Proposed design

### Slice 1 — Codex per-call token usage

#### Invocation

`CodexCliIntelligenceProvider.evaluate` switches from `execFile`-with-buffer to **streaming spawn**, invoking:

```
codex exec --json --output-last-message <abs out-dir>/last-message.txt <existing hygiene args> -
```

- **The streaming spawn helper is a new export of `src/providers/adapters/openai-codex/transport/codexSpawn.ts`** — the single home for codex child mechanics — consumed by the provider; NOT a hand-rolled second spawn path inside the provider (this exact file was missed once in the Spec-12 env audit; the funnel must not fork). Kill sequence on timeout: SIGTERM, then SIGKILL after the house-standard 2s grace.
- **The `--json` invocation is ADDITIVE over the existing argument set.** All clean-call hygiene properties are preserved in both modes, asserted by a both-modes unit test: `--cd <scratch>` (cached mkdtemp 0700 scratch dir), `-c project_doc_max_bytes=0`, `--skip-git-repo-check`, env from `buildCodexChildEnv()` (never `process.env`). (Lessons: CODEX-INTELLIGENCE-PROVIDER-CLEAN-CALL-SPEC.md; Spec 12 Rule 1a.)
- **The prompt moves OFF argv in `--json` mode**: it is written to the child's stdin and stdin is then ended (the positional becomes `-`). Process argv is world-readable (`ps`); cartographer-sweep-class prompts embed repo content, and macOS ARG_MAX (~1 MB) would hard-fail large prompts. The stdin-hygiene property is amended accordingly: *write prompt, then end* (plain mode keeps positional + bare `end()`, unchanged). The `--output-last-message` path is pinned **absolute** (a relative path would resolve against `--cd` — i.e., into the long-lived cached scratch dir, violating the rule below). Both pinned in the args unit test.

**Why spawn, not execFile:** `execFile`'s `maxBuffer` overflow does not truncate — it kills the child and fails the call. The `--json` event stream is strictly larger than today's plain output, so a buffered approach makes long calls — the cartographer-sweep class this directive targets — *more* likely to fail than today. Streaming bounds memory instead:

- stdout uses `setEncoding('utf8')` (or a StringDecoder) so multibyte chars never split; lines are assembled across chunk boundaries via a **carry buffer capped at 2 MB** — an over-cap line is discarded unparsed and counted (usage events are <1 KB, so the cap cannot lose usage; discards increment a counter surfaced to the drift tripwire). The carry is flushed on `close` before final usage accounting. Unit fixtures: a usage event split mid-JSON across two `data` chunks; a single >cap line followed by a usage event.
- **stderr is consumed continuously** via a `data` handler for the child's lifetime (rolling 600-char tail, chunks discarded) — an undrained pipe wedges a chatty-stderr child on the 64 KB OS buffer, a hang class `execFile` did not have. Fixture: ≥256 KB stderr, normal exit, call completes without timeout.
- **stdin error contract:** `child.stdin` gets an `'error'` handler that swallows EPIPE/ERR_STREAM_DESTROYED — the spec's own designed failure paths (old-CLI unknown-flag fast-exit; SIGTERM mid-drain) close the pipe while a cartographer-class multi-MB prompt may still be writing, and an unhandled stream `'error'` is an uncaughtException that takes down the whole agent server. The failure is already surfaced via the non-zero-exit reject path. Fixture: child exits immediately while a ≥1 MB prompt write is in flight → call rejects loudly with the both-levers message, process does not crash.
- **Settlement ordering (timer never settles):** the timeout timer only *initiates* the kill sequence (`options.timeoutMs ?? 30_000`, SIGTERM → 2s → SIGKILL); the promise **settles from the `close` handler** — after the carry flush and after `onUsage` — matching `spawnCodexAndWait`'s reject-at-close house pattern AND `execFile`'s own callback-at-exit semantics. This is what makes "error rows carry cost" actually reach the ledger: the funnel's catch block writes the error row synchronously at rejection, so rejection must happen after the final post-SIGTERM `token_count` flush was parsed, or the most expensive failure class (long timeout-killed sweeps) records null tokens forever. Because `close` can be held back indefinitely by a grandchild keeping the inherited stdout fd open past SIGKILL, a **bounded post-`exit` grace (5s)** applies: at `close`-or-grace, whichever first, the carry is flushed, `onUsage` fires from whatever was parsed, handlers are removed, streams destroyed, and the promise settles — exactly once. Fixtures: timeout-mid-stream with a final `token_count` after SIGTERM → the funnel **error row** (end-to-end, not just the callback) carries the final flushed tokens; child exits but stdout fd held open → accounting completes within the grace.

#### Result extraction — file only

The result is read from the `--output-last-message` file, **and only from there** — stdout events never become the result (injection surface; Signal vs Authority: events are observability signal, the file is the authority).

- **Result = `fileContent.trim()`** — preserving today's `stdout.trim()` semantics so the kill-switch toggles between byte-equivalent result behaviors. An **empty-after-trim file with exit 0 resolves `''`** (matching today's behavior for an empty answer); only a **missing file or non-zero exit** rejects. Mode-equivalence of result semantics is asserted in the both-modes unit test. **Documented asymmetry, decided:** if codex omits the file entirely on an empty answer (file-creation-on-empty-answer is verified once against the real CLI during implementation, on the canary machine), json mode rejects where plain mode resolves `''` — accepted, because the alternative (missing-file+exit-0 resolves `''`) would silently mask `--output-last-message` argument rot, which is the worse failure.
- The file is size-checked before read (`stat`): above 16 MB → loud reject (fail-loudly contract; the old 1 MB ceiling is gone and an unbounded read into a string is the replacement hazard otherwise).
- Reject errors carry a bounded, scrubbed stderr tail only — JSONL payloads are never concatenated wholesale into thrown Errors. When the tail matches an unknown-flag signature, the message names BOTH recovery levers: `intelligence.codexExecJson=false` and `INSTAR_CODEX_EXEC_JSON=0` (see kill-switch wiring — the env lever exists precisely for config-less construction sites, so the error text is honest everywhere it can appear).

**Out-dir lifecycle.** The output file lives in a **fresh per-call `mkdtempSync` directory** (prefix `instar-codex-out-`, 0700, unguessable suffix) — NOT in the long-lived cached scratch dir, whose `existsSync` reuse branch does not re-verify ownership/mode and must never have model output written into or read from it. The provider reads the file post-exit only (the `close` event fires after exit, so a half-written file is impossible on the success path). The per-call dir is removed in a `finally` covering ALL outcomes, **via `SafeFsExecutor`** (L12 — `lint-no-direct-destructive.js` bans raw `rmSync` in `src/`).

**Stale sweep (bounded, P19).** Belt-and-suspenders against crashed processes leaving model output on disk: an opportunistic sweep of `instar-codex-out-*` dirs from `os.tmpdir()`, with ALL of the following brakes, each unit-tested:
- **Off the event loop:** the sweep is fire-and-forget async (`fs.promises`), never awaited by `evaluate()`; `lastSweepAt` is set before the pass starts so concurrent calls cannot double-trigger. (`os.tmpdir()` population scales with the whole machine, not with instar's dirs — a synchronous readdir+lstat pass inside a judgment call would block the single server event loop on foreign state.)
- **Rate floor:** at most once per hour per process (module-level `lastSweepAt` gate — two back-to-back `evaluate()` calls perform exactly one `readdir`).
- **Per-pass caps:** at most 20 deletions AND at most 200 candidates examined (`lstat`-ed) per sweep — the deletion cap alone would not bound the verification pass on a crash-looping machine that accumulates dirs faster than one process drains them.
- **Age threshold 6 hours** (≫ any supported judgment-call timeout), AND an in-process `Set` of in-flight out-dirs that the sweep always skips — so a live long call's dir cannot be deleted from this process, and the 6h threshold makes a cross-process race require a >6h call (none exists; `timeoutMs` callers are bounded well below). The `Set` entry is **removed in the same `finally` that deletes the dir** (all outcomes), with the nesting pinned — `finally { try { safeRm(dir) } finally { set.delete(dir) } }` — so a throwing deletion (EPERM, sandbox-revoked tmp access) cannot leak the entry; fixture asserts the Set is empty after success/reject/timeout AND after a delete-throws case.
- **Per-candidate verification before deletion:** `lstat` — must be a directory, NOT a symlink, owned by the process uid, mtime older than the threshold. `os.tmpdir()` is hostile on Linux (world-writable `/tmp`; the provider's own threat-model comment says so) — a planted symlink or foreign dir is skipped, and EPERM/ENOENT during deletion is swallowed, never thrown into the call path.
- Deletion via `SafeFsExecutor`.

**Audit-volume consequence (engaged, not discovered at CI):** routing per-call `finally` deletions through `SafeFsExecutor` makes `.instar/audit/destructive-ops.jsonl` a hot-path log (~1,550 appends/day on a codex-routed agent) with — today — no rotation anywhere, growing without bound AND drowning the rare wipe-class entries the audit exists to surface. The same PR therefore adds a **size-capped rotation to `appendAuditEntry`** (rotate at 16 MB, keep one predecessor) — benefiting every SafeFs/SafeGit caller, not just this one. **Rotation posture pinned:** rename-based atomic rotation (`current → .1`) with a **re-stat immediately before the rename, skipping if below cap** — without it, two writer processes racing the threshold can double-rotate and the second rename clobbers the FRESHEST just-rotated 16 MB segment (the loss lands on the newest audit history, the worst direction for an audit log; the re-stat shrinks that window to effectively zero, and the live log is never at risk either way). The **first entry of each fresh segment is a rotation-marker entry** (timestamp, aged-out entry count, predecessor name), so "the audit log shrank" stays auditable — legitimate rotation must never be camouflage for audit-log loss (Observation Needs Structure). Fixture covers rotation + marker + the below-cap-skip. L12's guarantee (every destructive op audited) is unweakened; only retention is bounded.

#### Usage extraction — line-wise, shape-validated, never lexical

Usage parsing operates **line-wise via `JSON.parse` with top-level shape validation; substring/regex matching over the raw stream is forbidden** — model content embedded inside event string fields is structurally incapable of matching a top-level parse. (A key-substring *pre-filter that only decides whether to attempt the parse* is permitted — it cannot extract values, so the injection rationale is untouched.) Malformed lines are skipped. Two known shapes, with an explicit precedence rule:

1. **Protocol shape (authoritative when present):** `{"msg":{"type":"token_count","info":{"total_token_usage":{...}}}}` (also matched under a `payload` key). Cumulative — last event wins within this shape. Verified against a real codex 0.136.0 rollout: `input_tokens` includes `cached_input_tokens`; `total_tokens = input_tokens + output_tokens`.
2. **Thread-event shape (only when no protocol event appeared):** `{"type":"turn.completed","usage":{...}}` is per-turn — **summed** across all `turn.completed` events.

Never mixed in one accounting. When BOTH shapes appear in one stream, the protocol total is recorded and a large divergence from the thread-event sum emits the drift event (`reason: shape-divergence`) — a free consistency oracle against partial-cadence drift (a CLI that keeps the shape but stops emitting mid-stream).

Token fields are clamped to non-negative finite integers; a sample whose `total_tokens` reconciliation fails is **dropped, not recorded** — and the drop **counts as zero-usage for the tripwire** (see below) with `reason: reconciliation-failed`, so systematic reconciliation failure cannot recreate silent token-blindness.

Mapping: `tokensIn = input_tokens`; `tokensOut = output_tokens + (reasoning_output_tokens iff total_tokens reconciliation shows they are additive)`; `tokensCached = cached_input_tokens`.

**`onUsage` ordering and contract:** invoked **exactly once, at close-or-grace settlement after the carry-buffer flush** (see settlement ordering above — `close` normally, the bounded post-`exit` grace when a held fd defers it), whenever usage was recorded — independent of whether the promise already settled (a timeout reject at timer-fire must not freeze a stale smaller cumulative; the final post-SIGTERM `token_count` flush is the one recorded; unit fixture asserts the larger value wins). This widens the documented contract — the `IntelligenceOptions.onUsage` JSDoc in `src/core/types.ts` is amended in the same PR: *"invoked exactly once per call whenever usage was parsed, including calls that subsequently reject."* The payload type widens to `{ inputTokens, outputTokens, cachedTokens? }` across all providers and the funnel's composing wrapper. The adjacent `onModel` JSDoc's "codex/gemini/pi cannot report tokens" claim is corrected in the same edit — it is already stale for pi (which invokes `onUsage` today) and becomes stale for codex with this PR.

#### Funnel change — error rows carry cost

`CircuitBreakingIntelligenceProvider.evaluate`'s error path forwards captured usage into the metric row: `tokensIn`, `tokensOut`, AND `tokensCached` (the variable is in scope at `:204-210`). Unit fixture: usage-event-then-extraction-failure → the `'error'` row carries real tokens.

#### Drift tripwire

- The drift condition is **zero usage RECORDED** (post-validation, post-reconciliation) from a successful `--json` call — not "zero events parsed," which a reconciliation-rot would never trip, and **including the empty-stream case**: exit 0 with an empty event stream under `--json` is itself anomalous (a future codex that succeeds while emitting nothing reproduces exact token-blindness), so it counts as drift with `reason: empty-stream` rather than falling between the tripwire and the canary. Emission: DegradationReporter **legacy `.report()`** (the only path with a live consumer today; `reportStructured` has no consumer until the Remediator ships) with the **fixed feature constant `codex-usage-parse-drift`** (the feature string is the Telegram dedup key — a fixed constant prevents the P17 unique-source dodge), `reason` payload distinguishing `no-events` / `empty-stream` / `reconciliation-failed` / `shape-divergence` / `oversized-lines-discarded`. **Emission is gated to once per process lifetime** (module-level flag) — DegradationReporter's legacy path files an external feedback report per event with no feedback-side cooldown, so an hourly emission would spam the maintainer webhook ~24×/day/machine forever (P17). Once-per-boot is sufficient: the durable, always-current surface is `usageCoverage`, not the event. (`scripts/lint-degradation-emit-sites.js` auto-discovers `.report()` sites by scanning `src/` — no manual catalogue exists to edit; the new site is picked up automatically.)
- `totals` in `/metrics/features` gains `usageCoverage`: rows keyed per framework, with NULL-framework rows bucketed `"unknown"` (mirroring `byFeatureModel`'s NULL-model rendering). **Denominator = successful llm rows only (`fired + noop`)** — error rows legitimately lack usage (claude parses usage only on success; codex timeouts can kill the child pre-flush), and including them would normalize coverage below 1.0 and train operators to ignore the tripwire. Error rows are reported alongside as `errorRows` / `errorRowsWithUsage` (which also surfaces the error-path recording directly). **Computability pin:** usage-presence is a NULL test, not a SUM (a recorded `0` must count as reported; one large row must not mask N null rows) — so the single `GROUP BY feature, model, framework` query carries the presence counts itself: `successRowsWithUsage = COUNT(CASE WHEN outcome IN ('fired','noop') THEN tokens_in END)` and `errorRowsWithUsage = COUNT(CASE WHEN outcome='error' THEN tokens_in END)` (SQLite `COUNT(expr)` counts non-NULL). `usageCoverage` is then derived in JS from the same partition — no second window scan. Exemption is keyed **per provider implementation, not per framework**: rows with `model: 'interactive-pool'` are excluded from the claude-code denominator (that provider NEVER invokes `onUsage` by documented contract — under `subscriptionPath.mode: auto|force`, pool-routed calls would otherwise indistinguishably depress claude coverage); `gemini-cli` rows are `exempt: true`; **pi-cli is NOT exempt** — `PiCliIntelligenceProvider` already invokes `onUsage`, and exempting it would mask a future pi parse-rot. Expected coverage for non-exempt providers is ~1.0 against the success-only denominator.
- **Live canary (L5):** an integration-tier test runs ONE real `codex exec --json` call asserting usage is recorded. **Skip-vs-fail boundary:** named-skip when the binary is absent OR the failure is environmental (auth error, rate-limit signature, network error, non-zero exit with empty stream). **FAIL on the discriminating cases: successful exit + zero recorded usage — whether the stream was non-empty (parse rot) or empty (emission rot).** A canary that reddens on weather gets skip-listed — which is the suppression path that re-opens the rot.

#### Kill-switch (not a dark gate)

Config `intelligence.codexExecJson` (default **true**). `false` restores today's plain-output invocation byte-for-byte (result semantics included — see file-only extraction above).

- `CodexCliIntelligenceProviderOptions` gains `resolveExecJson?: () => boolean`. `evaluate()` reads it **per call** (the IntelligenceRouter caches built providers; a per-call closure read survives that caching). **Closure body pinned** — the two naive implementations are both wrong: a closure over the boot-time in-memory config object never sees a disk flip (no config hot-reload exists), and a per-call `loadConfig()` drags a synchronous SecretStore keychain `execFileSync` (~50–150 ms) onto every judgment call. The closure does a **TTL-cached (30 s) raw `readFileSync` + `JSON.parse` of `.instar/config.json` reading ONLY `intelligence.codexExecJson`** — never `loadConfig()`, never the secrets merge. **Resolution order:** the config key wins when the file parses AND the key is present; otherwise — including parse/read failure — fall through to `process.env.INSTAR_CODEX_EXEC_JSON !== '0'` (itself default-on). This keeps BOTH levers honest on every path (the error text names both; an env lever that closure-threaded paths ignored would be a broken rollback contract), and un-strands the realistic incident case: an operator hand-editing config under pressure who leaves a JSON slip gets a once-per-process parse-failure warning and a still-working env lever, not a silently-ignored kill-switch. Fixture: corrupt config + env=0 → plain mode. A config flip applies within 30 s, no restart.
- `BuildIntelligenceProviderOptions` gains the same closure, threaded at: `src/commands/server.ts` (main provider AND the IntelligenceRouter `buildProvider` closure — the per-component routing path the cartographer sweep actually uses), `src/commands/route.ts`, and `src/commands/reflect.ts` (note: `resolveIntelligence()` there doesn't currently receive config — its signature widens or construction moves; named so the implementer isn't surprised).
- **Default resolution carries an env fallback:** absent the closure, the provider resolves `process.env.INSTAR_CODEX_EXEC_JSON !== '0'`. This gives config-less construction sites — `src/core/crossModelReviewer.ts:432` (spec-converge's own codex reviewer) — a working rollback lever, so the unknown-flag error text is honest everywhere (a kill-switch whose recovery instruction is false on the path emitting it is a broken rollback contract).
- Multi-machine note: `.instar/config.json` is per-machine; a rollback flip must be applied per machine.
- Dark-gate machinery: **does not apply, definitively** — not an `enabled` key, no `ConfigDefaults.ts` entry (absence = on), an operational kill-switch rather than a feature gate; the dark-gate lint's assertions cannot match it (verified against `scripts/lint-dev-agent-dark-gate.js`).
- When OFF: one construction-time log line (`codex exec-json disabled — codex calls are token-blind`), and `usageCoverage` for codex-cli reads 0 with `exempt: false` — visible, never silent.

**Version risk:** `--json`/`--output-last-message` exist in codex CLI ≥ 0.20; fleet runs 0.136.x. Older CLIs fail loudly at spawn with both recovery levers named. No version sniffing.

### Slice 2 — Feature×model breakdown in `/metrics/features`

`FeatureMetricsLedger`:
- **New column** `tokens_cached INTEGER` via the existing `ADDED_COLUMNS` idempotent-ALTER pattern. **Pinned semantics (P18 — the schema is the perception):** `tokensCached ⊆ tokensIn` on every framework — `tokensIn`'s meaning is UNCHANGED (historical row continuity; claude↔codex comparability), `tokensCached` is an informational subset, and fresh cost is derivable as `tokensIn − tokensCached`. For claude, `tokensCached = cache_read_input_tokens` ONLY (cache *creation* costs ~1.25× fresh and stays plain input; cache *reads* cost ~0.1× — collapsing them would point the cost signal in two directions at once). For codex, `cached_input_tokens` maps directly. A per-provider unit fixture asserts the subset invariant. `FeatureMetricRecord` gains `tokensCached?: number`; the funnel's local `FeatureMetricsRecorder` interface widens to match.
- New aggregation `byFeatureModel(opts)` — `GROUP BY feature, model, framework`: `{ feature, model, framework, calls, realCalls, tokensIn, tokensOut, tokensCached, fired, noop, errors, shed, successRowsWithUsage, errorRowsWithUsage }[]`. NULL model renders `"unknown"`. **No latency percentiles on this dimension** (the percentile pass is the one full-row-materializing query and stays single-keyed; a future per-model percentile must reuse one composite-key query, never a second full-window load). **`summary()` calls `byFeatureModel` ONCE and partitions rows in JS** — per-feature slicing queries are forbidden (N× full-window scans on synchronous better-sqlite3). **Scan-neutral:** the per-feature `frameworks[]`/`models[]` arrays that `byFeature` currently computes via a separate DISTINCT window scan are derivable from this partition (its group keys ARE feature×framework×model) — `summary()` sources them from the partition and drops the now-subsumed DISTINCT scan, so the slice lands at net-zero window scans instead of +1. No new DDL beyond the one ADDED_COLUMNS entry; identical failure envelope to `byFeature` on a DB whose ALTER was swallowed.
- `summary()` gains per-`FeatureRollup` `byModel` (that feature's partition), and `totals` gains `byModel`, `usageCoverage` (Slice 1), `unlabeledTokenShare`, AND `unlabeledCallShare` (unlabeled realCalls ÷ total realCalls — token-blind unlabeled calls contribute 0/0 to the token share, so a token-weighted metric alone reads 0.00 while unlabeled traffic runs at volume). Zero denominators → 0.
- Read cost bounded by existing retention (default 30d, pruned boot + 6h); `retentionDays: 0` shifts the bound to the operator — pre-existing, noted.

Route `/metrics/features` returns the enriched summary — purely additive. `?feature=` filters `byModel` rows too. Human surface this slice = curl/agent reads; dashboard tab explicitly out of scope. Double-count note: codex rollout files also feed `TokenLedger.codexSummary` at `/tokens` — the same tokens legitimately appear in both (account-level vs per-feature views). FeatureMetricsLedger is per-machine; pool-wide audit = read each machine.

### Slice 3 — Attribution enforcement ratchet

**The baseline is driven to zero in this PR.** The verified unattributed funnel callsites — `src/threadline/WarrantsReplyGate.ts:295`, `src/threadline/PipeSessionSpawner.ts:158` + `:228`, `src/security/LLMSanitizer.ts:107`, `src/providers/uxConfirm/OverrideDetector.ts:128`, `src/providers/uxConfirm/TaskClassifier.ts:123`, `src/providers/adapters/anthropic-interactive-pool/index.ts:200` (canary completion judge), `src/messaging/slack/SlackAdapter.ts:827` — each gain `attribution: { component: '<Name>' }`. (CoherenceGate/SendGateway/AutoDispatcher were checked and are NOT violators — their LLM calls route through already-attributed components.) This is not scope creep; it is the directive itself — those calls are exactly the unaudited spend the operator named. **`LLMSanitizer` note:** its options come from a constructor *property assignment* (`this.defaultOptions`), which the lint's same-file-declaration rule does not cover — the fix there is the sanctioned inline shape at the callsite (`{...this.defaultOptions, attribution: {…}}`), not a constructor edit that would leave the lint red. **Every newly-attributed component is REGISTERED in `COMPONENT_CATEGORY` (`src/core/componentCategories.ts`) in the same PR** — an unregistered name resolves to `'other'` → the default framework, silently excluding it from per-component framework routing and under-reporting `GET /intelligence/routing` coverage. A wiring test asserts every literal `attribution.component` in `src/` resolves to a registered (non-`'other'`) category or a documented exclusion — and **that exclusion list joins the same ratchet test** (which therefore pins THREE lists: violations allowlist, FUNNEL_FILES, wiring-test exclusions; additions to any fail CI), because an unpinned exclusion surface would be the same dodge channel one layer over.

New lint `scripts/lint-llm-attribution.js` (house pattern: `lint-no-direct-llm-http.js`):
- Scans `src/` for funnel callsites: `.evaluate(` on intelligence providers/router handles (receiver matches `/intelligence|provider|llm/i` or threads `IntelligenceOptions`). Out of scope by construction: `src/providers/parity/scenarios/` and `_smoketest`/`_stresstest` files call one-shot adapters directly (they bypass the funnel and record no ledger rows); a lint self-test asserts the heuristic does not flag them, since the empty allowlist would otherwise redden CI on non-funnel calls.
- **Infrastructure pass-through exemption class (`FUNNEL_FILES`), distinct from the violations allowlist:** the funnel's own forwarders match the heuristic but MUST NOT carry inline attribution — stamping a component there would clobber every routed caller's real tag, corrupting the exact audit this spec builds. Exempt files: the IntelligenceProvider implementations, `CircuitBreakingIntelligenceProvider.ts`, `IntelligenceRouter.ts`, `AnthropicSubscriptionRouter.ts`, and the `createQueuedIntelligence` wrapper (`TopicIntentCapture.ts`). The list is pinned by the same ratchet test as the allowlist (entries asserted; additions fail CI; entries must be provider implementations or option-forwarding funnel components by documented constraint) — a second exemption surface that the ratchet doesn't watch would be exactly the dodge channel the ratchet exists to close.
- A non-exempt callsite passes when the options argument **lexically contains `attribution:` with a `component:` whose literal value is non-empty after trim and does not case-insensitively equal `unlabeled`** (inline, or on a same-file variable declaration with the same literal rule). `attribution: {}`, missing `component:`, `component: ''`, `component: 'Unlabeled'` — all violations.
- **Violations allowlist seeded EMPTY** (the baseline-zero work above makes grandfathering unnecessary). Stated in the lint header, with the known accepted limitations (conditional attribution and helper-wrapper indirection pass lexically; runtime backstop below).
