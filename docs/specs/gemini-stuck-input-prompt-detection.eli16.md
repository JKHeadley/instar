# ELI16 — recognize Gemini's input box so stuck messages get auto-recovered

When instar forwards a Telegram message into an agent's CLI session, it types the text into the input and presses Enter. Sometimes that Enter doesn't register (a known race with bracketed paste / a busy turn), so the message just sits in the input box, unsubmitted — the agent never sees it. To handle that, instar has a safety net: `verifyInjection` watches the pane for a few seconds after injecting, and if it sees the message still sitting at the prompt, it re-sends Enter to push it through. The function that decides "is the message still stuck at the prompt?" is `isMarkerStuckAtPrompt`.

That function looked for the cursor by checking the line for a prompt character: `❯` (Claude Code) or `›` (codex). **Gemini CLI uses neither.** Gemini draws its input inside a rounded border, and the active input line looks like `│ * <your text>` (a box pipe plus a bullet). Because `isMarkerStuckAtPrompt` didn't recognize that shape, it always concluded a Gemini message was *not* stuck — so the Enter-recovery never fired for Gemini, and forwarded prompts stalled in the box until something else nudged them. This is the recurring "Gemini auto-submit" friction the mentee layer kept hitting (rc=2 in the framework-issue ledger); it bit the loop live this session — a message to Codey sat unsubmitted until I manually sent Enter.

The fix is one line: also treat `│ *` (Gemini's box-bullet input line) as a prompt indicator, alongside `❯` and `›`. Now an injected message sitting in a Gemini input box is correctly detected as stuck, and the existing Enter-recovery kicks in automatically — no manual nudge needed.

Why this is safe and not noisy: the detection requires BOTH the prompt indicator AND the exact injected marker (the first ~40 chars of the message we typed) to be on that line. So `│ *` alone never trips it — only `│ *` together with the specific text we injected. A normal bulleted output line from the agent can't contain our injected marker, so it can't false-fire. And Gemini's empty-box placeholder ("Type your message or @path…") is not the injected marker either, so an idle empty box is never mistaken for stuck (the same placeholder-immunity property the codex fix relies on).

Tested (modeled on the existing codex-awareness tests): a Gemini box pane holding an injected marker is detected as stuck; the empty-box placeholder is NOT; Claude `❯` and codex `›` still match (no regression); and a `│ *` line WITHOUT the marker does not false-fire — 27/27 green, `tsc` clean. This is a sibling to the earlier codex `›` fix, finishing the framework coverage so all three CLIs (Claude, codex, Gemini) get the same stuck-input auto-recovery.
