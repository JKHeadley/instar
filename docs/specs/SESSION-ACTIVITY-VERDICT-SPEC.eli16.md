# Session-Activity Verdict — ELI16

Ask the system "is this session actually working right now?" and you get five different answers, because five different components each invented their own way of checking. One looks at whether a child process exists (so an idle session holding an open helper looks "busy" forever — we caught a log line that literally said "actively working" and "idle at prompt" about the same session in the same breath). One was properly fixed last week to check whether child processes are actually burning CPU — but only that one component got the fix. One counts idle sessions as reasons to delay urgent updates. One watches the terminal for changes. And the one that talks to the user is the most optimistic of all.

The user-visible result: you send a message, get told "received, actively working on it," later get told it was never delivered, while a watchdog simultaneously declares the same session stuck. Three contradictory stories about one session. Internally it's just as bad — every fix to one component's heuristic leaves the other four wrong.

The fix is boring on purpose: compute the answer ONCE, in one place, from the measurements the components already collect (child-process CPU progress, whether the prompt is idle, whether the transcript is growing, whether the screen actually changed once you ignore the spinner). The answer is one of five words — working, idle-at-prompt, stalled, dead, unknown — decided by a small table where every row is tested on both sides. "Unknown" is a real answer: when the measurements can't be read, the service says so instead of guessing, and each consumer falls back to whatever conservative thing it does today.

Then, one consumer at a time, the five components switch from their homemade heuristics to reading the shared verdict. The age-limit check stops deferring forever on idle sessions. Updates stop being blocked by sessions that aren't doing anything. The user-facing "actively working" is only said when the session is actually working — and when it's stalled, the wording says that honestly. Importantly, nothing about KILLING sessions changes here: each component keeps its own authority over what to do; this only unifies the facts they act on.

It starts enabled on the two development agents only, each consumer migration behind its own flag, so any wrong verdict shows up loudly on the dogfooding machines before the fleet ever sees it.

After adversarial review (Codey, on the PR), three things got tightened. First, the answer is computed fresh but remembered for five seconds — and everyone asking within those five seconds gets the SAME measurement, because one of the inputs (CPU progress since last look) is consumed by the act of looking; two components measuring independently would corrupt each other's readings. Anything making a destructive decision can refuse a stale answer instead of trusting it. Second, "unknown" now has a written contract per consumer (the reaper keeps, updates defer, the user-facing voice stops claiming "actively working") so it can't quietly turn back into five private guesses. Third, for session types whose idle-prompt shape we haven't characterized (codex, gemini), the system is explicitly forbidden from concluding "idle" — measurements can prove a session is working or stalled, but never that it's safely idle, so nothing destructive fires on a guess.
