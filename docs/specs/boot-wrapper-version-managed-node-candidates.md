# Boot-wrapper node-heal — version-managed node candidates

## Background — the instar-codey outage (2026-05-31)

instar-codey was offline for hours. A homebrew **node 25.6.1** install caused the
boot wrapper (`instar-boot.cjs`, generated by `installBootWrapper`) to self-heal
`.instar/bin/node` **forward** to ABI 141, but shadow-install `better-sqlite3` was
ABI 127 → server couldn't load it → launchd-throttled restart loop. PR #607
(v1.3.148) made all three better-sqlite3 heal paths prebuilt-first so a heal can
fetch a matching-ABI binary. This spec fixes the **deeper root**: why the boot
wrapper drifted to the wrong node and could not recover.

## Root cause

The generated boot wrapper's `selfHealNodeSymlink()` builds its node-heal
candidate list as `[currentNode, ...wellKnown]` where
`wellKnown = ['/opt/homebrew/bin/node', '/usr/local/bin/node', '/usr/bin/node']`.
**asdf / nvm / volta version-managed nodes are never candidates** (they live under
`~/.asdf/installs/...`, not the well-known list). So when `currentNode` drifts to
homebrew node 25 (ABI 141) and the existing better-sqlite3 is ABI 127:

- The ABI-compatibility loop tries each candidate, but the only candidate that can
  load the ABI-127 module — the asdf node 22 the rest of the system uses — is not
  in the list.
- No candidate is compatible → `best` stays `currentNode` (homebrew 25) → the
  wrapper self-heals **FORWARD to the wrong ABI** and the agent cannot recover.

Notably, `setup.ts`'s own `resolveNodeCandidates()` (the symlink-creation path) is
already correct — it includes `which node` (line ~824). Only the **generated boot
wrapper template** (a separate embedded copy) lacked it.

## Design

In the boot-wrapper template (`installBootWrapper` in `src/commands/setup.ts`),
after the well-known candidates are added, also add the PATH-resolved node:

```
try {
  const resolved = require('child_process')
    .execFileSync('which', ['node'], { encoding: 'utf-8', timeout: 5000 }).trim();
  if (resolved && fs.existsSync(resolved) && !candidates.includes(resolved))
    candidates.push(resolved);
} catch { /* best-effort */ }
```

`which node` resolves through asdf/nvm shims (and the launchd plist PATH places
the asdf install dir on PATH), so the version-managed node the system actually
uses becomes a candidate. The existing ABI-compatibility loop then picks it iff it
can load the existing native module. Purely **additive**: it only expands the pool
that is already ABI-filtered before selection — a `which node` that can't load the
module is skipped exactly like any other incompatible candidate.

## Migration parity

The regenerated wrapper carries a new marker string, `version-managed node
candidates`. `PostUpdateMigrator.migrateBootWrapperAbiCheck` previously treated a
wrapper as current if it merely contained the ABI-drift marker — which is why
instar-codey (ABI check present, asdf candidates absent) was never regenerated.
The sniff now requires **both** markers; an install with the ABI marker but not
the version-managed marker is regenerated via `installBootWrapper`. Idempotent
(both markers present → skip). Darwin-only (the boot wrapper is launchd-specific).

## Safety
- Additive only — never removes/reorders existing candidates, so it cannot make a
  heal worse; worst case a `which node` candidate that can't load the module is
  skipped (existing behavior).
- Best-effort — `which` failures are swallowed; no throw on the boot path.
- Mirrors the already-correct `resolveNodeCandidates()` in the same file.

## Test plan
- Unit (`tests/unit/PostUpdateMigrator-bootWrapperAbiCheck.test.ts`):
  - the generated wrapper contains the `which node` discovery + the new marker,
    placed BEFORE the ABI-compatibility loop (cross-platform content assertion).
  - regenerates when the ABI marker is present but the version-managed marker is
    absent (the instar-codey case).
  - idempotent skip when both markers are present.
- Regression: the boot-wrapper cjs/plist + durable-node-selection +
  shadow-install-self-heal + launchd-handoff suites (46 tests) stay green.
