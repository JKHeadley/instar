# Account × Machine matrix

## Problem statement

There is no operator surface for "put account X on machine Y." The account-follow-me Approve cards are DEPTH/quota-driven (`AccountFollowMeService.scanAndOffer`) — they only surface an account when load-balancing heuristics decide it's beneficial, so they never offer accounts on demand. When the operator (Justin, topic 13481, 2026-06-18) wanted to add his remaining subscriptions to the Mac Mini, there was no UI; the agent fell back to a clunky chat dance (DM a sign-in link, the operator pastes the code back in chat) — exactly the off-dashboard, route-the-code-through-chat experience the operator-surface standards forbid. The operator proposed the right design: a matrix.

## Proposed design (the operator's design, made concrete)

A **machine × account matrix** on the Subscriptions tab. Rows = accounts, columns = machines (the operator's machines in the pool). Each cell is either:
- a **✓** — that account is an active pool account on that machine; or
- a **"Set up" button** — that account is not yet on that machine.

Clicking "Set up" runs the whole auth IN THE DASHBOARD: it starts a fresh sign-in on the target machine (that machine re-mints its own login — Mechanism B, ToS-safe, nothing copied), opens the provider auth page, and the cell turns into a **code input + Submit**. The operator authorizes, gets the provider's code, pastes it into the cell, taps Submit; the code rides to the target machine, completes the login, and the cell flips to **✓**. At-a-glance "who's active where," and the auth never leaves the dashboard.

This REUSES already-shipped plumbing:
- **Matrix data** = the existing `GET /subscription-pool?scope=pool` — it already returns one row per (account, machine) tagged with `machineId`/`machineNickname`/`remote`. The frontend pivots these rows into the grid (union of account ids × union of machines; a cell is ✓ when a row exists for that pair with `status: active`, "needs-reauth" shown distinctly, else empty → button).
- **Code delivery + completion** = the existing `POST /subscription-pool/follow-me/submit-code` (shipped in PR #1228): it carries the code one authed hop to the target, types it into the waiting pane (fail-closed readiness), and drives the S7 email-gate → `SubscriptionPool.add`.

The ONE new backend piece, a thin PIN-GATED orchestrator over the already-shipped chain:
- **`POST /subscription-pool/matrix/start-cell`** `{ accountId, machineId, pin }` (dev-gated): **requires the dashboard PIN** (`checkMandatePin`), exactly like `/mandate/issue-for-machine`. With a valid PIN it does, server-side: resolve the agents pair + the account's email from pool state (NEVER from the body — the same `resolveFollowMeEnrollTarget` path `enroll/start` uses), issue the per-`(account, target-machine)` mandate via the existing PIN-gated issuance, then drive the existing mandate-gated `enroll/start` (delivered-mandate path for a peer target — the proven cross-machine flow that enrolled adriana). Returns `{ verificationUrl, loginId, machineId }`. Audited: `{accountId, machineId, outcome}`, never a code or token. Partial-failure semantics (codex r3 #3): idempotent — if mandate issuance succeeds but enroll-start fails/times out, start-cell returns an honest retryable error and the unused mandate lapses by its existing TTL (an unconsumed mandate enrolls nothing — harmless + bounded); re-tapping first CHECKS for an existing valid pending login for that `(accountId, machineId)` (correct expectedEmail + target + un-expired) — if present it is REUSED (its `verificationUrl` returned, no new mandate minted); only if there is none does it mint a fresh scoped mandate + enroll-start. So a retry never duplicates a pending login or stacks mandates. Why PIN, not Bearer (the load-bearing correction): the AGENT also holds the Bearer token (skills/scripts call the local API with it), so Bearer-only would let any agent session start a cross-machine account login with no operator present — re-creating the exact hole the PIN gate on `issue-for-machine` exists to close (Know Your Principal). The PIN is the only thing in this system that proves operator presence; the matrix tap collects it inline (Operators-act-in-taps). start-cell adds NO new authority — it is a UI-convenience wrapper that drives the existing PIN→mandate→enroll-start chain in one call so the frontend never handles fingerprints or raw mandate JSON.

## Decision points touched
- New PIN-gated route `start-cell` — orchestrates the EXISTING PIN-gated mandate-issuance + mandate-gated enroll-start (no new authority; the PIN gate is preserved, not bypassed). Reuses the shipped submit-code relay for completion. Per Signal-vs-Authority: the load-bearing authorization is the PIN (operator presence); the load-bearing account-safety is the existing S7 email-gate at COMPLETION. S7 gates the ADD, not the START — it is NOT a substitute for the start authorization (that is the PIN).
- Dev-gated like the rest of account-follow-me (503 when off).

## Frontloaded Decisions
- **FD1 — Matrix data is the existing pool-scope read, pivoted client-side.** No new data route; `GET /subscription-pool?scope=pool` already returns per-(account,machine) rows. The frontend builds the grid from them. A dark/unreachable peer is shown as a column with an honest "machine offline" state (from the pool-scope `failed` list), never a fabricated ✓ or a silent omission.
- **FD2 — Completion reuses the shipped submit-code relay unchanged.** The cell's code-submit calls `POST /subscription-pool/follow-me/submit-code {machineId, id, code}` — already live (PR #1228), already S7-gated, already fail-closed on pane readiness. The matrix adds no new code-delivery path; it is a frontend over shipped routes + one start route.
- **FD3 — start-cell sets expectedEmail = the account's pool email; S7 gates the ADD (confirmed), not the START.** Setting up account X on machine Y means Y must authenticate AS X; start-cell stamps `expectedEmail` from X's pool record (server-resolved, never the body) so the existing S7 gate validates the minted email == X's email at completion. Confirmed: if the operator signs into the WRONG provider account, the minted email ≠ expectedEmail → `completeFollowMe` returns `held`, no pool add, HIGH attention item — so a cell flips to ✓ ONLY when the machine logged into the RIGHT account. But S7 is the COMPLETION safety; it is NOT the START authorization (start-cell still spins a real login pane on a remote machine regardless of S7) — the start authorization is the PIN (FD4).
- **FD4 — Authorization is operator-presence via the PIN, because the agent shares the Bearer (the load-bearing decision).** This was the spec's biggest decision and the reviews corrected the first draft: "operator-direct, Bearer is enough" was UNSOUND — the agent also holds the Bearer token, so Bearer-only would let an agent session start a cross-machine account login with no operator present (the exact agent-acts-as-operator hole the PIN gate on `issue-for-machine` closes; Know Your Principal). So start-cell REQUIRES the dashboard PIN (`checkMandatePin`) and drives the existing PIN-gated mandate-issuance + mandate-gated enroll-start — the PIN gate is preserved end-to-end, never bypassed. The operator entering their PIN inline on the cell IS the operator-presence proof (and is a tap, per Operators-act-in-taps — never raw JSON/fingerprints). Each start-cell call's PIN authorizes EXACTLY ONE `(accountId, machineId)` mandate (scoped, single-use, inheriting the existing mandate TTL) — one presence proof is never broad batch authority. PIN-caching is explicitly OUT OF SCOPE for v1 (codex r3 #4): each 'Set up' tap requires the PIN, memory-only, no caching across taps or page reloads — so there is no unbounded-cache surface to reason about. (If re-typing ever proves painful, a bounded memory-only short-TTL cache that still issues one scoped mandate per tap is an additive enhancement, not part of this spec.)
- **FD5 — Multi-machine posture.** Inherently cross-machine and correct by construction: matrix data is the pool-scope fan-out (already merges peers, dark-peer-tolerant); start-cell drives the proven delivered-mandate cross-machine enroll path and the shipped submit-code relays one authed hop to the target via `resolvePeerUrls()` (self → loopback); an offline target column shows "offline" and its buttons are disabled (no false start). No durable state beyond the existing per-machine pools + transient pending logins.
- **FD6 — Honest cell states, each from a real signal.** A cell is one of: **✓ active** (a pool-scope row for that `(accountId, machineId)` with `status: active`, FROM A CURRENTLY-REACHABLE machine) · **⟳ needs-reauth** (row present, status flagged) · **▢ empty** → the "Set up" button (no row, machine reachable) · **◷ in-progress** (a pending url-code-paste login exists for that pair — from pending-logins, awaiting code) · **— offline / last-known** (the machine is in the pool-scope `failed`/stale set — its whole column is offline + buttons disabled; the WHOLE column reads offline at the machine level and per-account cells are 'unknown' — NOT a per-account 'last known active' (codex r3 #1: pool-scope queries LIVE and a dark peer lands in `failed` with NO account rows, so there is no per-account last-known data to honestly show; machine-level offline is the truthful ceiling). The **◷ in-progress** signal is read from `GET /subscription-pool/pending-logins?scope=pool`; each pending login carries its `id` (== `accountId` for matrix enrollments) AND machineId, so a cell correlates unambiguously on `(login.id === accountId, machineId)` even when several accounts are set up on the same machine (codex r3 #2). INVARIANT (codex r4 #1): the backend enforces at most ONE pending matrix login per `(accountId, machineId)` — the configHome is per-account (`.claude-followme-<accountId>`) on a given machine, so the pane identity (framework + configHome) and the matrix identity (accountId) are 1:1 for matrix enrollments. The code-submit payload carries the concrete `loginId` (not just accountId), so even if the invariant were ever relaxed the submit targets the exact login · **✗ can't-resolve** (start-cell returned the 409 "cannot resolve approved account email", e.g. the account's home machine is dark) · **⚠ held** (the LAST submit on this cell returned `outcome: held` — email mismatch). Held + in-progress are CLIENT-SIDE last-attempt/transient states (the client knows the response it just got for that cell); active/needs-reauth/empty/offline are derived from pool-scope; can't-resolve from the start response. No state is fabricated.
- **FD7 — Cell pivot key + integrity.** The grid pivots pool-scope rows on `(accountId, machineId)` (NOT `accountId` — the same account on two machines is the normal, individually-visible case). Cells are computed ONLY after separating reachable machines from the pool-scope `failed`/stale set; an account row from a peer that went dark is rendered under that machine's offline column, never as a confident ✓. Tested both ways (offline peer column vs genuinely-empty cell).
- **FD8 — Account identity key.** An account is keyed by its pool `id` (the stable per-account identifier the pool already uses), displayed by its `email`; emails are provider-scoped and resolved from the pool's existing identity model (not re-canonicalized here). The matrix never invents an account key — it uses the pool's.
- **FD9 — URL is opened in the OPERATOR's browser; the CLI pane stays on the target.** start-cell returns `verificationUrl`; the cell renders it as a link the operator opens locally (the auth happens in the operator's browser). The waiting `claude auth login` pane lives on the TARGET machine; the operator's pasted code rides the submit-code relay back to it. "On the target machine" never means a browser opens remotely. PRECONDITION (codex r2 #1): the matrix supports ONLY `url-code-paste` auth — the flow where the provider page yields a COPYABLE code. A framework/provider whose login requires a local loopback callback on the target host is NOT offered in the matrix (its cell shows an honest 'not supported here'), because opening that URL in the operator's browser would bind the callback to the wrong host. Claude's `url-code-paste` flow is copyable-code and satisfies this.

## Rejected alternatives
- **A queued multi-cell job model** (set up many cells in one batch with resumable per-cell state). Rejected as over-engineering for the expected usage (an operator sets up a handful of cells, one tap at a time): the existing transient pending logins already give per-cell start/await/complete state, the submit-code relay already completes each independently, and a cell's state is recoverable from pool-scope + pending-logins on refresh. A durable job/queue would add state + recovery machinery for no real gain at this scale; if batch setup ever becomes a real need it is an additive layer over the same per-cell routes.
