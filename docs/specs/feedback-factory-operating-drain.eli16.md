# Feedback Factory Operating Drain — Plain-English Overview

Instar already has most of a feedback factory. Agents send reports, one canonical store receives them, and a processor groups similar reports into clusters. But the factory currently stops there. It can turn twelve thousand reports into roughly a hundred and fifty organized piles while creating zero pieces of work. The code explicitly says that dispatch is not invoked. On another development install, even the clustering endpoint can be dark because the intended “live on development agents” setting is not covered by the registry and wiring test that protect other development-only features.

This spec closes the class of mistake before fixing that one pipeline. It adds a standing rule: a canonical pipeline is only complete when every declared stage has real runtime wiring, a bounded trigger, durable handoff, named consumer, useful progress metrics, and an end-to-end test that begins with accepted input and proves the consumer receives the output. A manifest and lint make that rule structural. Future pipelines will not be able to ship as a collection of individually tested stages with nothing operating the final connection.

For the feedback factory itself, clustering remains a signal, not an authority. A cluster does not become work just because it is old, large, or has a high score. An operator or registered policy authority first marks it ready. The drain then creates one durable work item, using a stable key so retries and concurrent ticks cannot duplicate it. A fenced consumer claims that item and creates or links one development Initiative and Action. It only acknowledges the handoff after those artifacts can be read back from their authoritative store. That means backlog work enters Instar’s normal work system instead of needing a person to hand-cluster and manually recreate it.

The first rollout is deliberately controlled. It is live only on development agents, with the work-creation side dry-run first. Existing historical clusters remain in a collecting state until an operator reviews and approves bounded batches. Fleet agents do not gain a new autonomous dispatch mechanism. Completing a queue item means “the development work was durably handed off,” not “the underlying bug is fixed.” Product-resolution claims still require their existing evidence authority.

The operating surface answers the practical question: is work moving? It shows how many reports are waiting, how many clusters are ready, how many work items are queued or claimed, their oldest ages, the last successful transition, and why progress stopped. Development-agent darkness is no longer a silent 503: expected fleet darkness is distinguished from a broken development configuration, and the system attempts a tightly bounded self-heal before escalating once. All logs and views are metadata-only; raw feedback bodies, credentials, transcripts, and prompts do not enter the queue or dashboard.

The main tradeoff is intentional operator control at the readiness boundary. This prevents noisy or poisoned clusters from spawning work automatically, but it means historical backlog activation needs reviewed batches. Once a batch is approved, the rest of the path is automatic, idempotent, observable, and bounded. Rollback simply disables the drain and consumer; durable records remain readable and no source feedback is deleted or rewound.
