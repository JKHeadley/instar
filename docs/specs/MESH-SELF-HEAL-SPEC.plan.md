## ⭐ SHARPENED THESIS (after grounding against existing specs — THE core insight)
**The awake-lease is decoupled from actually doing the job (polling Telegram + serving).** The laptop was RENEWING its lease perfectly (nonce/expiry advancing every cycle) while being a muted silent-standby that does NOT poll Telegram. So the existing renewal-aware features behaved "correctly" and made things WORSE for the user:
- F2 (staleHolderTakeover) judges the holder by nonce-watermark advance = "is it renewing?" The laptop WAS renewing → F2 correctly refused takeover (no live-holder steal). But a renewing-yet-not-serving holder is exactly the failure.
- soloCaptainHold needs the peer aged-out of the heartbeat window — the laptop was intermittently reachable, so never aged out → never fired.
- F3 (silentStandbyRelinquish) IS the right fix — a machine that becomes observe-only must RELINQUISH the lease it holds — BUT it ships DARK and must be enabled ON THE HOLDER (the laptop). It wasn't.
- My mistake: I enabled F2+soloCaptainHold on the MINI (the wrong machine + wrong feature). The correct lever was F3 on the LAPTOP. Enabling takeover against a live-renewing peer caused the tug-of-war → the laptop spawned its own 29099 session → duplicate.

**So the spec's load-bearing principle: LEASE OWNERSHIP MUST BE BOUND TO JOB-PERFORMANCE.** Holding the awake-lease must REQUIRE actually owning the Telegram poll + serving. A machine that stops polling/serving must automatically lose the lease (F3, made automatic + enabled), and the system must DETECT "nobody is actually polling" (independent of lease-renewal) and force a resolution. Renewal-liveness ≠ job-liveness — the existing features only check the former.

## The 4 confirmed gaps (REFRAMED after grounding — see memory [[mesh-zombie-lease-holder]])
- **G1 → lease/job decoupling:** the holder renews the lease while not polling/serving. Existing F2/soloCaptain check renewal, not job. FIX: bind lease-hold to job-performance; a non-polling holder auto-relinquishes (F3 made automatic+default-on after verify); add a "job-liveness" signal distinct from "renewal-liveness".
- **G2 → nobody-polling detector (CONFIRMED no existing detector):** when no machine owns the Telegram poll, nothing fires. FIX: explicit poll-ownership liveness heartbeat → forces handoff/escalation. THE key missing piece.
- **G3 → ownership-checked spawn (CONFIRMED out of scope of existing lease specs):** both machines can spawn a session for the same topic; killing a session does NOT clear its topic→session binding. FIX: refuse to spawn for a topic another machine owns; clear binding on close/transfer. (SpawnRequestManager/SpawnLedger exist but don't gate on lease ownership.)
- **G4 → seize-from-live (already conservatively protected by F2/soloCaptain; the DUP was G3 not takeover):** keep the conservative protection; the real fix for today's dup is G3, not changing takeover aggressiveness.

## Original 4 gaps (pre-grounding wording, superseded by G1–G4 above)
1. **Liveness-by-reachability bug.** Takeover/freshness judges the holder by "did the lease PULL succeed," not "is the holder actually RENEWING (epoch/nonce advancing)." A muted holder that serves a frozen-epoch lease on intermittent pulls looks fresh forever → `staleHolderTakeover` can NEVER fire. FIX: judge holder liveness by real renewal (epoch/nonce advance), not pull-success.
2. **Nobody-polling blindness.** When both machines end up not polling Telegram (one muted silent-standby holding the lease, the other send-only standby), NO detector fires → messages silently drop. FIX: a "are inbound messages being claimed by anyone?" heartbeat that forces a handoff/escalation when nobody owns polling.
3. **Duplicate-spawn.** Inbound delivered to BOTH machines (exactly-once ingress ledger failing cross-machine during flaps) → both spawn a session for the same topic → double-replies. FIX: ownership-checked spawn (refuse to spawn for a topic another machine owns) + clear the topic→session binding on close/transfer (killing a session does NOT clear its binding today).
4. **Seize-from-live tug-of-war.** Takeover fired against a flaky-but-ALIVE peer caused the duplicate (my mistake today). FIX: only ever seize from a PROVABLY-gone peer, never a flaky-alive one; never bidirectionally.

## ⭐⭐ ELEVATED STANDARD (Justin, 2026-06-27 ~15:09 PDT) — make this a first-class CONSTITUTIONAL standard in the spec
**"No data the agent depends on may be machine-dependent."** ANY data the agent relies on is EITHER shared-by-default across machines OR transparently lazy-loaded on demand from the machine that has it. The user NEVER deals with machine-dependence; the UX is smooth and coherent regardless of which machine serves. Browser/auth login sessions are just ONE instance — the standard is general: secrets (already synced), browser sessions (gap), operator identifiers like phone number (gap — I didn't have Justin's number on hand), relationships, learnings, KB, playwright profiles, etc. The spec must define: (a) the DEFAULT-SHARE set, (b) the LAZY-LOAD-ON-DEMAND channel (a go-to fetch path any machine uses when it's missing something), (c) an ENFORCEMENT mechanism so new agent-depended-on data can't silently ship machine-local. This pairs with the lease/job-binding thesis: together they make the multi-machine experience seamless. Existing groundwork to build on: MULTI-MACHINE-SEAMLESSNESS-SPEC, the replicated-store foundation (WS2.*), secret-sync, working-set handoff (lazy file pull).
