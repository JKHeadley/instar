# ELI16 — `instar worktree create` now checks all remotes, not just origin

`instar worktree create <branch>` is the safe, blessed way for an agent to make a git worktree of the instar repo inside its own home (the one place the macOS sandbox can't yank away mid-session). Before it creates anything, it validates that the checkout it's about to build from is really the canonical instar repo — it reads the repo's `remote.origin.url` and checks it against an allowlist of trusted instar URLs.

That check had a blind spot. Fleet agents don't use the canonical repo as their `origin` — each agent **forks** instar to its own remote (e.g. `origin = github.com/owner/instar-<name>.git`) and keeps the canonical repo as a *second* remote (e.g. `upstream` → `instar-ai/instar.git`), which is what the worktree actually builds against. Because the validator only looked at `origin`, it saw the fork URL, didn't find it in the allowlist, and **refused** — so every agent's own checkout was rejected from the very helper that's supposed to serve it. Both Echo and Codey hit this exact wall this session ("remote.origin.url … not in worktree.repoUrlAllowlist"), which defeats the whole agent-worktree convention.

The fix: validate against **all** of the checkout's remotes, not just `origin`. If origin is allowlisted, accept (unchanged fast path). Otherwise, run `git remote -v` and accept the checkout if *any* remote URL is in the allowlist — that's the canonical remote (e.g. `upstream`/`instar-ai`), the proof this is a real instar checkout. The trust boundary is unchanged: the allowlist is still the gate; we just stopped ignoring the canonical remote when it isn't named `origin`. The rejection messages and behavior for genuinely-untrusted repos (no allowlisted remote at all) are byte-for-byte the same, so the existing security tests pass unchanged.

Why this is safe, not a loophole: the allowlist still decides what's trusted. A repo with a hostile `origin` but a real canonical remote was already a legitimate instar checkout (it's literally how fleet agents are set up). A repo with NO allowlisted remote is still refused with the same error as before. The change only stops a false-negative that was blocking the fleet's own agents from a convention built for them.

Tested: a new unit test sets `origin` to a non-allowlisted fork plus an `upstream` pointing at the canonical URL and asserts the checkout is accepted (with `remoteUrl` resolving to the canonical one); the existing "rejects unset origin" and "rejects non-allowlisted origin (no other remote)" tests still pass — 31/31 green.
