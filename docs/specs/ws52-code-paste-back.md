# WS5.2 Account Follow-Me — operator code paste-back

## Problem statement

The account-follow-me flow now works end-to-end EXCEPT its last step. The Claude `url-code-paste` login is a TWO-part flow: (1) the operator opens the verification URL + authenticates, then (2) the provider hands them an **authorization code** that must be pasted back into the waiting `claude auth login` process (the tmux pane shows `Paste code here if prompted >`).

Today the dashboard card surfaces the "Sign in" link (part 1) but has **no field or instructions for part 2** — the code. In the live proof (2026-06-18, topic 13481) the operator authenticated, got a code, and had no idea what to do with it; he sent it via Telegram and the agent relayed it by hand (`tmux send-keys`). That is broken UX (the operator shouldn't route a code through chat) AND a security smell (an OAuth authorization code in chat history).

The login pane lives on the TARGET machine (e.g. the Mac Mini); the operator uses their SINGLE (fronting) dashboard (e.g. the laptop). So the code must travel operator → fronting dashboard → target machine's login pane, over the authed API/mesh, NEVER through chat.

## Proposed design

Three pieces (mirrors the existing follow-me cross-machine pattern):

1. **Target-local route** `POST /subscription-pool/follow-me/enroll/:id/submit-code` (body `{ code }`):
   - Look up the `PendingLogin` for `:id`. Derive its enroll pane session name through the **single shared helper** `enrollPaneSessionName(framework, configHome)` (`src/core/FrameworkLoginDriver.ts`) — the SAME helper enroll-start's spawn uses, so the two derivations can never drift or collide differently (codex review #1).
   - **Narrow the authority to the paste-back flow** (codex #4): refuse with 409 unless `login.kind === 'url-code-paste'` (a device-code login never prompts for a pasted code).
   - **Validate the code shape** (codex #2/#4): the route types the value into a live login pane, so it is clamped to a single token — any whitespace, newline, or control char is rejected 400 (a newline would submit the pane prematurely / split the code), and an over-long value is rejected. Delivery itself rides `SessionManager.sendInput`, which is already argv-safe (`tmux send-keys -l --`), so the code can never be interpreted as a tmux flag.
   - Verify the pane exists AND is still at the code prompt — capture the live frame and require it to show the paste-code prompt (both "paste" and "code" present); refuse 409 if the pane dropped to a shell (the login exited/crashed/timed out), so the code is never typed into a shell (FD13). Then deliver the code via `sendInput` (types it literally, then `Enter`), and best-effort `tmux clear-history` on the pane afterward (FD1 residual).
   - The code is NEVER stored, NEVER logged by instar state/logs. Only a redacted line (`submitted operator code to login <id> (value redacted)`) and a greppable **terminal-outcome** line (`[follow-me] submit-code outcome=validated|held|submitted id=<id>`) are written, so success/timeout/error rates are auditable without the value (Observability standard).
   - After delivery, briefly poll (bounded, ≤30s) for the credential the login writes to its `configHome` (file existence is ONLY a trigger, never the validator — codex #5). The decision is made by the existing follow-me **complete** path: `completeFollowMe` runs the **S7 email-gate** (the identity oracle resolves the freshly-minted account's email and validates it against the operator-approved `expectedEmail`) before anything is added — a mismatch returns `held` and emits a HIGH attention item. On a validated result, `SubscriptionPool.add`. On timeout, return `submitted` (the reissue/complete sweep finishes it) — never a false success.
   - **One-shot by construction** (codex #3): `completeFollowMe` removes the pending login on completion, so a successful submit makes any repeat submission 404; while a login is still pending the pane-existence + kind guards reject a code that no longer applies. A genuine double-tap before completion is idempotent at the provider (the auth code is single-use server-side).
   - Dev-gated (same `multiMachine.accountFollowMe` gate as the rest); 503 when off.

2. **Fronting relay route** `POST /subscription-pool/follow-me/submit-code` (body `{ machineId, id, code }`):
   - `machineId === self` → call the local submit-code directly.
   - Else resolve the target peer URL via `resolvePeerUrls()` and POST `{ code }` to the peer's local `…/enroll/:id/submit-code`, Bearer-authed, bounded timeout, dark-peer-tolerant (honest 502 on unreachable, never a false ok).
   - This is the route the operator's single dashboard calls; it carries the code one authed hop to the machine that owns the pane.

3. **Client (dashboard pending-login card)** — for `kind: url-code-paste` logins:
   - After the "Sign in" link, render a short instruction ("After you sign in, paste the code the page gives you here") + a `code` input + a "Submit code" button. The copy names *the code the page gives you* to reduce the user-code-vs-authorization-code ambiguity (codex #6); a rejected/expired code returns a plain "re-tap the sign-in link" message, not an opaque error.
   - On submit, POST to the fronting relay with `{ machineId: login.machineId ?? self, id: login.id, code }`. Show inline status: submitting → "Finishing setup…" → "Done — <account> is set up on <machine>" (or an honest error). Clear the input on success.

## Response schema (single source of truth — codex r5 #4)

The target route (and the relay, which forwards the target's response verbatim) returns exactly one of:

| Situation | HTTP | Body |
|-----------|------|------|
| feature dark / wizard absent | 503 | `{ error }` |
| missing/empty code | 400 | `{ error: 'code is required' }` |
| code has whitespace/control chars or > 512 chars | 400 | `{ error }` |
| code is a URL, not the raw code | 400 | `{ error }` |
| no pending login for `id` | 404 | `{ error }` |
| login kind ≠ url-code-paste | 409 | `{ error }` |
| a submit already in flight for this login | 409 | `{ error }` |
| pane not at the code prompt | 409 | `{ error }` |
| send to pane failed | 502 | `{ error }` |
| relay: target machine unreachable | 502 | `{ error }` |
| account minted + S7 email match | 201 | `{ enabled:true, outcome:'validated', account }` |
| account minted but S7 email mismatch | 200 | `{ enabled:true, outcome:'held', reason }` |
| code delivered, credential not yet observed within bound | 200 | `{ enabled:true, outcome:'submitted', message }` |

There is no `pending` outcome — the only success-pending state is `submitted` (the client renders it "finishing sign-in"). The client maps `validated`→"Done", `submitted`→"finishing sign-in", `held`→"didn't match — check with operator", any 4xx/5xx→ the plain `error` text.

## Decision points touched
- New routes (target submit-code + fronting relay) — both dev-gated, deny-by-default inherited from the existing enroll gate. They carry **no new credential-returning authority**; they add a **narrow paste-back authority** (typing the operator's own single-use auth code into the operator's OWN waiting login), bounded structurally by the pending-login lookup, the `url-code-paste` kind guard, the pane-existence check, the single-token code-shape validation, and the per-login in-flight mutex. The provider validates the code; instar never reads or returns the resulting credential (codex r2 #5).
- No block/allow gate added.

## Frontloaded Decisions
- **FD1 — Security of the code in transit + at rest.** The OAuth authorization code travels over the existing Bearer-authed API + the authed mesh hop, identical trust to every other `/subscription-pool/*` call. It is single-use + short-lived, typed into the waiting login, and NEVER stored or logged **by instar application state/logs** (value redacted from all logs; only a greppable terminal-outcome key is recorded). It must NOT go through any chat/messaging surface. Honest residual (codex r3 #2): typing into an interactive pane echoes the value into that pane's tmux scrollback on the TARGET machine — so after a successful send the route best-effort `tmux clear-history` on the pane; combined with the code being single-use + short-lived, the residual is accepted. Accepted: authed-API transit, never chat.
- **FD2 — No credential ever returned.** submit-code returns only a status (`validated`/`held`/`submitted`/error). The credential is written by `claude auth login` into the account's own configHome; instar never reads or returns it (unchanged Secret-Drop discipline).
- **FD3 — Cross-machine via resolvePeerUrls** (the proven follow-me path), not lastKnownUrl. Single-machine → the relay calls local. Dark target → honest 502, never a false ok.
- **FD4 — Pane targeting is reconstructed, not stored.** The session name is derived from the PendingLogin's framework + configHome (the same formula enroll-start uses), so no new state. If the pane is gone (login already done/expired), 409 with a plain message ("this login isn't waiting for a code — re-tap Approve").
- **FD5 — Completion feedback.** submit-code drives to a real outcome where possible (poll for the credential, then call follow-me complete) so the operator sees "Done", not just "sent". Bounded; on timeout returns `submitted` (the sweep finishes), never a false "done". The credential file is ONLY the trigger to run the S7 email-gate; the gate (identity-oracle email match against the operator-approved `expectedEmail`) is the validator, never file-existence alone (codex #5).
- **FD6 — Pane targeting is a single source of truth.** The enroll pane session name is derived ONLY through the shared `enrollPaneSessionName(framework, configHome)` helper, used by both enroll-start's spawn and submit-code's lookup, so the two can never drift apart or normalize/collide differently (codex #1). The slug is the per-credential `configHome` (unique by construction), normalized + tail-clamped, with `framework` included.
- **FD7 — The route is a narrow, validated input surface, not raw plumbing.** Typing operator-supplied text into a live login pane IS an authority, so it is bounded structurally: dev-gated + Bearer-authed; only `kind: url-code-paste`; only an active pending login; a single-token code shape (no whitespace/newline/control chars, length-capped); argv-safe delivery (`send-keys -l --`). It carries no authority beyond delivering the operator's own single-use auth code into the operator's own login (codex #2/#4).
- **FD8 — One-shot / replay posture.** The auth code is single-use provider-side; on the instar side, `completeFollowMe` removes the pending login on success so any repeat is a 404, and the kind + pane-existence guards reject a code that no longer applies. A pre-completion double-tap is provider-idempotent (codex #3). Note (internal review): `completeFollowMe` transitions the login to terminal `completed` BEFORE the S7 gate runs, so BOTH a validated AND a held (email-mismatch) outcome consume the pending login — recovery from a held mismatch is re-tap Approve (a fresh login), not re-submit. A heavier per-login `waiting→submitted` state machine was considered and rejected as gold-plating given the provider's single-use guarantee + the login-removal one-shot.
- **FD9 — Multi-machine posture (Cross-Machine Coherence).** Machine-local execution, proxied-on-WRITE by design: the login pane lives on the TARGET machine's disk (a tmux pane is inherently machine-local), the operator's SINGLE fronting dashboard calls the relay, and the relay forwards one authed hop to the pane's owning machine via the proven `resolvePeerUrls()` path (self → loopback). No durable state is introduced that could strand on a topic transfer (the only state is the existing follow-me `PendingLogin`); the relay generates no URLs that must survive a machine boundary; an offline target returns an honest 502, never a false ok.
- **FD10 — Concurrent-submit safety (codex r2 #1).** A per-login in-flight mutex (`followMeSubmitInFlight`, instance-scoped) holds while the first submit delivers + polls; a concurrent second POST for the same login returns 409 immediately rather than typing a second (stale) code into the same pane. The client also disables the button on submit, so the server mutex is the backstop, not the only guard. Honest scope (internal review): the mutex covers concurrency DURING one submit; after a `submitted` (timeout) return it releases, so a retry re-enters — but the readiness check is the backstop there (it 409s if the prompt advanced). A heavier durable state machine remains unwarranted (FD8/FD20).
- **FD11 — Code character class + length, explicitly (codex r2 #2/#3).** The accepted value is any single printable token with NO whitespace and NO control chars (regex reject `/[\s -]/`), capped at **512 chars** (lowered from 4096 per codex r5 — observed Claude codes are <200; 512 leaves margin without inviting a punctuation blob into the terminal), and a pasted URL (`http(s)://` or containing `://`) is rejected with plain copy. The validation is **reject-unsafe, not enumerate-allowed**: provider auth-code formats vary (URL-safe base64-ish, punctuation), so allow-listing a specific charset risks rejecting a valid code — we only reject what is unsafe to type into a pane (whitespace would submit early / split; control chars are never legitimate). The **server** is authoritative and canonicalizes in this exact order (codex r6 #3): accept a string only → `trim()` leading/trailing whitespace → reject if empty → reject remaining whitespace/control chars → reject a URL shape → length-check the trimmed value (≤512). The client ALSO trims before submitting (so a mobile/password-manager paste with surrounding whitespace is forgiving), but server validation does not depend on the client — it re-trims and re-validates.
- **FD12 — "Submitted" is a normal success-pending state, not a failure (codex r2 #4).** Completion polling watches for the credential file (`<configHome>/.claude.json`) only as a TRIGGER to run the S7 gate. If the file hasn't appeared within the ≤30s bound (atomic/delayed write, version differences), the route returns `outcome: submitted` and the client shows "finishing sign-in" — the existing reissue/complete sweep is the primary completion path and finishes it. The operator never sees a false "done", and "submitted" is never rendered as an error.
- **FD13 — Pane readiness, not just existence (codex r3 #1).** Before typing, the route requires the live pane frame to still show the paste-code prompt (both "paste" and "code" present in the captured frame). A `claude auth login` that exited/crashed/timed out leaves the pane at a SHELL; pane-EXISTS is not pane-WAITING, and typing the code into a shell would execute it (leak + garbage). The check **FAILS CLOSED** (internal security review): we must be able to CAPTURE + verify the pane before typing — if `captureOutput` is unavailable the route refuses 503, and a missing/empty/not-ready frame refuses 409 (the empty-frame case subsumes the old has-session existence test). The code is NEVER typed unless a non-empty frame positively shows the prompt AND is not a shell. This closed an open bypass: the readiness check previously SKIPPED when `captureOutput` was absent, blind-typing the code — a fail-OPEN safety guard (the exact Signal-vs-Authority anti-pattern). (Tested: captureOutput absent → 503, empty frame → 409, shell-prompt frame → 409, `sendInput` never called in any.)
- **FD14 — CSRF / browser-origin posture (codex r3 #3).** Auth is **header-only Bearer** — the dashboard injects `Authorization: Bearer <token>` where the token lives in JS memory (from the PIN exchange), NOT a cookie. A cross-origin attacker page cannot read that token and cannot attach a custom `Authorization` header on a cross-origin request without a CORS grant the server does not give, so there is no CSRF surface. This is the IDENTICAL posture as every existing dashboard-write route (`/mandate/issue-for-machine`, `/subscription-pool/follow-me/enroll/start`, …); these routes add no new auth surface.
- **FD15 — Concurrency model is single-process-per-machine (codex r3 #4).** instar runs ONE server process per machine (`instar server`); it is not clustered, so the instance-scoped in-flight mutex (FD10) is sufficient. The cross-process backstop is independent of the mutex anyway: the provider's auth code is single-use server-side, and `completeFollowMe` removes the pending login on success (a replay is a 404). If instar ever moved to a multi-process model, the mutex would need to become shared state — noted, but out of the current deployment model.
- **FD16 — Prompt detection is a deliberate heuristic, not a brittle exact-match (codex r4 #1).** The readiness check requires both "paste" and "code" in the captured frame rather than an exact prompt string (`Paste code here if prompted >`). This is intentional per Signal-vs-Authority: the CLI prompt wording varies across versions (the same variance that caused the URL line-wrap fix), so a brittle exact-match would false-NEGATIVE and strand valid codes. The realistic failure mode the check defends against — the login process EXITED and the pane dropped to a shell — never shows both words, so the heuristic is adequate; it sits on top of has-session (it is OUR enroll pane, not an arbitrary shell). The accepted residual: a frame whose scrollback happens to contain both words could pass; the downside is bounded (the code is single-use and goes to our own login pane). Mitigated further by capturing only the last ~12 lines (near the live prompt, codex r5 #1) AND a NEGATIVE shell-prompt check (codex r6 #1): the last visible line must NOT end in a shell sigil (`$`/`%`/`#`) — an exited login drops to a shell prompt ending in one of those, while the real Claude prompt ends in `>`. Both the positive (paste+code present) and negative (not-a-shell) checks must hold before a single char is typed. A PTY-state wrapper that records an explicit `waiting_for_code` was considered and rejected as a large rebuild disproportionate to a screen-scrape that already gates on has-session + positive + negative checks.
- **FD17 — "Pane not ready" vs "bad code" are distinct, recoverable UX (codex r4 #2/#3).** A pane-not-at-prompt returns 409 with "re-tap Approve to start a fresh sign-in" (which regenerates the login); a malformed code returns 400 with "paste only the code"; a pasted URL returns 400 with "paste just the code, not the whole link". A stranded single-use code is recovered by re-tapping Approve (a fresh login + fresh code) — the client copy says so, distinctly from a bad-code message.
- **FD18 — The relay is an operator-authenticated surface, not an anonymous one (codex r4 #4).** submit-code is reachable only with the dashboard's Bearer token (FD14); it is not a credential-guessing surface (the provider validates the code, instar does not), so anonymous brute-force is not the threat model. Repeated submits are bounded by the in-flight mutex (no concurrency), the pane-readiness check (a typed-garbage attempt requires the pane still at the prompt), and the single-use code + login-removal one-shot. A per-login attempt counter was considered (raised again at r6) and REJECTED, for a concrete reason: a MALFORMED submit is rejected at validation BEFORE `sendInput`, so it can never type garbage into the pane; the only thing that reaches the pane is a well-formed code, which is the operator's own legitimate retry (single-use, provider-validated). A blind total-submit cap would therefore mostly LOCK OUT a legitimate fat-finger retry while adding no protection the validation + readiness + single-use code don't already give — brittle infra against a non-threat (Signal-vs-Authority). The surface is operator-Bearer-authed, not anonymous. Should real abuse ever be observed it is a cheap additive guard at that time.
- **FD19 — Authorization inherits the existing follow-me/mandate gates unchanged (codex r4 #5).** Live-User-Channel-Proof: because this feature exists precisely because a defect reached the operator on first use (2026-06-18), it is NOT 'done' until a user-role drive through the REAL dashboard card on the real machines completes the paste-back end-to-end (the completion gate's live-test artifact). This feature adds NO new authorization decision: the enroll gate (deny-by-default mandate evaluation) authorized the follow-me in the first place; submit-code only delivers the operator's own code into the already-authorized, already-pending login. The S7 email-gate still independently validates the minted account before it is added to the pool. On authority specifically (codex r6 #4): possession of the dashboard Bearer token IS full operator authority — the operator is the single principal across all THEIR machines, so there is no per-pending-login ACL to check beyond Bearer possession + the dev-gate; the relay verifies the target machine via `resolvePeerUrls()` and the pending login via `id`, and S7 re-anchors the account identity to the operator-approved `expectedEmail` at completion.

- **FD20 — Crash-during-submit recovery is bounded and safe (codex r7 #2).** The in-flight mutex is in-memory, so a server crash in the narrow window AFTER `sendInput` but BEFORE `completeFollowMe` could leave the code consumed provider-side while the pending login still looks active. This degrades safely with the EXISTING machinery, no new durable state required: on restart the reissue/complete sweep re-examines pending logins — if the credential landed it completes (validated/held via S7); if not and the TTL expired it reissues; and the operator can always re-tap Approve to start a fresh login (a fresh single-use code), which the kind + readiness guards make safe. We deliberately do NOT add a durable `codeSubmittedAt` phase marker: it would introduce per-login durable state (against this feature's machine-local, no-new-durable-state posture, FD9) to smooth a crash inside a ~30s window whose worst case is one operator re-tap. If that window ever proves painful in practice it is a cheap additive marker at that time.
