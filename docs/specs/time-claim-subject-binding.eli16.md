# TIME_CLAIM subject binding — plain-English overview

TIME_CLAIM exists to stop an agent from guessing how long its active autonomous session has been running. Before a message is delivered, it recognizes a statement such as “seven hours elapsed” or “two hours remaining” and compares that number with the real session clock. On July 23, that useful check interrupted two correct Echo messages for the wrong reason. One message described the length of an offline test window; another described detection latency. Neither said how long the session had been running, but both looked enough like duration phrases to the old extractor that it compared them with the session clock. Echo had to use the audited advisory override twice.

The fix adds the missing semantic boundary: a quantity and unit are not enough; the verifier must also bind the thing being measured. An anchored duration remains a session-clock candidate when the text explicitly names the run, session, clock, or time box. Existing unqualified shapes such as “seven hours elapsed” also remain candidates because the caller supplies an active session clock and that behavior is the original safety contract. However, when the local clause names a competing duration subject—such as a test window, latency, ETA, timeout, CI queue, outage, or recovery—the candidate is dropped instead of being compared with the session.

This change does not weaken verification of actual session-clock claims. Grossly wrong elapsed, remaining, and percentage claims still produce the same TIME_CLAIM advisory; accurate claims still pass with the existing tolerance. The change only prevents the checker from manufacturing a contradiction by measuring the wrong subject.

The class-level repair is larger than the two phrases. The standards registry now requires every quantitative verifier to bind both measurement and subject. Its testing rule requires both sides in the same decision table: real subject-bound claims that must verify, and realistic competing subjects that must pass through. Focused unit, route, and live-path E2E tests exercise that boundary.
