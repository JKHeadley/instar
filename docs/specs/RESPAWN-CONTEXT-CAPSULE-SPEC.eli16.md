# Respawn Context Capsule — ELI16

When an agent's server restarts for an update, it usually takes the agent's working session down with it. The replacement session gets a summary of the recent CONVERSATION, so it remembers what was talked about — but not where the WORK stood. It has to re-figure out which task it was on, which build folder it was using, how far through the quality checks it was, and what it was about to do next. We have watched this go wrong three times in two days: once the replacement rebuilt its workspace against the wrong code and burned fifty minutes on a cascade of confusing errors; once a supervisor had to hand-type the missing context back in; and the workspace-rebuilding half of the problem produced folders with no safety checks wired at all.

The fix is a "capsule": a tiny structured note, one per session, saved to disk at natural milestones — workspace created, quality gate passed, pull request opened. It records just the facts: task id, build folder and branch, how far through the gate, and one line saying what comes next. No secrets, no conversation text, size-capped. Two things write it: the session itself at milestones (one cheap command), and the workspace-creation tool automatically (it already knows every workspace fact — so even a session that never checkpoints gets the most valuable part for free).

When a killed session is respawned, the spawner reads the capsule and puts a short "RESUMING WORK" note at the top of the new session's instructions, right next to the conversation summary. The capsule is a hint, never gospel: if reality has moved on — the branch already merged, the folder deleted — the new session trusts what it can verify and says so. Old capsules are flagged as possibly stale instead of silently used. The feature starts enabled only on the two development agents until it proves itself, and clearing or ignoring it is always safe because everything in it can be re-derived — it just saves the fifty minutes and the hand-holding.
