# Build-Session Yield Safety

## 1. The failure this closes

A background build / autonomous session edits files in its worktree, says "standing by for tests," and then **yields** — and because a `claude -p`-style session ends the moment the model stops emitting, it dies with the work **uncommitted and unpushed**. Real work, zero delivery, invisible for hours (the 2026-06-12 topic-22367 incident).

Three mechanisms already exist around this failure; this spec is scoped to the gap **none** of them close:

| Mechanism | Shipped | Covers | Gap |
|-----------|---------|--------|-----|
| PromiseBeacon escalation ladder (#1097) | yes | a dead session's open **commitment** is revived/escalated | uncommitted **git** work (no commitment represents it) |
| Mid-work ResumeQueue | yes | revives a session reaped with work-evidence (`midWork`) | a dirty worktree is **not** an evidence type, and a revived session has no durable follow-through to actually **commit** the work |
| OrphanedWorkSentinel (#1113) | in review | **detects + surfaces** an already-dead session's dirty worktree (records + one notice; optional non-destructive preservation patch) | post-hoc only; does not feed revival, no follow-through |

**The delta:** (A) make "uncommitted worktree work" a first-class **resume signal**, and (B) give a revived session a **durably-tracked obligation** to commit/preserve that work — closing the loop — with a **non-destructive preservation patch** as the safety net. Honest scope: this is **loss-REDUCTION, not a guarantee** — a revived session can still die; the design makes that case loud and recoverable, never silent.

## 2. Design

### R1 — `uncommitted-worktree-work` as STRONG work-evidence (collected PRE-kill)

Add one STRONG signal to `src/core/WorkEvidence.ts`: the string literal `uncommitted-worktree-work`.

**Where it is computed — NOT on the kill chokepoint.** The git dirty-check is **collected by the killer, before `terminateSession` is called**, exactly like the other evidence the chokepoint already receives and clamps. The chokepoint stays synchronous-git-free: a per-kill `git status` on `SessionManager.terminateSession` would block the event loop for EVERY termination and serialize under quota-shed (many concurrent kills) — rejected. Killers (`SessionReaper`, quota-shed migrator, boot reconcilers) run the dirty-check in their own decision loop, where blocking is already expected and pressure-aware. **De-duplicated under concurrent pressure:** results are cached per resolved-worktree-path for a short TTL (default 30s) so that N killers reaping overlapping sessions in the same burst do NOT each spawn a git subprocess on the same worktree — the first computes, the rest read the cache (a worktree's dirty state does not meaningfully change inside one shed burst).

**Dirty-check definition (deterministic):** `git status --porcelain` over the session's resolved worktree, **non-empty** after dropping ignored files (porcelain excludes `.gitignore`/`core.excludesFile` by default). Staged, unstaged, and untracked-but-not-ignored changes all count. The check **reuses the OrphanedWorkSentinel / `agentWorktreeGit` dirty-check — ONE implementation**, invoked through `SafeGitExecutor.readSync(['-C', <resolvedPath>, 'status', '--porcelain'], <resolvedPath>)` with **array args only, never string interpolation** (a worktree path/branch with spaces, quotes, or a leading `-` can never become a git option or shell token).

**Bounded + fail-open:** the dirty-check runs under a hard **5s** timeout that **kills the subprocess** on expiry; on timeout, git error, missing/unresolvable worktree, or a non-git path the signal is simply **absent** (never a spurious revive, never a wedge). The worktree path is `realpath`-normalized before the check to defeat symlink stalls/escapes (and the normalized path is rejected if it still begins with `-`); the normalization itself is **time-bounded** (a circular/pathological symlink chain that would hang `realpath` is abandoned under a short 2s budget → signal absent), so no part of the dirty-check — resolution or git — can hang the killer loop unbounded. A timeout is not silent: it bumps a per-worktree consecutive-timeout counter (reset on any successful check) and is logged at WARN; **≥3 consecutive timeouts on the same worktree raises ONE HIGH Attention item** ("this worktree can't be dirty-checked — work may be silently accumulating"), so a worktree large/slow enough to permanently defeat the check can't silently disable the safety net.

**Authority — explicit kills are never overridden (closes the operator-revive hazard).** `uncommitted-worktree-work` being STRONG makes a reap resume-eligible **only on the eligibility paths the ResumeQueue already admits** — it does **not** override the existing origin veto. An **operator kill, an explicit per-topic user stop, or an emergency stop is NEVER auto-revived on a dirty worktree alone** (the ResumeQueue's `origin === 'operator'` exclusion remains final; this spec adds no `includeOperatorKills` bypass). The signal earns revival for *autonomous/pressure* reaps that the queue already revives — never against a human's explicit "stop."

**Work-quality floor (no junk revives).** Ignored files are already excluded. Additionally, a reap whose ONLY dirty entries match a build-residue denylist (`dist/`, `node_modules/`, `*.log`, `.cache/`, coverage output) does NOT raise the signal — a build that rewrites artifacts every start can't drive an endless revive loop. The denylist is config-overridable.

### R2 — Close-the-Loop obligation (a SIGNAL + a tracked beacon, NOT a blocking gate)

The first draft proposed a hard "block the revived session's first yield until the worktree is clean." **Rejected on Signal-vs-Authority**: a brittle check holding blocking authority over a session's yield is the exact anti-pattern instar fights, and the need for a fallback proves a block is not a guarantee. (An external review suggested instead *caging* the revived session into a "may only commit/stash/discard" state — that is the same brittle-block by another name and is **deliberately declined**; the residual risk that a non-compliant session ignores the directive is accepted and covered by the durable beacon + preservation net below.) R2 is therefore **structural ENABLEMENT, not a block**:

1. **Signal (directive).** A revived session carrying `uncommitted-worktree-work` gets this exact directive as the **first line of its continuation prompt** (the existing `ResumeQueueDrainer` continuation-prompt path, made evidence-specific), verbatim: *"You were revived because your worktree had uncommitted changes from a prior session. Before any other work: review the dirty files and either commit them with a real, descriptive commit, or deliberately preserve/discard them."* When `build-or-autonomous-active` is ALSO present, a second sentence names the build too, so neither obligation is hidden. This informs the mind; it does not act for it.

2. **Close the Loop (durable beacon).** On that revival the system registers ONE **revival-scoped commitment** through the existing `CommitmentTracker` (beacon-enabled), counterparty = self/system: *"commit or deliberately resolve the uncommitted worktree."* The shipped **PromiseBeacon** re-surfaces it on cadence and **escalates if unresolved** — so the obligation survives even if the revived session stalls (not just if it dies). **Dedup:** a later revival of the SAME session+worktree with an already-open such commitment **refreshes that row's atRisk timestamp — it never opens a second** (no beacon flooding); a fresh row is opened only after the prior one is delivered/dismissed.
   - **Delivery is driven by the EXISTING OrphanedWorkSentinel background dirty-check — no new polling loop.** The agent cleans the tree; the next scheduled sweep observes the result and delivers. **Anti-gaming (resolution has teeth):** delivery requires the worktree clean **AND ≥1 non-empty commit on the branch since the revival** (a `--allow-empty`/no-op commit does NOT satisfy it). A worktree that goes clean with **zero new commits** — i.e. the work was `reset`/`stash`/discarded away — does **NOT** auto-deliver: it triggers the preservation patch first and escalates the commitment at HIGH as "resolved without saving," so destruction can never read as "done."

3. **Preservation patch (the safety net — never an auto-commit to history).** If the revived session dies again before resolving (revive-of-revive), the queue's give-up path — bounded by the ResumeQueue's existing resurrection cap (default surfaced as **2**; after the cap, preserve instead of revive again) — writes a **non-destructive preservation patch** via the OrphanedWorkSentinel `preserveWork` path: a read-only `git diff` of tracked changes **+** a copy of untracked-non-ignored files into `state/preserved-work/<sessionId>-<ISO8601>.patch`, **never** touching the index/ref/published history. (A patch file is used rather than `git stash` deliberately: it lives OUTSIDE repo state, so it cannot be accidentally `pop`-dropped or clobbered by a later stash, and it survives a worktree removal.) Safety bounds on the write (anti-DoS): a per-file cap (**50 MB**) and a total cap (**100 MB**) — over either, the tracked diff is still preserved but the untracked copy is skipped and a HIGH item is raised; symlinks are NOT dereferenced; the destination is asserted to stay under `state/preserved-work/` (no path-escape). **Secret handling, both halves:** untracked files matching a hardcoded denylist (`.env*`, `*.key`, `*.pem`, `id_rsa*`, `id_ed25519*`) are excluded from the copy, AND the tracked-`diff` half is line-scrubbed against a secret-pattern regex (`(?i)(password|secret|token|api[_-]?key|credential|private[_-]?key|aws_secret)`), each match replaced with `[REDACTED]` before the patch is written (an accidentally-committed `.env` can't leak through the diff). **Each** patch-write raises **one** Attention item (honest cardinality — a session that keeps dying surfaces each cycle, never deduped into a single muted row). If the patch **write itself fails** (disk/permission), that is raised at HIGH and the entry goes to a terminal `preserve-attempt-failed` state — the worktree is treated as unsafe and is NOT revived again (no infinite re-attempt on a full disk); the safety net never fails silently.

### Deconfliction (mandatory)

- **Extends, never duplicates.** R1 adds one value to the existing evidence enum and feeds the existing eligibility classifier; R2 reuses `CommitmentTracker` + `PromiseBeacon` + the OrphanedWorkSentinel preserve path + the existing resurrection cap. No parallel queue, no new beacon engine, no new cap.
- **vs #1097 (PromiseBeacon ladder):** orthogonal. A session with BOTH an open commitment AND a dirty worktree enqueues **once**; the continuation prompt surfaces both; the dirty-work obligation is its own commitment row, independent of any pre-existing one.
- **vs `build-or-autonomous-active`:** when both signals are present on one reap, dedupe to a **single** queue entry (the entry carries both evidence names); revival happens once.
- **vs #1113 (OrphanedWorkSentinel):** complementary. #1113 is the post-hoc detector for sessions that died **outside** the reap chokepoint (crash, OOM) — it still surfaces those. R1/R2 cover the **reaped-with-evidence** path and add the revival + follow-through #1113 lacks.
  - **Ship-order dependency (declared):** R2's commitment delivery is observed by #1113's background sweep, so this spec ships **after** #1113 is merged AND on a machine where #1113 is live (both are dev-gated dark — on the dev agent both resolve live together; on the fleet both stay dark, so there is no machine where R2 is live but the sweep is not). **Fallback** if the sweep is unavailable/disabled where R2 is live: the ResumeQueue's own drain tick performs the delivery dirty-check (the same bounded check as R1) — delivery never hard-depends on #1113 being healthy, it only prefers its existing sweep to avoid a second loop.

### Multi-machine posture (mandatory declaration): machine-local-by-design

A worktree's dirty work physically exists on **one** machine's disk, and the ResumeQueue is per-machine. Posture: **machine-local by design.** The evidence is collected and the entry enqueued on the machine that holds the worktree. If the topic has since **moved to another machine**, the revival on the new machine MUST detect the worktree is **inaccessible/non-existent locally** (an early `pathExists` + git-repo check in `ResumeQueueDrainer.validateReality`) and **fail cleanly** — invalidate with reason `worktree-not-on-this-machine` and **never** attempt to revive or commit work on a machine that does not have it. **The work is not lost by the refusal, because it never left the holder.** The dirty worktree physically remains on the machine that holds it, and *that* machine runs its own R1/R2 against it on its own reaps — so the holder is the recovery path, with no cross-machine coordination required. The new machine simply **refuses to act on work it cannot see** and records the refusal (`state/cross-machine-invalidations/<topicId>.jsonl`, raised, never silently dropped) purely so the no-op is auditable — not because the new machine is responsible for the recovery. A single-machine agent hits none of this. (Cross-machine *carry* of the patch artifact is out of scope and tracked separately — this spec guarantees **no false action on the wrong machine**, not teleport of the diff.)

## 3. Safety / posture

- Ships **dark + dev-gated** via the `developmentAgent` gate (config `enabled` OMITTED, resolved through the gate — same shape as OrphanedWorkSentinel #1113); live on the dev agent first. Registered in `DEV_GATED_FEATURES` (lint-enforced) and `ConfigDefaults` (`monitoring.resumeQueue.yieldSafety`), backfilled to existing agents by `PostUpdateMigrator.migrateConfig` (existence-checked, idempotent). Agent-awareness: a CLAUDE.md-template line + an `upgrades/next` fragment + a side-effects artifact ship with the implementation PR.
- **R1 is read-only.** R2's only mutations are loss-REDUCING: the preservation patch (reversible; never touches the index/ref/published history) and, if the session itself chooses to, its own commits. The system **never auto-commits on the agent's behalf** — that would pollute git history with low-quality non-semantic commits, possibly broken code, under a misattributed author; a side patch-file is the correct artifact. **Secret-handling is a curated best-effort, not a guarantee:** it denylists secret-NAMED untracked files (`.env*`, `*.key`, `*.pem`, `id_rsa*`, `id_ed25519*`) and redacts a curated set of secret-pattern lines from the tracked diff — it covers the common cases, NOT every secret shape (a novel connection-string or header format can slip through), so a preserved patch is not a secrets-management substitute and an operator should review one before surfacing it. The claim is narrow and honest: the feature does not *introduce* a new plaintext-secret artifact for the common patterns, it does not promise to scrub a determined or exotic secret.
- **Fail-open everywhere:** git error/timeout/hang, missing worktree, wrong machine, or a disabled gate all degrade to today's behavior (worst case: #1113 flags it post-hoc) — never a wedge, never a false block, never a silent loss.
- **Observability:** the new evidence value + the revival commitment appear in the existing `GET /sessions/resume-queue` and `GET /commitments`; the preservation patches are listed under the OrphanedWorkSentinel surface. No new bespoke route required. A dirty-check that **times out** is logged at WARN (not silently dropped) — so a worktree large/slow enough to consistently time out the 5s check, which would silently disable the feature for that session, is visible to the operator rather than invisible.

## 4. Frontloaded Decisions

These are resolved here with defaults so the building agent never stops mid-run; all ship behind the dark dev-gate and are config-overridable (cheap-to-change), EXCEPT the two marked **durable** (a user-visible/identity surface — deliberately fixed now, contested-and-accepted as non-cheap):

- **Dirty-check timeout:** 5s, hard subprocess-kill, fail-open (absent signal), timeout logged at WARN. `monitoring.resumeQueue.yieldSafety.dirtyCheckTimeoutMs`.
- **Dirty definition:** `git status --porcelain` non-empty after ignored-exclusion; staged+unstaged+untracked-non-ignored count.
- **Build-residue denylist (non-exhaustive — a starting default, per-project tuning expected):** `dist/`, `build/`, `out/`, `.next/`, `.nuxt/`, `.turbo/`, `node_modules/`, `coverage/`, `*.log`, `*.tsbuildinfo`, `.cache/` — overridable at `...yieldSafety.residueDenylist`. Documented as a tuning parameter: an over-broad entry can false-negative a real generated file (no revive); an under-broad list can let a build-artifact-only worktree revive once (bounded by the cap). Neither loses work.
- **Resurrection cap:** **default 2** revives before the give-up path preserves instead of reviving again — the ResumeQueue's existing cap mechanism, value surfaced here. `...yieldSafety` inherits it; recommend keeping it small (2–3).
- **Commitment resolution (frontloaded):** satisfied ONLY by (worktree clean **AND** ≥1 non-empty commit on the branch since revival) **OR** an explicit operator acknowledgment. A clean-via-`reset`/`stash`/discard with no new commit does NOT satisfy it (→ preservation + HIGH escalation). Delivery is observed by the existing OrphanedWorkSentinel sweep, not a new poll.
- **Preservation patch content (frontloaded):** tracked changes as a unified `git diff`; untracked-non-ignored files copied raw (no deref of symlinks), excluding a hardcoded secret denylist (`.env*`, `*.key`, `*.pem`, `id_rsa*`, `id_ed25519*`); per-file cap 50 MB, total cap 100 MB (over → diff-only + HIGH item).
- **Evidence enum literal (durable):** `uncommitted-worktree-work` (appears in logs/`/sessions/resume-queue` — renaming later is a migration, so fixed now).
- **Config root (durable):** `monitoring.resumeQueue.yieldSafety` with `enabled` OMITTED (dev-gate resolved).
- **Preservation patch location (durable):** `state/preserved-work/<sessionId>-<ISO8601>.patch` (reuses #1113's path; ISO-8601 timestamp for stable sort/parse — operators and tooling locate patches by this path, so it is a published artifact contract, fixed now).
- **Revival directive (durable):** the verbatim first-line text in §2 R2.1 (agents adapt to its exact phrasing/placement — a user-visible contract, fixed now). When `build-or-autonomous-active` is also present, append this exact second sentence: *"A build/autonomous run was also active when the prior session ended — after resolving the worktree, check on or restart that work too."*
- **Secret scrubbing (durable, hardcoded — not config-overridable):** untracked-file denylist `.env*`, `*.key`, `*.pem`, `id_rsa*`, `id_ed25519*`; tracked-diff line-redaction regex `(?i)(password|secret|token|api[_-]?key|credential|private[_-]?key|aws_secret)` → `[REDACTED]`. Hardcoded (not a config knob) so it can't be weakened by config; curated/non-exhaustive by design (see §3).
- **Attention cardinality + terminal state:** each preservation patch-write raises exactly ONE Attention item, never deduped across a session's repeated deaths; a patch-write FAILURE → HIGH item + the entry's terminal `preserve-attempt-failed` state (no further revival of that entry).
- **Directive delivery + injection point (durable):** prepended by the existing `ResumeQueueDrainer` continuation-prompt builder as the first line of the **user-facing** continuation message (not a system prompt, env var, or hidden file); the §2 R2.1 text appears verbatim, and any refactor of continuation-prompt assembly must preserve this prepend.
- **Evidence-strip ordering:** an entry's evidence is cleared ONLY AFTER the continuation prompt has been injected. A pre-injection `validateReality` failure (worktree vanished mid-flight) invalidates the entry with a named reason + (if it had real evidence) a notice — so a revival can never silently drop the directive and lose the dirty-work signal.
- **Config schema (durable shape):** `monitoring.resumeQueue.yieldSafety: { /* enabled OMITTED — dev-gate resolved */ dirtyCheckTimeoutMs: 5000, resurrectionCap: 2, residueDenylist: ["dist/","build/","out/",".next/",".nuxt/",".turbo/","node_modules/","coverage/","*.log","*.tsbuildinfo",".cache/"], preservationMaxFileBytes: 52428800, preservationMaxTotalBytes: 104857600 }`.
- **Operator/explicit-kill policy:** never auto-revived on dirty-worktree-alone (origin veto final).
- **Dedup key + multi-worktree cardinality:** the signal + commitment are keyed on `(sessionId, resolved-worktree-path)`. A session with TWO dirty worktrees gets TWO independent commitment rows (one per worktree); a re-revival of the SAME `(sessionId, worktree)` refreshes its single row, never opens a second.
- **`preserve-attempt-failed` is a do-not-revive blocklist entry:** once an entry reaches it (patch write failed), that `(sessionId, worktree)` is blocklisted per-machine — a later reap is immediately invalidated with `preserve-attempt-failed-prior` and never re-enters the preservation path (no full-disk retry loop); cleared on a later successful preservation or operator override.
- **Cross-machine invalidation is terminal on the refusing machine:** an entry invalidated `worktree-not-on-this-machine` is not revalidated there — recovery is the holder's own local R1/R2, never a retry on the wrong machine.
- **Dirty-check cache TTL:** 30s per resolved-worktree-path (`...yieldSafety.dirtyCheckCacheTtlMs`); realpath budget 2s; both fail-open.

## 5. Test plan (3-tier)

- **Unit:** `uncommitted-worktree-work` clamps + classifies STRONG; dirty-check both sides (clean → absent, dirty → present), ignored-only → absent, residue-only → absent; fail-open on git error/timeout (and the timeout logs WARN); array-arg invocation survives a worktree path with spaces/quotes/leading-dash; operator-origin reap with dirty tree → NOT enqueued; dedupe when both build-active + uncommitted present → one entry; resolution gate: clean+non-empty-commit → satisfies, clean+empty-commit → does NOT, clean+no-commit (reset/stash) → does NOT (→ preserve+HIGH); preservation bounds: a >50 MB untracked file → skipped + diff preserved + HIGH; secret-denylisted untracked file (`.env`) → never copied; destination stays under `state/preserved-work/`.
- **Integration:** a pressure reap with a real dirty temp worktree enqueues with the evidence; revival registers ONE beacon-enabled commitment (visible in `/commitments`); a second revival of the same worktree refreshes that row, does NOT open a second; `/sessions/resume-queue` shows the entry; a moved-topic revival on a machine lacking the worktree invalidates with `worktree-not-on-this-machine`, does NOT act, and (holder offline) writes the `cross-machine-invalidations` record.
- **E2E:** boot → pressure-reap a session with a real dirty worktree → revive → assert the continuation prompt carries the verbatim commit-first directive AND a commitment row exists → a real non-empty commit lands → the existing sweep delivers the commitment; separately, revive-of-revive past the cap → assert a preservation patch exists at the documented ISO-named path + one Attention item per patch-write; a patch-write failure raises a HIGH item (never silent).
