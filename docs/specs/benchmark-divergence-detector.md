# Benchmark-Divergence Detector — real grades vs benchmark predictions

**Origin:** operator, topic 11960, 2026-07-12: *"our current benchmark assumes some models perform very well in certain scenarios; if the real-life data suggests otherwise then we need to re-evaluate all aspects of that scenario, including whether the model is getting enough context, whether the model has an appropriate prompt, and whether the benchmark is appropriately representing the scenario."* Operator approved building the loop ("yes, please proceed") and gave two first-class requirements (R1, R2 below).

**This spec = Increment A** of the benchmark-feedback loop: the divergence DETECTOR. Observe-only, in-repo. Increment B (bench prompt-parity CI ratchet, ACT-1195) and Increment C (representative-cases feed) are **explicitly non-scope** (see Non-scope). <!-- tracked: ACT-1195 --> The A build proceeds identically regardless of the operator's pending A-only vs A+B scope steer — that choice sequences B, it never blocks A.

**Parent standard:** *Decision Provenance & Outcome Review* — whose rule text explicitly includes "graded real cases feeding its bench battery"; this detector is the review-the-outcomes-against-the-promise layer. Dual lineage acknowledged: the compounding-improvement signal it produces is squarely *Never-Waste Feedback — corrections compound*.

**Dependency status (honest):** the LLM-Decision Quality Meter (`docs/specs/llm-decision-quality-meter.md`) — operator-approved, **PR #1458 MERGED 2026-07-12T20:46Z** — the build-ordering constraint (this spec extends the meter's annotate-chokepoint and reads its tables) is now SATISFIED. Live *function* remains gated on the meter's `provenance.uniformSeam` being live and grades accumulating.

## Problem statement

The **benchmark** (INSTAR-Bench v2) is a PREDICTION: per (task, model), a battery-run pass-rate asserting "model M is good at scenario S." The **quality meter** is GROUND TRUTH: per decision-point, real production grade outcomes (right/wrong/unknown). Nothing compares the two today, and the grade data ages out on retention clocks before anything learns from it. This feature compares real grade-rates to predicted rates per (decision-point × model), guarantees the comparison consumes the data before retention deletes it (R1), across every machine (R2) — and never lets a stale or unfaithful benchmark blame (or credit) a model.

## Data flow — THREE distinct stages

1. **RECORD (continuous, per-decision):** the meter's correlation spine + provenance. No benchmark involvement.
2. **GRADE (evidence-triggered):** the meter's graders (the grading pass AND the sentinel/realcheck annotate arms) stamp right/wrong/unknown through ONE chokepoint — `upsertOutcome → recomputeQualityBucketInTx` — which already maintains the meter's `decision_quality_rollup` by recompute-from-raw, plus a boot/6h reconcile. **Increment A extends exactly that chokepoint** (and the reconcile) to ALSO maintain a NEW per-model table (`decision_quality_rollup_by_model`, below). Grades land in the DECISION's UTC day, and a day's buckets legally keep mutating as late evidence matures (the meter's design) — the detector's consumption model is built for that (FD7).
3. **ANALYZE (periodic — THIS feature):** on the serving-lease holder only, the detector re-reads a **rolling window** of matured days from the per-model table (never raw rows), pool-collects every machine's aggregates (R2), compares to the mirrored benchmark passRate, and idempotently upserts findings carrying the ranked questions.

The improvement fork: a divergence routes to **fix-the-SYSTEM** (context/prompt) vs **fix-the-BENCHMARK** (drift/unrepresentative). The detector points; humans and later increments fix.

## Frontloaded decisions

- **FD1 — Prediction source (in-repo mirror), grounded against the live batteries.** The "prediction" is the battery pass-rate: `aggregate.mjs` computes `passRate = passes / deterministic` per `(taskId, model)`. The mirror is a small, content-free, git-tracked file: `{ taskId → { perModel: { <batteryModelId> → { passRate, passes, deterministic } }, benchedPromptSource, promptFidelity, benchedPromptHash, capturedAt } }` — **counts, not just the rate**, so the prediction's own sampling uncertainty is weighable (FD3). `benchedPromptHash` = sha256 over the task's resolved `promptTemplate` under ONE pinned canonicalization (exact template string, LF line endings, no trim) used identically by the off-repo pull and the in-repo live hasher. `benchedPromptSource` (e.g. `src/core/CompletionEvaluator.ts:137`) is **annotation/cross-check ONLY — it is NEVER dynamically imported or resolved as a filesystem path** (a mirror field is untrusted data; path characters pass any id clamp, so the only safe resolver is static — FD6). Mirror population is an operational pull step; staleness handling is FD4 (incl. the missing-file fail-closed state); refresh *ownership* is a tracked residual (Non-scope) — softened by the structural backstop that an unrefreshed mirror degrades EVERY finding to `precondition-failed` + chronic, a failing status no reader can mistake for health. Runtime residency stated honestly: the mirror is meaningful on a source checkout (the dev agent — consistent with the dev-gated ship posture); a dist-only install without the JSON simply reports `mirror.present: false` (the same honest-absence pattern as the doorways registry). Posture: **unified** (git-tracked, identical on every checkout).
- **FD2 — Comparison unit and arithmetic (per decision-point × model, settled-grades denominator, honesty bounds).** The runtime join is an **enrolled-pair exact-match table `decisionPointId → taskId`** seeded for the two wave-1 pairs (`LLM_BENCH_COVERAGE` stays the CI coverage ratchet — its keys are COMPONENT names with compound task strings like `completion-judge+p13-stop-judge` and are NOT the runtime join); no entry ⇒ pair not enrolled (fail-closed). For each enrolled pair: `realGradeRate = right_n / (right_n + wrong_n)` over settled grades; `gradedN = right_n + wrong_n ≥ minSample`; `unknownShare = 1 − (right_n + wrong_n) / decided_total` where **`decided_total` is a per-bucket column counting ALL recorded decisions (graded or not)** — the meter's rollup `unknown_count` only counts explicit unknown outcome ROWS, so without `decided_total` the unsettled-stream gate would be decorative (a point-day with 25 settled grades atop 500 ungraded decisions must NOT sail through). `unknownShare > maxUnknownShare` (0.5) ⇒ `insufficient-evidence`. The meter's per-point orphan share (its rollup `orphan_outcomes` column, aggregated per point across the window; orphan buckets key on the OUTCOME's day — tolerated skew) above `maxOrphanShare` (0.10) ⇒ verdict `partial` with `orphanTainted: true` until the meter's ACT-1199 routing lands. <!-- tracked: ACT-1199 -->
- **FD3 — Divergence test (deterministic, two-sided CI-aware — a product HEURISTIC, stated honestly).** Flag only when `|realGradeRate − predictedRate(model)| > max(divergenceThreshold, wilsonHalfWidth95(real, gradedN), wilsonHalfWidth95(passRate, deterministic))` — BOTH proportions carry sampling noise (a 10-case battery has ±~0.26 at p=0.5; treating its rate as exact manufactures divergence out of battery noise). This is a deliberately conservative detector threshold, NOT a formal two-proportion significance test (a Newcombe-style difference interval is a possible refinement; the max-of-half-widths form is chosen for simplicity and its conservative bias). Every finding carries `gradedN`, `unknownShare`, `ciHalfWidth`, `benchN` (= deterministic), `benchCiHalfWidth`. Deterministic formulas, no new dependencies.
- **FD4 — Verdict enum (frozen, direction-split, precondition-first).** `verdict ∈ { divergent-worse, divergent-better, aligned, insufficient-evidence, no-benched-baseline, precondition-failed, partial }`, `preconditionReason ∈ { prompt-drifted, prompt-drifted-within-window, hash-unverifiable, stale-mirror }`:
  - **Q0 — the template-hash precondition** (named honestly: it verifies the PROMPT TEMPLATE only; runtime context assembly / tool schemas / retrieval are Increment C's representativeness question, and non-hashed inputs are listed in finding metadata, never implied covered). Benched hash ≠ live hash ⇒ `prompt-drifted` — a benchmark bug routed to Increment B, NEVER a model verdict; suppresses divergent AND aligned (a false `aligned` on a drifted benchmark is as wrong as a false `divergent`). Live hash uncomputable ⇒ `hash-unverifiable` (never treated as faithful). The analysis window mixing prompt identities (any contributing bucket `prompt_id = '__mixed__'`, or differing recorded ids across the window — the decisions RAN on different prompts, P20 verify-the-state) ⇒ `prompt-drifted-within-window`. The staleness/drift predicates are DECIDABLE from specified state only: benched hash ≠ live hash ⇒ `prompt-drifted`; mirror age > `mirrorStalenessMaxDays` ⇒ `stale-mirror`; BOTH ⇒ `stale-mirror` (refresh the mirror first — an aged mirror makes the drift comparison moot). A mirror file that is MISSING or unparseable at analysis time ⇒ `stale-mirror` suppression for ALL enrolled pairs with the envelope's `mirror.present: false` — an operational failure is NEVER misreported as `no-benched-baseline` (a per-model coverage claim). Staleness SUPPRESSES actionable verdicts; it never merely annotates.
  - **Direction split:** `divergent-worse` carries questions 1–3; `divergent-better` leads with "is this grade-rate inflated? (evidence rule too lenient / recurrence-suppressible / denominator)" + the meter's evidence-strength reference — over-performance is the signature of a gamed/lenient grade stream and must never read as "promote this model."
  - `no-benched-baseline` (deduped per distinct normalized model; `unmapped: true|false` distinguishes a mapping miss from a genuinely-unbenched model).
- **FD5 — Model-id normalization (fail-closed, versioned, in Increment A).** A versioned in-repo exact-match table maps production model ids ↔ battery model ids; **fuzzy/substring matching forbidden**; entries must map IMMUTABLE ids (a known-mutable alias like `*-latest` is refused at table validation); unmapped ⇒ `no-benched-baseline (unmapped: true)`. Ships seeded for the wave-1 pairs; a near-empty table degrades to the safe inert default.
- **FD6 — Live prompt hash (static registry — no dynamic resolution).** A compile-time in-code registry `taskId → templateExport` (static imports of the wave-1 prompt builders' exported templates; **never a file path, never a line number** — line refs rot and paths invite traversal). The detector hashes the exported template under the FD1 canonicalization. Registry-vs-mirror `benchedPromptSource` mismatch ⇒ `hash-unverifiable`. Q0 additionally checks the window's RECORDED prompt identity (FD2's bucket `prompt_id`): uniformity is required (FD4), because the current source's hash says nothing about what days-old decisions actually ran. Tests: every enrolled pair statically resolves; no dynamic import from mirror fields.
- **FD7 — R1: rolling-window consumption + loss-accounting watermark (NO coupling to the meter's raw-row lifecycle).** The detector **never reads or writes raw decision_quality/outcome rows and never gates any meter prune** — the meter's stores and retention are byte-identical whether the detector is dark, dryRun, live, or stalled (wiring-tested in all three states). Because a day's buckets legally mutate for up to the meter's reconcile window (~30d: late evidence, unknown-regrade backoff, sentinel supersede-wrong grades — which are disproportionately the WRONG grades), **exactly-once-per-day is the wrong primitive**: each pass recomputes verdicts over the trailing `maxDaysPerAnalysis` (35) matured days — a day is matured when `day ≤ today − analysisMaturityLagDays` (default 2) — with idempotent finding upserts, so a late grade flip updates the latest view on the next pass (unit-tested). The per-machine watermark is **pure R1 loss-accounting**: PK `(machineId, decisionPointId, model)` with `analyzedThroughDay` as a VALUE column (bounded rows; stale-machine/retired-point hygiene pruning mirrors the meter's cursor pruning); a day that ages past the by_model retention having never been covered by any pass increments a content-free `unanalyzedLoss` counter **on the GET envelope summary per machine** (never a finding row — in the loss scenario there may be none). It is honestly **holder-local**: a lease move forfeits the old holder's watermarks; the new holder re-scans ≤ the window — duplicates never skips, harmless by idempotency (tested). To prevent a months-long FALSE-loss drip after churn (days the old holder analyzed reading as "never covered" on the new holder), a first-seen `(machineId, decisionPointId, model)` key seeds `analyzedThroughDay = today − maxDaysPerAnalysis`; the honest undercount for the churn gap is accepted and named (churn-then-age-out unit case). Blunt scope statement: this watermark is NOT a processing checkpoint and guarantees NO coverage and NO exactly-once — it exists solely to COUNT unanalyzed loss.
- **FD8 — R2: single analyzer, pool-for-verdict-only, chronic honesty.** The ANALYZE pass runs on **the serving-lease holder only** — the cadence tick AND `POST /analyze` are both lease-gated (non-holder ⇒ no-op / 409 naming the holder; the lease is re-checked once before the findings upsert to bound a mid-pass flap). It pool-collects every machine's matured-window aggregates from the declared peer route (FD10) via a bounded, parallel, peer-URL-guarded fan-out under one deadline. Offline machine ⇒ `partial` with `coverage: { machinesReporting, machinesKnown, byMachine }`; re-collected when it returns; **a missing peer never stalls any watermark or any retention** (verdict-only coupling; local-prune-advances-with-all-peers-offline is tested). Per-finding `analysisWindow` is the INTERSECTION of days actually covered by all reporting machines. **Chronic counter (union semantics, enumerated):** the per-key streak increments on ANY non-actionable verdict (`partial`, `insufficient-evidence`, AND `precondition-failed`) and resets only on an **actionable** one, where `actionable = { divergent-worse, divergent-better, aligned }`; `no-benched-baseline` neither increments nor resets (a coverage gap is not a conclusion attempt). At `chronicCycles` (3) consecutive, `chronic: true` names why (persistently-offline machine / graded-n stuck / mirror stale N days) — a permanently stale mirror may not hide forever. Unit test includes aligned-resets-streak. The streak is **persisted on the finding row and seeded at pass start from the pool-merged latest view**, so lease churn cannot reset it. Analysis time carries bounded jitter (±10% of cadence) so a fixed sleep schedule can't systematically exclude a machine.
- **FD9 — Peer + mirror inputs are untrusted (clamped, bounded, attributed).** Type-clamped allowlist (counts finite non-negative integers; ids `^[A-Za-z0-9._/-]{1,128}$`; `day` strict `YYYY-MM-DD`, not in the future, within retention age; enums clamped; unknown fields dropped). Volume ceilings: `maxAggregateRowsPerPeer` (points × models × window with a hard absolute cap; excess ⇒ the peer's response classified `pool.suspect`, truncated-and-flagged) and `maxNewFindingKeysPerPass` (excess collapses into ONE deduped `unmapped-flood` finding). Implausible aggregates (rate outside [0,1], counts exceeding plausible volume) ⇒ `pool.suspect`, excluded and surfaced, never silently merged. Every finding records per-machine contribution counts plus a derived `dominantMachineShare` (flagged ≥ 0.8) — attribution is a surfaced signal, not just forensics. The peer envelope carries each machine's `retentionEdgeDay` (so loss-accounting compares against real bounds) AND, per decision point, the window-aggregated orphan counts + decided base from the meter's rollup (through the same clamps) — `maxOrphanShare` is evaluated over the POOL-MERGED aggregate, not holder-local (R2 applies to the honesty gates too; cross-machine test). **Free-text never crosses the pool:** on `?scope=pool` merge, peer `questions` are DROPPED and regenerated locally from the verdict/direction enum (static deterministic text — lossless); `pool.failed[]`/`suspect[]` carry normalized enum reasons + clamped machine ids only, never peer URLs or raw error text; any residual free-text field is length-clamped and treated as quoted untrusted data (test: hostile peer text never survives the merge). All queries parameterized.
- **FD10 — Surfaces (observe-only, frozen envelopes).**
  - `GET /benchmark-divergence` (Bearer; 503 dark): `{ enabled, dryRun, analyzer: { isHolder, holderMachineId, lastLocalAnalysisAt, stale }, mirror: { present, capturedAt, staleDays, stale }, summary: { byVerdict, unmappedModels, missingModelShare, unanalyzedLoss: { byMachine } }, findings: [FINDING], scope }`; `FINDING = { taskId, decisionPointId, model, verdict, preconditionReason?, realGradeRate, predictedRate, delta, gradedN, unknownShare, ciHalfWidth, benchN, benchCiHalfWidth, orphanTainted, chronic, chronicStreak, coverage: { machinesReporting, machinesKnown, byMachine }, dominantMachineShare, unmapped?, benchedPromptHash, mirrorCapturedAt, analysisWindow: { fromDay, toDay }, firstSeenAt, lastSeenAt, advisory: true, questions: [static ranked TEXT] }`. A plain-scope read on a non-holder is tagged `analyzer.stale: true` — never dead data presented as current. **Content-free guarantee:** never joins/inlines raw decision context; question 1 is answered by correlation-id POINTERS into the meter's own surfaces; a test asserts no payload field. `?scope=pool` merges peers' findings through the FD9 allowlist, fronted by the shared PoolPollCache, with per-key order **`(analysisWindow.toDay DESC, lastSeenAt DESC)`** — window recency wins, wall-clock breaks ties (a stale-holder catch-up pass can never shadow a fresher-window finding) — plus a `pool: { peersQueried, peersOk, failed[], suspect[] }` block.
  - `POST /benchmark-divergence/analyze` (Bearer, rate-limited, single-flight, idempotent, lease-gated per FD8).
  - `GET /benchmark-divergence/rollup-aggregates?fromDay=&toDay=` (Bearer; 503 dark) — the peer collection route: serves this machine's matured-window per-model aggregates; **the SERVING peer clamps the range** to min(requested, its own `maxDaysPerAnalysis`, its retention); bounded response.
  - **`advisory: true` on every finding row**: no consumer may gate/route/escalate on a finding except as a SIGNAL into a proper authority (Signal vs. Authority).
- **FD11 — History without unbounded growth or silent erasure.** Latest-view upsert per `(taskId, decisionPointId, model)` + content-free history rows keyed `+ (analysisWindow.toDay, mirrorCapturedAt)` (same-day re-analysis upserts, never appends). Cap `maxHistoryPerKey` (50) prunes oldest **except the FIRST history row per key, retained permanently** (the first detection is never evicted); the ~cap×cadence horizon is stated honestly: history is a bounded trajectory, not a forever archive.
- **FD12 — Execution + engineering pins.** In-process on the shared FeatureMetricsLedger connection (the meter's execution model; `busy_timeout` 5000ms). The window read is one grouped indexed query over the by_model table — O(points × models × window-days), thousands of rows worst case, never O(raw). Driving job: built-in template `benchmark-divergence-analysis`, `enabled: false` fleet-wide, `supervision: tier0` (justified: fully deterministic aggregation + comparison, no LLM step), wired through BOTH `installBuiltinJobs` and `migrateBuiltinJobs`; **the rollout step enables the job manifest on the development agent** (fleet manifests stay false) so the dryRun soak actually runs on cadence. Micro-constants (fan-out deadline, POST rate-limit, jitter bound) are named code constants in the build PR.
- **FD13 — Ship posture + honest rollback.** Dark fleet, live-in-dryRun on a development agent (`benchmarkDivergence.enabled` omitted ⇒ dev-agent gate). **dryRun (true) = ZERO detector-owned durable writes** (no findings, no watermark, no history; would-analyze summaries logged content-free). Routes 503 dark. **Rollback honesty:** the flag governs the DETECTOR only. The stage-2 by_model write is METER-side — it ships inside the meter's annotate chokepoint and is exactly as enabled as the meter's grading (dark meter ⇒ no grades ⇒ no writes; live meter ⇒ per-model essence accumulates REGARDLESS of detector state — deliberately, so a later detector enable has history rather than an R1 cold-start hole). Its rollback lever is therefore the meter's seam, stated here so an operator never flips `benchmarkDivergence.enabled` expecting that write to stop. Migration is truly additive (a NEW table + registry/config entries; nothing existing rebuilt): rollback = flags off, tables sit inert.

## Durable schema (additive — the meter's existing tables are BYTE-IDENTICAL)

- **`decision_quality_rollup_by_model` (NEW parallel table — deliberately NOT an extension of the meter's `decision_quality_rollup`,** whose inline `PRIMARY KEY (decision_point, day)` cannot be altered in SQLite and whose three writers + reconcile + read surfaces stay untouched): PK `(decision_point, model, day)`; columns `right_n, wrong_n, unknown_n, decided_total` (INTEGERs), `prompt_id` (the window-uniformity source: the recorded prompt identity when uniform within the bucket, else the reserved `'__mixed__'`), `last_write_at`. Maintained at the SAME annotate chokepoint (`recomputeQualityBucketInTx` recomputes both tables in one transaction — recompute-from-raw is naturally supersede-correct and replay-idempotent, so an unknown→right regrade nets counts and a replayed pass changes nothing: both unit-tested against a SENTINEL-landed wrong grade, not just the pass) and by the reconcile, extended with `GROUP BY model` — **for the by_model table the reconcile rebuilds ONLY days whose raw truth still exists (freeze, don't zero, beyond the raw floor)**, because the meter's rollup is a derived cache of raw and blind zeroing would silently destroy not-yet-analyzed essence. An empty/unparseable model id is REFUSED into `''`: written under reserved `'__missing__'` (excluded from comparisons, surfaced as `missingModelShare`). **`decided_total` is recomputed from raw DECISION rows (the correlation spine) grouped by model — NEVER from outcome rows** — and a bucket row is MATERIALIZED for every (point, model, matured-day) with ≥1 raw decision, even with zero outcomes (counts 0/0/0, decided_total > 0, prompt_id recorded): a fully-ungraded day must weigh in the window's unknownShare, or the unsettled-stream gate is decorative at window scale. Tests pin the data source through BOTH arms: N recorded decisions + 1 outcome ⇒ decided_total = N after the chokepoint recompute AND after a reconcile rebuild; a decisions-only day ⇒ a bucket with decided_total > 0 and zero grade counts; a window with fully-ungraded days ⇒ `insufficient-evidence`. Secondary index `idx_decision_quality_rollup_by_model_day ON (day)` (the meter's own rollup ships the same) backs the window read, the retention prune, and the reconcile's day-range scans. Retention: its own knob `byModelRetentionDays` (180, bounded — P19). Backfill: one post-migration reconcile pass seeds per-model history for days still within raw retention; older days are lost and that is accepted, stated.
- `benchmark_divergence_findings` — latest view + capped history (FD11), all fields content-free, `advisory: true` stamped.
- `benchmark_analysis_watermark` — PK `(machineId, decisionPointId, model)`, `analyzedThroughDay` VALUE + `updatedAt`; hygiene pruning.
- Terminology: `decisionPointId` everywhere; "component" is never a key.

**Why a rollup table and not an event log — and why in-process, not an external warehouse:** the counts ride the meter's existing substrate and its recompute/reconcile machinery (corrections = recompute-from-raw within retention); an append-only grade-event stream + materialized views would introduce a second source of truth, replay machinery, and its own retention story for marginal auditability — the FD11 history rows already record the verdict trajectory. A central data-warehouse/TSDB push would be the conventional analytics shape, but instar agents are deliberately self-contained (file-based/SQLite, no external service dependencies); adding a warehouse dependency for an observe-only detector would invert that posture, and the peer-collect reuses mesh machinery (lease, PoolPollCache, peer-URL guard) that already exists and is already tested. The git-tracked mirror and SQLite's PK immutability (which forced the parallel-table shape) are acknowledged constraints of that choice — accepted trade-offs, stated rather than hidden.

**Backup/restore:** all three content-free tables ride the meter's SQLite DB backup — included, NOT excluded; a backup→restore round-trip preserving a finding + by_model counts is tested.

## Multi-machine posture

- by_model rollup + watermark: **machine-local** — `machine-local-justification: operator-ratified-exception` — inherits the quality meter's ratified machine-local grades posture (meter spec §FD10/§5.5; PR https://github.com/JKHeadley/instar/pull/1458 is the machine-verifiable ref; marker validity is conditioned on that merge, which the build-ordering constraint already requires). Watermark rows are additionally holder-local per FD7 (honest duplicates-never-skips semantics on lease churn).
- Findings: **written on the lease-holder, proxied-on-read** (`?scope=pool`, FD10) — one authoritative writer per computation, merged read ordered `(toDay DESC, lastSeenAt DESC)`, tolerant of holder churn.
- Mirror: **unified** (git-tracked file).
- No user-facing notices; no generated URLs; nothing strands on topic transfer.

## The ranked questions (per flagged finding)

`divergent-worse`: 0 — precondition (enforced as FD4 verdicts) → 1 — enough context? (correlation-id pointers into the meter's provenance) → 2 — right prompt? (live vs benched template) → 3 — does the battery represent reality? (→ Increment C).
`divergent-better`: 1 — is the grade-rate inflated? → 2 — is the battery too hard / unrepresentative? → 3 — genuine improvement (only after 1–2 are excluded).

## Glossary (one-liners)

**Battery/task** — an off-repo INSTAR-Bench test set for one decision surface. **passRate** — the battery's per-model pass share over deterministic cases. **Grade** — the meter's per-decision right/wrong/unknown outcome stamp. **Matured day** — a UTC day old enough (≥ `analysisMaturityLagDays`) that its grade counts are mostly settled. **Serving-lease holder** — the one machine currently authoritative for pool-wide actions. **Q0 / template-hash precondition** — "did the benchmark test the prompt template production actually runs?" **Mirror** — the in-repo content-free copy of the battery's predictions. (Complexity is acknowledged as the accepted trade-off for cross-machine honesty: several small deterministic state machines, each independently tested, rather than one clever one.)
