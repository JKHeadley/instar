# Claim Verification — the plain-English version

Today an agent can write a confident sentence from memory—“the tests passed,” “that handoff landed,” or “we are capped at four lanes”—even when it has not checked the real source. Claim Verification gives Instar a reusable way to notice those factual statements and ask: what exactly is being claimed, how harmful would an error be, and is there trustworthy current evidence?

The first release is intentionally dark. It watches and measures; it does not stop, rewrite, delay, or send a message, so it does not directly reduce errors in messages already delivered. Instar already has a completion-claim observer that looks at a Claude Code response after a turn, plus a small deterministic check that compares time claims with the live session clock. V1 extends that one existing pipeline instead of creating another checker, while preserving the legacy Action-Claim projection so existing commitment behavior cannot change. One bounded model pass returns ordinary schema-validated JSON that separates factual clauses and suggests their type and importance. A deterministic protected-cue lane records a conservative high-risk gap when the model misses an obvious protected shape, but it never invents a factual verdict. Fixed policy applies minimum safety levels: wording like “I think” cannot make a completion, approval, capacity, or irreversible-action premise unimportant. The Stop hook keeps delivery nonblocking and stores no raw response or attempt ledger. Every metric is labeled “server-admitted only”; total transport coverage and real production recall remain unknown.

Where a reliable source already exists, a small read-only verifier checks it. A time claim uses the live clock; a same-turn completion can use scrubbed structural tool evidence; selected state facts can use fresh registry snapshots. Every result is one of “supported,” “refuted,” or “unverifiable,” and it names the source version and observation time. Missing, stale, ambiguous, conflicting, or partial data is never called a refutation. The system does not pretend that documentation or model agreement is ground truth. In particular, Instar has no canonical structured capacity registry today, so capacity claims are detected as important but remain unverifiable. A prose operating-model document is not invented to paper over that gap.

The feature also starts a privacy-scrubbed local corpus. It stores structural claim shapes and bounded metadata—claim type, verifier and model versions, verdict, eventual correct answer when a deterministic source establishes one, cost, and latency—not raw private message text or raw evidence. V1 ingests deterministic-source labels only; an operator-adjudication interface is deliberately deferred. Private identifiers use keyed pseudonyms rather than reversible plain hashes. These local records are useful for dark observation but are explicitly untrusted for cross-machine learning, promotion, routing, or policy calibration, and saturated messages cannot support shape-frequency conclusions. Those consumers remain disabled until a separately converged Verified Claim Evidence Ledger provides origin authentication, anti-rollback, fencing, validated receipts, complete replication, and installation-wide cohorts collected prospectively. V1 also does not create a raw-message review vault: an independently privacy-reviewed live recall-audit substrate is required before any user-visible advisory can graduate. Feedback can never make the safety policy looser: uncertainty still rounds upward, and drift may only increase checking or demote a shortcut.

V2 is designed but not authorized to build. It has two pieces. First, a miner looks for claim shapes that recur often and have a stable cheap source. The current time-versus-clock check is the prototype. A proposed shortcut is promoted only after enough trusted settled examples and a later holdout prove very high accuracy and recall. The shortcut is a registered read-only recipe, never generated executable code. It stays shadow-tested and is automatically demoted to the general path when agreement drifts. V2 remains disabled until both the Verified Claim Evidence Ledger and a separately proven replicated recipe registry exist; the current local SharedState ledger is not misrepresented as either foundation. V1 data must also prove that recurring shapes are costly enough to justify a miner instead of simpler manual predicate additions.

Second, important claims without one sufficient source can enter a bounded due-diligence workflow. That workflow reuses Instar’s durable TaskFlow records and shared intelligence queues, but it remains disabled until TaskFlow has replicated, lease-fenced ownership across the machine pool. It has one current owner, stable idempotency, strict queue/team/token/tool/time/spend limits, cancellation and restart rules, and three finite read-only verifier roles. Verifiers receive typed evidence handles, not shell access, arbitrary network access, secrets, or permission to send or change anything. The server signs scope- and revision-bound read receipts, and the deterministic aggregator reads those receipts—not a model’s prose verdict. Several models agreeing is not proof; canonical evidence is what counts. A correction additionally requires an authenticated receipt joining the authored attempt to the exact delivered channel message. A fenced durable outbox reserves one stable transport identity before sending and reconciles crashes or ownership transfer, so retries do not become duplicate corrections. Claim Verification itself never becomes an action authority.

V3 uses only trusted settled benchmark cohorts from the prerequisite evidence ledger for conservative calibration. It may recommend a better model door, more sampling, a higher criticality tier, or demotion of a cheap checker. It cannot silently lower a tier, reduce verification, change an oracle, or promote a new checker. Every change is versioned, reversible, privacy-bounded, and subject to the same dark rollout and operator review.

The important shape is simple: one existing claim pipeline produces evidence; one existing contextual authority remains responsible for messaging decisions; real sources outrank recollection; uncertainty stays visible; and the system learns to become cheaper only after it has proved that “cheaper” still means correct.
