# WS5.2 Account Follow-Me — Security Convergence Spec (CMT-1413)

## 1. Summary

WS5.2 designs the secure mechanism by which a subscription account (an Anthropic Claude login, or any-provider account) that the operator enrolled on one machine becomes **usable** by the same agent on its other machines — the most sensitive deferral in the parent multi-machine seamlessness spec, because it borders directly on credentials/logins crossing machines. The parent spec (`docs/specs/MULTI-MACHINE-SEAMLESSNESS-SPEC.md` §WS5.2) deferred this with a hard boundary: cross-machine enrollment is OPERATOR-INITIATED and PIN/operator-authenticated (a peer machine can NEVER enroll an account onto itself via the mesh), it is explicitly NOT login-file sync (OAuth config-homes never cross machines), and the flag `multiMachine.accountFollowMe` is reserved-dark until designed in its own round. This is that round (CMT-1413).

The load-bearing realization, grounded in `SubscriptionPool.ts` (the registry stores each account's login LOCATION — `SubscriptionAccount.configHome` — NEVER a token; `FORBIDDEN_CREDENTIAL_FIELDS` + `assertNoCredentialFields()` make that structural at `add()`/`update()`), is that **"account follow-me" is two separable transfers that must NEVER be conflated**:

- **(A) Registry follow-me — metadata replication only.** A redacted, non-credential projection of `SubscriptionAccount` (id, nickname, email, provider, framework, status, quota) replicates so a peer machine *knows the account exists and how loaded it is*. `configHome` is STRIPPED — a config-home path is meaningless on another machine and replicating it would invite exactly the "just point at the path" leak the boundary forbids. This is the cheap, always-on half.
- **(B) Per-account credential availability on a recipient.** The recipient machine must hold a *valid login it produced itself for that account*. The default and ships-now-eligible mechanism is **re-mint per machine**: the operator drives a fresh, per-machine `EnrollmentWizard` flow on the recipient (public device-code/URL, never a token), pre-authorized by an operator-issued mandate. Credentials are re-minted per machine, **never transported**.

A second, fully-specced-but-default-off mechanism — **sealed-credential transport** (encrypt the credential blob to the recipient's X25519 key via `SecretSync` and write it into the recipient's config-home) — is designed here so the secure form exists on the shelf, but it is gated behind a **separate, per-provider allowlist flag** and **REFUSED for Anthropic by default** (Anthropic's ToS prohibits relocating Claude OAuth tokens out of Claude Code's own store — `SubscriptionPool.ts` lines 17–21). The two mechanisms are referred to throughout as **Mechanism B (re-mint, default)** and **Mechanism A (sealed-transport, gated)**.

This spec covers: the decision (what replicates), the threat model and named attacker scenarios, enrollment & operator consent, integration & failure semantics, Phase-C scaling (N cloud VMs, headless, no-LAN/no-2-peer assumption, bounded per-account budget), a three-tier test plan with named attack tests, and a dark/dry-run/receive-only rollout. No credential-bearing code is authorized by this document.

### 1.1 Reconciled contradiction (load-bearing — resolved in writing here)

The `SubscriptionPool.ts` header (lines 11–16) explicitly *architects for* shipping each account's credential blob over E2E secret-sync ("a future cross-machine sync … ships each account's credential blob over the existing E2E secret-sync"). The parent spec's WS5.2 invariant says the opposite: "OAuth config-homes never cross machines." These are in **direct contradiction** and the contradiction is load-bearing — it decides whether follow-me is credential transport (the pool header's model) or re-enrollment orchestration (the parent-spec model).

**Resolution (this spec is authoritative):** the parent-spec invariant wins for OAuth (Anthropic). Follow-me defaults to **re-mint per machine (Mechanism B)**; OAuth config-homes never cross. The pool header's "ship the credential blob" path survives ONLY as Mechanism A — a per-provider allowlist, default-empty, with `anthropic` excluded. **Build-round obligation R0 below requires the `SubscriptionPool.ts` header comment to be corrected in the same PR that wires any follow-me path**, so the two documents stop contradicting each other.

## 2. Decision — what replicates (pointer vs sealed-token), with the recommendation

There are two candidate definitions of "follow-me," scored below; the **recommendation is the pointer/re-mint mechanism as the default and only ships-now-eligible path**, with sealed-transport designed but gated default-off and Anthropic-excluded.

### 2.1 Mechanism B — pointer / re-mint per machine (RECOMMENDED DEFAULT)

The mesh carries a non-credential enrollment descriptor `{provider, framework, nickname, email, intended account-id}` plus the metadata projection of §1(A). Each recipient machine runs its OWN `EnrollmentWizard.start()` headless flow against a fresh local `configHome`; the operator approves the login per machine from their phone. No OAuth token, refresh token, or config-home file content ever crosses the mesh. The new credential is minted by the provider's own flow, on the machine that will use it. The two machines then hold two independent logins for the same operator account — exactly what `SubscriptionPool`'s "store LOCATION not token" model wants.

### 2.2 Mechanism A — sealed-credential transport (DESIGNED, GATED, ANTHROPIC-EXCLUDED)

The live credential blob (the contents of the account's `configHome`, e.g. `.credentials.json`) is sealed per recipient via `encryptForSync(secrets, recipient.encryptionPublicKey)` (`SecretStore.ts` line 499 — X25519 ECDH, ephemeral key per payload → forward secrecy, AES-256-GCM), shipped over the `secret-share` mesh verb, and written into the recipient's config-home so a session can spawn there without a fresh login.

### 2.3 Threat-delta scoring (why B is the default)

| Axis | Mechanism B (re-mint) | Mechanism A (sealed-transport) |
|---|---|---|
| **Wire content** | Non-secret descriptor + redacted meta. Zero credential material on the wire, ever. | Live OAuth/session credential (sealed). Confidentiality in transit is the same proven `SecretSync` property — but it IS a credential on the wire. |
| **Token-at-rest delta** | The target holds its OWN real, refreshable login (S7) — so a target compromise yields a live credential for THAT login (not a transportable one for the other). A one-machine compromise yields no transportable credential for the other. | Credential now lives on N machines, each in a **framework-readable plaintext config-home file** (NOT inside instar's AES-256-GCM `SecretStore`, and NOT made inert by key-rotation/epoch advance — gap 6). One recipient-machine disk compromise yields a live credential. Exposure delta 1 → N, directly-actionable. |
| **Provider ToS** | Strongest — each tool instance logs in itself, exactly as a human on a new laptop. | instar reads + moves + persists an OAuth token the ToS says must stay in Claude Code's own store. Even sealed in transit, instar becomes the courier of a credential it is told not to relocate. Highest non-technical risk; unremovable by crypto. |
| **Rotation/expiry** | Each machine refreshes its own short-lived token locally (the pool already tracks `lastRefreshAt`, `SubscriptionPool.ts` line 118). | A sealed credential goes stale the instant the source refreshes → needs either a long-lived refresh-token replicated N-wide (worse) or re-seal-on-refresh (a recurring credential-bearing mesh flow). Strictly more dangerous. |
| **Revocation** | Local & total ONLY for a still-paired cooperative machine (`SubscriptionPool.remove()` + framework logout). For a de-paired/non-cooperative holder the B-minted login is independently refreshable and un-reachable — revocable ONLY provider-side (S8/gap 10). | A delivered credential cannot be un-delivered; key-rotation makes only the SEALED copy inert, not the plaintext file (gap 6); the only complete revocation is **rotating at the provider**. |
| **Operator cost** | N phone approvals for N machines (each a ~30s device-code approval). | One credential push per (account, target) — fewer touches, far higher residual risk. |

### 2.4 Recommendation

**Default to Mechanism B (re-mint per machine) for ALL providers; ship-now eligibility is Mechanism B only.** Mechanism A is fully designed (§§4, R9–R14) but stays behind a distinct, default-off, per-provider allowlist flag (`multiMachine.accountFollowMe.credentialTransport`, default empty) and is **REFUSED for Anthropic by default**. Ranked reasoning:

1. **It preserves the one invariant the whole subscription stack is built on** — "a leaked registry file never leaks a credential" / "instar never extracts a token" (`SubscriptionPool.ts` lines 15, 19). B extends that invariant across machines unchanged; A carves an exception into it.
2. **The exposure delta is categorically smaller** — B widens only *metadata* exposure (already public within the pool via WS5.1); A widens *directly-actionable credential* exposure 1 → N with an unavoidable plaintext-on-disk residual in each config-home.
3. **ToS** — B is what the provider intends; A has instar moving a token the ToS forbids relocating, a standing risk no sealing removes.
4. **B's cost is already specced and acceptable** — "N logins for N machines" is the WS5.1 honest-disclosure flow (the agent OFFERS to drive `POST /subscription-pool/enroll`, phone-friendly, headless-ready, auto-reissue on expiry), fully Phase-C-compatible, and never creates an irreversible credential spread.
5. **A is kept on the shelf, secure** — IF a future provider explicitly permits token movement AND an operator explicitly opts in for that provider, the secure mechanism (per-recipient seal with AAD binding, expiry, rotation prompt, revoke verb) is already designed — but it never becomes the path of least resistance and never touches Anthropic by default.

## 3. Threat Model

### 3.1 Foundational finding (drives scenarios 1, 3, 4 and Mechanism A's design)

The existing transport envelope `EncryptedSecretPayload` (`SecretStore.ts` lines 79–88: `{ephemeralPublicKey, iv, ciphertext, tag}`) is **plain ECIES** — ephemeral ECDH + AES-256-GCM. It has **no recipient-fingerprint binding, no expiry, no epoch counter, no replay nonce, and no GCM AAD**. The AES-GCM `tag` authenticates *ciphertext integrity only* — not *who the payload was sealed for*, nor *when*. `decryptFromSync` (`SecretStore.ts` line 554) will decrypt ANY payload sealed to a machine's long-term X25519 key, forever. That is acceptable for a Telegram token (idempotent; the operator can rotate it trivially); it is **NOT acceptable for a live OAuth credential**.

**Therefore: Mechanism A MUST NOT reuse `encryptForSync`/`decryptFromSync`/the `secret-share` verb/handler as-is.** It requires a hardened, account-credential-specific envelope whose AES-GCM **AAD** = `(recipientFingerprint ‖ accountId ‖ grantId ‖ issuedAt ‖ pairingEpoch)`, so the GCM tag authenticates *who/what/when* and a replay to a different machine, account, or epoch fails the tag check at decrypt. This is the single most important crypto decision in the round and is mandatory wherever Mechanism A is implemented. (Mechanism B carries no credential and is unaffected — its descriptor reuses the mesh Ed25519 envelope unchanged.)

**Critical implementation finding (gap-confirmed — drives R3, I2, and the build-order obligation): `encryptForSync` as written CANNOT carry AAD, and the legacy decrypt path is reachable for any payload.** Verified in code: `encryptForSync` (`SecretStore.ts` line 499) has NO AAD parameter and never calls `setAAD`; `decryptFromSync` (`SecretStore.ts` line 562) takes only `(payload, ownPrivateKey)`, derives its key via HKDF with a fixed `HKDF_INFO='instar-secret-sync-v1'` (line 35/574), and never calls `setAAD` — so it will decrypt ANY payload sealed to a machine's long-term X25519 key, with no concept of AAD/expiry/epoch/recipient binding. Worse, `SecretShareHandler.handle` (`SecretSync.ts` lines 146–148) decrypts ANY inbound `secret-share` command with that bare `decryptFromSync` and writes every key into the vault. **Consequence:** if a Mechanism-A credential were ever marshaled as a `secret-share`-class command, an attacker who captures a sealed blob could replay it through the EXISTING unbound handler, which has no concept of AAD and would happily decrypt+store it — defeating the entire epoch/recipient binding. Therefore "seal via `encryptForSync` with AAD binding" is **not achievable on the existing function**; a NEW `encryptAccountCredential(secrets, recipientPublicKey, aad)` + a NEW `decryptAccountCredential(payload, ownPrivateKey, expectedAAD)` (the latter FAILS CLOSED if AAD is absent or mismatched) MUST be built in the first PR (R3/I2/§9 build-order). The legacy `secret-share`/`decryptFromSync`/`SecretShareHandler` path MUST be structurally unreachable for credential-class data (R3a).

### 3.1a Foundational finding #2 — the mandate authorship proof is a SYMMETRIC HMAC keyed on the per-machine authToken, so it is LOCAL-ONLY and a peer CANNOT verify a mandate issued on another machine (gap-confirmed — drives R4, R4a, I3)

This is the single most dangerous gap in the round, because R4/I3/§5.2's central authority-vs-reach control rests on a capability the codebase does not provide. R4 (as originally drafted) required a target machine, before acting on a *replicated* follow-me mandate, to "verify the mandate's issuance signature chains to the operator's PIN-gated issuance (the `MandateAudit` hash-chain, `chain.ok`)." But the real authorship proof is **not** chain-verifiable across machines and **not** asymmetric. Verified in code: `AgentServer.ts` lines 1038–1046 set `issuanceKey = this.config.authToken` and `mSign = HMAC-SHA256(issuanceKey, canonical)`; `MandateStore.verifyAuthorship` (`MandateStore.ts` lines 201–204) calls `verifySig(canonical, authProof)` which recomputes the SAME HMAC with the SAME key. This is a symmetric MAC keyed on the **per-machine** server `authToken`. A target/recipient machine holds a DIFFERENT `authToken`, so it structurally CANNOT verify a mandate issued elsewhere — `mVerify` recomputes a different HMAC and returns false for every legitimate cross-machine mandate.

Both "easy" ways to make the original R4 work are unacceptable:
- **(a) Share the issuance `authToken` across machines** — this turns a single-machine disk compromise into a forge-any-mandate-on-any-peer key: the exact catastrophe this round exists to prevent.
- **(b) Trust the replicated `MandateAudit` hash-chain as authority** — that chain only proves internal append-consistency on the ISSUER's disk (`MandateAudit` is a local file, T12-out-of-scope per `AgentServer.ts` lines 1031–1032). It is *reach*, not cross-machine-verifiable *authority*.

**Therefore cross-machine mandate verification is an explicit, first-PR primitive of any build round (alongside the AAD envelope), NOT a free inherit from `MandateGate`.** The existing HMAC-over-authToken proof is a named LOCAL-ONLY limitation. Any mandate that crosses the mesh MUST carry an ASYMMETRIC issuance signature: the canonical mandate is signed with the issuing machine's Ed25519 *identity* private key (the same `MachineIdentity` key used for mesh envelopes, `types.ts` lines 1692–1698), so any peer holding the registered public key can verify it, and the mandate is bound to the issuer's machine fingerprint so a peer checks "this mandate was issued by my trusted operator-machine." The operator's PIN authority is proven to a remote verifier via the trust root: **PIN-gated issuance happens on the operator-machine M → M signs with its Ed25519 identity key → the peer verifies M is the registered operator machine** (the verified-operator binding, not a content name). Specified as **R4a** below. If a session is on a machine that is itself the operator-machine, the chain collapses to local verification; for every other machine the asymmetric path is the ONLY one that can hold deny-by-default. Until R4a is built, R4/§5.2 cannot claim deny-by-default holds across machines.

### 3.2 Named attacker scenarios

**S1 — Compromised or newly-added peer harvests a synced credential (Mechanism A catastrophe case).** An attacker with root on a paired machine, or a rogue machine added to the pool, receives the broadcast `secret-share` fan-out (`SecretProvisioner.provisionAll`, `SecretSync.ts` lines 109–125 — it pushes to *every* online registered peer) and decrypts a credential sealed to its X25519 key (`SecretShareHandler.handle`, line 146). Because v1 SecretSync "syncs ALL secret-marked fields" and fans out to all peers, one rogue peer harvests every synced account at once. Blast radius: full takeover of one (or every) account until provider-side rotation.
*Mitigations:* account credentials are NOT `{secret:true}` vault fields and NEVER ride `provisionAll`'s broadcast (I1); per-account explicit destination (a new `provisionAccountTo(accountId, targetMachineId, mandateId)` path sealing to exactly one named machine, never collection-wide); a `credential-sync` trust floor at the destination (newly-added peers start below it); Phase-C: the destination is an operator-PIN-authenticated grant per (account, machine), independent of pool size — a self-added VM never auto-qualifies.

**S2 — Forged "enroll this account" instruction.** An attacker forges a mesh command (a fake `secret-share`/`account-share`, or a fake "enroll account X onto machine Y") to make a victim adopt an attacker-controlled account (silently routing the operator's prompts/responses through attacker billing+logging via `AnthropicSubscriptionRouter`) or trigger an unwanted push.
*Mitigations:* enrollment is operator-PIN-authenticated, never mesh-authenticated (a peer signature proves *which machine*, the mandate proves *the operator intends it* — requester ≠ authorizer); `EnrollmentWizard.start/complete` is reachable from NO mesh verb (a build test enumerates the mesh-verb→handler table and FAILS if any verb routes to enrollment or credential-push); the (Mechanism-A) push carries a signed, operator-issued grant naming `(accountId, targetMachineFingerprint, issuedAt, expiresAt)` and the receiver refuses any account-credential payload lacking a verifiable, unexpired, machine-matched grant.
*RBAC finding (gap-confirmed — drives R3a, I1):* the existing `secret-share` verb's RBAC is **open to ANY registered peer** — `checkCommandRBAC` (`MeshRpc.ts` lines 235–249) classifies `secret-share` in the "any registered peer" read/observe/e2e-encrypted bucket and returns `{ok:true}` for any peer that passed `verifyEnvelope`, with NO per-recipient, no mandate, and no destination-allowlist check at the RBAC layer. If Mechanism A rode `secret-share` as written, a rogue-but-registered peer (S1's exact attacker) would be authorized at the gate to RECEIVE-class the verb, and the ONLY thing stopping it would be the AAD recipient-binding at decrypt — i.e. authorization would have silently collapsed onto crypto alone, the exact "reach ≠ authority" failure R4/I3 forbid. **Therefore Mechanism A MUST use a DISTINCT mesh verb (`account-credential-share`) with its OWN `checkCommandRBAC` case** that is NOT the "any registered peer" class — verifying at the gate, before decrypt is even attempted, that (a) the sender is the operator-mandated source, (b) THIS recipient == the mandate's single `targetMachineId`, and (c) the grant is unexpired + single-use. This is the credential-domain analog of why `commitment-mutate` got its own RBAC case (`MeshRpc.ts` lines 219–227); the same precedent applies. Specified as **R3a**.

**S3 — Replay of a sealed credential to a de-authorized machine (Mechanism A).** The operator legitimately pushes a credential to machine B; B is later de-paired. An attacker re-delivers the captured `EncryptedSecretPayload` to B (or a machine still holding B's key). With no expiry/epoch/recipient-binding, `decryptFromSync` happily decrypts the stale payload and B re-acquires a credential it was supposed to lose — defeating revocation entirely, and the captured payload is valid forever.
*Mitigations:* the AAD-bound envelope (§3.1) ties the payload to `recipientFingerprint ‖ accountId ‖ grantId ‖ issuedAt ‖ pairingEpoch` — a replay to a different machine/account/epoch fails the GCM tag; a monotonic per-pairing epoch (incremented on de-pair) so a credential sealed at epoch N is rejected at epoch N+1 (receiver checks its *current* epoch, durably, surviving reboot); grant TTL + single-use `grantId` recorded as consumed.
*Epoch finding (gap-confirmed — drives R4b, OQ3 promotion): the pairing-epoch does NOT exist in the codebase and is not yet designed; it is only named.* Verified: there is `FencedLease.leaseEpoch` (who-is-awake) and a CoherenceJournal `topic-placement` epoch — neither is a per-PAIRING epoch, and neither increments on de-pair. De-pair today (`MachineRegistryEntry`, `types.ts` lines 1740–1754) only flips `status:'revoked'` + sets `revokedAt`/`revokedBy`; it does NOT rotate the machine's static `encryptionPublicKey` (`types.ts` line 1698) and does NOT bump any epoch. A revocation guarantee built on an undesigned counter is a wish, not a mechanism — so the epoch is **promoted from an Open Question to a first-class designed primitive in this round** (this is the security round, before code; designing it here is the mandate). **Lean (R4b): rotate the machine's X25519 `encryptionPublicKey` on de-pair INSTEAD of inventing a new counter.** A rotated recipient key makes EVERY old sealed payload undecryptable, achieving S3 (replay-defeat) and S4 (forward-revocation of sealed blobs) at once, reusing existing identity machinery — leaving only R10's plaintext config-home residual (S4/gap-6) to disclose honestly. The AAD's `pairingEpoch` term then binds to the key-rotation generation. Either the key-rotation lean or an explicit anchored counter MUST be specified (R4b); both must define where the value is durably anchored against rollback by a hostile local user and how the receiver learns the current value across reboot.

**S4 — Credential residue surviving on a de-paired machine (Mechanism A; passive, no active attacker needed).** After de-pairing, two residues remain on the de-paired machine: (i) the instar-held sealed blob in the vault, and (ii) — the dangerous one — the **framework's own plaintext token file in config-home** (e.g. `~/.claude/.credentials.json`, the real config-home content per `SubscriptionPool.ts` lines 99–103) that R10 admits MUST be written for a session to spawn. The plaintext residue *itself* is the breach. SecretSync today has no revoke verb (`SecretSync.ts` line 18: "a dedicated revoke verb is a deliberate follow-up"). For a live OAuth credential this is a hard requirement.

*Mitigations (honesty-corrected — gaps 3 + 6 confirmed):* revoke-on-de-pair is multi-layer, and the layers are honestly scoped to the artifact each can actually reach:
- **(1, active) `credential-revoke`/`secret-revoke` mesh verb** wipes BOTH residues — the sealed blob via `SafeFsExecutor` AND the framework's plaintext token file (the framework's own logout against that config-home) — audited. This is the only layer that reaches the plaintext file, and it requires the target to be reachable.
- **(2, passive backstop — scoped, NOT a complete backstop)** advancing the pairing-epoch / rotating the recipient X25519 key (R4b) renders the **instar-held SEALED blob** cryptographically inert. **Explicit limitation (gap 6):** this does NOT and CANNOT make the framework's plaintext config-home token file inert — that file is what a session reads, the epoch never touched it, and a hostile local root on a de-paired machine reads it directly. The "epoch-inert at rest" property is therefore a backstop for the sealed copy ONLY, never for the plaintext copy. It is **not** a complete revocation against the precise threat this scenario names (root on a de-paired machine after the active verb never arrived).
- **(3) short credential TTL** (lease, not a standing copy) so a dark machine's plaintext token EXPIRES on its own — making R10's refresh-on-arrival + short-TTL the real containment for the plaintext residue, with the residual-plaintext-window stated explicitly to the operator.
- **(4, the ONLY complete answer) provider-side rotation**, surfaced phone-first on suspected compromise (honest disclosure, P8 no-dead-ends — it cannot be auto-done). Because layers 2–3 cannot make the plaintext residue inert, provider rotation is not a "nice extra" — it is the genuinely-complete revocation, and the spec must say so rather than implying cryptographic inertness already solved it (the prior draft's contradiction: it both claimed inertness AND admitted plaintext residue, which cannot both be true).

**S5 — Confused-deputy: a machine uses another's account beyond intended quota/scope.** Machine B is granted account A for a bounded purpose (one topic, one window) but instead drains A's full quota, routes background sentinel/gate traffic through it, or uses it for unrelated work. `AnthropicSubscriptionRouter` today has no per-account scope concept — it routes by availability/credit, not by grant scope.
*Mitigations:* the grant carries explicit scope `{accountId, targetMachine, allowedComponents|allowedTopics, maxSpend|maxQuotaFraction, expiresAt}` (the grant — never the machine — is the authorizer); the router consults the grant at selection time (it already threads `options.attribution.component`, `AnthropicSubscriptionRouter` lines 87/109) and falls back to the machine's own account when out-of-scope or over the per-grant ceiling, never silently overspends the borrowed one.
*Partition finding (gap-confirmed — drives R7a): the per-account ceiling is a distributed-counter-under-partition problem the prior draft asserted as solved without specifying the consistency model.* Verified: `AnthropicSubscriptionRouter.evaluate` (lines 86–125) is a per-call LOCAL decision with no notion of a global account budget; "the authoritative machine refuses further grants as the aggregate nears the ceiling" depends on each VM REPORTING spend and an authoritative machine aggregating — but under this spec's own Phase-C assumptions (no LAN, partitions expected, the authoritative machine can be offline per §6.4), if the authoritative machine is partitioned, naïve enforcement either lets VMs keep spending against last-known local allowance (unbounded overshoot → a real money/quota-exfil vector: a compromised VM under a legitimate grant drains the borrowed account during a partition) or stalls the feature (contradicting "credentials are non-quorum and fully local at execution"). **Safe-direction resolution (R7a — LEASE-SLICE, pick option 1):** each VM gets a pre-allocated **sub-budget — a lease slice of the ceiling** — from the authoritative machine. A VM spends only within its slice; the worst case is bounded by **sum-of-leases, never N×ceiling**. A VM that exhausts its slice **fails closed to its own account** until it can renew its slice from the authoritative machine. The aggregate of all outstanding slices never exceeds the grant's `maxSpend`, so even with the authoritative machine partitioned the bound holds. The grant's `maxSpend` is set with this slicing in mind. (The rejected alternative — best-effort-eventually-consistent with a named bounded overshoot — is documented as worse for a money/quota credential and is NOT chosen.) So N VMs cannot collectively exceed the single account's grant, by construction, under partition.

**S6 — Headless-enrollment phishing (fake code/URL).** Enrollment is mobile-first and headless on cloud VMs (the VM has no browser; `EnrollmentWizard` surfaces the provider's PUBLIC `verificationUrl`/`userCode`, lines 35–49, 124–144). An attacker substitutes a fake `verificationUrl`/`userCode` (MITM on the display surface, a poisoned `driveLogin` driver, a lookalike URL) so the operator approves the attacker's login at a fake page — handing over an authenticated session before any sync. The known two-code Claude sequence (`flowNotice`, lines 106–115) is exactly where a phished second code slips by, and a headless VM has no human at it to notice a wrong host.
*Mitigations:* validate `artifact.verificationUrl` against the provider's canonical host allowlist before showing it (a URL off-allowlist is refused, not relayed); deliver the pending-login artifact only over the PIN-gated dashboard / authenticated Telegram path (never an unauthenticated link), treating `driveLogin` output as untrusted (domain-allowlist + shape validation); prefer the device-code flow on a cloud VM (operator approves on their own trusted device against the provider's real page) and bind the enrollment to the operator's authenticated session; post-enroll, verify the enrolled `SubscriptionAccount.email` (auto-populated from the account's own login, `SubscriptionPool.ts` lines 89–93) matches what the operator expected and raise an attention item on a surprise email before the account is selectable.

**S7 — Mechanism B weaponized for credential THEFT-by-redirection on a compromised/forgotten TARGET VM (gap-confirmed — Mechanism B is the DEFAULT, so this is the default mechanism's blind spot).** S6 phishes the OPERATOR with a fake URL; S7 makes the TARGET VM the attack surface. Mechanism B deliberately MINTS a fresh live credential and stores it in plaintext on the target by design (§5.3, R10). If the target VM is compromised (S1 already puts a rooted peer in-scope), its `driveLogin` (`LoginDriver`, `EnrollmentWizard.ts` lines 42–49) — which runs ON the target (R6) — can (a) return an attacker's `verificationUrl`/`userCode` so the operator approves the ATTACKER's account (the wizard "NEVER handles a credential" per its header, but the attacker controls what login actually completes), or (b) after a legitimate login, exfiltrate the freshly-minted token from the target's own config-home. The prior draft framed Mechanism B as strictly safer ("a compromise of one never yields a transportable credential for the other") — true for the OTHER machine, but Mechanism B by design CREATES a real, harvestable credential on the (possibly hostile or simply forgotten — the "VM I forgot I spun up" of §5.1) target. **The key correction: operator-driving an enrollment onto a machine implicitly TRUSTS that machine with a real, refreshable login it will hold in plaintext.**
*Mitigations:* authorizing Mechanism B onto a target is **equivalent to granting that machine a live credential** — so the target MUST clear the SAME trust floor as a Mechanism-A recipient (R9's "established, trust-gated mesh peer"), NOT a lower bar just because "nothing crossed the wire" (specified in R6, R9). Apply S6's host-allowlist + completed-account-email validation to the target-side `driveLogin` EXPLICITLY and BEFORE the account is selectable: a target-supplied `verificationUrl`/`userCode` is untrusted (host-allowlist) even when minted on the target, and the completed account's email is validated against operator expectation before the account becomes usable. State plainly: Mechanism B is safer for *blast-radius* (no transportable credential leaves) but NOT for "is this machine trustworthy enough to hold a real login" — **both mechanisms require the target to be trusted with a live credential.**

**S8 — Mechanism B revocation under a NON-COOPERATIVE / departed holder (gap-confirmed — the DEFAULT mechanism has the weakest revocation).** A B-minted credential is a fully-valid, independently-minted OAuth login (its own refresh token) the provider sees as legitimate, that instar on the holder can keep refreshing indefinitely. The §2.3 claim "removing the recipient's login leaves the source untouched — rollback is local and total" is true ONLY for a **still-paired, cooperative machine running instar that obeys the logout**. It is FALSE for a de-paired or hostile machine: instar cannot reach into it to force the logout, and R12.i's "surface an attention item" understates this — for the DEFAULT path the de-paired machine keeps a live, long-lived, independently-refreshable login. B's per-machine-independent-login property (sold as a FEATURE in §7) is exactly what makes its credentials independently long-lived and un-reachable after de-pair.
*Mitigations:* a B-minted credential on a de-paired/non-cooperative machine is revocable **ONLY by provider-side action** — the operator de-authorizes that session/device at the provider, or rotates. This is the explicit, documented revocation path for B-on-a-departed-machine (R12.i corrected) — NOT an attention item that implies instar will handle it. Scope the "rollback is local and total" claim to "a still-paired, cooperative machine"; for any other case the honest answer is provider-side de-authorization, surfaced phone-first (P8 no dead ends).

### 3.3 Cross-cutting threat-model invariants

- **I1 — No credential ever rides the existing broadcast `SecretSync.provisionAll` path.** Account credentials (Mechanism A) are a distinct sync class with destination allow-lists, never collection-wide fan-out. (Guards S1, S3.)
- **I2 — The transport envelope AND verb/handler/decryptor are NEW, not reused (Mechanism A).** Account credentials use the AAD-bound envelope (`recipientFingerprint ‖ accountId ‖ grantId ‖ issuedAt ‖ pairingEpoch`) via a NEW `encryptAccountCredential`/`decryptAccountCredential` pair (the latter fails-closed on absent/mismatched AAD); the current `encryptForSync`/`decryptFromSync`/`secret-share`-handler (no AAD param, no AAD check, permissive RBAC) are insufficient and MUST be structurally unreachable for credential-class data. (Guards S1, S3, S4; built first per §9; see R3, R3a, §3.1.)
- **I3 — Deny-by-default, grant-is-authorizer, PIN-authenticated, requester ≠ authorizer, cross-machine-VERIFIABLE** (Coordination-Mandate precedent, parent spec line 418). No mesh peer signature alone authorizes enrollment or credential receipt — AND the operator's PIN-gated authority must be verifiable by a REMOTE peer via an asymmetric issuance signature, because the existing HMAC-over-authToken proof is LOCAL-ONLY (§3.1a). (Guards S2, S5; see R4, R4a.)
- **I4 — Revocation must work against an offline/hostile machine**, but the layers are honestly scoped to the artifact each reaches: the active delete verb wipes the plaintext config-home file (reach-dependent); key-rotation/epoch-advance makes only the instar-held SEALED blob inert (NOT the plaintext file); short TTL ages out the plaintext residue; and provider-side rotation is the ONLY complete revocation. Cryptographic inertness is a scoped backstop for the sealed copy, never a complete backstop for the plaintext credential. (Guards S3, S4; see R4b, R10, R12; corrected per gaps 3+6.)
- **I5 — Phase C: every bound is independent of pool size** — destination allow-lists are explicit (not "all peers"), per-account budgets are lease-sliced with a sum-of-leases bound under partition (not per-VM, not best-effort overshoot), trust/quorum math never assumes exactly 2 peers, and enrollment is headless device-code with no interactive-browser assumption AND a defined cross-machine enroll-drive transport. (Guards S1, S5, S6, S7; see R6, R6a, R7a.)
- **I6 — Honest disclosure of residual risk** — the active wipe + sealed-blob inertness + short TTL reduce but do NOT eliminate a leaked-OAuth credential's reach (the plaintext config-home residue survives an epoch advance, and a B-minted credential on a departed machine is independently refreshable); only provider-side re-auth/rotation/de-authorization fully kills it, and the operator surface must say so plainly. (Guards S4, S8; P8 no-dead-ends.)

## 4. Numbered Requirements

> R1–R8 (plus R4a, R4b, R6a, R6b, R7a) are mandatory for BOTH mechanisms or for the shared primitives. R3a and R9–R14 apply only where Mechanism A (sealed-transport) is ever enabled (R4b's key-rotation backstop is built once and serves both). R0 is a documentation-coherence precondition.

**R0 — Resolve the `SubscriptionPool.ts` header ↔ WS5.2 contradiction in writing, in the same PR.** Any build PR that wires a follow-me path MUST correct the `SubscriptionPool.ts` header comment (lines 11–16) so it no longer asserts the default is "ship each account's credential blob." It must instead state: default is re-mint per machine; credential-blob transport is the gated, per-provider-allowlist, Anthropic-excluded Mechanism A. (§1.1.)

**R1 — Operator-initiated, operator-authenticated, deny-by-default.** A peer machine can NEVER initiate account follow-me onto itself via the mesh. Initiation requires the operator's dashboard-PIN/mandate authority (parent spec lines 418–419). Authorization is expressed AS a coordination `Mandate` (`src/coordination/types.ts`) — an `Authority` (line 18) with `action: 'account-follow-me'` and `bounds` pinning `accountId` (a specific `SubscriptionAccount.id`, never "all"), `targetMachineId` (a specific machine, never "all"), and `mechanism` (`'re-mint'` | `'credential-transport'`; only `'re-mint'` is permitted by default config). This inherits, for free, from `MandateGate.evaluate()` (`MandateGate.ts` lines 85–134): deny-by-default (no mandate → deny), expiry (`mandate.expiresAt`, line 96), revocation (`mandate.revoked`, line 101), named-party binding, and the hash-chained `MandateAudit`. The agent literally cannot mint or widen this — authorities are "produced ONLY by the PIN-gated issuance path" (`types.ts` lines 37–38, 80). Phone-first (Mobile-Complete Operator Actions): the operator drives this from the dashboard Mandates tab, never a terminal command.

**R2 — Pointer/re-mint is the wire default (Mechanism B).** When `multiMachine.accountFollowMe` is enabled, the replicated artifact is the non-credential enrollment descriptor + the redacted metadata projection. No OAuth token, refresh token, or config-home file content crosses the mesh under the default path. The descriptor and the metadata projection BOTH pass `assertNoCredentialFields()` (`SubscriptionPool.ts` line 359) at the outbound chokepoint before they can be enqueued — re-running the existing structural guard on the wire as defense-in-depth, so a future field added to `SubscriptionAccount` can never silently ride the wire. `configHome` is STRIPPED, not replicated.

**R3 — Per-recipient sealing is mandatory for ANY credential-bearing payload, via a NEW envelope+function (Mechanism A).** Where Mechanism A is enabled (R9), the credential is sealed per recipient to the recipient's X25519 key (from its `MachineIdentity` / `MachineRegistry` `encryptionPublicKey`, `types.ts` line 1698) via a **NEW `encryptAccountCredential(secrets, recipientPublicKey, aad)`** with the AAD binding of I2/§3.1 — **NOT `encryptForSync`**, which has no AAD parameter and cannot carry the binding (verified, §3.1). Decryption at the recipient is via a **NEW `decryptAccountCredential(payload, ownPrivateKey, expectedAAD)`** that FAILS CLOSED if AAD is absent or mismatched — **NOT `decryptFromSync`**, which never calls `setAAD`. Both functions are built in the first PR (§9 build-order). Sealing is per-recipient via the `provisionAccountTo(accountId, targetMachineId, mandateId)` path of S1's mitigation — NEVER `provisionAll`'s broadcast, never a group key, never unsealed even on a "trusted" LAN.

**R3a — Mechanism A uses a DISTINCT command type + handler + RBAC case; the legacy `secret-share` path is barred for credentials.** Account-credential payloads MUST use a DISTINCT mesh command type **`account-credential-share` (never `secret-share`)** routed to a DISTINCT handler that calls `decryptAccountCredential` (R3). The new verb gets its OWN `checkCommandRBAC` case — NOT the "any registered peer" read/observe class that `secret-share` falls into (`MeshRpc.ts` lines 235–249, verified permissive) — verifying AT THE GATE, before decrypt: (a) the sender is the operator-mandated source, (b) THIS recipient == the mandate's single `targetMachineId`, (c) an unexpired, single-use grant. Precedent: `commitment-mutate`'s dedicated RBAC case (`MeshRpc.ts` lines 219–227). The legacy `SecretShareHandler`/`decryptFromSync`/`secret-share` verb MUST be structurally unreachable for credential-class data — a Tier-2 test asserts `SecretShareHandler.handle` rejects (or never receives) an account-credential payload, and that `decryptAccountCredential` throws on a payload with no/wrong AAD. Reusing `secret-share` for credentials is FORBIDDEN (it would put account credentials in the broadcast-eligible read class and collapse authorization onto crypto alone). (Guards S1, S2; gaps 2, 12.)

**R4 — Reach is not authority; the receiver revalidates (L15).** Acceptance at the recipient requires the full mesh acceptance gate (`MeshRpc.acceptCommand` = `verifyEnvelope` + `checkCommandRBAC`, lines 164/199/265): a valid Ed25519 signature from a registered ACTIVE peer, correct recipient binding, an unreplayed nonce, AND the follow-me/credential RBAC case (R3a for Mechanism A). **Critically (S2, S3 dual of WS1.3 forged-pin theft):** before any target machine acts on a *replicated* follow-me mandate, it MUST verify the mandate's ASYMMETRIC issuance signature per R4a — NOT merely that "a mandate record appeared in the journal," and NOT the HMAC-over-authToken proof, which a remote peer structurally cannot verify (§3.1a). A replicated journal record is *reach*; R4a's signature verification is *authority*. A peer-supplied account descriptor is revalidated against the recipient's own operator binding (the `topic-operator` verified-principal pattern); a peer can never assert a recipient must enroll an account it has not locally consented to.

**R4a — Cross-machine mandate verification is an asymmetric, first-PR primitive (NOT a free inherit from `MandateGate`).** The existing mandate authorship proof is a SYMMETRIC HMAC keyed on the per-machine server `authToken` (`AgentServer.ts` lines 1038–1046; `MandateStore.verifyAuthorship`, `MandateStore.ts` lines 201–204) — named here as a **LOCAL-ONLY limitation that CANNOT authorize a peer** (a target machine has a different authToken and would reject every legitimate cross-machine mandate). Sharing the authToken across machines is FORBIDDEN (one disk compromise → forge-any-mandate-on-any-peer). Therefore any mandate that crosses the mesh MUST carry an **asymmetric issuance signature**: the canonical mandate is signed with the issuing machine's Ed25519 *identity* private key (the `MachineIdentity` key used for mesh envelopes, `types.ts` lines 1692–1698), bound to the issuer's machine fingerprint, so any peer holding the registered public key verifies "this mandate was issued by my trusted operator-machine." The operator's PIN authority is proven to a remote verifier through the trust root: **PIN-gated issuance on operator-machine M → M's Ed25519 signature → the peer verifies M is the registered operator machine** (verified-operator binding, never a content name). Built in the first PR alongside R3. **Tier-2 test (proves the gap):** a mandate issued on machine A with A's authToken is REJECTED by machine B's HMAC verifier today, and ACCEPTED only via the new asymmetric path. (Guards S2, S3; gap 1.)

**R4b — De-pair forward-revocation: rotate the recipient X25519 key (the chosen pairing-epoch mechanism).** The "pairing-epoch" named across S3/S4/I2/I4/R12.iii does NOT exist in the codebase (verified: only `FencedLease.leaseEpoch` and a placement epoch exist; de-pair flips `status:'revoked'` and rotates nothing — `types.ts` lines 1740–1754). It is promoted from Open Question to a designed primitive here. **Chosen lean: on de-pair, ROTATE the machine's X25519 `encryptionPublicKey`** (reusing existing identity machinery) — this makes EVERY old sealed payload undecryptable, delivering S3 (replay-defeat) and S4's sealed-blob inertness at once, with no new counter to anchor. The AAD's `pairingEpoch` term binds to the key-rotation generation. The build MUST define: (1) what a "pairing" is and that the rotation is durably persisted against rollback by a hostile local user, (2) that de-pair triggers the rotation (and may also bump an explicit generation counter if a counter is preferred over key-rotation — but key-rotation is the lean), (3) how the receiver learns the current key/generation durably across reboot. **Honest limitation:** key-rotation makes the SEALED blob inert; it does NOT touch the framework's plaintext config-home token (S4 layer-2 limitation, gap 6) — that residue is handled by the active wipe verb + short TTL + provider rotation, not by rotation. **Tier-1 test:** a sealed blob fails to decrypt after key-rotation — proven ONLY under the new key/AAD scheme, NOT under the current single-key store. (Guards S3, S4; gaps 3, 7.)

**R5 — No LAN trust assumption (Phase C).** No path may assume peers share a LAN or that exactly 2 peers exist. All Mechanism-A sealing is per-recipient (R3), all auth is the mesh Ed25519 envelope (works identically across N cloud VMs over the tunnel/relay), and no code path reads a credential "because we're on the same network." Authorization is **operator-rooted, NOT peer-quorum-rooted** — authority flows from the operator's single PIN-gated mandate, so it holds identically at pool size 1, 2, or 50. The only peer-trust check is R4's receiver-revalidation of the issuance signature (single-signer, not a quorum). No future contributor may add a "N peers must approve" assumption.

**R6 — Headless enrollment on cloud VMs (Phase C — the core of Mechanism B).** Mechanism B's per-machine login MUST complete with no interactive browser on the recipient. The operator-driven flow targets a specific recipient VM: the operator picks "enroll account `<id>` on machine `<nickname>`" on the dashboard; the drive reaches the target via R6a's defined transport, where `EnrollmentWizard.start()` (whose `driveLogin` / `LoginDriver`, `EnrollmentWizard.ts` line 42) runs ON the target VM, so the device-code/URL is minted by the machine that will hold the credential and never transits. `EnrollmentWizard.start()` already yields a PUBLIC `verificationUrl` + `userCode` with a visible TTL (line 124) and auto-reissues on expiry (`reissueExpired`, line 153) — the operator approves from their phone; the cloud VM never opens a browser. The VM-side `LoginDriver` MUST use the device-code/URL flow exclusively — never a flow assuming a local browser/redirect. `complete()` MUST run `ensureInteractiveReady(configHome)` (the 2026-06-09 headless-`claude auth login` wedge fix, imported at line 30) so the freshly-enrolled cloud home does not wedge the first session pinned onto it. A cloud VM with no display is the assumed default, not an edge case. **The completed login's host + email are validated per S7 BEFORE the account is selectable** (a target-supplied `verificationUrl`/`userCode` is untrusted even when minted on the target), and **authorizing Mechanism B onto a target is equivalent to trusting it with a live credential — the target MUST clear the same trust floor as a Mechanism-A recipient (R9), not a lower bar (S7).**

**R6a — The cross-machine enroll-drive transport MUST be defined before Mechanism B is claimed Phase-C-ready (gap-confirmed: it does not exist today).** Verified: the only production `LoginDriver` is `FrameworkLoginDriver` (`src/core/FrameworkLoginDriver.ts` line 43), whose `deps.spawn` opens a LOCAL tmux pane in the server's own process and scrapes it (`drive()` lines 107–138). There is NO cross-machine spawn/exec primitive — the `MeshCommand` union (`src/core/MeshRpc.ts` lines 28–105) contains no `remote-spawn`/`remote-exec`/`enroll-on-peer` verb (the only cross-machine command-execution is `remote-close`, a relayed session kill, not a login spawn). So "operator on machine A taps 'enroll account X on VM B' → A makes B run its own `EnrollmentWizard`" has no mechanism today; R6's "minted ON the target, never transits" guarantee is currently about a code path that isn't there. The build MUST pick ONE in writing:
- **(1) A NEW operator-authorized mesh verb (`enroll-drive`)** carrying the mandate id + accountId, RBAC'd as **operator-mandate-gated** (its OWN `checkCommandRBAC` case — NOT the "any registered peer" read class, same discipline as R3a), whose handler on the TARGET invokes its OWN local `EnrollmentWizard.start()` and returns the public artifact back to A's pending-login surface; OR
- **(2) Reframe Mechanism B as operator-side machine selection**: each VM runs its own instar server, and the operator opens the dashboard scoped to the TARGET VM's own server instance and drives that VM's local enroll route directly — i.e. the orchestration is operator-side, per-server, NOT A-drives-B over the mesh.
Whichever is chosen, the spec names the REAL symbol that spawns the login on the target. (Guards the §5.3/§7 Phase-C claim; gap 11.)

**R6b — Phase-C headless-enrollment reliability contract (gap-confirmed: the device-code scrape has no remote-VM timeout/recovery contract).** Verified: on a headless VM the public artifact reaches the operator only via `FrameworkLoginDriver.drive()` scraping a tmux pane (`parseArtifact`, lines 85–100) within a 60s `scrapeTimeoutMs` (line 76), throwing on timeout (lines 132–137); `EnrollmentWizard.start()` does NOT catch that throw (lines 122–144) — only `reissueExpired` swallows driver errors (lines 172–176). So a slow/differently-formatted/network-stalled remote login throws an uncaught error through the (R6a) enroll-drive handler, and the two-code Claude flow (`flowNotice`, lines 106–115) can have code #2 arrive after the 60s single-scrape window. The build MUST specify: (1) a configurable, **larger scrape-timeout budget for remote/cloud drives** (cloud→provider latency is not the local-LAN assumption); (2) an **honest failure surface** — a remote drive throw produces an operator message ("couldn't start the login on `<nickname>` — retry?"), NOT a silent stuck pending-login, since `EnrollmentWizard.start` does not currently swallow the throw; (3) the **two-code Claude sequence handled headlessly** — the poll window MUST cover the FULL two-code interaction (re-scrape/re-poll AFTER the operator submits code #1, since code #2 is emitted later in the SAME pane), or split it into two artifacts, rather than one 60s deadline from spawn. **Prefer the device-code single-code flow on cloud VMs where the provider supports it** (`EnrollmentWizard.defaultKind` already does this for openai, lines 92–94) — name it as the Phase-C default. (Guards S6; gap 13.)

**R7 — Bounded per-account budget, independent of pool size (Phase C).** Follow-me replication / enrollment-offer / credential-push rate is capped PER ACCOUNT (not per pool), so adding the Mth machine cannot multiply traffic by M. Specific bounds: a per-account **max-follow-machines cap** (config-declared, small default e.g. 5) — issuing a mandate that would exceed it is refused at issuance with an honest reason; **one mandate per (account, target) pair**, never one-message-per-machine spam — a depth-zero detection across K new VMs raises ONE aggregated consent item listing all K (P17 Bounded Notification Surface), not K topics; the metadata projection is fixed-size per account and rides the WS5.1 small heartbeat summary (count + blocked-bool per provider), with the holder-authorized inventory pull on demand → N VMs cost O(N) fixed-size summaries, not O(accounts×N); and at most one in-flight enrollment per (account, target-machine) (the wizard's single-flow-per-id discipline plus a per-target in-flight cap). Mechanism-A credential operations are serialized and rate-capped, structurally bounded by the mandate's single `targetMachineId`, inside the parent spec's ONE aggregate replicated-journal budget for descriptor records, with backoff + a sustained-failure breaker (P19).

**R7a — Per-account spend ceiling is LEASE-SLICED (bounded under partition), not a naïve aggregate (gap-confirmed).** `AnthropicSubscriptionRouter.evaluate` (lines 86–125) is a per-call LOCAL decision with no global account budget; a distributed ceiling that depends on reporting + an authoritative aggregator overshoots if that machine is partitioned (the spec's own Phase-C assumption). The chosen consistency model is **option 1 — lease slices**: each granted VM gets a pre-allocated SUB-budget (a slice of the grant's `maxSpend`) from the authoritative machine; a VM spends only within its slice; on slice exhaustion it **fails closed to its own account** until it renews the slice; the sum of all outstanding slices never exceeds `maxSpend`, so the worst case under partition is bounded by **sum-of-leases, never N×ceiling**. The grant's `maxSpend` is set with the slicing in mind. The rejected alternative (best-effort-eventually-consistent with a named overshoot) is documented as worse for a money/quota credential and NOT chosen. **Phase-C test:** authoritative machine partitioned → aggregate spend is bounded by sum-of-leases, NOT N×ceiling. (Guards S5; gap 5.)

**R8 — Observable + audited (P7).** Every follow-me decision — mandate evaluate (allow/deny), offer-made, enrollment-started, completed, refused (with RBAC/grant reason), sealed-credential-pushed (Mechanism A), revoke-triggered, wipe-completed — emits a feature metric + an audit line. The `MandateAudit` hash-chain covers the authorization decisions; the enrollment/data-plane actions need their own audit trail. Audit lines carry NAMES only (account id, recipient machine id, descriptor fields) — never a token, never a credential value, never even a sealed blob's plaintext length that could leak structure. Mirrors `secretKeyPaths` (`SecretSync.ts` line 53): the read surface returns key-paths/names, never values. A credential that moved (or a login that was enrolled) without an audit line is the unaccountable-action failure mode this whole round exists to prevent.

**R9 — Mechanism A is a separate, provider-gated, opt-in flag; BOTH mechanisms require a trusted target.** Sealed-credential transport is NOT reached by `multiMachine.accountFollowMe` alone. It requires a distinct `multiMachine.accountFollowMe.credentialTransport` **per-provider allowlist** (default EMPTY — never a global boolean, so a future operator cannot flip Anthropic into a ToS violation by accident), AND the mandate's `mechanism: 'credential-transport'`. It is REFUSED for any provider whose ToS prohibits token movement — **Anthropic is excluded from the default allowlist** per `SubscriptionPool.ts` lines 17–21. For a refused provider, the system falls back to Mechanism B with an honest operator notice ("this provider requires a per-machine login; I'll walk you through it"). No dead end (P8). Mechanism A also REFUSES a target machine that is not an established, trust-gated mesh peer with a verified `encryptionPublicKey` (the bootstrap-paradox guard: a fresh cloud VM has no key to seal to until it is paired+trusted — never auto-trust a new VM to receive a credential). **The SAME established-trusted-peer floor applies to Mechanism B (S7):** because Mechanism B mints and holds a real, refreshable live credential on the target, authorizing it onto a machine IS trusting that machine with a live credential — a target may NOT clear a lower bar just because "nothing crossed the wire." Both mechanisms gate the target at the same trust floor.

**R10 — At-rest containment for sealed credentials, with the plaintext residual stated honestly (Mechanism A; gaps 3+6 reconciled).** A credential delivered via R3 is written to the recipient's config-home ONLY at the moment of use; the design MUST minimize the plaintext-on-disk window. **Refresh-on-arrival is MANDATORY where the framework permits it** (not merely "preferred"): the recipient immediately mints its OWN short-lived token from a sealed refresh artifact, then drops the sealed artifact — and the credential's TTL is set short enough that an offline/de-paired machine's plaintext token EXPIRES before a wipe verb realistically matters. The residual-plaintext-window is stated explicitly to the operator. The sealed-blob-in-transit and any instar-held copy live in the AES-256-GCM `SecretStore` (`SecretStore.ts` line 7), never a bare file. **The unavoidable residual — the framework's own plaintext token file in config-home — CANNOT be made cryptographically inert by R4b's key-rotation/epoch advance** (that only protects the instar-held sealed blob, which is NOT what a session reads — gap 6). Against a hostile root on a de-paired machine after the active wipe verb never arrived, the plaintext file is readable; the spec does NOT claim otherwise. The honest containment for the plaintext residue is the active wipe verb (reach-dependent) + short TTL + **provider-side rotation as the only complete answer (I6/R12.ii)**. This explicit accepted exposure delta is documented to the operator at enable time.

**R11 — Rotation / expiry (Mechanism A).** Sealed credentials carry an explicit expiry and are stale the instant the source refreshes its local token (`lastRefreshAt` transition, `SubscriptionPool.ts` line 118). The recipient MUST NOT use a stale sealed credential; on staleness it falls back to a fresh per-machine login (Mechanism B) rather than presenting an expired token. Re-seal-on-refresh, if implemented, is rate-capped under R7 and audited under R8. A refresh token is NEVER replicated unless R9's provider allowlist explicitly permits it for that provider.

**R12 — Revocation / de-authorization is a real un-merge, not a flag wish (both mechanisms).** "Stop following to machine X" is mandate revocation (control plane: PIN-gated revoke on the Mandates tab; the agent cannot revoke; every subsequent `MandateGate.evaluate` denies) PLUS a real data-plane effect:
   (i) **Mechanism B — local & total ONLY for a still-paired, cooperative machine; provider-side for any other (S8/gap 10).** For a still-paired machine running instar that obeys the logout: the target logs the account OUT of its config-home (the framework's own logout against that `CLAUDE_CONFIG_DIR`), deletes the per-account config slot, and calls `SubscriptionPool.remove(accountId)` (`SubscriptionPool.ts` lines 334–341) on the target — removing the recipient's login leaves the source untouched, rollback is local and total. **But this is FALSE for a de-paired or hostile machine:** a B-minted credential is a fully-valid, independently-minted OAuth login with its OWN refresh token that instar on a non-cooperative holder can keep refreshing indefinitely, and instar cannot reach in to force the logout. So for a de-paired/non-cooperative holder, the **documented revocation path is provider-side de-authorization** — the operator de-authorizes that session/device at the provider, or rotates — surfaced phone-first as a real instruction, NOT an attention item implying instar will handle it. When a machine is de-paired its `MachineStatus` flips to `'revoked'` with `revokedAt`/`revokedBy` (`types.ts` lines 1711, 1750–1752); the agent surfaces "machine X de-paired — it still holds its own live, refreshable login for account Y. instar cannot revoke it remotely; de-authorize/rotate at `<provider>` to kill it." The "rollback is local and total" claim is scoped to the cooperative case in writing (gap 10).
   (ii) **Mechanism A:** revocation triggers the `credential-revoke`/`secret-revoke` mesh verb (the one `SecretSync.ts` line 18 flagged as a "deliberate follow-up" — now IN SCOPE and REQUIRED before any Mechanism-A credential ships) instructing the target to wipe the transported blob via `SafeFsExecutor` (audited), plus the same logout + `SubscriptionPool.remove`. Because A relocated a credential, the design MUST also trigger an operator-facing **credential-rotation prompt** at the provider (the only real revocation for a delivered credential), surfaced phone-first. A delivered credential cannot be un-delivered — this asymmetry is itself a reason A is not the default.
   (iii) **Offline-honest pending state — BOUNDED, with a loud give-up (Phase C; gap 9).** If the target is offline at revoke time, the mandate is revoked immediately on the control plane; the data-plane wipe is a **durable pending action** fired when the target reconnects (reuse the cross-machine-secret-sync "write it down durably, fire on return" pattern). Until the target confirms the wipe, the dashboard shows the account on that machine as `revocation-pending`, NOT `removed`; the operator is told the truth ("Authorization revoked. `<nickname>` is offline — its copy will be removed the moment it reconnects"), never a false "removed everywhere" claim (L8 active follow-through, P8 no dead ends). **The pending wipe is NOT unbounded:** it inherits the resume-queue's give-up discipline — after a fixed reconnect-deadline (operator-tunable, small), it **escalates from `revocation-pending` to a LOUD `revocation-FAILED — rotate at provider NOW` attention item** (tied to R12.ii's provider-rotation prompt), surfacing that the credential on `<nickname>` could not be confirmed destroyed. The honest end-state of an offline-forever / terminated / reimaged VM is "rotate at the provider," never a silently-aging "pending." This is load-bearing precisely because the cryptographic backstops cannot deliver against the plaintext residue: a terminated VM holding a live (Mechanism A) credential is otherwise an indefinitely-live, UI-says-"pending"-forever leak (gaps 1, 6, 9 compounded). For Mechanism A, R4b's key-rotation makes the offline machine's instar-held SEALED blob inert at the next de-pair — but it does NOT touch the plaintext config-home token, so provider rotation, not inertness, is the load-bearing answer when the wipe verb never arrives.

**R13 — Seal-time status revalidation is the load-bearing guard; detection is necessary but insufficient (gap-confirmed retarget).** The original wiring to `PeerVisibilityGuard.detectImproperRevocations` (`PeerVisibilityGuard.ts` lines 49–67) was mis-targeted: that helper is a PURE hygiene check that flags entries which ARE revoked but are MISSING `revokedBy`/`revokeReason` audit fields — it does NOT detect a peer that still treats a de-paired machine as ACTIVE (a peer whose registry never received the revocation), which is the `peer-missing`/`peer-flapping` surface, a separate code path. A peer with a stale registry showing machine X `active` would pass `detectImproperRevocations` cleanly and STILL seal a credential to the de-paired X — the exact hole. **Therefore the load-bearing guard is at SEAL TIME:** the credential-seal path (`provisionAccountTo`) MUST re-validate the destination's CURRENT status from authoritative registry state (status === `'active'` AND not revoked AND key/epoch-current per R4b) — NEVER from a possibly-stale cached peer view — before sealing. **Safe-direction default: if the seal-time status is unknown or stale beyond a freshness bound, REFUSE the seal (fail-closed), do not seal optimistically.** `detectImproperRevocations` remains wired as a complementary monitoring layer (it catches non-propagated/under-audited revocations after the fact), but it is NOT the guarantee — refusing-to-seal-to-a-machine-whose-revocation-this-machine-hasn't-heard-of is. (Guards S3, S4; gap 8.)

**R14 — Single-machine + mixed-pool no-ops, with flag coherence.** On an agent with no registered peers (pool size ≤ 1), no follow-me code path is ENTERED (parent invariant 6). Per the parent flag-coherence invariant (invariant 5): a follow-me descriptor or credential verb is emitted ONLY to peers advertising the matching `multiMachine.accountFollowMe` flag state in their authenticated `MachineCapacity.seamlessnessFlags` heartbeat — an absent flag = pre-spec = non-participant = no follow-me traffic, never a silent partial enrollment across a skewed pool. The applier silently drops unknown journal kinds, so a mixed pool degrades to "no meta replication," never a half-applied account view. A target VM with the flag dark REFUSES the operator-driven enroll instruction with an honest "enroll directly on that machine" reason (no dead end, P8).

## 5. Enrollment & Operator Consent

### 5.1 Authorize an account to follow you to a new machine

Authorization is per-machine, per-account explicit consent expressed AS a coordination mandate (R1) — NOT a standing global "sync everything" toggle. The operator's act is on the dashboard **Mandates tab**, phone-first (constitutional Mobile-Complete Operator Actions; matches the existing Coordination Mandate UX): pick the account (from the registered `SubscriptionPool` list), pick the target machine (from the registered machine/nickname list — reuse `NicknameAssigner`), pick the mechanism (re-mint default; credential-transport only when the provider is on the allowlist), set a duration, type the PIN, tap **Authorize**. Issuance is PIN-gated server-side (`types.ts` line 13) — the agent's Bearer token is structurally insufficient to mint it.

Because `paramsSatisfyBounds` is exact-match (`MandateGate.ts` line 112), a mandate authorizing follow-me to machine X does NOT authorize machine Y — the operator consents once per (account, machine) pair, closing the "I approved my laptop, why is it now on a VM I forgot I spun up?" surprise in an N-VM fleet.

### 5.2 The agent's role: request, never self-authorize

When the agent detects depth-zero on a new machine (WS5.1 already does this), the flow is:
1. The agent calls `MandateGate.evaluate({ action: 'account-follow-me', params: { accountId, targetMachineId, mechanism }, agentFp, mandateId })`.
2. **deny (no mandate yet)** → the agent surfaces a phone-first consent request: a message *"Machine `<nickname>` has no Claude account yet. Want it to use your `<account email>` subscription? [Authorize on dashboard →]"* — a tap-through to the pre-filled Mandates form (account + target pre-selected). NEVER "run `instar subscription-pool enroll`" (parent spec line 416: never a bare CLI instruction; B2 no CLI instructions to users). It does NOT proceed.
3. **allow** → the agent drives the chosen mechanism (§5.3 / §5.4).

This preserves "a peer machine can NEVER initiate enrollment onto itself via the mesh": the *gate* is the authorizer, and only PIN-gated dashboard issuance can create the gate's allow. A peer mesh message claiming "the operator said enroll me" hits a deny — no mandate exists (reach ≠ authority, L15). The mandate replicates over the WS2 store as a NEW JournalKind with its retention/rate-cap entry in the same PR, and R4a's receiver-revalidation of the **asymmetric** issuance signature (NOT the local-only HMAC proof — §3.1a) applies before any target acts.

### 5.3 Mechanism B step-by-step (re-mint, default)

0. The operator's "enroll account X on VM B" reaches the target via the **R6a-defined transport** (a new operator-mandate-gated `enroll-drive` mesh verb, OR operator-side per-server selection) — NOT an assumed-but-absent A-drives-B primitive. The target machine must already clear the established-trusted-peer floor (R9/S7) — authorizing this is trusting it with a live credential.
1. On the target, `EnrollmentWizard.start()` runs (the wizard's own `driveLogin` executes ON the target VM, R6), with `configHome` set to a fresh per-account slot. The wizard already isolates each login to its own `CLAUDE_CONFIG_DIR`, handles the two-code Claude flow (`flowNotice`, lines 106–115) — subject to R6b's full-two-code poll-window contract — and auto-reissues on expiry (lines 153–179). A remote drive throw surfaces an honest "couldn't start the login on `<nickname>` — retry?" (R6b), never a silent stuck pending-login. Follow-me adds nothing secret to the wizard.
2. The PUBLIC code/URL surfaces back to the operator's phone via the existing PendingLoginStore / `GET /subscription-pool/pending-logins` surface, scoped with the target `machineId`. The target-supplied `verificationUrl`/`userCode` is host-allowlist-validated even though minted on the target (S7). The operator approves at the provider's page from their phone.
3. On `complete()` — which already seeds interactive-readiness for claude-code homes (the 2026-06-09 wedge fix, lines 189–200) — follow-me **validates the completed account's email against operator expectation BEFORE the account is selectable** (S7; surprise email → attention item, not auto-use), then calls `SubscriptionPool.add()` on the target with the local `configHome`. The two machines now hold two independent logins for the same operator account: exactly the "store LOCATION not token" model, and a compromise of one never yields a transportable credential for the other (though the target itself now holds a real, refreshable login — S7/S8 revocation applies).

### 5.4 Mechanism A step-by-step (sealed-transport, gated, non-default)

Reachable only when the mandate's `mechanism: 'credential-transport'` AND the provider is on the `credentialTransport` allowlist (R9). The blob is sealed to the target's X25519 key via the NEW AAD-bound `encryptAccountCredential` (I2/R3 — NOT `encryptForSync`) over the per-named-recipient `provisionAccountTo(accountId, targetMachineId, mandateId)` path — NOT `provisionAll`'s broadcast. It rides the **DISTINCT `account-credential-share` verb with its own operator-mandate-gated RBAC case (R3a) — NEVER the permissive `secret-share` verb/handler**, which is structurally barred for credential-class data. The command carries `mandateId` + `targetMachineId` + an expiry/nonce (audience-bound, single-use, the WS4.4 proxy-assertion pattern); the receiver's RBAC case verifies sender-is-mandated-source + recipient-matches-mandate + unexpired-single-use grant AT THE GATE (before decrypt), re-validates the destination's CURRENT active/key-current status at seal time (R13), then `decryptAccountCredential` fails-closed on absent/wrong AAD and writes the credential only on success (R10). The real revoke verb (R12.ii) is a prerequisite — Mechanism A cannot ship without "stop following" being able to destroy the relocated copy.

### 5.5 De-authorization

Mandate revocation on the Mandates tab (PIN-gated; the agent cannot revoke) plus the real data-plane effect of R12 — Mechanism B logs out + `SubscriptionPool.remove` on a still-paired cooperative target (a de-paired/non-cooperative B holder is revocable ONLY provider-side, surfaced as a real instruction — S8/R12.i); Mechanism A also fires the wipe verb + provider rotation prompt; offline targets get the **bounded** durable pending wipe + `revocation-pending` honest state that escalates to a LOUD `revocation-FAILED — rotate at provider` after the reconnect deadline (R12.iii).

## 6. Integration & Failure Semantics

**6.1 `SubscriptionPool` (registry follow-me = metadata only).** A redacted `SubscriptionAccount` projection replicates over the WS2 store as a new `subscription-account-meta` JournalKind (HLC ordering, receiver-side schema validation before merge, aggregate journal budget, per-peer sustained-failure breaker). The outbound projector RE-RUNS `assertNoCredentialFields()` (R2 defense-in-depth) and STRIPS `configHome`. The recipient stores the meta in a foreign namespace its local reads UNION (WS2 union-reader discipline), so `GET /subscription-pool?scope=pool` (WS5.1) shows a peer's account depth/quota without holding its login.

**6.2 `AnthropicSubscriptionRouter` (routing never silently crosses a credential boundary).** The router (`AnthropicSubscriptionRouter.evaluate`) selects an Anthropic path for an *internal* call on *this* machine; it MUST NOT route a call to a peer's account. Add a `locallyExecutable` gate to account selection: an account is executable on machine M iff M's `SubscriptionPool` holds it with a real local `configHome` AND a valid login (status active/warming, not needs-reauth). A meta-only account (replicated-in from a peer, no local config-home) is `locallyExecutable: false` and is invisible to `evaluate()`'s fallback/`force` paths — closing the `force`-mode "use an account I have metadata for but no credential" hole at *selection* time, not at call time.

**6.3 Rate-limit on a followed account.** Two cases, because "followed" means metadata:
- **6.3a — the recipient holds its OWN valid login (operator drove enrollment there).** Normal pool account; existing swap machinery applies unchanged — `ProactiveSwapMonitor` (lag-aware 80% measured threshold, covers the untagged interactive session) pre-empts the wall, `SessionMigrator` / `QuotaAwareScheduler.onQuotaPressure` performs the reactive `--resume` swap. **Critical: the swap target set is FILTERED to `locallyExecutable` accounts only** — a rate-limited account on M can NEVER auto-swap onto a peer's account (that credential lives only on the peer). If M has no other locally-executable account, it does NOT silently degrade: WS5.1 honest-disclosure (the agent OFFERS to drive `EnrollmentWizard` for an additional account on M), and `PlacementExecutor` marks M `quotaState.blocked` (`all-machines-quota-blocked` flagged if every VM is walled).
- **6.3b — the recipient holds only metadata.** `locallyExecutable: false`; never a swap candidate. A rate-limit on it is the *peer's* concern, surfaced read-only in `?scope=pool`. User-visible answer: "that account is busy on `<peer-nickname>`; this machine isn't enrolled on it" — never a stall.

**6.4 Source machine offline (partition / dark peer).** "Source" = the machine where the login physically lives. Because credentials never cross, an offline source does NOT strand a recipient that already has its own login (6.3a is fully local — no source dependency at execution time). The offline source withholds only (i) fresh metadata/quota updates (recipients serve last-known meta with the WS2 staleness tag, mirroring `GET /guards?scope=pool`'s last-known-posture pattern) and (ii) the holder-authorized on-demand inventory (WS5.1), which returns an honest "temporarily unavailable from `<machine>`." A recipient must NEVER attempt to "recover" the account by reconstructing a login from replicated state — there is none, by construction. Credential availability is deliberately a **non-quorum** property: reachability of N−1 peers can never manufacture a usable login on the Nth — the security-correct asymmetry.

**6.5 `SecretSync` vs the credential verb — tightly fenced, OAuth-excluded, distinct path.** Mechanism A does NOT ride the existing `secret-share` verb/`SecretShareHandler`/`decryptFromSync` (permissive RBAC, no AAD — R3a, §3.1); it uses the DISTINCT `account-credential-share` verb + `decryptAccountCredential`. The existing `SecretSync` machinery (per-recipient X25519, receive-only-by-default via `pushEnabled`) contributes its *posture* — receive-only default, per-recipient encryption, registered-peer-only, decrypt-fails-closed — which the new credential path inherits and TIGHTENS (operator-mandate-gated RBAC, seal-time status revalidation, AAD fail-closed). For Anthropic the config-home is the credential location and never serializes into a `Secrets` value — so there is literally nothing to encrypt, and the boundary is enforced by **absence of a serializer**, not a runtime check that could regress. **Explicit rules:** (1) the build round MUST NOT add a "push the config-home" or "push the OAuth token" path to the legacy `secret-share`/SecretSync flow; (2) a lint/test MUST assert no `configHome`/OAuth field is ever marked `{secret:true}` (so it can never ride `provisionAll`'s broadcast); (3) a Tier-2 test asserts `SecretShareHandler.handle` rejects (or never receives) an `account-credential-share` payload — the legacy decryptor is unreachable for credential-class data.

**6.6 Failure-mode summary (each with safe direction).**
- *Forged peer tries to enroll itself / inject a credential* → impossible: enroll instruction is operator-authenticated and carries no secret; the registry guard + projector re-assertion reject any credential field on the wire; SecretSync has no OAuth serializer.
- *Recipient executes against a peer's metadata-only account* → impossible: `locallyExecutable` gate excludes it from router selection.
- *Rate-limit with no local alternate* → honest disclosure + agent-offers-enrollment + placement marks blocked; never a silent stall.
- *Source offline* → recipients with their own login unaffected; meta/inventory degrade to last-known / temporarily-unavailable; no login reconstruction attempted.
- *Mixed-version / mixed-flag pool* → invariant-5 coherence → meta simply not replicated; credential half refuses with an honest "enroll directly" path. Conservative degrade, never half-applied.
- *Clock skew gaming the meta merge* → WS2 HLC + skew-suspicious quarantine; a fast-clock peer can't win every account-meta merge.
- *Rollback leaving residue* → namespaced un-merge + local-only `remove()` (cooperative target) or provider-side de-authorization (de-paired/non-cooperative B holder — S8) + flag-dark refusal; for Mechanism A, key-rotation makes the SEALED blob inert but NOT the plaintext config-home token (gap 6) — the active wipe verb + short TTL + provider rotation handle the plaintext residue; each verifiable by test.
- *Offline target never returns* → BOUNDED pending wipe escalates to a LOUD "revocation-FAILED — rotate at provider" after the reconnect deadline (R12.iii) — never a silently-aging "pending."
- *Forged cross-machine mandate / replayed sealed credential* → R4a asymmetric issuance-signature verification refuses a mandate not signed by the trusted operator-machine; R3a's gate-level RBAC + AAD fail-closed refuse a credential not bound to THIS recipient/account/epoch.
- *Stale-active peer seals to a de-paired machine* → R13 seal-time status revalidation refuses the seal (fail-closed on unknown/stale status); detection is the complementary monitoring layer, not the guarantee.

## 7. Phase-C Scaling (N cloud VMs, headless, no-LAN/no-2-peer, bounded budget)

Held in every requirement; consolidated specifics:

- **N cloud VMs / no LAN** — R3 (per-recipient seal via the NEW envelope, no group key), R3a (distinct credential verb + RBAC), R5 (no network-trust path; mesh Ed25519 auth identical over tunnel/relay), R4a (asymmetric cross-machine mandate verification — works over the relay where the local-only HMAC proof cannot), 6.5 (per-recipient encryption posture).
- **Headless enrollment** — R6a (the cross-machine enroll-drive transport MUST be defined — it does not exist today), R6 (public device-code/URL flow minted ON the target VM, operator approves from phone, `LoginDriver` never assumes a local browser/redirect, `ensureInteractiveReady` on the cloud home), R6b (remote-VM scrape-timeout budget + honest failure surface + full-two-code poll window; device-code single-code flow preferred on cloud). S7 (the target is trusted with a live credential; host+email validated). Mechanism A's bootstrap paradox (R9: a fresh VM has no key to seal to until paired+trusted) is itself why B is the safe headless default — B works the instant the VM can reach the provider endpoint AND clears the trust floor.
- **Quorum/trust math not assuming 2 peers** — R5 (operator-rooted authority, NOT peer-quorum; holds at pool size 1, 2, or 50; the only peer check is R4a receiver-revalidation of the single-signer asymmetric issuance signature). 6.4 (credential availability is explicitly non-quorum).
- **Bounded per-account budget independent of pool size** — R7 (per-account max-follow-machines cap; one mandate per (account, target); fixed-size meta on the heartbeat → O(N) not O(accounts×N); one in-flight enrollment per (account, target); serialized + breaker for Mechanism A; aggregated consent item for K new VMs, not K topics). R7a (LEASE-SLICED per-account spend ceiling — sum-of-leases bound holds even with the authoritative machine partitioned; a VM exhausting its slice fails closed to its own account; a 6th VM does not raise the ceiling).
- **Per-VM independent quota is a FEATURE** — with Mechanism B each followed account has its OWN login → its OWN quota window as the provider sees it; across N VMs this is the desired multi-account behavior the pool already exploits (auto-swap, depth tie-breaker, WS5.1). The spec notes this explicitly so no one "optimizes" toward a single shared credential (Mechanism A) thinking it cleaner — it is worse for BOTH quota distribution AND blast radius.

## 8. Testing (three tiers + named attack tests)

Per the Testing Integrity Standard, every requirement here lands with all three tiers. The Phase-1 "feature is alive" E2E test is the single most important test.

### 8.1 Tier 1 — Unit
- `provisionAll()` over a secret set containing an account-credential-class field emits ZERO credential payloads (credentials never ride the broadcast path — I1).
- The metadata projector STRIPS `configHome` and `assertNoCredentialFields()` rejects any credential-shaped field on the outbound record (R2).
- (Mechanism A) `decryptAccountCredential` THROWS on a payload with no AAD or wrong AAD (fail-closed — R3, gap 2); the NEW envelope fails to decrypt after the recipient X25519 key is rotated on de-pair (R4b key-rotation, NOT a counter advance under the old single-key store — gaps 3, 7); a replayed consumed `grantId` is rejected; a payload sealed for machine X fails on machine Y (recipient-fingerprint binding via AAD).
- (Mechanism A) after R4b key-rotation, the at-rest *sealed* blob fails to decrypt — sealed-blob inertness proven independent of any delete reaching the machine (S4); AND a companion test asserts the FRAMEWORK plaintext config-home token is NOT made inert by rotation (gap 6 honesty — proves the residual the active wipe + TTL + provider rotation must handle).
- (Mechanism A) `MandateStore.verifyAuthorship` REJECTS a mandate issued on machine A (signed with A's authToken-HMAC) when verified on machine B (proves the §3.1a local-only gap); the NEW asymmetric path (R4a Ed25519 issuance signature) is ACCEPTED on B and a tampered/forged signature is rejected (gap 1).
- `MandateGate` deny-by-default: no `account-follow-me` mandate → `evaluate` denies; exact-bounds: a mandate for (accountA, machineX) does NOT satisfy (accountA, machineY) or (accountB, machineX) (R1, drives `paramsSatisfyBounds`).
- The router's account selection excludes a `locallyExecutable: false` meta-only account from fallback/`force` (R6.2 wiring-integrity).
- `EnrollmentWizard.start` with a `driveLogin` returning an off-allowlist `verificationUrl` is refused — no pending login issued, no code surfaced (S6); `flowNotice` guidance present for the two-code path.

### 8.2 Tier 2 — Integration (full HTTP pipeline)
- A registered-but-untrusted destination machine returns `refused: trust-floor` from `provisionAccountTo` (applies to BOTH mechanisms — S7: Mechanism B's target must clear the SAME floor); the `account-credential-share` RBAC case rejects a payload from a non-mandated sender or to a non-matching recipient AT THE GATE before decrypt (R3a, S1, S2).
- The legacy `SecretShareHandler.handle` rejects (or never receives) an `account-credential-share` payload — the legacy decryptor is unreachable for credential-class data (R3a, gap 2); `decryptAccountCredential` is the only path that yields a credential.
- There is NO mesh verb whose handler reaches `EnrollmentWizard` or `provisionAccountTo` EXCEPT the explicitly operator-mandate-gated `enroll-drive`/`account-credential-share` (R6a/R3a) — a test enumerates the mesh-verb→handler table and FAILS the build if any *non-operator-gated* verb routes to enrollment/credential-push, and asserts those two verbs carry their own RBAC case, not the "any registered peer" class (S2).
- A forged journal-injected follow-me mandate whose ASYMMETRIC issuance signature does NOT verify against the registered operator-machine's Ed25519 key is refused at the target (R4a — the credential-domain dual of WS1.3 forged-pin theft); a mandate carrying only the legacy authToken-HMAC proof is refused cross-machine (gap 1).
- Revocation-removes-credential: revoke → target logs out + `SubscriptionPool.remove` fires (still-paired cooperative target); Mechanism A also wipes the transported blob AND the framework plaintext token, both in the `SafeFsExecutor` audit log; an *offline* machine that never received the verb still cannot decrypt its SEALED residue after key-rotation, but the test ALSO asserts the operator surface tells the truth about the plaintext residue (R12, gap 6).
- De-paired non-cooperative B holder: the operator surface documents provider-side de-authorization as the revocation path, NOT a claim instar removed it (S8/R12.i, gap 10).
- Offline-revoke-honesty + give-up: revoke against an offline target → durable `revocation-pending` action + honest message; after the reconnect deadline it ESCALATES to a LOUD `revocation-FAILED — rotate at provider` attention item, never a perpetual "pending" (R12.iii, gap 9).
- Seal-time status revalidation: `provisionAccountTo` to a destination whose CURRENT authoritative status is revoked/unknown/stale-beyond-bound is REFUSED (fail-closed), even if a cached peer view still shows it active (R13, gap 8).
- The pending-login surface is unreachable without PIN/authenticated-channel auth (anonymous fetch 401/403); a target-supplied off-allowlist `verificationUrl` is refused even when minted on the target; post-complete, an enrolled `email` mismatching the operator's stated expectation raises an attention item before the account is selectable (S6, S7).
- A remote enroll-drive that times out / throws surfaces an honest "couldn't start the login on `<nickname>` — retry?" message, NOT a silent stuck pending-login (R6b, gap 13).
- A leased Mechanism-A credential past its TTL is refused by the router (R11).
- A routing decision for a component outside the grant's `allowedComponents` is refused and falls back to the local account; a VM that exhausts its lease slice fails closed to its own account, never overspending the borrowed one (R7a, S5).

### 8.3 Tier 3 — E2E lifecycle (feature is alive)
- Production initialization mirroring `server.ts`: with `multiMachine.accountFollowMe` dark, all follow-me routes return 503 and no follow-me code path is entered (R14). With the flag on in a ≥2-machine pool, the depth-zero detect → phone-first consent deep-link → operator authorizes → target runs `EnrollmentWizard` headless → account becomes `locallyExecutable` on the target — end-to-end, returning 200, not 503.
- Single-machine / pool-size-1 no-op: with no peers, the whole follow-me path is inert (parent invariant 6).
- Budget cap: issuing past the per-account max-follow-machines cap is refused at issuance with an honest reason (R7).

### 8.4 Phase-C named attack tests
- **5-VM destination isolation (S1):** operator grants account→VM2 only; assert VM3/VM4/VM5 never receive the blob and a self-added VM6 is refused at the `account-credential-share` RBAC gate (R3a).
- **Cross-VM grant replay (S2/S3):** a grant naming VM2's fingerprint replayed to VM3 is rejected (AAD recipient-fingerprint mismatch, before decrypt).
- **Cross-machine mandate verification (S2, gap 1):** a mandate issued+HMAC-signed on the operator-machine is REJECTED on a peer's HMAC verifier (proves the gap) and ACCEPTED only via the R4a Ed25519 issuance-signature path; a peer cannot forge an issuance signature without the operator-machine's identity key.
- **Key-rotation durability across reboot (S3, R4b):** the rotated recipient X25519 key / generation survives a VM reboot (durable, not in-memory) so a reboot cannot resurrect the old key and replay an old sealed blob.
- **Plaintext-residue honesty (S4, gap 6):** after de-pair + key-rotation on a still-online VM, the SEALED blob is inert but the framework plaintext token is still present until the wipe verb runs / TTL expires — the test pins the honest containment story (active wipe + TTL + provider rotation), not a false inertness claim.
- **Lease-slice under partition (S5, R7a):** 5 VMs each granted the same account; partition the authoritative machine; assert aggregate spend is bounded by sum-of-leases, NOT N×ceiling, and a VM exhausting its slice fails closed to its own account; adding a 6th VM does not raise the ceiling.
- **Headless no-browser + two-code (S6/R6/R6b):** on a headless VM, no interactive-browser code path is invoked, the device-code artifact is the only thing surfaced, and the two-code Claude flow completes within the full-interaction poll window (code #2 after the operator submits code #1).
- **Cross-machine enroll-drive (R6a):** "enroll on VM B" reaches B via the chosen operator-mandate-gated transport (named symbol), B runs its OWN `EnrollmentWizard`, and a non-operator-gated peer cannot trigger an enroll-drive.
- **Departed B holder (S8, gap 10):** de-pair a VM holding a B-minted login; assert instar reports provider-side de-authorization as the revocation path and does NOT claim it removed the credential.
- **Offline de-pair honesty + give-up (R12.iii, gap 9):** de-pair an offline cloud VM; the surface reports "wipe pending — SEALED copy inert after rotation; plaintext residue handled by TTL; rotate at provider if compromise suspected," never a false "removed," and escalates to "revocation-FAILED — rotate now" after the reconnect deadline.

### 8.5 Wiring-integrity (per the standard)
- The `locallyExecutable` predicate is new surface on the account-selection path — a meta-only account is provably unselectable by the router (not a no-op, not null).
- `enroll-drive`/`account-credential-share` mesh verbs delegate to the real `acceptCommand` gate AND their own operator-mandate-gated RBAC case (not a stub, not the "any registered peer" class); the verified-operator binding is wired to the `topic-operator` principal, not a bare peer assertion.
- `encryptAccountCredential`/`decryptAccountCredential` are real, distinct from `encryptForSync`/`decryptFromSync`, and the credential path provably never calls the legacy pair (wiring-integrity for the new envelope — gap 2).
- R4a's asymmetric verification is wired to the machine's real Ed25519 identity key and the registered operator-machine fingerprint, not a no-op that returns true.
- R13's seal-time status check reads authoritative registry state, not a cached peer view — proven by a test where the cached view diverges from authoritative state.

## 9. Rollout

- **Dark by default.** `multiMachine.accountFollowMe` ships reserved-dark and lands via `migrateConfig()` with an existence check, default `false` — single-account and pre-spec agents are strict no-ops (parent invariant 6). `multiMachine.accountFollowMe.credentialTransport` (per-provider allowlist) defaults to EMPTY; `multiMachine.accountFollowMe.maxFollowMachines` defaults small (e.g. 5).
- **Dry-run first** (Graduated Feature Rollout track). When first enabled, the agent logs intended follow-me offers/enrollments/decisions WITHOUT performing them; the operator reviews the dry-run audit (R8) before flipping to live, mirroring SecretSync's and the sentinels' dry-run discipline.
- **Receive-only-by-default for Mechanism A** (mirrors SecretSync's `pushEnabled`). `multiMachine.accountFollowMe.credentialTransport` enabling a provider only lets a machine RECEIVE a sealed credential; outbound push needs the explicit push grant on the authoritative machine, and a receive-only/stale machine refuses to push (preventing a divergent store from clobbering good logins on peers). The status read surface reports `mode` (`full` | `receive-only`) like `GET /secrets/sync-status`.
- **Build-order obligation.** The FIRST PR of any build round builds the SHARED security primitives + their unit tests, with NO wiring to a live credential — proving the secure mechanism before any credential-bearing code path exists, exactly as CMT-1413 requires. That first PR MUST include: (1) the NEW `encryptAccountCredential`/`decryptAccountCredential` AAD-bound pair (R3/I2 — `encryptForSync` cannot carry AAD; gap 2); (2) the distinct `account-credential-share` verb + its operator-mandate-gated RBAC case (R3a; gap 12); (3) the R4a asymmetric (Ed25519 identity-key) cross-machine mandate issuance-signature + verification (the existing HMAC-over-authToken proof is local-only; gap 1); (4) the R4b de-pair key-rotation (the chosen pairing-epoch mechanism — gaps 3, 7); (5) the grant/lease-slice primitives (R7a; gap 5). R0's `SubscriptionPool.ts` header correction lands in the same PR as the first follow-me wiring. The cross-machine enroll-drive transport (R6a) MUST be designed-and-named before Mechanism B is claimed Phase-C-ready (gap 11).
- **Migration & awareness (P3 + P5).** `migrateConfig()` adds the flags with existence checks (default-off / default-empty preserving today's behavior). `generateClaudeMd()` + `migrateClaudeMd()` add the Mandates-tab follow-me form + the depth-zero deep-link consent flow with proactive triggers ("user says 'put my Claude account on the new VM' → drive the phone-first consent flow, never a terminal").
- **Guard posture.** Every flag flip + every follow-me decision emits feature metrics + an audit line (R8); a pool-wide downgrade or an unexpected enrollment instruction is detectable via the existing guard-posture surfaces, never silent.

## 10. Open Questions

1. **Refresh-token replication for an allowlisted provider (Mechanism A, R11).** Is refresh-on-arrival (recipient mints its own short-lived token from a sealed refresh artifact, then drops it) feasible for every provider that might be allowlisted, or do some force a long-lived bearer? If forced, that provider should likely be re-classified out of the allowlist. Needs per-provider investigation before any allowlist entry beyond a documented example.
2. **`credential-sync` trust floor definition (S1).** What exactly qualifies a machine as a credential destination above the floor — operator grant alone (R1), or also a minimum pairing tenure / successful-operation history? The grant is necessary; is it sufficient, or do we want a tenure backstop against an attacker who phishes a single grant?
3. **Key-rotation vs counter anchoring (S3/R4b — now a NARROWED question, no longer "where does the epoch live").** R4b promotes the pairing-epoch from an Open Question to a designed primitive and chooses the **lean: rotate the recipient X25519 `encryptionPublicKey` on de-pair** (reuses identity machinery; makes every old sealed blob undecryptable). The remaining narrow question: confirm the rotated key (or, if a counter is preferred, the generation counter) is durably anchored against rollback by a hostile local user — encrypted `SecretStore` vs OS-keychain anchoring — and that the receiver learns the current key/generation across reboot. This is an anchoring-detail decision for the first PR, not an open "does the mechanism exist" question (it now does). The honest residual stands regardless: rotation never touches the framework's plaintext config-home token (gap 6) — that is the active wipe + TTL + provider-rotation's job.
4. **Anthropic allowlist trigger.** Under what *documented* ToS change would Anthropic ever move onto the `credentialTransport` allowlist? The default is "never until the ToS explicitly permits token relocation"; this question records that the bar is a documented provider permission, not an operator's convenience.
5. **`max-follow-machines` default + per-account override.** Is a single global default (e.g. 5) right, or should it be per-account (a personal account vs a team account may want different ceilings)? Lean: global default with a per-account override in the mandate's `bounds`.
6. **Aggregated consent UX for K simultaneous depth-zero VMs (R7).** One attention item listing K VMs with per-VM authorize controls — confirm the Mandates-tab form can express "authorize these 3 of 5" in one phone interaction without spawning K mandates' worth of friction.
7. **Cross-machine enroll-drive transport choice (R6a — gap 11).** Pick in writing before Mechanism B is Phase-C-claimed: a new operator-mandate-gated `enroll-drive` mesh verb (A drives B over the mesh), OR operator-side per-server selection (the operator drives the TARGET VM's own local enroll route, each VM running its own instar server). Lean: the per-server selection is simpler and avoids a new credential-adjacent mesh verb, but it requires every VM's dashboard to be reachable by the operator (tunnel per VM). Decide based on the real fleet topology.
8. **Offline-wipe reconnect deadline (R12.iii — gap 9).** What is the default reconnect-deadline before `revocation-pending` escalates to the loud `revocation-FAILED — rotate at provider` item? Lean: short (hours, not days) for a live credential, operator-tunable. Confirm it aligns with the resume-queue's resurrection-cap discipline.
9. **R6b remote scrape-timeout budget (gap 13).** What scrape-timeout budget is right for a cloud VM → provider device-code endpoint (vs the local 60s)? And is the two-code Claude flow better handled by extending the single poll window or splitting into two artifacts? Lean: prefer the device-code single-code flow on cloud VMs wherever the provider supports it, sidestepping the two-code problem entirely.
