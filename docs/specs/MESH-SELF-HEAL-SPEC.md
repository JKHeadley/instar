# Mesh Self-Heal — Lease↔Job Binding, Nobody-Polling Detection, and the Machine-Independence Standard

> **North star (operator directive, 2026-06-27):** A user must NEVER have to hand-fix a zombie-lease / duplicate-session / nobody-polling / missing-data scenario. The multi-machine experience must be smooth and coherent and NOT machine-dependent. The agent self-heals from ANY such scenario.

> **Round-2 note:** this spec was rewritten after a 6-reviewer convergence round (~30 material findings; see `MESH-SELF-HEAL-SPEC.findings-round1.md`). The two blocking findings (the overloaded lease; gating spawn on a placement view) are resolved in §1.3 and §3.3. Cross-model external review was unavailable (codex not installed) — internal panel only, recorded honestly.

## 0. Why this spec exists (the incident)

On 2026-06-27 the operator's Telegram messages silently stopped reaching the agent while the agent's replies still went out. Diagnosis revealed a class of failures the existing self-heal features (F1–F4, soloCaptainHold; see `multi-machine-lease-self-heal.md`, `multi-transport-mesh-comms.md`) do NOT cover, and a manual-recovery attempt that made it worse (a duplicate session). Empirically-confirmed facts:

- The **laptop held the awake-lease and renewed it perfectly** (nonce/expiry advancing every cycle) **while being a muted silent-standby that did not poll Telegram.** Both machines ended up not polling; nobody fetched inbound messages; they dropped with no alarm.
- Because the holder kept *renewing*, every renewal-aware feature correctly judged it "healthy" and refused takeover — **renewal-liveness is not job-liveness.**
- A manual takeover attempt (enabling F2/soloCaptainHold on the standby against a live-but-flaky peer) caused a tug-of-war; the non-holder laptop spawned its OWN session for the operator's topic → **two machines served one topic → double-replies.**
- Killing the duplicate session did **not** clear its topic→session binding; only an explicit topic pin stopped re-spawn.
- The operator's own phone number was not available on the serving machine — a small instance of the broader **machine-dependent-data** problem.

## 1. The two load-bearing principles + the foundation they sit on

### P1 — Lease ownership MUST be bound to job-performance
Holding the awake-lease must REQUIRE actually doing the job it represents: owning the Telegram poll and serving the topics the machine owns. A machine that holds the lease but is not polling/serving is a **zombie holder** and must lose the lease automatically. Renewal alone must never be sufficient evidence of fitness to hold.

### P2 — No data the agent depends on may be machine-dependent (the Machine-Independence Standard)
ANY data the agent relies on is EITHER **shared-by-default** across the agent's machines OR **transparently lazy-loaded on demand** from the machine that holds it, through a single go-to channel. The user never encounters a seam. New agent-depended-on data may not silently ship machine-local; an enforcement mechanism makes that structural.

### 1.3 Foundation notes (resolving the two BLOCKING review findings)

**Foundation note A — the lease is overloaded; G1 is a deliberate interim binding (finding C1).** Today the awake-lease conflates two distinct responsibilities into one record: (i) *"I am the single coordinator/router"* and (ii) *"I am actually polling Telegram and serving topics."* The incident is precisely these two diverging. The STRUCTURAL fix is the `MULTI-MACHINE-SESSION-POOL-SPEC` **L1/L3 split** — an L1 router-lease (who coordinates) distinct from L3 per-conversation ownership (who serves what). This spec does NOT re-litigate that split; it ships the **incident-scoped interim binding** (P1/G1) that makes the awake-lease *behave* as if poll-ownership were bound to it, **defined to CONVERGE with the L1/L3 model, never to become a third overloaded signal on the same record.** Concretely: G1's job-liveness predicate gates *poll-ownership* specifically (the L3-serving responsibility), so when the pool's L1/L3 split lands, G1's predicate migrates onto L3 ownership unchanged rather than being thrown away. The retirement of the `telegramPolling`-as-lease-role overload (parent spec M3) is the precedent this follows.

**Foundation note B — gate spawn on the FENCED LEASE, not a placement view (finding C2).** Per-topic ownership-as-a-CAS-record is the session-pool **L3 SessionOwnership registry**, which ships DARK. In the *current single-lease foundation* (the one the incident ran on) there is no trustworthy per-topic ownership view — and a placement/route view is explicitly **not reliably readable from a non-router machine** (memory: *multi-machine placement inference trap* — "pin/route state lives on the lease-holder"). Therefore G3's spawn gate keys on the one authority that IS fenced and trustworthy: **the awake-lease itself.** The rule is "**spawn iff I hold the awake-lease (fenced), else forward to the holder.**" This is shippable on the single-lease foundation today, genuinely fixes the incident (a non-holder can no longer spawn), and forward-converges: when L3 lands, "hold the lease" generalizes to "hold this topic's L3 ownership."

These two principles + the foundation are complementary: P1 makes *serving* follow the user; P2 makes *the data serving needs* follow the user.

## 2. Scope

In scope: **G1** (P1 lease↔job binding; promote F3 to job-liveness-triggered, default-on only post-live-verify), **G2** (nobody-polling detector), **G3** (lease-gated spawn + topic-binding single-writer lifecycle), **G4** (preserve the conservative seize gates — F2/soloCaptainHold unchanged), and the **P2 standard** (default-share set + lazy-load channel + enforcement).

Out of scope (tracked separately): the L1/L3 pool split itself (this spec converges WITH it); non-Telegram channel mechanics beyond §3.6's generalization clause.

## 3. Design

### 3.1 G1 — Lease↔job binding via a THREE-SIGNAL liveness model

The incident proved one watermark is not enough. G1 defines three distinct, locally-clocked signals (all machine-local, monotonic, evaluated by the machine about ITSELF — never a peer-observed value driving a relinquish; finding Sec-F1):

- **`pollAttemptedMonoMs`** — stamped whenever the poll loop *attempts* a `getUpdates` cycle (success OR error). Proves the machine is alive and trying.
- **`pollSucceededMonoMs`** — stamped on a *successful* `getUpdates` response. Proves the external channel is reachable from here.
- **`serveProgressedMonoMs`** — stamped when a fetched update is dispatched/served/durably-enqueued. Proves end-to-end progress (closes the "fetched-and-dropped" gap, finding Adv-F4).

**Truth source (finding DC-D19 — THE original-bug root):** these signals are sourced from the **lifeline's actual-poll truth** (`state/lifeline-poll-active.json` — the in-flight `getUpdates` record), NOT the server's *intent* to poll. Server-intent ≠ lifeline-actual is exactly what let the incident hide; G1 reads the actual.

**Fitness + the relinquish rule (findings Adv-F1, Sca-F6, DC-D13; round-2 Adv2-F2, Adv2-F5):** a holder is a **zombie** (must relinquish) ONLY when ALL hold: (a) it is in the `active` leaseRole (an `observe-only`/`deferential` machine is handled by F3's existing path, not G1 — finding Int-LOW); (b) `serveProgressedMonoMs` (when updates are pending) OR `pollSucceededMonoMs` (when idle) is stale past `jobStaleThresholdMs` on its OWN clock, confirmed across N evaluator ticks that are themselves fresh (finding Adv-F8 — defeats an evaluator-resume false trip). **"Pending" is determined as `lastFetchedUpdateId > lastServedUpdateId`** — two poll-side counters (the highest update id `getUpdates` has FETCHED vs the highest the dispatch seam has confirmed SERVED/durably-enqueued) — **NOT from the serve-side queue depth, and NOT from the Telegram-acked offset** (round-2 Adv2-F5 + round-3 Adv3-F-D: the standard poller advances the Telegram offset on FETCH to avoid re-fetching, so "offset not advanced" would falsely read non-pending even when serve silently dropped the updates; the fetched-vs-served counter pair is the serve-blind-spot-proof signal because the served counter only advances on confirmed serve); AND (c) **the staleness is LOCAL, not global.** The GLOBAL branch — HOLD, do not relinquish — requires **POSITIVE evidence** that the outage is shared: a fresh, signed heartbeat RECEIVED from ≥1 live peer in-window that ALSO reports `pollSucceeded`-stale. **"I can't hear any peer" must NEVER satisfy "Telegram is down for everyone"** (round-2 Adv2-F2 — a heartbeat-transport partition would otherwise be misread as global and HOLD would protect a real zombie, re-creating the incident). Absent positive peer evidence, a `pollAttempted`-fresh + `pollSucceeded`-stale holder is treated as a LOCAL failure → proceed toward relinquish (the safe direction), letting the fenced CAS + G2 backstop pick a server. Only when `pollAttemptedMonoMs` is fresh AND a live peer confirms it is equally blind does the holder HOLD + route to G2's escalation. If `pollAttemptedMonoMs` is ALSO stale, the loop itself is wedged → relinquish unconditionally.

**Auto-relinquish mechanism (promote F3).** A confirmed zombie runs F3's `relinquishAndBroadcast()` (signed tombstone). **Tombstone-nonce race fix (finding Sec-F6):** because G1 can fire while the holder's renewal path is healthy and actively bumping nonces (unlike F3's muted-standby case), the relinquish first **quiesces the local renew path** and draws the tombstone nonce `>` the highest nonce the holder will ever emit for that epoch — closing the renewal-vs-tombstone resurrection race. A higher-epoch normal/F2 takeover still strictly dominates.

**Prompt-handoff to avoid a drop window (finding Adv-F7; round-2 Sec2-F1).** With F2 dark, a peer otherwise reacquires only on its next tick (drop window). The tombstone carries a targeted "you take it" hint to the designated fit peer (the G2 single-claimant, §3.2). **The hint is a field INSIDE the signed tombstone envelope and is advisory-only** — the recipient still independently runs the §3.2 fitness check + wins the fenced epoch-CAS before serving; a forged/replayed hint can never substitute for the single-claimant determination or nudge an unfit machine to serve.

**Truth-source — honest cross-process read (round-2 Int2-D; round-3 Les3-F1).** `pollAttempted/SucceededMonoMs` come from the lifeline's `lifeline-poll-active.json` (FD10), written by the lifeline PROCESS. `serveProgressedMonoMs` is a DIFFERENT process's truth — the server/session process's WS1.1 dispatch seam / durable-inbound-queue enqueue point. These cannot share one file: the lifeline atomically rewrites `lifeline-poll-active.json` (tmp+rename) on every poll-state change and the record has no serve field, so a second-process write into it is a clobber, NOT reuse. Therefore: `serveProgressedMonoMs` is written to a **separate server-process record** (`state/serve-progress.json`, single-writer = the dispatch seam) and the relinquish evaluator performs an explicit **cross-process read of BOTH records** — this is net-new, deliberately-acknowledged wiring (a schema extension + the second record + an atomicity test), NOT inherited reuse. The two records carry monotonic stamps from their own processes; the evaluator never subtracts one process's monotonic value from another's — it reads each record's own freshness within its own clock domain. **Boot-epoch staleness guard (round-4 Adv4-B):** because `serve-progress.json` is a persisted FILE (it must be, for the cross-process read — one process cannot read another's memory), a stamp written by a PRIOR process incarnation must never read as fresh after a crash/restart. The record carries a **boot-identity marker** (a boot id / wall-clock boot stamp) alongside the monotonic value; the reader discards any stamp whose boot-identity ≠ the writer process's current incarnation (a boot-epoch fence — NOT a raw monotonic-vs-monotonic comparison, which would itself be an invalid cross-domain subtraction), treating it as "no progress yet," so a crash-stale on-disk stamp cannot mask a non-serving new process. The `serve-progress.json` field is a single machine-global monotonic-MAX liveness watermark (not per-topic state), so concurrent dispatch-seam writers are benign — the stamp only ever advances (round-4 Adv4-A). On an owner-death reclaim of a `claimed`-but-unconfirmed message (§3.3), re-delivery is chosen over loss consistent with §8's fail-direction; a wedge that served-but-lost-the-confirmation can therefore re-inject once (a possible duplicate, the accepted safe-direction tradeoff vs silent loss — round-4 Adv4-D).

**Posture (round-4 Adv4-B reconciliation):** all three watermarks are **machine-local, never replicated** (finding Sca-F1), read only by the owning machine for its OWN relinquish decision (skew-immune — no foreign-monotonic subtraction). Backing differs by source: `pollAttempted/SucceededMonoMs` live in the lifeline's `lifeline-poll-active.json` record; `serveProgressedMonoMs` lives in the separate `state/serve-progress.json` record (it is file-backed, NOT in-memory-only, because the relinquish evaluator reads it cross-process) under the boot-epoch staleness guard above. What peers SEE is the §3.2 advertised poll-freshness, not these raw stamps.

### 3.2 G2 — Nobody-polling detector (single fenced authority)

**Advertised signal (findings Sca-F2, Sec-F2, DC-D18):** each machine advertises, in the **machine-auth-SIGNED heartbeat body** (reusing the multi-transport replay-proof signed envelope — forged-origin rejected), a small `pollOwned: bool` plus a **locally-computed** `pollFresh: bool` (is my own `pollSucceededMonoMs` within threshold, by MY clock). No raw monotonic stamp crosses the wire and no machine compares foreign monotonic values — the detector reduces over booleans only.

**Detector reuse — do NOT reinvent (round-2 Int2-A).** G2's nobody-polling condition reduces over the EXISTING `src/core/pollerCount.ts` (B5) detector — already wired in `routes.ts`, computing `ok`/`dual`/`silence`/`indeterminate` from the same `pollingActive` truth — NOT a new boolean fold. Mapping: B5 `silence` (persisting across `nobodyPollingConfirmObservations`, finding Adv-F9 so a normal handoff gap never trips it) = the nobody-polling condition; B5 `indeterminate` (a dark/unknown peer) ⇒ **fail-CLOSED, no claim**; B5 `dual` (two pollers already) ⇒ **VETO any G2 claim** (claiming into a dual state IS the 409 poll-war G2 exists to prevent).

**Resolution is SINGLE-CLAIMANT through the fenced lease (findings Adv-F2, Adv-F6, Les-A1; round-2 Adv2-F1, Int2-B).** The claim is NOT each machine independently deciding "am I fit?" — that is the split-brain double-poll bug. Exactly one machine claims, chosen deterministically: **the F4-preferred-awake machine if it is itself fit, else the lowest-machineId fit machine** (finding DC-OQ2). The claimant acquires poll-ownership by **winning the fenced epoch-CAS** (the same authority that gates the lease). **CAS-win is necessary but NOT sufficient (round-2 Adv2-F1):** because fitness was assessed from heartbeat-cadence-LAGGING advertised booleans, after winning the CAS the claimant MUST RE-VERIFY its OWN live `pollSucceededMonoMs` freshness (current, local) before committing to serve; on self-unfit it immediately relinquishes the epoch (signed tombstone + the G1 quiesce) AND immediately advertises `pollFresh:false` + a `selfExcludedThisEpisode` flag in its signed heartbeat (round-3 Adv3-F-A: re-electing "excluding itself" is locally-scoped and invisible to peers — without this, peers re-run their election over the lagged advertised set and re-nominate the just-relinquished machine until cadence propagates `pollFresh:false`, leaving a fit peer unselected for ~one heartbeat interval). The immediate self-demotion advertisement makes peers' next election skip it at once. Bounded by `confirmObservations` so it cannot ping-pong. **Actuation seam (round-2 Int2-B):** winning the CAS is not self-actuating — the claim drives the EXISTING `poll-follows-lease` / `effectivePollIntent` lever (`TelegramLifeline.ts`) to START the winner's `getUpdates` loop and STOP the loser's, and a **post-claim live-verify** confirms the winner's `lifeline-poll-active.json` actually ADVANCES (FD10's server-intent ≠ lifeline-actual rule, applied to actuation — winning the lease without the lifeline actually polling is the same bug on the output side). The loser stands down on observing the higher epoch (≤1 tick). This makes the lease epoch the **single authority that atomically gates poll-ownership AND spawn** (§3.3), so G1/G2/G3 cannot reach divergent conclusions.

**Escalation (one voice; findings Int-HIGH one-voice, Les-B3; round-2 Adv2-F4).** If NO machine is fit (global outage), exactly ONE operator escalation fires, on the **deterministic delivery path** (`telegram.sendToTopic` / Cold-Start Lifeline Fallback — never the LLM tone gate, which fails closed under this pressure). The single-speaker is **the lowest-machineId among machines proven HEARTBEAT-LIVE this window** — NOT the absolute lowest machineId, which is frequently the crashed/offline machine that caused the trouble (round-2 Adv2-F4: a dead nominee would swallow the only alarm → one-voice becomes no-voice). A bounded "no escalation observed for this episode within T" backstop lets the next-lowest live machine fire if the nominee fails. The message carries a **pool-deduped episode key keyed on (episode), not (machine)** (reusing the WS3.3 dedup pattern + the universal topic-creation budget) so N machines seeing the condition produce ONE message. **The dedup is INTRINSIC to the send — the episode-key collapse happens AT the `telegram.sendToTopic` chokepoint, and there is NO separate pre-send suppress marker** (round-3 Adv3-F-C: a write-marker-then-send ordering would, on a crash between, leave the marker with no message sent → the backstop suppresses → ZERO escalation, re-creating the one-voice→no-voice failure at finer grain). The safe direction is explicit: **send-FIRST, dedup-at-chokepoint** — a rare duplicate from the backstop collapses to one via the episode key, choosing duplicate-over-silence. Fixed template (no peer-controlled interpolation).

### 3.3 G3 — Lease-gated spawn + single-writer topic-binding lifecycle

**Spawn gate (finding C2):** before a machine spawns (or resumes) a session for a topic, it MUST hold the **fenced awake-lease**. If it does, it serves. If it does NOT, it **forwards the inbound to the holder** via the existing **WS1.1 dispatch-to-owner seam** (`deliverMessage`/`dispatchInbound`) — this spec does NOT reinvent forwarding; it inherits WS1.1's dependency on the durable-inbound-queue merge. The ownership read is a **LOCAL read** of the fenced lease (finding Sca-F4 — never a synchronous per-spawn cross-machine RPC). **Fail direction: CLOSED** (finding Adv-F2, DC-D15, resolving the round-1 §3.3↔§4.4 contradiction): under any ambiguity (can't confirm I hold the lease) the machine does NOT spawn — it forwards or defers, and G2 is the bounded backstop guaranteeing someone ends up serving within the recovery window. "Don't spawn on ambiguity" is the rule; G2, not a local spawn, breaks the tie.

**Single-writer binding lifecycle (findings Les-C3, Adv-F5, DC-D16).** A topic→session binding is durable state with a **single-writer chokepoint** (a `SpawnLedger` invariant: *a binding exists IFF a live session exists for it*), enforced by a wiring-integrity test, NOT a cleanup bolted onto one of several spawn paths. On close/transfer the binding is cleared **as part of the existing post-transfer closeout, AFTER in-flight inbound for the topic has drained** (drain-then-clear at a turn boundary — never clear-then-strand; the durable inbound queue carries any in-flight message to the new owner). **The drain is bounded (round-2 Adv2-F3):** a wedged session (context-wedge / AUP-rejection — documented permanent-death modes) would never reach a turn boundary, so the drain has a deadline AND is short-circuited the moment the wedge-sentinel flags the session dead; on expiry/short-circuit the binding is force-cleared and the inbound is handed to the new owner via the durable queue. **This includes reclaiming the dead session's `claimed`-but-undelivered messages back to the durable queue (owner-death claim-reclaim), not only the still-queued set** (round-3 Adv3-F-B: a message the wedged session already claimed from the queue but did not durably serve is neither still-queued nor served — without owner-death reclaim it falls into the pre-existing durable-queue `possiblyNotInjected` thin window). Where that razor-thin claimed-but-uninjected window genuinely cannot be closed, it is surfaced via the EXISTING `possiblyNotInjected` loss-notice (an honest "I may not have gotten this — resend"), never silently asserted as "no loss." A drain-never-completes case must never strand the binding on a dead-as-server session. The binding is **machine-local by design** (it names a local tmux session — it cannot be meaningfully replicated); its lifecycle binds to the local session.

### 3.4 G4 — Preserve conservative seize (unchanged)

F2 (`staleHolderTakeover`) and soloCaptainHold keep their conservative gates verbatim. The incident's harm was G3 (a non-holder spawning) + a human enabling seize against a live peer — NOT timid takeover. G1 adds only the *self-yield* path; the fit machine acquires after the unfit one has signed away its claim (self-yield + conservative-seize). G4 is explicitly NOT made more aggressive.

### 3.5 P2 — The Machine-Independence Standard

1. **Default-share set.** Builds on the existing replicated-store foundation (WS2.*) + secret-sync. The **operator-identifier class** (phone/contact) is added as a **field-class on the EXISTING WS2.6 user-registry store** (finding Int-MEDIUM, DC-OQ3, Les-C4) — NOT a new store/flag. It inherits that store's full hardening verbatim (type-clamp on receive, `<replicated-untrusted-data>` envelope, channel-set identity, tombstone-on-delete, WS2.3 at-rest-honesty disclosure). **Identity RESOLUTION of an inbound principal stays LOCAL-authoritative** (a replicated operator-identifier is advisory reference, never the authoritative "who is my operator" answer — Know Your Principal). **At-rest sensitivity (round-2 Sec2-F3):** a phone number is a higher-sensitivity, directly-actionable identifier (SIM-swap / direct-contact) than channel-set data; enabling its replication lands it plaintext on EVERY pool machine including any rented cloud VM. The operator-identifier field-class is therefore opt-in and, by default, EXCLUDED from replication onto non-physically-controlled (cloud) ropes; where it does replicate, the WS2.3 disclosure explicitly names that enabling it pushes the operator's phone to those machines.
2. **Lazy-load-on-demand channel.** ONE uniform "fetch X from whichever machine has it" primitive generalizing **Working-Set Handoff** (verified-slice pull) + **pool proxied-on-read** (`?scope=pool`). **Security (finding Sec-F5 — the biggest new surface):** targets are resolved ONLY through the machine-auth-verified registry (never a free-form peer-asserted URL); every fetch carries machine-auth; URL-shape/allowlist validation identical to remote-close §2.1; it inherits the **credential-enumeration exclusion** (never ships credential material to a private-IP rope). **Caching (finding Sca-F9; round-2 Sec2-F2):** cache the LOCATION resolution with a TTL (generalize the WS4.4(f) pool poll-cache), single-flight/coalesce concurrent identical fetches, negative-cache a known miss. **Cache integrity:** ONLY a machine-auth-verified registry resolution may populate the cache; the negative-cache TTL is short (a poisoned "known miss" is a denial vector — P2 would falsely report data unreachable); and if a cache-resolved target fails auth at fetch time, the entry is invalidated and re-resolved ONCE rather than failing the whole request. **Fail direction: CLOSED** — an honest "fetching from the other machine, one moment" (or guided re-auth), never a silent degraded answer.
3. **Browser-session sub-case (the motivating instance; findings Int-MEDIUM, DC-OQ4).** The **login-location registry IS the existing Playwright Profile Registry** (finding Les-C4 — do NOT reinvent; reconcile P2 with it). Its name-only map (machine ↔ which account, by vault-key NAMES) must REPLICATE (or proxy-on-read) so P2 doesn't fail its own motivating example; it is a **typed/closed schema** whose receive-side clamp REJECTS any `cookies`/`token`/`storageState` field (finding Sec-F4 — promote the no-credential property from prose to structure + a test). Cookie material is **machine-local by design** (a session lives on one disk). The action ladder (DC-OQ4): holder online → transparently route the action to the holder (lazy-load/proxy); holder offline → **one guided re-auth** on the serving machine (deduped per (resource, episode), deterministic path — never per-action; finding Les-B3); never copy cookies, never a silent dead-end.
4. **Enforcement (findings Int-HIGH, Les-C4, Adv-F10).** NOT a parallel ratchet. **Register the Machine-Independence Standard in `docs/STANDARDS-REGISTRY.md`** with a named structural guard so the EXISTING `/conformance/coverage` audit grades it like every other standard. The guard is a **CI lint** (precedent: `lint-dev-agent-dark-gate.js`) that flags a new agent-depended-on data path (a new store class / a new `state/*.json` read on the serving hot path) not registered in the default-share set nor reachable via the lazy-load channel. **Honest scope (finding Adv-F10):** the lint covers DECLARED stores/paths; it cannot catch every ad-hoc local read, so it is paired with a runtime probe (when serving, a required resource resolving to an unregistered machine-local path is logged/flagged). **The probe logs only the registration-STATUS + a path CLASS, never raw resource contents/values** (round-2 Sec2-F4 — mirroring the established "matched-pattern names only, never raw pane text" posture), and its logging is sampled/deduped so a repeatedly-hit misregistration cannot flood. Stated as a known blind spot, not hidden.

### 3.6 Generalization clause (finding Int-LOW)
`pollAttempted/Succeeded` and `serveProgressed` are defined as advancing on ANY owned-channel poll/serve signal; the Telegram `getUpdates` is the canonical instance only because it is the incident's channel. A Slack/iMessage-primary agent stamps them from its own channel, so P1 never silently assumes Telegram is the only awake-job.

## 5. Verification (test-as-self — mandatory, live, on the real pair, with DETERMINISTIC injected faults)

No mechanism is enabled by default until **live-verified on the real Mini+Laptop pair** — synthetic symmetric tests give false confidence (live-verify-multimachine lesson). Faults are **forced via injectable seams** (finding Les-B2), never waited for:
- **Zombie-holder:** an injectable seam FREEZES `serveProgressedMonoMs` while renewal continues → the fit peer takes over, messages flow, **ZERO drops** (asserted, not "messages flow").
- **Global-outage:** force `pollSucceeded=false` on BOTH machines → neither relinquishes (HOLD), exactly ONE escalation fires on the deterministic path.
- **Split-brain double-poll:** partition + both observe nobody-polling → exactly ONE wins the fenced CAS and serves; the other stands down.
- **Duplicate-spawn:** inbound during a flap to a non-holder → the non-holder forwards, exactly one machine serves (verify in the MIXED-version config too — finding Int-MEDIUM).
- **Binding-strand:** transfer with an in-flight inbound → message is drained to the new owner, not stranded.
- **Machine-independence:** an action needing data/login on the other machine completes with no user-visible seam (online → routed; offline → one guided re-auth).
- Evidence: **timestamped `/health` from BOTH machines** + the audit JSONL. North-star metric: **operator-found escapes → 0.**
- **Prerequisite (finding Int-LOW):** the Mini operator Playwright login exists (done 2026-06-27); the **laptop operator login is a tracked prerequisite** for the both-machine scenarios — gates the rollout, called out so it isn't discovered at graduation.

## 6. Migration Parity, Agent Awareness, Observability (mandatory standards)

**Migration Parity (finding Les-B4, Int-HIGH).** New flags default dark, added as COMPLETE blocks to `ConfigDefaults.ts` via `applyDefaults`/`deepMerge` add-missing (NOT hand-written `migrateConfig`; clobber-safe for operator-set values), each range-validated at startup (FD2). Each flag → false restores prior behavior byte-for-byte. **Per-flag live-read vs restart-gated (round-3 Les3-F2 — do NOT blanket-claim "read live matching F1b"; F1b is live ONLY because its watchdog explicitly re-reads `enabled` each fire, and a flag falsely claimed live-disable-able but actually restart-gated is a false rollback guarantee under incident pressure):** `ownershipCheckedSpawn` is read LIVE (a per-spawn local lease read, no boot-bound loop). `jobLivenessBinding` and `nobodyPollingDetector` run INSIDE the lease/heartbeat tick (armed once at boot) — they are read live ONLY if their evaluator explicitly re-reads `enabled` each tick (matching F1b's pattern); the spec REQUIRES that re-read so the disable is genuinely live, and the rollback lever is verified by an integration test that toggles the flag on a running tick. Any flag whose live-read is not implemented is documented as restart-required, never assumed live. **F3 default-on is a POST-live-verify config flip on the Echo pair, NOT a same-PR fleet default** (finding Les-B1); the signed-`released` canonicalize must be fleet-deployed (every peer upgraded) BEFORE the flip — default-on graduates only on a fully-upgraded pool, never mid mixed-version window (FD8 covers the conservative old-peer behavior). A migration-parity test + a version-skew test are required. **DARK_GATE_EXCLUSIONS:** `jobLivenessBinding`, `nobodyPollingDetector`, `ownershipCheckedSpawn` each get a `DEV_GATED_FEATURES`/`DARK_GATE_EXCLUSIONS` entry (lint `scripts/lint-dev-agent-dark-gate.js`); the existing `soloCaptainHold` exclusion is preserved/extended, not silently promoted.

**Agent Awareness (finding Les-B4, Int-MEDIUM).** `generateClaudeMd()` + content-sniffed `migrateClaudeMd()` entries: "why did my machine relinquish / why is nobody-polling recovering," the new `/health` status fields, and the kill-switches.

**Observability (finding Int-MEDIUM).** `/health → multiMachine.syncStatus` gains `pollOwned`, `pollFresh`, per-machine job-liveness ages, `nobodyPollingEpisodes`; a `logs/mesh-selfheal.jsonl` audit line per transition (relinquish-fired, nobody-polling detected/recovered, spawn-forwarded-because-not-holder, binding-cleared); a **self-disarm + ONE Attention** for auto-relinquish OR G2-re-claim churn mirroring F1b's `maxReArmsPerHour` (round-4 Adv4-C — covers a chronically-unfit-but-CAS-winning machine re-nominated across episodes, not only within-episode relinquish; an over-relinquishing/over-claiming detector is itself an incident). Ownership-forwarded-spawn counts are operator-visible.

**Dependency on F1 + out-of-process backstop (finding Les-C5).** G1's relinquish and G2's detection run INSIDE the lease/heartbeat tick; if that tick wedges, they wedge too. They therefore DEPEND on F1's bounded-await + monotonic watchdog to keep the tick running, and the out-of-process fleet/launchd watchdog (§Version-Skew) is the final backstop — stated explicitly so the detectors don't share the failure mode they catch.

Each feature ships dark, graduates only after its §5 scenario passes live, with flag-coherence gating (a machine engages the new behavior only when every pool peer advertises the same flag — `MachineCapacity.seamlessnessFlags`; mixed pools degrade to today's behavior — finding Int-MEDIUM):
