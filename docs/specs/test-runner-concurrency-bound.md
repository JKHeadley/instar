# Test-Runner Concurrency Bound — a host-wide vitest semaphore (Bounded Blast Radius)

## Constitutional fit

This spec implements the **Bounded Blast Radius** constitutional standard (the same standard that mandates the fork-bomb LLM-spawn cap). An unbounded per-actor resource-spawn pattern whose host-wide aggregate is catastrophic is precisely what that standard exists to prevent; the test-runner storm is the second instance of that class (the first, LLM subprocesses, is already bounded). It also serves **Responsible Resource Usage** (OS resource hygiene) and directly closes the load-stall kill cascade documented in the 2026-07-02 respawn root-cause analysis.

**But it is NOT a verbatim copy of the spawn cap.** Round-1 convergence review established that the spawn cap's core reclaim policy and fail-direction were tuned for a *fundamentally different consumer* (one long-lived server process, second-scale holds, an OOM-catastrophe fail-direction). Reusing them unmodified would re-create the exact meltdown this feature prevents and inherit the 2026-07-01 stale-holder wedge. This spec therefore **extracts a parameterized core** and derives a *consumer-specific* policy for the test lane. §1.1 is the load-bearing section: the fail-direction inverts.

## 1. Problem

A vitest ROOT process is not one process: it forks a worker pool sized to the machine (~cores threads/forks). N concurrent roots ≈ N × cores workers competing for CPU, memory, and the filesystem. On an Instar dev host, many actors start suites independently and simultaneously:

- several agent build sessions, each running `pnpm test` / `pnpm test:integration` in its own worktree;
- every `git push` (husky pre-push runs a suite);
- `/build` phase gates and scratch verification runs;
- occasional operator-driven runs.

Nothing bounds the SUM. The existing fork-bomb spawn cap (`hostSpawnSemaphore`) bounds LLM subprocesses only — test runners ride completely outside it. 29 concurrent roots on a 16-core host ≈ 300+ runner threads: the event loops of the co-resident agent servers starve, their supervisors interpret the stall as ill-health, and healthy servers get killed. The safety mechanism becomes the outage.

This is the same *class* as the 2026-06-20 LLM fork-bomb, one layer over: an unbounded per-actor spawn pattern whose aggregate is catastrophic, needing a HOST-WIDE counting bound at a single structural chokepoint.

### 1.1 The fail-direction INVERTS from the spawn cap (the load-bearing design decision)

The spawn cap fails **closed** on uncertainty (can't confirm host-local disk → refuse to reclaim → keep counting; can't acquire → don't spawn). That is correct *for an OOM safety floor*: a delayed LLM spawn is cheap, an OOM is catastrophic, and its background consumers have a heuristic-degradation fallback when a call is shed. **None of that holds for a test cap.** The harms invert:

- A false **BLOCK** (over-counting, a stale/phantom holder, a wedged lock) has NO degradation path — it wedges *every* `git push` (husky pre-push), *every* `/build` phase gate, and *every* scratch run host-wide behind a loud timeout wall. On a host that can't push, the agent can't ship the fix for the very reachability/load incidents this family exists to prevent — the bound blocks the release valve.
- A false **PASS** (bounded over-admission) is one extra concurrent suite — transient, self-limiting CPU pressure that resolves in minutes.

Therefore the least-harmful direction on *uncertainty* for THIS detector is **fail-OPEN — admit the run**. Every uncertainty branch in this spec (df misclassification, pid-reuse, corrupt holders file, provably-wedged lock, unresolvable ancestry) biases toward *admitting* a run over *wedging* the dev loop. This is the single most important divergence from the spawn cap and it is applied consistently below. (State-Not-Symbol: "unknown fails toward the least-harmful action *for that detector*" — and this detector's least-harmful is the opposite of the spawn floor's.)

**Fail-open granularity matters (round-2 finding):** fail-open is a verdict about *persistent, provable* uncertainty — never a per-poll shortcut. A single missed lock attempt is "keep polling", not "admit"; the mass-admit failure mode (20 queued roots each treating one contended lock attempt as fail-open → the exact fork-storm this spec prevents) is designed out in §2.4.

### 1.2 Prior incident this design must not repeat (2026-07-01) — root-cause is IN SCOPE

On 2026-07-01 a crashed agent server left **unreaped holders** in `host-spawn-holders.json` → **false saturation** → the tone gate (a fail-closed consumer) held → **the agent went unreachable**. The inherited "dead-holder auto-heal" did NOT fire; recovery was a manual dead-PID prune of the holders file. That incident is *this exact holder-set reclaim logic failing to self-heal in production.* Building a SECOND semaphore on the same core doubles that surface, and the test lane makes each stale holder worse (cap 1–2 vs 8 → one phantom is 50–100% of capacity; minute-scale suites vs second-scale spawns → wider stale window + more pid-reuse exposure).

**Mandatory work item of the extraction PR (Distrust Temporary Success):** root-cause WHY the 2026-07-01 auto-heal did not fire, against the real holders file and host state (the two live suspects in the foundation: `pruneDead`'s fail-closed df gate silently disabling all reclaim when `df` can't confirm host-local, and the `pidDead && heartbeatStale` conjunction making a reused pid's holder immortal). The finding is recorded in the extraction PR's description and this spec's §6 table is re-checked against it. The spawn-lane FIX (back-porting the prune lever and evaluating a spawn-lane max-hold TTL) is **out of scope for this PR** to keep the extraction reviewable — but it is NOT a loose "considered": a tracked commitment for the back-port decision is opened at build start (Close the Loop; see §4).

This design answers the incident directly with: fail-OPEN on *provable* uncertainty (§1.1), a max-hold TTL that reclaims even a pid-alive holder (§2.4), immediate reclaim of a provably-dead pid (§2.4), a first-class manual + passive prune lever (§2.6), and a durable event ledger so none of it is invisible (§2.8). §5 requires tests that specifically exercise crashed / SIGKILL / pid-reuse / hung holders on the *test* file.

## 2. Design

### 2.1 Primitive — extract a PARAMETERIZED core; do NOT reuse spawn semantics verbatim

Extract the holder-set mechanics of `src/core/hostSpawnSemaphore.ts` into a shared internal core `src/core/hostSemaphoreCore.ts`. The extraction seam is explicit (integration review M2):

- **Moves into the core:** the lock primitive (`O_CREAT|O_EXCL` + stale-lock reclaim), holders read/parse/write (atomic temp+rename), the host-local `df -P` determination, and the prune-dead pass — but the prune-dead pass takes a **ReclaimPolicy** parameter (see §2.4), not the spawn cap's hardcoded `pidDead && heartbeatStale && dfLocal`.
- **`acquire()` admission is parameterized** via an optional `admit(live, cap) => boolean` predicate. The spawn cap passes its F5 interactive/background reservation predicate; the test lane passes a plain `live.length < cap`. The F5 lane logic is NOT hoisted into the core — it stays in `hostSpawnSemaphore`'s predicate.
- **The singleton + config layer stays per-module** (`configureHostSpawnSemaphore` / `getHostSpawnSemaphore` / `_resetForTest` are duplicated thin per module — two holders files, two caps, two independently-tunable policies). It is NOT extracted (two caps can't share one singleton).
- **`hostSpawnSemaphore.ts`'s public export list AND its on-disk holders-file byte format are unchanged.** A golden/characterization test (§5) pins the spawn holders format (including the optional `lane` field's presence/absence) byte-identical pre/post extraction, and the 10 importers (4 src, 6 test) are verified against the unchanged export surface. The extraction is treated as its OWN reviewed change with its own tests — not an assumed no-op.
- **Doc-contract fidelity:** the foundation carries a doc-code contradiction (`HOLDER_STALE_MS`'s comment says pid-dead OR heartbeat-stale; `pruneDead` implements AND). The extraction fixes the comment to the AND semantics and states the pid-primary/heartbeat-secondary contract once, authoritatively, in `hostSemaphoreCore`'s ReclaimPolicy docs — the golden test pins the behavior either way.

New `src/core/hostTestRunnerSemaphore.ts` — holders file `~/.instar/host-test-runner-holders.json`, cap from `INSTAR_HOST_TEST_MAX`, the plain `admit` predicate, and the test **ReclaimPolicy** (§2.4). The test lane's holder record adds an **identity fingerprint** at acquire time — `{ pid, hostname, acquiredAt, startedAt, cmd }` where `startedAt` is the process start time (`ps -p <pid> -o lstart=`) and `cmd` a clamped copy of the command line — consumed by the §2.4 signal-corroboration gate. Two resource pools stay strictly separate (separate holders + lock files, no cross-talk); the two caps are **independent and additive**, not jointly bounded (§2.7).

*Acknowledged trade-off:* the file-lock holder-set is not an OS-native semaphore and has known subtle failure modes on exotic filesystems; it is chosen for zero dependencies and proven production history in the spawn cap. Migrating `hostSemaphoreCore` to an OS-native primitive is a possible future hardening, out of scope here.

### 2.2 Chokepoint — vitest globalSetup, with hard performance constraints

A `tests/setup/test-runner-semaphore.globalSetup.ts` is added to the `globalSetup` of all five vitest configs (`vitest.config.ts`, `vitest.integration.config.ts`, `vitest.e2e.config.ts`, `vitest.contract.config.ts`, `vitest.push.config.ts`). Only the integration config already has a `globalSetup` array (`build-dist.globalSetup.ts`) — there the semaphore is **prepended** so `setup()` acquires before the dist build and `teardown()` releases after it (globalSetup teardown runs in reverse); the slot is deliberately held through the build (acceptable at this cap). The other four are a plain add. Integration review confirmed these 5 are exhaustive (no `vitest.workspace.ts`, no programmatic `startVitest`; every `package.json` path + husky pre-push routes through one of the 5). A §5 guard-test asserts every `vitest.*.config.ts` includes the globalSetup so a future 6th config can't silently escape the bound.

**Why globalSetup and not a wrapper script as the primary chokepoint (cross-model finding):** a `guarded-vitest.mjs` wrapper only guards the `package.json` script paths that call it — a direct `npx vitest` / editor-integration / ad-hoc invocation bypasses it silently. The globalSetup rides INSIDE the config, so every invocation path that loads the config is bound, with zero script rewiring. The wrapper remains the named FALLBACK if the acquire-before-fanout invariant fails validation (item 5).

**Hard implementation constraints (each closes a round-1 finding — a naive reuse re-creates the meltdown):**

1. **Async yielding wait, never a busy-spin.** The between-poll wait MUST be `await new Promise(r => setTimeout(r, interval))`. The spawn cap's in-module `busyWaitTiny()` synchronous spin, multiplied across 20+ queued cold-starting roots, pegs every core for the whole wait — re-introducing the exact starvation this spec prevents. Only the sub-millisecond lock critical section may spin, and its deadline MUST exceed the critical-section duration. **Pinned defaults (cheap-to-change-after behind the ratified dry-run phase):** poll interval 5s ± up to 1s uniform jitter; lock-acquire deadline 250ms (≫ the sub-ms critical section).
2. **`df -P` OUTSIDE the lock.** The host-local determination is computed BEFORE taking the holders lock and passed in, so the critical section is file-read + count + write (sub-ms). (In the spawn cap a single long-lived server memoizes `df` once; here every root is fresh, so `df`-under-lock would hold the exclusive lock for up to 3s while all others spin-and-timeout.) The determination is cached to an on-disk marker at the frozen path `~/.instar/host-test-runner-dflocal.json` (`{ device, local, checkedAt }`) so cold-start roots skip the probe entirely.
3. **Jitter, not backoff, on the poll; write only on change.** The poll is cheap (0.2/s) so backoff isn't needed and would only delay noticing a freed slot; instead add ±jitter to desynchronize the thundering herd, and skip the holders rewrite on a failed acquire when prune removed nothing (the spawn cap rewrites on every failed poll → O(waiters) churn). This is the correct reading of the conformance gate's "no backoff" flag: **jitter-not-backoff**. P19 mapping, explicitly: *bounded ingress* = the hard wait ceiling caps total attempts; *breaker* = the typed-error exit at the ceiling; *cost cap* = constant-cost attempts + write-skip + bounded per-minute logging. All three brakes are present; backoff is the one substituted element and the substitution is reasoned (the poll queries capacity, not a failing target — growing the interval only delays noticing a freed slot).
4. **Throw a typed error on timeout, never `process.exit()`.** `process.exit` bypasses vitest cleanup and any prepended-after teardowns and can leak the slot; a thrown typed error is reported cleanly by vitest and leaves no holder. Confirm no slot is held on the timeout path.
5. **Acquire BEFORE worker-pool fanout — a load-bearing invariant that MUST be verified, not assumed.** The whole design rests on `globalSetup` running before vitest materializes the worker pool (and its transform/project-graph/reporter state). If a vitest version initializes any of that before `globalSetup`, the semaphore would bound only part of the blast radius. §5 carries an explicit validation item that instruments each pinned config/version to prove the slot is acquired before worker fanout; if a version cannot guarantee it, the chokepoint moves to the `guarded-vitest.mjs` wrapper (which then must ALSO be wired into husky pre-push and documented as not covering ad-hoc invocations). The spec does not ship on the *assumption* — it ships on the verified invariant or the wrapper fallback.

### 2.3 Run-class differentiation (resolves the coarse-cap tension) — with a bounded, conservative exemption

A single coarse root-count cap is caught between two real needs: the operator directive "NEVER run builders' full **suites** concurrently" argues for a very low cap on suites, while an inner-loop `npx vitest run one.test.ts` (1–2 workers, trivial load, nowhere near a fork-bomb) should not be blocked behind a full suite for up to the wait ceiling. So the globalSetup classifies the run:

- **suite-class** — the integration / e2e / contract / push configs (always full suites), AND any `vitest.config.ts` run that does not meet the strict targeted-run test below. Counted against the cap.
- **targeted-run** — a `vitest.config.ts` invocation whose argv filters are **exclusively positional, existing file paths** (each resolves to a real file on disk — not a directory, not a glob, not `-t`/`--project`/pattern flags, and at most K files, default 5). **Exempt** from the cap. The classifier is deliberately conservative because the exemption is a gameable surface (round-2 adversarial finding: `vitest run tests/` or a broad glob carries "a filter" but collects the whole suite — an agent blocked at cap=1 is exactly the actor incentivized to find that hole). Anything ambiguous classifies suite-class — misclassification toward suite-class costs one wait; toward targeted-run it re-opens the fork-storm.
- **Exempt runs are still clamped:** the same globalSetup injects a worker-pool clamp (maxWorkers ≤ 4) into targeted runs, so even a mis-classified or gamed exemption cannot fork-storm — the clamp is the compensating control for any residual classifier porosity.
- Every exemption is logged to the event ledger (§2.8) with the resolved file count, so the dry-run soak review can flag any exemption pattern that looks like cap-dodging.

This lets the suite-class cap sit at the low value the directive wants without punishing the inner loop, and it makes "bound every root" no longer the design (closing the scope decision by construction — see Frontloaded Decisions).

### 2.4 ReclaimPolicy for the test lane (parameterized, NOT the spawn defaults)

The test lane's ReclaimPolicy diverges from the spawn cap's on every axis where the workload differs:

- **Immediate reclaim of a provably-dead pid.** On a df-confirmed-local disk, a holder whose pid is dead is reclaimed IMMEDIATELY — no 5-minute `heartbeatStale` gate. That gate exists to protect a *live-but-heartbeat-quiet* spawn from reclaim; the test lane has no heartbeat-refresh thread, so a live suite is already fully protected by pid-liveness alone, and the gate would only add up-to-5-min reclaim latency after every test crash/Ctrl-C (a frequent dev event that would otherwise stall the next push).
- **Max-hold TTL (the single most important test-lane addition), with corroborated, escalating signaling.** A holder older than the ceiling (`INSTAR_HOST_TEST_TTL_MS`, default 60 min — measured against the dry-run soak's observed p99 suite+dist-build duration and raised with headroom if real runs approach it; a warning line is emitted at 80% of TTL so a legitimate long run is never killed without notice) is reclaimable EVEN IF its pid is alive. This closes the "hung-but-alive vitest root never reclaimed → two hangs wedge all host testing indefinitely" failure that pid-liveness alone cannot. **Signaling is gated by two mandatory checks (round-2 security/adversarial findings — a holder row is data, not a warrant):**
  1. **Pid sanity clamp** — the target must be an integer ≥ 2 and not the reclaimer's own pid or process group. A row carrying 0, 1, a negative, or a non-integer is treated as corrupt (quarantine path below); the slot is reclaimed WITHOUT signaling. (`process.kill(-1, …)` signals every process the user can reach — a malformed row must never become a host-wide massacre.)
  2. **Identity corroboration** — before any signal, the live process at that pid is verified against the holder's recorded fingerprint: process start time ≤ `acquiredAt` (+skew allowance) AND command line matching a vitest/node test-runner shape. A reused pid fails the start-time check deterministically; corroboration failure ⇒ reclaim the SLOT (capacity restored — the actual goal), send NO signal, log the mismatch loudly to the ledger.
  Corroborated signaling escalates: SIGTERM to the process group → bounded grace (30s) → SIGKILL, each step audited to the ledger. The slot is freed at SIGTERM time; the bounded over-admission while a SIGTERM-ignoring zombie's workers drain is accepted and stated (the SIGKILL escalation bounds it).
- **Fail-OPEN on reclaim uncertainty** (§1.1): if `df` can't confirm host-local, reclaim is disabled for the pass (runs still admit). If the holders file is corrupt/unparseable, the design **admits AND quarantines**: under the lock, the corrupt file is renamed aside to `host-test-runner-holders.corrupt-<ts>.json` for inspection (never silently discarded; only the newest 5 quarantine files are retained), a fresh empty holders file is created via the single repair path, the event is written to the ledger at WARN level, and the run is admitted. A "poison sanity ceiling" (liveHolders beyond an absurd bound) is treated as corruption and takes the same quarantine path. A **foreign-hostname holder on a df-confirmed-local disk is bogus by this design's own host-local contract** — it is dropped (not counted forever) and surfaced loudly as a likely synced-`~/.instar` signal (§2.7).
- **Wedged-lock handling (fail-open granularity — round-2 finding).** A missed lock attempt within the poll loop is "retry next poll", NEVER an admit. The fail-open admit for lock trouble fires only on *provable wedge*: the lock file exists and is older than 10s (the critical section is sub-ms, so a 10s-old lock is wedged regardless of its holder's pid-liveness) — in which case the lock is age-reclaimed (removed + retaken; paralleling the max-hold reasoning for slots) — or the lock has been continuously unavailable for the entire wait budget, in which case the run admits with a `fail-open-admit` ledger record. An admitting root that never took the lock best-effort appends a lockless **witness record** (an `O_EXCL` side-file in a `witness/` subdir) so admitted-open runs remain countable by the route and the soak review; witness files are swept by the next successful prune pass.

### 2.5 Re-entrancy — ancestry + holders cross-check, NOT a bare inherited env flag

A test can itself spawn a child vitest; without a skip it would deadlock waiting for a slot its parent holds. The spawn-boundary makes a bare inherited `process.env` flag unsafe on two counts: (a) vitest runs the *test* in a WORKER whose env is a snapshot — a `process.env` mutation made in the root's globalSetup is NOT guaranteed to reach the worker (pool/version-dependent; threads copy env unless `SHARE_ENV`), so the child's globalSetup may see no flag and deadlock against its own parent; (b) a leaked/exported flag in a persistent shell silently disables the bound host-wide (fail-open in the *wrong* place — a real second root skips).

The skip is therefore **corroborated, holder-scoped, and cross-checked** — the ancestry+holders check is the AUTHORITY; the env marker is an optimization hint:

- The marker is `INSTAR_TEST_SEMAPHORE_HELD=<root-pid>:<slot-id>`, injected into worker env explicitly via each config's `test.env` (not left to ambient inheritance), so it reliably reaches the worker that spawns the child.
- **The actual skip algorithm (works with OR without the marker):** the child walks its own ancestor pid chain (`ppid` recursion via `ps`), intersects it with the pids in the live holders file, and skips iff a live ancestor holds a current slot. When the marker is present it only narrows the check (the named pid must be in the ancestor chain AND hold a slot); a stale/foreign/leaked marker fails the cross-check and does NOT skip. A scrubbed-env child therefore still skips via pure ancestry — the §5 meta-test asserts exactly this.
- A **process-global in-memory flag** ensures one process never takes more than one slot even if multiple prepended globalSetups run in it (workspace/aggregate runs): once this process holds a slot, all further globalSetups in it skip.
- §5 meta-test spawns a REAL nested `vitest run` from inside a WORKER (not two roots) and asserts the child skips, and a second meta-test spawns a child with a SCRUBBED env and asserts the ancestry path still skips it (env alone would fail).

### 2.6 Exemptions, kill switch, and the recovery lever

- **Kill switch — env only.** `INSTAR_HOST_TEST_SEMAPHORE=off` is the SOLE chokepoint kill switch. Config `enabled:false` is NOT a chokepoint lever (the globalSetup runs in a bare test process that can't know which agent's config to read — the config lever would silently no-op, worse than no lever). The config `intelligence.testRunnerCap` exists only to tune the *route's report* and any server-launched tooling; §6 and this section state env-only rollback honestly. **An `off` skip is never silent beyond its own process:** every skip appends a ledger event (§2.8), the route surfaces a skip-reason histogram, and a sustained 100% `off` rate on a dev host grades the limiter `diverged` in the guards inventory (§4) — the 2026-06-05 lesson (a silently-disabled guard explains more incidents than a broken one) applied to this guard from day one.
- **CI exemption — hardened.** Skip only when `CI === 'true' || CI === '1'` AND a positive CI signal is present (`GITHUB_ACTIONS` / `RUNNER_OS`) — a stray local `CI` export (or `CI=false`, which is truthy) must not disable the dev-host bound. Documented assumption: GitHub-hosted ephemeral single-suite runners; a self-hosted shared runner running matrix jobs concurrently is out-of-scope (stated, not silently unbounded).
- **Watch / list exemption.** Detect via the globalSetup context `config.watch` (vitest v2+ context arg) with an argv fallback; a `vitest --watch` session skips (a long-lived interactive tool must not hold a slot for hours). This is an explicit **operator-trusted exemption, outside the bound** — NOT enforcement (a printed notice is not a cap): a watch session can still run expensive roots, and the design accepts that as an operator's deliberate long-lived tool rather than bounding it (bounding a watch session with its own small long-lived slot was considered and rejected as complexity out of proportion to the risk — watch sessions are few and operator-initiated). A collect/list invocation (`vitest list`, used by `pre-push-smoke.mjs`) is no-op'd on the same seam so a mere listing doesn't wait for or consume a slot.
- **Every skip prints one deterministic line** naming the reason (`off` / `CI` / `watch` / `list` / `reentrant` / `targeted-run` / `fail-open-admit`) AND the resolved posture (`dry-run` / `enforcing`), and appends the same to the event ledger — an inert bound is never silent (a fork-storm debugged with the guard off must be observable, and a mixed-posture host must be detectable per-run).
- **Distinct capacity-timeout signal.** The timeout throws a typed error with a DISTINCT exit code and an unmistakable message: "could not START within budget — this is NOT a test failure; N holders: [pid, age]; levers: INSTAR_HOST_TEST_SEMAPHORE=off, INSTAR_HOST_TEST_MAX". The per-minute wait line goes to STDERR and carries an active-work indicator recognized by the silence/load-stall sentinels (§2.10). CLAUDE.md tells agents a rejected push may be contention, not red tests — and that a `git push` run under an outer command timeout needs that timeout ≥ the pre-push acquire budget (a correctly-waiting push must not be killed by its own caller; see §2.9 budgets).
- **Priority/timeout split.** Interactive- and pre-push-launched runs get the longer wait budget (user-blocking, few); background `/build` verification runs get the short fail-loud budget. The class is derived from the launching config (push config, or `INSTAR_HOST_TEST_RUN_CLASS=background` set by the server when it launches build tooling), never from user input. Defaults in §2.9. Admission remains barging (no FIFO claim, §3); the starvation risk this creates for the push class at cap=1 is a named soak-review item — the ledger's would-wait data (§2.8) is collected precisely so a starvation signature (push-class waits trending toward ceiling) is visible BEFORE the enforce flip, and the pre-authorized cheap mitigation is an aging/waiter-ticket refinement or push-class priority admission, buildable post-soak without a new convergence round.
- **Recovery lever (the 2026-07-01 lesson).** `POST /test-runner-limiter/prune` forces a full reclaim pass — the surfaced action replacing hand-edits of JSON. Bearer-auth, POST-only, never on any unauthenticated allowlist, single-flight (concurrent calls coalesce), rate-limited (one forced pass per 5s), and its response enumerates what was reclaimed vs would-be-reclaimed. This route and the globalSetup acquire path are the ONLY two places persistent reclaim (row removal, corroborated signaling) happens — the GET route is pure (§2.7).

### 2.7 Observability route (parity + no-lie constraint)

`GET /test-runner-limiter` (Bearer-auth, GET-only, never on the `/health` allowlist) → `{ cap, posture, liveHolders: [{pid, hostname, acquiredAt}], available, saturated, recentEvents, skipHistogram }`, matching the `/spawn-limiter` shape decision where applicable (documented divergence: `/spawn-limiter`'s `waiters`/`acquireMs`/`waitersMax` are in-process-singleton stats the out-of-process test globalSetup doesn't share, so they are omitted rather than faked; `liveHolders` is an array here — the honest live-state read). `posture` is `'off' | 'dry-run' | 'enforcing'`, resolved through the IDENTICAL resolver the globalSetup uses (§2.9), honestly labeled as the server-process view (a shell whose env diverges is exactly what the per-run ledger posture stamps detect). A CapabilityIndex entry for the `/test-runner-limiter` prefix ships with the route (parity with `/spawn-limiter`; `instar dev:preflight` checks discoverability).

**No-lie constraint (integration H1):** the route resolves `cap` and `posture` through the IDENTICAL resolvers the globalSetup uses (env → flip-file → code-default), reading the holders file only for live state — NEVER from `intelligence.testRunnerCap.maxConcurrent` (which the enforcing process doesn't read; reporting a config cap while runs are bounded at the code-default would make the "why is my run waiting?" answer a lie). The status read is **pure — lock-free, write-free, signal-free** (read + parse + count-with-pid-liveness computed in memory for display; dead/TTL-expired holders are excluded from the reported live count as a VIRTUAL prune, but the file is never written and nothing is ever signaled from a GET). All persistent reclaim is confined to the acquire path and `POST /prune` (§2.6) — this supersedes any reading of "prune-on-read" as a physical write; a GET that could kill a process is an unacceptable surface even Bearer-authed.

**Security on the route (untrusted holder fields):** `hostname`/`id`/`cmd` are untrusted (a poisoned holders file can carry markup). They are clamped to a sane charset + length at the route projection and HTML-escaped on any render; verify whether the existing `/spawn-limiter` dashboard render escapes and fix the sibling if not. `~/.instar` is created `0o700` (owner-only holder metadata). Docs state: do NOT place `~/.instar` on a userspace-synced path (Dropbox/iCloud/Syncthing) — `df -P` reports such a home as local, and a synced holders file cross-counts foreign-hostname holders between machines (the §2.4 foreign-on-local-disk drop + loud surface is the mitigation).

### 2.8 Durable event ledger (the soak's evidence store — and the audit trail for every signal)

Every consequential decision appends one JSON line to `~/.instar/host-test-runner-events.jsonl` (frozen path; `0600`; size-bounded with rotation ~5MB; written OUTSIDE the holders lock; best-effort — a ledger write failure never blocks or fails a run). Event kinds: `would-block` (dry-run; carries the live holder set that would have blocked), `block` (enforcing), `would-signal` / `signal-term` / `signal-kill` (each with target pid, holder row, corroboration result), `reclaim-dead`, `reclaim-mismatch` (corroboration failed — slot freed, no signal), `fail-open-admit` (with cause), `skip` (with reason + posture), `targeted-exemption` (with file count), `quarantine` (corrupt file), `acquire`/`release` (with timestamps, so per-class would-wait/realized-wait distributions are derivable offline).

This single ledger is simultaneously: (a) the **soak evidence** — "zero false would-blocks over the window" is read from here, not from scrolled tmux stderr (bare test processes are ephemeral; without a durable sink the ratified flip criterion is unmeasurable); (b) the **kill audit** — a wrong SIGTERM is forensically visible (which pid, which holder row, which corroboration outcome, fired by whom), answering "loud, audited" with a mechanism the out-of-process chokepoint can actually reach (no server logger, no Bearer token needed — it's a file append); (c) the **abuse surface** — exemption gaming and `off`-lever abuse show up as ledger patterns the soak review and the route's histogram surface. The GET route serves a bounded recent tail; the ledger is machine-local like everything else here.

### 2.9 Config surface, env authority, the enforce flip, and deployment

`.instar/config.json` → `intelligence.testRunnerCap`: `{ enabled, maxConcurrent, acquireWaitMs }` (tunes the route report + server-launched tooling only; NOT the chokepoint kill switch — §2.6). Env is the chokepoint authority for tuning (test processes don't read agent config, and multiple agents on one host must agree):

- `INSTAR_HOST_TEST_MAX` — suite-class cap, code-default **1** (ratified §4).
- `INSTAR_HOST_TEST_ACQUIRE_MS` — background-class wait budget, default **120000** (2 min, fail-loud). The interactive/pre-push budget is a fixed 5× multiple (default **600000**, 10 min), overridable via `INSTAR_HOST_TEST_ACQUIRE_MS_INTERACTIVE`. Queue-depth math behind the defaults: at cap=1 the expected wait is (queue depth) × (suite + prepended dist build); the 10-min interactive ceiling covers a depth-1 queue of the p95 suite; deeper sustained queues are a signal to raise `INSTAR_HOST_TEST_MAX` to 2 (the named operator response), not to grow budgets unboundedly.
- `INSTAR_HOST_TEST_TTL_MS` — max-hold ceiling, default 3600000 (§2.4).
- `INSTAR_HOST_TEST_SEMAPHORE=off` — kill switch (§2.6).
- `INSTAR_HOST_TEST_RUN_CLASS=background` — the server-set class hint (§2.6).
- `INSTAR_TEST_SEMAPHORE_HELD=<pid>:<slot>` — internal re-entrancy marker (§2.5).

**The enforce flip is a host-local FILE, not a fleet of env exports (round-2 integration finding).** Test roots launch from heterogeneous env contexts (interactive shells, husky pre-push inheriting an agent tmux env, launchd-spawned server children that never source shell profiles) — a per-surface env var yields a split-posture host and corrupts the soak evidence. The posture authority is therefore the frozen-path marker file `~/.instar/host-test-runner-enforce.json` (`{ enforcing: true, flippedAt, by }`): one file touch flips the WHOLE host uniformly, by the same rendezvous logic as the holders file. `INSTAR_HOST_TEST_ENFORCE` remains a per-process OVERRIDE (highest precedence; `1` forces enforcing, `0` forces dry-run) for testing and emergency scoping. Resolution order: env override → flip file → code-default (dry-run). Every run stamps its resolved posture into its ledger events, so a divergent-env process is visible in the soak data.

**Migration wiring (Migration Parity):** type in `src/core/types.ts` under `intelligence`; default in `src/config/ConfigDefaults.ts` (the codebase's only-add-missing mechanism, matching the `spawnCap` treatment — no hand-written `migrateConfig` block); CLAUDE.md awareness in BOTH `src/scaffold/templates.ts` (new agents) AND `PostUpdateMigrator.migrateClaudeMd` (existing agents), including the "a rejected push may be contention" + outer-timeout guidance (§2.6); CapabilityIndex entry (§2.7). **Audience split:** the chokepoint files (5 vitest configs, the globalSetup, the core, the new semaphore module) are instar-repo SOURCE — they reach the incident population (builder/worktree hosts) by `git` pull into a checkout, NOT via `PostUpdateMigrator`; the route + config knob + CLAUDE.md awareness ship to deployed agent servers via the update/migrate path.

**Coverage honesty during rollout:** a worktree on a branch that predates this change has NO globalSetup — its suites run unbounded and invisible to the holders file until the branch rebases onto a base containing the change. On a dev host with many live worktrees this is the common case for a while; merge-to-main + normal rebase cadence is the closure path. The soak review treats old-branch runs as UNCOVERED, never as evidence about the cap.

### 2.10 The wait must not read as a hang (closes the very cascade this spec targets)

An agent `/build` session blocked in globalSetup for the wait budget shows a tmux frame with a once-a-minute line and no spinner — a plausible false-positive for the `ActiveWorkSilenceSentinel` (30-min threshold) and load-stall watchers whose false kills MOTIVATED this spec. Before shipping, the wait's frame signature is validated against the live silence/load-stall predicates; the printed wait line carries an active-work indicator those predicates recognize, OR the waiting run registers as known-blocked so no watcher kills it. Otherwise the fix re-triggers the exact layer-3 kill it targets (adversarial F10).

### 2.11 Dry-run semantics (the ratified soak posture, made precise)

While the resolved posture is dry-run (§2.9), the chokepoint performs **FULL bookkeeping with ZERO enforcement side-effects**:

- Acquire/release runs for real: holders are written, counted, and released; dead-pid reclaim and quarantine run normally (pure bookkeeping, no signals). This is load-bearing for the soak — if dry-run skipped bookkeeping, the holders file would always be empty and every would-block verdict vacuous.
- A run that WOULD block instead logs `would-block` (with the live holder set) to the ledger and **admits**.
- The TTL reclaim path logs `would-signal` and frees the stale slot but sends **NO signal** — SIGTERM/SIGKILL are armed ONLY in enforcing posture. The feature's most dangerous action must not be live while the operator has been told the posture is observe-only; the soak thereby also validates the signal-corroboration classifications (the highest-risk predicate) before any signal can fire.
- All ledger kinds flow in both postures, each stamped with the resolved posture.

## 3. What this is NOT

- NOT a change to any test's content, worker/pool settings (except the §2.3 clamp injected into exempt targeted runs), or vitest version.
- NOT a scheduler with priorities/preemption. Admission is barging (whoever wins the lock when a slot frees), bounded by the wait ceiling — **the spec makes NO FIFO fairness claim** (the holder-set primitive has no arrival ordering). Push-class starvation under barging is a named soak-review item with a pre-authorized refinement path (§2.6). A monotonic arrival ticket is a possible future refinement, explicitly out of scope.
- NOT cross-machine: holders file host-local by the `df -P` fail-closed contract. Multi-machine posture: **machine-local BY DESIGN** — a test run consumes THIS host's cores; the route needs no `?scope=pool` (nothing to aggregate). The flip file, df marker, ledger, and witness records are machine-local for the same reason. "Host-wide" means **across actors sharing one OS-user home** (`os.homedir()`); a multi-OS-user host (e.g. an operator's `git push` as `justin` vs an agent's `pnpm test` as `justin_instar_1`) is not bounded across users — stated, with single-user dev host as the covered case. The test cap and the LLM spawn cap are **additive, not jointly bounded** (worst case ≈ suite workers + 8 spawns) — the combined ceiling to sanity-check against core count.
- NOT a gate on WHAT code may run: a deterministic resource bound (counting, no content judgment), same class as the shipped spawn cap — Signal-vs-Authority compliant (no brittle blocking authority over behavior).
- NOT zero-friction at cap=1: full suites serialize, and on a busy multi-agent host that is a real, accepted throughput cost (the operator directive priced it deliberately). The ledger's wait distributions make the cost measurable; `INSTAR_HOST_TEST_MAX=2` is the named relief valve.

## 6. Rollback & failure table

**Rollback:** `INSTAR_HOST_TEST_SEMAPHORE=off` (env, immediate, no release — the sole chokepoint switch; ledger-visible, never silent). Posture rollback from "enforcing wedged something": delete the flip file (or `INSTAR_HOST_TEST_ENFORCE=0` per-process) — back to dry-run, not off. Full rollback = revert the PR. **On-disk artifacts, all inert and safe to delete after revert:** the holders file, the lock file, witness records, the df marker (`host-test-runner-dflocal.json`), the flip file (`host-test-runner-enforce.json`), the event ledger (`host-test-runner-events.jsonl`), and up to 5 quarantined corrupt files. Stale `INSTAR_HOST_TEST_*` env exports are harmless no-ops once the globalSetup is gone.

| Failure | Behavior | Direction |
|---|---|---|
| Holders file corrupt/unparseable | quarantine aside (keep newest 5) + fresh file via single repair path + WARN ledger event; admit | **fail-OPEN** (admit), loud |
| `df -P` can't confirm host-local | admit the run; reclaim disabled for that pass | **fail-OPEN** (admit) |
| Lock miss within a poll | retry next poll — NEVER an admit | bounded wait |
| Lock file older than 10s (provably wedged) | age-reclaim the lock, retake | self-heals |
| Lock continuously unavailable for the full wait budget | admit + `fail-open-admit` ledger event + witness record | **fail-OPEN** (admit), countable |
| Holder pid dead (df-local) | slot reclaimed IMMEDIATELY (no 5-min gate) | self-heals fast |
| Holder past TTL, fingerprint MATCHES, enforcing | SIGTERM group → 30s grace → SIGKILL; slot freed at SIGTERM; every step ledgered | self-heals, bounded, audited |
| Holder past TTL, fingerprint MISMATCH (pid reuse) | slot reclaimed, **NO signal**, `reclaim-mismatch` ledgered | self-heals, innocent process safe |
| Holder row pid 0 / 1 / negative / non-integer | treated as corrupt ⇒ quarantine path, **NO signal** | fail-safe |
| Holder past TTL, dry-run posture | slot freed + `would-signal` ledgered, **NO signal** | observe-only honored |
| SIGKILL'd root | pid-dead ⇒ immediate reclaim next pass | self-heals fast |
| Foreign-hostname holder on local disk | dropped + surfaced loudly (synced-home signal) | self-heals |
| Wait timeout | typed error, DISTINCT exit code, "NOT a test failure" + levers | visible, never silent |
| Re-entrant child (verified ancestor+slot) | skips acquisition | no deadlock |
| Stale/leaked HELD marker | fails ancestry+holders cross-check ⇒ does NOT skip | bound stays live |
| Ledger write fails | run proceeds normally (best-effort append) | never blocks |
| `off` kill switch exported host-wide | every run skips BUT ledgers the skip; route histogram + guards posture surface it | visible, never silent |
