# Test-Runner Concurrency Bound — a host-wide vitest semaphore (Bounded Blast Radius)

## Constitutional fit

This spec implements the **Bounded Blast Radius** constitutional standard (the same standard that mandates the fork-bomb LLM-spawn cap). An unbounded per-actor resource-spawn pattern whose host-wide aggregate is catastrophic is precisely what that standard exists to prevent; the test-runner storm is the second instance of that class (the first, LLM subprocesses, is already bounded). It also serves **Responsible Resource Usage** (OS resource hygiene) and directly closes the load-stall kill cascade documented in the 2026-07-02 respawn root-cause analysis.

**But it is NOT a verbatim copy of the spawn cap.** Round-1 convergence review established that the spawn cap's core reclaim policy and fail-direction were tuned for a *fundamentally different consumer* (one long-lived server process, second-scale holds, an OOM-catastrophe fail-direction). Reusing them unmodified would re-create the exact meltdown this feature prevents and inherit the 2026-07-01 stale-holder wedge. This spec therefore **extracts a parameterized core** and derives a *consumer-specific* policy for the test lane. §1.1 is the load-bearing section: the fail-direction inverts.

## 1. Problem

A vitest ROOT process is not one process: it forks a worker pool sized to the machine (~cores threads/forks). N concurrent roots ≈ N × cores workers competing for CPU, memory, and the filesystem. On an Instar dev host, many actors start suites independently and simultaneously:

- several agent build sessions, each running `pnpm test` / `pnpm test:integration` in its own worktree;
- every `git push` (husky pre-push runs a suite);
- `/build` phase gates and scratch verification runs;
- occasional operator-driven runs.

Nothing bounds the SUM. The existing fork-bomb spawn cap (`hostSpawnSemaphore`) bounds LLM subprocesses only — test runners ride completely outside it. 29 concurrent roots on a 16-core host ≈ 300+ runner threads: the event loops of the co-resident agent servers starve, their supervisors interpret the stall as ill-health, and healthy servers get killed. The safety mechanism becomes the outage.

This is the same *class* as the 2026-06-20 LLM fork-bomb, one layer over: an unbounded per-actor spawn pattern whose aggregate is catastrophic, needing a HOST-WIDE counting bound at a single structural chokepoint.

### 1.1 The fail-direction INVERTS from the spawn cap (the load-bearing design decision)

The spawn cap fails **closed** on uncertainty (can't confirm host-local disk → refuse to reclaim → keep counting; can't acquire → don't spawn). That is correct *for an OOM safety floor*: a delayed LLM spawn is cheap, an OOM is catastrophic, and its background consumers have a heuristic-degradation fallback when a call is shed. **None of that holds for a test cap.** The harms invert:

- A false **BLOCK** (over-counting, a stale/phantom holder, a wedged lock) has NO degradation path — it wedges *every* `git push` (husky pre-push), *every* `/build` phase gate, and *every* scratch run host-wide behind a loud timeout wall. On a host that can't push, the agent can't ship the fix for the very reachability/load incidents this family exists to prevent — the bound blocks the release valve.
- A false **PASS** (bounded over-admission) is one extra concurrent suite — transient, self-limiting CPU pressure that resolves in minutes.

Therefore the least-harmful direction on *uncertainty* for THIS detector is **fail-OPEN — admit the run**. Every uncertainty branch in this spec (df misclassification, pid-reuse, corrupt holders file, provably-wedged lock, unresolvable ancestry) biases toward *admitting* a run over *wedging* the dev loop. This is the single most important divergence from the spawn cap and it is applied consistently below. (State-Not-Symbol: "unknown fails toward the least-harmful action *for that detector*" — and this detector's least-harmful is the opposite of the spawn floor's.)

**Fail-open granularity matters (round-2 finding):** fail-open is a verdict about *persistent, provable* uncertainty — never a per-poll shortcut. A single missed lock attempt is "keep polling", not "admit"; the mass-admit failure mode (20 queued roots each treating one contended lock attempt as fail-open → the exact fork-storm this spec prevents) is designed out in §2.4.

**Honest naming (codex external finding):** because it fails OPEN, this is a **best-effort host-wide bound with fail-open recovery**, not a hard mutual-exclusion guarantee. Under a persistent lock wedge the bound is temporarily lost in the safe direction until the 10s age-reclaim fires (§2.4); the witness records make those windows auditable. Read every "bound"/"cap"/"host-wide" in this spec with that qualifier — the design deliberately prefers a brief over-admission to ever wedging the dev loop.

### 1.2 Prior incident this design must not repeat (2026-07-01) — root-cause is IN SCOPE

On 2026-07-01 a crashed agent server left **unreaped holders** in `host-spawn-holders.json` → **false saturation** → the tone gate (a fail-closed consumer) held → **the agent went unreachable**. The inherited "dead-holder auto-heal" did NOT fire; recovery was a manual dead-PID prune of the holders file. That incident is *this exact holder-set reclaim logic failing to self-heal in production.* Building a SECOND semaphore on the same core doubles that surface, and the test lane makes each stale holder worse (cap 1–2 vs 8 → one phantom is 50–100% of capacity; minute-scale suites vs second-scale spawns → wider stale window + more pid-reuse exposure).

**Mandatory work item of the extraction PR (Distrust Temporary Success):** root-cause WHY the 2026-07-01 auto-heal did not fire, against the real holders file and host state (the two live suspects in the foundation: `pruneDead`'s fail-closed df gate silently disabling all reclaim when `df` can't confirm host-local, and the `pidDead && heartbeatStale` conjunction making a reused pid's holder immortal). The finding is recorded in the extraction PR's description and this spec's §6 table is re-checked against it. The spawn-lane FIX (back-porting the prune lever and evaluating a spawn-lane max-hold TTL) is **out of scope for this PR** to keep the extraction reviewable — but it is NOT a loose "considered": a tracked commitment for the back-port decision is opened at build start (Close the Loop; see §4).

This design answers the incident directly with: fail-OPEN on *provable* uncertainty (§1.1), a max-hold TTL that reclaims even a pid-alive holder (§2.4), immediate reclaim of a provably-dead pid (§2.4), a first-class manual + passive prune lever (§2.6), and a durable event ledger so none of it is invisible (§2.8). §5 requires tests that specifically exercise crashed / SIGKILL / pid-reuse / hung holders on the *test* file.

## 2. Design

### 2.1 Primitive — extract a PARAMETERIZED core; do NOT reuse spawn semantics verbatim

Extract the holder-set mechanics of `src/core/hostSpawnSemaphore.ts` into a shared internal core `src/core/hostSemaphoreCore.ts`. The extraction seam is explicit (integration review M2):

- **Moves into the core:** the lock primitive (`O_CREAT|O_EXCL` + stale-lock reclaim), holders read/parse/write (atomic temp+rename), the host-local `df -P` determination, and the prune-dead pass — but the prune-dead pass takes a **ReclaimPolicy** parameter (see §2.4), not the spawn cap's hardcoded `pidDead && heartbeatStale && dfLocal`.
- **`acquire()` admission is parameterized** via an optional `admit(live, cap) => boolean` predicate. The spawn cap passes its F5 interactive/background reservation predicate; the test lane passes a plain `live.length < cap`. The F5 lane logic is NOT hoisted into the core — it stays in `hostSpawnSemaphore`'s predicate.
- **The singleton + config layer stays per-module** (`configureHostSpawnSemaphore` / `getHostSpawnSemaphore` / `_resetForTest` are duplicated thin per module — two holders files, two caps, two independently-tunable policies). It is NOT extracted (two caps can't share one singleton).
- **`hostSpawnSemaphore.ts`'s public export list AND its on-disk holders-file byte format are unchanged.** A golden/characterization test (§5) pins the spawn holders format (including the optional `lane` field's presence/absence) byte-identical pre/post extraction, and the 10 importers (4 src, 6 test) are verified against the unchanged export surface. The extraction is treated as its OWN reviewed change with its own tests — not an assumed no-op.
- **Doc-contract fidelity:** the foundation carries a doc-code contradiction (`HOLDER_STALE_MS`'s comment says pid-dead OR heartbeat-stale; `pruneDead` implements AND). The extraction fixes the comment to the AND semantics and states the pid-primary/heartbeat-secondary contract once, authoritatively, in `hostSemaphoreCore`'s ReclaimPolicy docs — the golden test pins the behavior either way.

New `src/core/hostTestRunnerSemaphore.ts` — holders file `~/.instar/host-test-runner-holders.json`, cap from `INSTAR_HOST_TEST_MAX`, the plain `admit` predicate, and the test **ReclaimPolicy** (§2.4). The test lane's holder record adds an **identity fingerprint** at acquire time — `{ pid, hostname, acquiredAt, startedAt, cmd }` where `startedAt` is the process start time (`ps -p <pid> -o lstart=`) and `cmd` a clamped copy of the command line — consumed by the §2.4 signal-corroboration gate. Two resource pools stay strictly separate (separate holders + lock files, no cross-talk); the two caps are **independent and additive**, not jointly bounded (§2.7).

*Acknowledged trade-off — alternatives considered (round-4, both externals):* the file-lock holder-set is not an OS-native semaphore and has known subtle failure modes on exotic filesystems. The alternatives and why each loses here: **`flock`/fcntl advisory locks** — released when ANY fd on the file closes in the same process (hazardous around vitest's fork/worker tree) and lost on NFS/synced homes in ways that still require the df-style detection this design already carries, while providing no holder METADATA (who, since when, which lane) that the TTL/corroboration/attribution machinery needs; **SQLite with busy_timeout** — buys a real mutex but adds a native dependency to the zero-dependency test path, and its busy-wait is the same polling loop in different clothes. This repo DOES ship SQLite server-side (PendingRelayStore et al.), so the objection is not "SQLite is foreign" — it is that the CHOKEPOINT must run in every bare worktree checkout, and coupling test-runnability to a native module's build health creates a brand-new wedge class (one broken `better-sqlite3` rebuild after a Node upgrade would block ALL testing host-wide — a §1.1 false-BLOCK with no degradation path), while the JSON path has zero build dependency; **POSIX named semaphores** — leak permanently on crash (the 2026-07-01 failure class with LESS visibility), carry no holder metadata, and have low macOS limits; **third-party lockfile libraries** (`proper-lockfile` etc.) — a new dependency for the same primitive `hostSpawnSemaphore` already ships with a production soak history and a byte-format the golden test pins; **a local broker daemon** — requires a running server on exactly the serverless builder/worktree population §2.6 identifies as the incident hosts. JSON+`O_EXCL`+atomic-rename is chosen for zero dependencies, holder metadata as first-class data, and continuity with the proven spawn-cap core. Migrating `hostSemaphoreCore` to an OS-native primitive is a possible future hardening, out of scope here.

*Build guidance (L12 — SafeFsExecutor funnel):* all new destructive-fs callsites this design adds (quarantine keep-newest-5 deletion, witness-record sweep, ~5MB ledger rotation, atomic lock age-reclaim) live in the `src/core` semaphore modules where `scripts/lint-no-direct-destructive.js` reaches them, routed through `SafeFsExecutor` — NEVER relocated into `tests/setup/…globalSetup.ts` (the lint may not reach it). The globalSetup stays a thin caller.

### 2.2 Chokepoint — vitest globalSetup, with hard performance constraints

A `tests/setup/test-runner-semaphore.globalSetup.ts` is added to the `globalSetup` of all five vitest configs (`vitest.config.ts`, `vitest.integration.config.ts`, `vitest.e2e.config.ts`, `vitest.contract.config.ts`, `vitest.push.config.ts`). Only the integration config already has a `globalSetup` array (`build-dist.globalSetup.ts`) — there the semaphore is **prepended** so `setup()` acquires before the dist build and `teardown()` releases after it (globalSetup teardown runs in reverse); the slot is deliberately held through the build (acceptable at this cap). The other four are a plain add. Integration review confirmed these 5 are exhaustive (no `vitest.workspace.ts`, no programmatic `startVitest`; every `package.json` path + husky pre-push routes through one of the 5). A §5 guard-test asserts every `vitest.*.config.ts` wires BOTH per-config seams — the globalSetup entry AND the shared config-eval helper (classification/clamp/reservation, §2.3/§2.5) — so a future 6th config can't silently escape the bound at either seam (round-5 integration: a config with the globalSetup but without the helper re-opens unclamped nested shelter for exactly that config), and FAILS on the introduction of any `vitest.workspace.*` file until the workspace path is reviewed.

**Why globalSetup and not a wrapper script as the primary chokepoint (cross-model finding):** a `guarded-vitest.mjs` wrapper only guards the `package.json` script paths that call it — a direct `npx vitest` / editor-integration / ad-hoc invocation bypasses it silently. The globalSetup rides INSIDE the config, so every invocation path that loads the config is bound, with zero script rewiring. The wrapper remains the named FALLBACK if the acquire-before-fanout invariant fails validation (item 5).

**Hard implementation constraints (each closes a round-1 finding — a naive reuse re-creates the meltdown):**

1. **Async yielding wait, never a busy-spin.** The between-poll wait MUST be `await new Promise(r => setTimeout(r, interval))`. The spawn cap's in-module `busyWaitTiny()` synchronous spin, multiplied across 20+ queued cold-starting roots, pegs every core for the whole wait — re-introducing the exact starvation this spec prevents. Only the sub-millisecond lock critical section may spin, and its deadline MUST exceed the critical-section duration. **Pinned defaults (cheap-to-change-after behind the ratified dry-run phase):** poll interval 5s ± up to 1s uniform jitter; lock-acquire deadline 250ms (≫ the sub-ms critical section).
2. **`df -P` OUTSIDE the lock.** The host-local determination is computed BEFORE taking the holders lock and passed in, so the critical section is file-read + count + write (sub-ms). (In the spawn cap a single long-lived server memoizes `df` once; here every root is fresh, so `df`-under-lock would hold the exclusive lock for up to 3s while all others spin-and-timeout.) The determination is cached to an on-disk marker at the frozen path `~/.instar/host-test-runner-dflocal.json` (`{ device, local, checkedAt }`) so cold-start roots skip the probe entirely. **The marker is revalidated, not trusted forever (round-2 security finding):** on each cold start the recorded `device` is compared against a fresh `fs.statSync(dir).dev` AND `checkedAt` is honored against a 24h TTL — on mismatch or expiry the `df` probe re-runs and the marker is rewritten. This closes the "home later moves onto iCloud/Syncthing/NFS → stale `local:true` permanently arms reclaim against a cross-machine holders file" hazard the whole host-local contract depends on. (An alternative — hostname-namespacing every rendezvous path, `~/.instar/<hostname>/…` — was considered and rejected: it would break the single-bound-per-host-user contract for multi-checkout hosts and change every frozen path; the cheap `statSync().dev` revalidation + the foreign-hostname drop (§2.4) are the chosen mitigation.)
3. **Jitter, not backoff, on the poll; write only on change.** The poll is cheap (0.2/s) so backoff isn't needed and would only delay noticing a freed slot; instead add ±jitter to desynchronize the thundering herd, and skip the holders rewrite on a failed acquire when prune removed nothing (the spawn cap rewrites on every failed poll → O(waiters) churn). This is the correct reading of the conformance gate's "no backoff" flag: **jitter-not-backoff**. P19 mapping, explicitly: *bounded ingress* = the hard wait ceiling caps total attempts; *breaker* = the typed-error exit at the ceiling; *cost cap* = constant-cost attempts + write-skip + bounded per-minute logging. All three brakes are present; backoff is the one substituted element and the substitution is reasoned (the poll queries capacity, not a failing target — growing the interval only delays noticing a freed slot).
4. **Throw a typed error on timeout, never `process.exit()`.** `process.exit` bypasses vitest cleanup and any prepended-after teardowns and can leak the slot; a thrown typed error is reported cleanly by vitest and leaves no holder. Confirm no slot is held on the timeout path.
5. **Acquire BEFORE worker-pool fanout — a load-bearing invariant that MUST be verified, not assumed.** The whole design rests on `globalSetup` running before vitest materializes the worker pool (and its transform/project-graph/reporter state). If a vitest version initializes any of that before `globalSetup`, the semaphore would bound only part of the blast radius. §5 carries an explicit validation item that instruments each pinned config/version to prove the slot is acquired before worker fanout; if a version cannot guarantee it, the chokepoint moves to the `guarded-vitest.mjs` wrapper (which then must ALSO be wired into husky pre-push and documented as not covering ad-hoc invocations). **Ship gate, stated plainly (round-4 codex):** failing this validation BLOCKS shipment until the wrapper fallback is wired and passing — there is no third path. The validation gate covers the CONFIG-EVAL seam identically (round-5 codex — config evaluation is not a stable concurrency-control boundary across editor integrations/config caching/future vitest versions, so the classification/clamp/reservation behaviors §2.3/§2.5 place there are proven per pinned version by the same §5 instrumentation, with the same wrapper fallback). Wrapper-as-PRIMARY was considered and rejected: it covers only the script paths that call it, silently un-bounding editor/ad-hoc invocations that the config-riding chokepoint bounds for free — a coverage regression traded for boundary stability the validation gate already buys. The wrapper's reduced coverage (ad-hoc shell `npx vitest` bypasses it) is explicitly ACCEPTED as the fallback posture because the constitutional target is the multi-actor storm, which flows through `package.json` scripts and the pre-push hook — both wrapper-covered and repo-lint-guarded (§5); ad-hoc single invocations are the documented residual either way. The spec does not ship on the *assumption* — it ships on the verified invariant or the wrapper fallback.

### 2.3 Run-class differentiation — TWO SYMMETRIC LANES, not a lane + a bespoke exemption

A single coarse root-count cap is caught between two real needs: the operator directive "NEVER run builders' full **suites** concurrently" argues for a very low cap on suites, while an inner-loop `npx vitest run one.test.ts` (1–2 workers, trivial load, nowhere near a fork-bomb) should not be blocked behind a full suite for up to the wait ceiling.

**Round-3 review (adversarial + scalability + integration + the codex external, converging) established that the earlier "exempt-and-post-count" design was the wrong shape** — its bespoke machinery (skip-then-count, a clamp at one evaluation point but lane-routing at another, an aggregate cap with no atomic reservation) generated a bug per round because it was NOT the proven holder-set primitive. The convergent fix is to stop treating targeted runs as an *exemption* and make them a **second SYMMETRIC semaphore lane** that reuses the identical, already-hardened acquire path — same lock, same atomic holder record, same throw-on-budget, same tuning-file cap authority, same reclaim policy. There is no "skip free" and no post-hoc counting; a targeted run ACQUIRES a targeted-lane slot exactly as a suite run acquires a suite-lane slot.

- **suite-class** — the integration / e2e / contract / push configs (always full suites), AND any `vitest.config.ts` run that does not meet the strict targeted test below. Acquires a **suite-lane** slot (cap `INSTAR_HOST_TEST_MAX`, default 1).
- **targeted-run** — a `vitest.config.ts` invocation whose argv filters are **exclusively positional, existing file paths** (each resolves to a real file on disk — not a directory, not a glob, not `-t`/`--project`/pattern flags, and at most K files, default 5). Acquires a **targeted-lane** slot (cap `INSTAR_HOST_TEST_TARGETED_MAX`, default 6) — a roomier lane so the inner loop isn't stuck behind the single suite slot, but STILL a bounded, throwing acquire: when the targeted lane is full for its budget the run **throws the same typed capacity-timeout as the suite lane — it NEVER fail-open-admits** (fail-open-admit is lock-wedge-ONLY per §1.1). This closes the shard-a-suite bypass (80 concurrent ≤5-file runs can't all admit — at most `TARGETED_MAX` do; the rest throw) by construction, not by a post-hoc cap. Worst-case targeted-LANE load ≈ `TARGETED_MAX × maxWorkers-clamp` ≈ 24 workers (the lane-slot term only — nested shelter under targeted slots is an additional additive term, counted in §3's budget table).
- **Single-source classification — VERIFY THE STATE, never a symbol (round-4: adversarial + security + decision-completeness + lessons-aware, converging).** Two evaluation points exist (config-eval sets the ≤4 maxWorkers clamp because only a config can reshape the pool before fanout; globalSetup does the lane acquire). They MUST NOT disagree in the dangerous direction (unclamped AND targeted-lane-routed). An earlier draft bound the two points with a per-run "clamp stamp"; round-4 review killed the stamp from four independent angles — a presence-based symbol leaks (env stamps inherit into descendant processes and persistent shells — the exact hazard §2.5 cross-checks for `INSTAR_TEST_SEMAPHORE_HELD`), strands (a marker file orphaned between config-eval and globalSetup satisfies a LATER run's check), forges, and forks an unresolved either/or for the builder. The binding is instead **direct observation of the live state**: globalSetup receives the RESOLVED config (the same context §2.6 reads `config.watch` from), so it routes to the targeted lane iff (a) its OWN strict argv classification says targeted AND (b) the resolved config's ACTUAL worker-pool bound (`maxWorkers`/pool equivalents) is ≤ 4. There is no symbol to leak or forge: a run whose final resolved pool exceeds 4 workers — because CLI pool flags outrank config values, because config-eval never clamped, for ANY reason — lands in the cap-1 suite lane BY CONSTRUCTION. The dangerous combination is impossible rather than merely tested-against. (State-Not-Symbol, applied literally.)
- **Pool-shaping argv disqualifies — the CLI-precedence bypass is closed at BOTH points (round-4 adversarial HIGH).** vitest resolves CLI options with HIGHER precedence than config values, so `npx vitest run one.test.ts --maxWorkers=32` would defeat a config-eval clamp. Two independent closures: (1) ANY pool-shaping argv flag (`--maxWorkers`, `--minWorkers`, `--pool`, `--poolOptions.*`, `--fileParallelism`, `--no-isolate`) disqualifies the run from the targeted class at BOTH evaluation points (configs are code and read `process.argv`); (2) even if a novel pool flag escapes the list, the resolved-state check above sees the >4 pool and routes suite-class anyway — the flag list is defense-in-depth, the state check is the guarantee. `VITEST_MAX_THREADS`/`VITEST_MAX_FORKS` env: config-SET values take precedence over those env vars, so the config clamp holds — §5 VERIFIES this on the pinned vitest version rather than assuming it.
- The maxWorkers ≤ 4 clamp is applied at CONFIG-EVAL time (`vitest.*.config.ts` are code; globalSetup receives an already-resolved config and per §2.2 item 5 cannot reliably re-shape a materialized pool). §5 asserts a targeted run's ACTUAL concurrent worker count ≤ 4 on the pinned vitest version; if the config seam can't guarantee it, the targeted class tightens (lower K, or targeted runs share the suite lane) rather than ship unclamped. §5's agreement matrix covers the full adversarial argv surface (positional file, directory, glob, `-t`, `--project`, mixed, AND each pool-shaping flag), plus the state check: a targeted-argv run carrying `--maxWorkers=16` must route suite-class, asserted at the routing decision AND by the meta-test's actual-worker-count measurement.
- Peak concurrent targeted-lane occupancy is a named soak-review metric (§4).
- Every targeted-lane acquire is ledgered with the resolved file count (§2.8), so the dry-run soak review can flag any classification pattern that looks like cap-dodging.

This lets the suite-class cap sit at the low value the directive wants without punishing the inner loop, and it makes "bound every root" no longer the design (closing the scope decision by construction — see Frontloaded Decisions).

### 2.4 ReclaimPolicy for the test lane (parameterized, NOT the spawn defaults)

The test lane's ReclaimPolicy diverges from the spawn cap's on every axis where the workload differs:

- **Immediate reclaim of a provably-dead pid.** On a df-confirmed-local disk, a holder whose pid is dead is reclaimed IMMEDIATELY — no 5-minute `heartbeatStale` gate. That gate exists to protect a *live-but-heartbeat-quiet* spawn from reclaim; the test lane has no heartbeat-refresh thread, so a live suite is already fully protected by pid-liveness alone, and the gate would only add up-to-5-min reclaim latency after every test crash/Ctrl-C (a frequent dev event that would otherwise stall the next push).
- **Max-hold TTL (the single most important test-lane addition) — DEFAULT is capacity-reclaim-ONLY; process-signaling is a separate, opt-in, post-soak arm.** A holder older than its ceiling is reclaimable EVEN IF its pid is alive — this closes the "hung-but-alive vitest root never reclaimed → two hangs wedge all host testing indefinitely" failure that pid-liveness alone cannot. **Round-2 review (three internal reviewers + the codex external pass, converging) established that KILLING the holder is a far riskier action than the design needs by default:** a semaphore's job is to restore *capacity*, not to execute processes, and every kill path (process-group blast radius, non-durable escalation, cross-process TTL divergence) adds a surface that can harm an innocent process. The reclaim therefore splits:
  - **Default — capacity-reclaim-only (no signal ever).** A holder past its TTL has its SLOT freed (capacity restored — the actual goal) and a loud `stale-holder-reclaimed` event is written to the ledger AND surfaced on `/guards`/the route. NO process is signaled. The over-admission while a genuine hung root's worker pool keeps draining is accepted and stated: bounded in practice by the host actor count, visible in the ledger, and the operator's lever (see below) exists if it bites. This is the shipped behavior through the entire soak and by default afterward.
  - **Opt-in — TTL process-signaling arm (`INSTAR_HOST_TEST_TTL_SIGNAL=1`, default OFF, never armed during the dry-run soak).** Only when an operator has explicitly turned this on (after the soak proved TTL classification is accurate) does reclaim escalate to killing a corroborated hung root. Even then it is gated by FOUR mandatory checks — a holder row is data, not a warrant:
    1. **Pid sanity clamp** — target must be an integer ≥ 2 and not the reclaimer's own pid/pgid. A row carrying 0, 1, negative, or non-integer is corrupt ⇒ quarantine path, slot freed, NO signal. (`process.kill(-1, …)` would signal every process the user can reach.)
    2. **Identity corroboration** — the live process at that pid must match the holder's recorded fingerprint: start time ≤ `acquiredAt` (+skew) AND a vitest/node test-runner command line. A reused pid fails the start-time check deterministically ⇒ slot freed, NO signal, `reclaim-mismatch` ledgered.
    3. **Group-leadership check (round-2 security/adversarial finding).** A group-targeted signal (`kill(-pid)`) is sent ONLY when `getpgid(pid) === pid` — the root is its own group leader, so the group is exactly its own fork tree. If it is NOT a leader (launched from a husky hook sharing `git push`'s group, a tmux pane job, an agent exec layer), signal the corroborated single PID only and ledger the downgrade — the group membership was never corroborated, so it is never the blast target. (To make group-signaling reliably available, the holding root's globalSetup best-effort `setsid`/`setpgid`s itself at acquire and records leadership in the fingerprint.)
    4. **Durable escalation (round-2 security/scalability/adversarial finding), with an independent timer for the quiesced host (round-3 adversarial finding).** SIGTERM does NOT delete the holder row. Instead the row transitions to a non-cap-counting `terminating{ signaledAt, fingerprint }` tombstone (capacity is freed for admission, but the kill obligation persists). The SIGKILL is driven by THREE completers, so it does not depend on one ephemeral process: (a) an **unref'd-but-loop-kept timer armed in the reclaiming process** for the grace window, which attempts the SIGKILL directly if that process is still alive; (b) ANY later acquire pass or `POST /prune`; (c) a lightweight **server-tick tombstone sweep** on a cadence WHERE a server exists. Each completer re-checks: grace elapsed AND fingerprint STILL corroborates AND leadership → SIGKILL the target, drop the tombstone; pid dead or fingerprint mismatch → drop the tombstone, no signal. This closes the round-2 "reclaimer exits mid-grace → SIGKILL never fires" hole. Honest residual: on a fully quiesced serverless host where the reclaiming process ALSO dies within the grace and no further test activity ever occurs, completion waits until activity resumes — bounded by activity, not unbounded time. This residual only bites an operator who armed `TTL_SIGNAL` on a quiet host (the DEFAULT reclaim never signals at all), and the CPU-draining root remains attributable in the ledger meanwhile.
  - **Per-row TTL, RE-CLAMPED on read (round-2 + round-3 adversarial findings).** The effective ceiling is stamped into the holder ROW at acquire time (`ttlMs`), read from `INSTAR_HOST_TEST_TTL_MS` (default 60 min — validated against the soak's observed p99 suite+dist-build hold, §4). Every reclaimer honors the ROW's `ttlMs` (not its own env) so a peer running the 1h default can't prematurely reclaim a run that legitimately raised its own TTL to 2h — BUT the reclaimer **re-clamps the row value to the sane host max on READ, not only at acquire**. A same-user poisoned/buggy/hand-edited row carrying `ttlMs=999999999999` parses fine and is NOT caught by the poison-sanity COUNT ceiling, so trusting it verbatim would make that slot's hung root immortal (the max-hold TTL silently disabled for that slot — the exact failure §1.2 calls the most important addition). Re-clamping on read bounds any row value by the host-max ceiling: the legitimate raise-my-own-TTL case is honored up to the ceiling, no row value can exceed it. **The ceiling is PINNED (round-4: decision-completeness + lessons-aware):** `HOST_TEST_TTL_MAX_MS` = 4 × the code-default TTL = **14400000 (4h)**, a **CODE CONSTANT** — deliberately NOT env- or tuning-file-tunable, so reclaimers can never diverge on it (a per-actor ceiling would let a divergent actor prematurely reclaim a peer's legitimately-raised row — the exact cross-actor split §2.9 exists to prevent; uniformity across checkout versions is covered by §2.9's coverage-honesty). Stated invariant: the constant must exceed any supported legitimate per-row raise (4h ≫ the 2h example). §5 asserts BOTH: a peer honors a row legitimately stamped 2h (no premature reclaim), AND a row with `ttlMs` beyond `HOST_TEST_TTL_MAX_MS` is reclaimed AT the constant, not at the row value — jointly satisfiable precisely because the ceiling is a distinct constant above the largest supported raise.
  - **The 80%-of-TTL warning** is emitted by an **unref'd timer armed in the holding root's own globalSetup** at acquire (fires against the row's re-clamped `ttlMs`, prints to its own stderr + ledger WARN); prune passes additionally ledger `approaching-ttl` as the durable backstop — the warning has a named owner, not a prose wish.
- **Fail-OPEN on reclaim uncertainty** (§1.1): if `df` can't confirm host-local, reclaim is disabled for the pass (runs still admit). If the holders file is corrupt/unparseable, the design **admits AND quarantines**: under the lock, the corrupt file is renamed aside to `host-test-runner-holders.corrupt-<ts>.json` for inspection (never silently discarded; only the newest 5 quarantine files are retained), a fresh empty holders file is created via the single repair path, the event is written to the ledger at WARN level, and the run is admitted. A "poison sanity ceiling" (liveHolders beyond an absurd bound) is treated as corruption and takes the same quarantine path. A **foreign-hostname holder on a df-confirmed-local disk is bogus by this design's own host-local contract** — it is dropped (not counted forever) and surfaced loudly as a likely synced-`~/.instar` signal (§2.7).
- **Wedged-lock handling (fail-open granularity — round-2 finding).** A missed lock attempt within the poll loop is "retry next poll", NEVER an admit. The fail-open admit for lock trouble fires only on *provable wedge*: the lock file exists and is older than 10s (the critical section is sub-ms, so a 10s-old lock is wedged regardless of its holder's pid-liveness) — in which case the lock is **race-safely** age-reclaimed — or the lock has been continuously unavailable for the entire wait budget, in which case the run admits with a `fail-open-admit` ledger record. **Race-safe age-reclaim (round-2 scalability/adversarial finding):** several queued pollers can observe the same >10s lock in one window; a naive unlink-then-retake lets B delete A's FRESH lock and both enter the critical section (a holders row is then silently lost to last-write-wins). Reclaim is therefore by ATOMIC rename of the specific observed lock (`rename(lock, lock.reclaim-<pid>-<nonce>)`) — only the single winner of the atomic rename proceeds to create the new lock; every loser sees its rename fail and re-polls. An admitting root that never took the lock best-effort appends a lockless **witness record** to a `witness/` subdir (frozen path §4) recording `{ pid, hostname, acquiredAt }` so admitted-open runs remain countable by the soak review. **Witness sweep is liveness-gated (round-2 integration finding):** a witness is swept only when its recorded pid is DEAD or its age exceeds the max-hold TTL — never unconditionally, so a still-running admitted-open suite's witness is not erased mid-run.
- **Honesty on the bound during a persistent lock wedge — with the worst case QUANTIFIED (codex external, rounds 3+4).** Witness records make fail-open admissions *auditable*; they do NOT re-impose the count while the lock stays wedged. If the lock is genuinely wedged for the full wait budget, multiple roots can admit together — the bound is temporarily LOST in the fail-open (safe) direction until the wedged lock ages out and is reclaimed. Worst-case analysis: over-admission is bounded by the number of roots that ATTEMPT within the wedge window — on this host's real actor population that is the handful of agent build sessions plus pre-push runs (single digits), NOT the 29-root storm (which was a queue draining unbounded over hours, exactly what admission normally prevents); each open-admitted root still carries its lane's per-run clamp (targeted ≤4 workers), its TTL, and a witness record, so the load is attributable and self-expiring. The original incident's kill cascade came from the watchdogs, which run in INDEPENDENT long-lived server processes — a wedged test-lane lock file does not impair them, and §2.10 separately ensures waiting runs don't read as hangs. The 10s age-reclaim bounds how long a provably-wedged lock can sustain the loss; only a lock that is continuously RE-contended-but-alive for the full budget yields the multi-admit, and that window equals the wait budget by construction. **And the loss has a lockless FLOOR — the storm ceiling (round-5 codex):** before a fail-open admit, the run counts LIVE (pid-alive) witness records; at or beyond `WEDGE_STORM_CEILING` (code constant, **8**) it REFUSES with the typed error naming the wedge + the witness count instead of admitting. This is a deliberate, narrow fail-direction flip: §1.1's harm comparison holds at small over-admission (one extra suite is minutes of CPU), but at 8+ simultaneous open-admitted suites the harm ordering inverts — that IS the meltdown this spec exists to prevent, and at that point wedging further pushes is the lesser harm. The ceiling is reachable only during a sustained lock wedge with storm-level attempt volume (never in normal operation, where the lane caps bind first), needs no lock (witness records are lockless by design), and is §5-tested + §6-tabled. This is the correct trade (a wedged-lock wedge must never block the dev loop — until the alternative is the storm itself), stated plainly rather than papered over.

### 2.5 Re-entrancy — ancestry + holders cross-check, NOT a bare inherited env flag

A test can itself spawn a child vitest; without a skip it would deadlock waiting for a slot its parent holds. The spawn-boundary makes a bare inherited `process.env` flag unsafe on two counts: (a) vitest runs the *test* in a WORKER whose env is a snapshot — a `process.env` mutation made in the root's globalSetup is NOT guaranteed to reach the worker (pool/version-dependent; threads copy env unless `SHARE_ENV`), so the child's globalSetup may see no flag and deadlock against its own parent; (b) a leaked/exported flag in a persistent shell silently disables the bound host-wide (fail-open in the *wrong* place — a real second root skips).

The skip is therefore **corroborated, holder-scoped, and cross-checked** — the ancestry+holders check is the AUTHORITY; the env marker is an optimization hint:

- The marker is `INSTAR_TEST_SEMAPHORE_HELD=<root-pid>:<slot-id>`, injected into worker env explicitly via each config's `test.env` (not left to ambient inheritance), so it reliably reaches the worker that spawns the child.
- **The actual skip algorithm (works with OR without the marker) — LANE-SCOPED (round-4 adversarial HIGH):** the child walks its own ancestor pid chain (`ppid` recursion via `ps`), intersects it with the pids in the live holders file, and skips iff a live ancestor holds a current slot **in the SAME LANE the child itself would acquire**. Lane-blindness was a cap-1 bypass: a suite-class child under a TARGETED-lane holder would have skipped the suite lane entirely — with TARGETED_MAX=6 sheltering up to 6×NESTED_MAX unclamped full-suite pools that never touch the cap-1 lane. The deadlock rationale for the skip holds only SAME-lane (a child blocking on a slot its own ancestor holds); a suite-class child under a targeted-only ancestor blocks on a slot held by UNRELATED runs — ordinary queuing that resolves — so it acquires the suite lane normally. When the marker is present it only narrows the check (the named pid must be in the ancestor chain AND hold a same-lane slot); a stale/foreign/leaked marker fails the cross-check and does NOT skip. A scrubbed-env child therefore still skips via pure ancestry — the §5 meta-test asserts exactly this.
- A **process-global in-memory flag** ensures one process never takes more than one slot even if multiple prepended globalSetups run in it (workspace/aggregate runs) — and it is **LANE-SCOPED like the ancestry skip (round-5 adversarial):** the flag records the held slot's LANE; a further globalSetup in the same process skips only SAME-lane (a targeted-first process whose later config is suite-class acquires the suite lane, or takes the nested-clamp path) — otherwise a first-evaluated targeted config would shelter a later unclamped suite-class project in the same process, the round-4 lane-blindness shape at the in-process surface. The §5 guard-test additionally FAILS on the introduction of any `vitest.workspace.*` file until the workspace path is reviewed against this flag.
- **Nested-amplification is bounded by an ATOMIC per-slot reservation (round-2 + round-3 adversarial findings).** The skip is all-or-nothing per process, so a single slot-holding root could otherwise shelter N nested full `vitest run` children (each skipping via ancestry, each fanning a full worker pool = N× amplification under one slot). A non-atomic "count nested skips then clamp the (N+1)th" is defeated by CONCURRENT nested launches (each reads count<cap and all skip free — round-3 finding). So the nested count is an ATOMIC reservation: a nested child that skips still increments a per-slot nested sub-count **under the holders lock** (a lightweight nested sub-record on the sheltering slot), and a child that atomically observes the count already at `INSTAR_HOST_TEST_NESTED_MAX` (default **2**) is CLAMPED (config-time maxWorkers ≤ 4) rather than skipping unclamped — it still skips (never blocks: blocking a nested child on a slot its ancestor holds is the deadlock the skip exists to prevent), but at ≤4 workers.
  - **The evaluation point is PINNED — the whole reserve-and-observe happens at CONFIG-EVAL time, under the holders lock (round-4 lessons-aware MEDIUM).** The clamp can only be applied at config-eval (§2.3), so a reservation observed anywhere later re-opens the two-point race this bullet exists to close: N concurrent nested launches each config-eval-reading count < cap before any sibling increments would ALL skip unclamped. Therefore the nested child performs the ancestry check AND the atomic reserve-and-observe inside its config evaluation (config files are code and may do synchronous fs/lock I/O; the §2.2 critical-section disciplines apply there identically), and its globalSetup merely CONSUMES the already-made reservation for the skip decision. Observe-and-clamp is one atomic step under the lock, at the only point that can still act on the answer.
  - **The reservation has a full LIFECYCLE, not just a create (round-5: integration + lessons-aware, converging).** `NESTED_MAX` is a CONCURRENT cap, so the reservation must release. Nested sub-records are **PID-KEYED** (one sub-record per nested child pid on the sheltering slot — idempotent, so a config re-evaluation for the same child cannot double-count), removed under the holders lock in the nested child's globalSetup **teardown** (best-effort), and **swept by any prune pass when the recorded pid is dead** (mirroring holder reclaim — capacity direction only, never a signal). Sequential nested runs therefore never burn a lifetime budget: §5 asserts the third SEQUENTIAL nested run under one slot is NOT clamped. Run-shapes that will not execute a suite reserve nothing: `off`/CI and argv-visible `--watch`/list short-circuit the config-eval reservation; a shape only detectable later leaks at most one pid-keyed record, reaped on pid-death. **Lock trouble at config-eval is pinned to the AT-CAP coercion** — a lock miss/wedge at the reservation point yields clamp-≤4 + WARN ledger, NEVER skip-unclamped (that would reopen the mass-amplification-under-contention hole the round-2 fail-open-granularity finding closed) and NEVER block (self-deadlock).
  - **A nested skip that finds NO reservation still skips — loudly (round-5 integration).** If globalSetup's skip path finds no reservation for this child (an unguarded config that omitted the config-eval helper, or a future reordering), it skips anyway (the deadlock rule leaves no alternative — clamping is no longer possible and blocking self-deadlocks) but emits `nested-skip` with `reserved:false` at WARN level, so an unguarded config shows up in the soak data and the route histogram rather than passing silently. The §5 config-list guard-test covers BOTH per-config seams (below).
  - **The sub-count is sanity-coerced on every read, under the lock (round-4 security — the ttlMs-poisoning class, applied to the field round 4 itself added).** The count must be an integer in [0, sane max]; any non-conforming value (negative, non-integer, NaN-ish, absurd) is treated as **AT-CAP** — all further nested children under that slot are clamped ≤4 workers — and a WARN is ledgered. The coercion direction is deliberately toward CLAMPING: a wrongly-clamped nested child just runs at ≤4 workers (the §2.3 safe-superset rationale), so this defense cannot wedge anything.
  - **Cross-lane honesty (round-5 lessons-aware):** the lane-scoped skip means cross-lane nested acquisition can, in a contrived population (suite holder spawning a targeted child while every targeted holder spawns suite children), form a bounded hold-and-wait cycle that no run's completion breaks — it resolves by the shortest acquire budget's typed capacity-timeout, which is the named cycle breaker. It cannot hang past that budget; a `block`/timeout ledger pattern with nested cross-lane fingerprints is the soak signature if it ever occurs in practice.

  Worst-case nested amplification under one slot is bounded to `NESTED_MAX` full pools + (excess × 4) clamped — attributable via the `nested-skip` ledger event (with sheltering root pid + slot-id) and a named soak-review signature; with the lane-scoped skip above, full-pool nested shelter exists only under the SUITE slot (a targeted ancestor shelters only targeted-class children, each already ≤4 workers). Honest scope: this CLAMPS unbounded nesting rather than BLOCKING it (blocking would deadlock); a pathological test spawning 50 concurrent nested runs still runs them, but at ≤4 workers each beyond the first two, and loudly attributed.
- §5 meta-tests: a REAL nested `vitest run` from inside a WORKER (not two roots) asserts the child skips; a SCRUBBED-env child asserts the ancestry path still skips it (env alone would fail); a nested FULL-SUITE child under a TARGETED-lane holder asserts it ACQUIRES (or waits on) the suite lane — never skips (the lane-scope test); CONCURRENT nested launches racing config-eval admit at most `NESTED_MAX` unclamped (the atomicity test); the clamped excess child's ACTUAL concurrent worker count is measured ≤ 4 (parity with the targeted-lane meta-test — a flag-level assertion is not enough); a poisoned nested sub-count fixture (negative/absurd) still clamps the next nested child.

### 2.6 Exemptions, kill switch, and the recovery lever

- **Kill switch — env only.** `INSTAR_HOST_TEST_SEMAPHORE=off` is the SOLE chokepoint kill switch. Config `enabled:false` is NOT a chokepoint lever (the globalSetup runs in a bare test process that can't know which agent's config to read — the config lever would silently no-op, worse than no lever). The config `intelligence.testRunnerCap` exists only to tune the *route's report* and any server-launched tooling; §6 and this section state env-only rollback honestly. **An `off` skip is never silent beyond its own process:** every skip appends a ledger event (§2.8), the route surfaces a skip-reason histogram, and a sustained 100% `off` rate on a dev host grades the limiter `diverged` in the guards inventory (§4) — the 2026-06-05 lesson (a silently-disabled guard explains more incidents than a broken one) applied to this guard from day one.
- **CI exemption — hardened, and graded like `off` when spoofed (round-2 adversarial finding).** Skip only when `CI === 'true' || CI === '1'` AND a positive CI signal is present (`GITHUB_ACTIONS` / `RUNNER_OS`) — a stray local `CI` export (or `CI=false`, which is truthy) must not disable the dev-host bound. Documented assumption: GitHub-hosted ephemeral single-suite runners; a self-hosted shared runner running matrix jobs concurrently is out-of-scope (stated, not silently unbounded). Because an agent could still export `CI=true GITHUB_ACTIONS=true` on a dev host to buy a labeled-innocent skip, a sustained `CI`-reason skip on a known-non-CI host is graded IDENTICALLY to sustained `off` — `diverged`, with the standard aggregated attention item.
- **Serverless-host honesty (round-3 adversarial finding).** The `diverged` GRADE and its attention item are server-side, but the chokepoint runs on bare builder/worktree checkouts that ARE the incident population and often have NO running agent server — so a server-only surface would be silent on exactly the hosts that matter (the 2026-06-05 "a silently-disabled guard explains more incidents than a broken one" lesson, unmet where it counts). Two out-of-process surfaces cover the serverless host: (a) the globalSetup ESCALATES the severity of its always-printed skip line to a loud stderr WARN specifically for the `off` and `CI-on-a-non-CI-host` reasons (a self-disable is never a quiet one-liner); (b) `instar dev:preflight` and the pre-push hook read the LOCAL ledger and WARN on a sustained self-disable pattern — **WARN-only on the pre-push hook, NEVER a push-blocking fail (round-4 security):** the detection surface for a self-disable must not itself become a new way to wedge the dev loop (§1.1's false-BLOCK inversion), and the ledger is a best-effort, concurrently-appended, same-user-writable file — its content may only ever ADD a warning, never block a push; the parser tolerates torn/malformed lines. `dev:preflight` — an advisory tool a human runs deliberately — MAY fail on the same pattern. The server-side `diverged` grade (which needs the source-of-truth "is this host a CI runner?" — a long-lived agent server knows it is not an ephemeral CI job) is the additional surface WHERE a server exists, stated honestly as such, not the only one.
- **Watch / list exemption — kept, but NEVER quiet (round-4 adversarial HIGH).** Detect via the globalSetup context `config.watch` (vitest v2+ context arg) with an argv fallback; a watch session skips (a long-lived interactive tool must not hold a slot for hours — and having its lingering watcher HOLD a suite slot would wedge the cap-1 lane, the §1.1 false-BLOCK direction, so the exemption stands rather than an initial-run acquire). But round-4 review established two ways this was the cheapest remaining cap-dodge, both now surfaced: (1) **bare `vitest` in a TTY DEFAULTS into watch mode** — agent tmux panes have TTYs, so the incident population reaches the exemption *accidentally*. A run that defaulted into watch (no explicit `--watch`/`-w` in argv) always gets the loud stderr WARN + a "use `vitest run`" hint, never the quiet one-liner. (2) Adversarially, `--watch` is a labeled-innocent full-suite skip — so ANY watch skip in an agent-launched/non-interactive context (the `INSTAR_HOST_TEST_RUN_CLASS` hint, agent env markers) also escalates to the loud WARN, watch skips are a named soak-review metric (§4), and a SUSTAINED watch-skip pattern on a dev host is graded like `off` — `diverged`, same attention item, and the same `dev:preflight`/pre-push (WARN-only) ledger pattern check as (b) above. A deliberate human `--watch` in an interactive shell keeps its quiet skip line. A collect/list invocation (`vitest list`, used by `pre-push-smoke.mjs`) is no-op'd on the same seam so a mere listing doesn't wait for or consume a slot.
- **Every skip prints one deterministic line** naming the reason (`off` / `CI` / `watch` / `list` / `reentrant` / `fail-open-admit`) AND the resolved posture (`dry-run` / `enforcing`), and appends the same to the event ledger — an inert bound is never silent (a fork-storm debugged with the guard off must be observable, and a mixed-posture host must be detectable per-run). **`targeted-run` is NOT a skip reason (round-4: three reviewers, converging):** a targeted run ACQUIRES a targeted-lane slot (§2.3) and is ledgered as an `acquire` event with run-class + file count (§2.8) — its per-run console line is the targeted-lane ACQUIRE announcement (mirroring the suite-lane acquire line), never a `skip` kind. Emitting both would double-count targeted activity in the skip histogram and pollute the §4 soak evidence.
- **Distinct capacity-timeout signal.** The timeout throws a typed error with a DISTINCT exit code and an unmistakable message: "could not START within budget — this is NOT a test failure; N holders: [pid, age]; levers: INSTAR_HOST_TEST_SEMAPHORE=off, INSTAR_HOST_TEST_MAX". The per-minute wait line goes to STDERR and carries an active-work indicator recognized by the silence/load-stall sentinels (§2.10). CLAUDE.md tells agents a rejected push may be contention, not red tests — and that a `git push` run under an outer command timeout needs that timeout ≥ the pre-push acquire budget (a correctly-waiting push must not be killed by its own caller; see §2.9 budgets).
- **Priority/timeout split.** Interactive- and pre-push-launched runs get the longer wait budget (user-blocking, few); background `/build` verification runs get the short fail-loud budget. The class is derived from the launching config (push config, or `INSTAR_HOST_TEST_RUN_CLASS=background` set by the server when it launches build tooling), never from user input. Defaults in §2.9. Admission remains barging (no FIFO claim, §3); the starvation risk this creates for the push class at cap=1 is a named soak-review item — the ledger's would-wait data (§2.8) is collected precisely so a starvation signature (push-class waits trending toward ceiling) is visible BEFORE the enforce flip, and the pre-authorized cheap mitigation is an aging/waiter-ticket refinement or push-class priority admission, buildable post-soak without a new convergence round.
- **Recovery lever (the 2026-07-01 lesson).** `POST /test-runner-limiter/prune` forces a full reclaim pass — the surfaced action replacing hand-edits of JSON. Bearer-auth, POST-only, never on any unauthenticated allowlist, single-flight (concurrent calls coalesce), rate-limited (one forced pass per 5s), and its response enumerates what was reclaimed vs would-be-reclaimed. This route and the globalSetup acquire path are the ONLY two places persistent reclaim (row removal, corroborated signaling) happens — the GET route is pure (§2.7).

### 2.7 Observability route (parity + no-lie constraint)

`GET /test-runner-limiter` (Bearer-auth, GET-only, never on the `/health` allowlist) → `{ cap, targetedCap, posture, ttlSignalArmed, liveHolders: [{pid, hostname, acquiredAt, ttlMs, state}], targetedHolders, admittedOpen: [{pid, acquiredAt}], suite: {available, saturated}, targeted: {available, saturated}, recentEvents, skipHistogram }` — availability is **PER-LANE** (round-4 integration: a singular `available`/`saturated` cannot honestly answer "why is my targeted run waiting?" when the targeted lane is full but the suite lane is free — the exact lie class the no-lie constraint below prohibits; §5's integration test asserts the targeted-lane saturation flips when ONLY that lane is full) — matching the `/spawn-limiter` shape decision where applicable (documented divergence: `/spawn-limiter`'s `waiters`/`acquireMs`/`waitersMax` are in-process-singleton stats the out-of-process test globalSetup doesn't share, so they are omitted rather than faked; `liveHolders` is an array here — the honest live-state read). `admittedOpen` is derived from live (pid-alive) witness records so a lock-wedge over-admission is *visible*, not merely ledgered (round-2 integration finding — the §2.4 "countable by the route" claim now has a backing field). `posture` is `'off' | 'dry-run' | 'enforcing'` and `ttlSignalArmed` reflects `INSTAR_HOST_TEST_TTL_SIGNAL`, both resolved through the IDENTICAL resolver the chokepoint uses (§2.9), honestly labeled as the server-process view (a shell whose env diverges is exactly what the per-run ledger posture stamps detect). A CapabilityIndex entry for the `/test-runner-limiter` prefix ships with the route (parity with `/spawn-limiter`; `instar dev:preflight` checks discoverability).

**No-lie constraint (integration H1):** the route resolves `cap` and `posture` through the IDENTICAL resolvers the globalSetup uses (env → flip-file → code-default), reading the holders file only for live state — NEVER from `intelligence.testRunnerCap.maxConcurrent` (which the enforcing process doesn't read; reporting a config cap while runs are bounded at the code-default would make the "why is my run waiting?" answer a lie). The status read is **pure — lock-free, write-free, signal-free** (read + parse + count-with-pid-liveness computed in memory for display; dead/TTL-expired holders are excluded from the reported live count as a VIRTUAL prune, but the file is never written and nothing is ever signaled from a GET). All persistent reclaim is confined to the acquire path and `POST /prune` (§2.6) — this supersedes any reading of "prune-on-read" as a physical write; a GET that could kill a process is an unacceptable surface even Bearer-authed.

**Security on the route (untrusted holder fields):** `hostname`/`id`/`cmd` are untrusted (a poisoned holders file can carry markup). They are clamped to a sane charset + length at the route projection and HTML-escaped on any render; verify whether the existing `/spawn-limiter` dashboard render escapes and fix the sibling if not. `~/.instar` is created `0o700` (owner-only holder metadata). Docs state: do NOT place `~/.instar` on a userspace-synced path (Dropbox/iCloud/Syncthing) — `df -P` reports such a home as local, and a synced holders file cross-counts foreign-hostname holders between machines (the §2.4 foreign-on-local-disk drop + loud surface is the mitigation).

### 2.8 Durable event ledger (the soak's evidence store — and the audit trail for every signal)

Every consequential decision appends one JSON line to `~/.instar/host-test-runner-events.jsonl` (frozen path; `0600`; size-bounded with rotation ~5MB; written OUTSIDE the holders lock; best-effort — a ledger write failure never blocks or fails a run). Event kinds: `would-block` (dry-run; carries the live holder set that would have blocked), `block` (enforcing), `stale-holder-reclaimed` (the DEFAULT TTL action — slot freed, NO signal), `approaching-ttl` (80%-of-TTL backstop), `signal-term` / `signal-kill` (only under the opt-in TTL-signal arm; each with target pid, holder row, corroboration + group-leadership result), `reclaim-dead`, `reclaim-mismatch` (corroboration failed — slot freed, no signal), `fail-open-admit` (with cause), `skip` (with reason + posture), `nested-skip` (with sheltering root pid + slot + `reserved:true|false` — `false` flags an unguarded config, §2.5), `would-clamp` (dry-run; targeted and nested-excess variants with the natural resolved pool bound, §2.11), `schema-unknown` (an unknown-state row excluded from the cap count, §2.9), `tuning-changed` (first observation of a tuning-file content-hash change, with the differing fields old → new, §2.9), `quarantine` (corrupt file), `acquire`/`release` (with timestamps + the resolved `cap`/`targetedCap` + run-class + the resolved targeted file count when run-class is targeted + the tuning-file content hash, so per-class would-wait/realized-wait distributions AND cap/posture/tuning divergence are derivable offline — round-2 integration finding: divergence in the effective cap must be as visible in the soak data as posture divergence).

**Warning-fatigue bound (round-5 codex):** every loud stderr WARN in this design is once-per-condition-per-RUN by construction (each is emitted at one decision point in a short-lived root process — there is no loop that re-prints it), the once-a-minute wait line is the only recurring line and is already rate-fixed, the ledger is rotation-bounded (~5MB), and the route's histogram — not raw stderr — is the aggregation surface for patterns. A busy host's steady state is therefore quiet-by-default with one line per consequential event, not a warning wall.

This single ledger is simultaneously: (a) the **soak evidence** — "zero false would-blocks over the window" is read from here, not from scrolled tmux stderr (bare test processes are ephemeral; without a durable sink the ratified flip criterion is unmeasurable); (b) the **kill audit** — a wrong SIGTERM is forensically visible (which pid, which holder row, which corroboration outcome, fired by whom), answering "loud, audited" with a mechanism the out-of-process chokepoint can actually reach (no server logger, no Bearer token needed — it's a file append); (c) the **abuse surface** — classification gaming and `off`-lever abuse show up as ledger patterns the soak review and the route's histogram surface. The GET route serves a bounded recent tail; the ledger is machine-local like everything else here.

### 2.9 Config surface, env authority, the enforce flip, and deployment

`.instar/config.json` → `intelligence.testRunnerCap`: `{ enabled, maxConcurrent, acquireWaitMs }` (tunes the route report + server-launched tooling only; NOT the chokepoint kill switch — §2.6). Env is a per-process tuning OVERRIDE; the host-uniform authority for EVERY lever that must agree across all actors (the three caps — suite, targeted, nested — plus posture and the signal arm) is the host-local tuning file (below). (The TTL host-max ceiling is deliberately NOT among them — it is a code constant, §2.4.)

- `INSTAR_HOST_TEST_MAX` — suite-lane cap, per-process override; host-uniform authority is the tuning file; code-default **1** (ratified §4).
- `INSTAR_HOST_TEST_TARGETED_MAX` — targeted-lane cap, per-process override; host-uniform authority is the tuning file; default **6** (§2.3).
- `INSTAR_HOST_TEST_NESTED_MAX` — per-slot concurrent-nested cap, per-process override; host-uniform authority is the tuning file; default **2** (§2.5).
- `INSTAR_HOST_TEST_ACQUIRE_MS` — background-class SUITE-lane wait budget, default **120000** (2 min, fail-loud). The interactive/pre-push budget resolves as: `INSTAR_HOST_TEST_ACQUIRE_MS_INTERACTIVE` if set, ELSE 5 × the resolved background budget (env or default; note 5×120000 already = the 600000 default — the third clause is a harmless floor). Queue-depth math: at cap=1 the expected wait is (queue depth) × (suite + prepended dist build); the interactive ceiling must cover a depth-1 queue of the p95 suite HOLD duration — an assumption §4 validates against real ledger data before the flip, not an assertion (round-2 scalability finding).
- `INSTAR_HOST_TEST_TARGETED_ACQUIRE_MS` — TARGETED-lane wait budget, default **60000** (1 min, fail-loud/THROW — §2.3; a targeted run whose lane is full throws the typed capacity-timeout, never fail-open-admits).
- `INSTAR_HOST_TEST_TTL_MS` — max-hold ceiling, default 3600000; stamped per-row at acquire (§2.4).
- `INSTAR_HOST_TEST_TTL_SIGNAL=1` — arms the opt-in process-signaling reclaim (default OFF — the DEFAULT reclaim never signals; §2.4).
- `INSTAR_HOST_TEST_SEMAPHORE=off` — kill switch (§2.6).
- `INSTAR_HOST_TEST_ENFORCE` — per-process posture override (§ below).
- `INSTAR_HOST_TEST_RUN_CLASS=background` — the server-set class hint (§2.6).
- `INSTAR_TEST_SEMAPHORE_HELD=<pid>:<slot>` — internal re-entrancy marker (§2.5).

**Host-uniform levers live in a FILE, not a fleet of env exports — for EVERY cross-actor lever, not just the suite cap (round-2 + round-3 integration findings).** Test roots launch from heterogeneous env contexts (interactive shells, husky pre-push inheriting an agent tmux env, launchd-spawned server children that never source shell profiles) — a per-surface env var yields a SPLIT host and corrupts the soak evidence: "multiple agents must agree" is willpower, not structure, if `INSTAR_HOST_TEST_MAX=2` only reaches the shell that exported it (leaving launchd/tmux descendants enforcing cap=1 against the same holders file — the bound splits into per-surface sub-bounds). This applies EQUALLY to every lever that is enforced against a SHARED holders lane: posture, the suite cap, the **targeted cap**, the **nested cap**, and the signal arm are all cross-actor (round-3 finding: a lone `INSTAR_HOST_TEST_TARGETED_MAX=100` export makes one actor keep admitting targeted runs while others enforce 6 — the effective host cap becomes the MAX across actors, re-uncapping the shard storm). The host-uniform authority is therefore ONE frozen-path tuning file `~/.instar/host-test-runner-tuning.json` (`{ v, enforcing?, maxConcurrent?, targetedMax?, nestedMax?, ttlSignal?, flippedAt, by }`): one file write sets any of them for the WHOLE host uniformly. The matching env vars remain highest-precedence PER-PROCESS overrides for testing/emergency scoping. Resolution order for each: env override → tuning file → code-default. Every run stamps its resolved posture AND all resolved caps into its ledger events, so a divergent-env process is visible in the soak data.

**Cap values are sanity-clamped on read, and cap-inflation is as LOUD as `off` (round-4 adversarial + security).** A lone `INSTAR_HOST_TEST_MAX=50` export (or an absurd tuning-file value) is the quiet twin of the kill switch — the effective host cap is the max across actors, yet it produces no skip event and dodges every `off`-keyed surface. Three closures: (1) tuning-file cap values are read through **PINNED sanity ceilings — code constants, 4× each cap's code-default (the `HOST_TEST_TTL_MAX_MS` precedent): suite ≤ 4, targeted ≤ 24, nested ≤ 8** — an out-of-range/non-integer value resolves to the code-default and ledgers a WARN (round-5 adversarial: an unpinned "sane max" would make the ceiling itself the unreviewed number); (2) the chokepoint prints the loud stderr WARN whenever its RESOLVED cap exceeds the host-uniform authority (tuning file, else code-default) by more than 4× — an env override stays legal for emergency scoping but is never silent beyond that factor; (3) `dev:preflight` and the pre-push hook (WARN-only, §2.6) flag a sustained resolved-cap-divergence pattern from the ledger's per-run cap stamps, and beyond-sane resolved caps join the server-side `diverged` grading where a server exists.

**Posture-weakening gets the IDENTICAL loudness treatment (round-5 adversarial MEDIUM — the quiet twin one slot over).** A per-process `INSTAR_HOST_TEST_ENFORCE=0` while the host authority says enforcing admits past the cap with full bookkeeping — so it is surfaced exactly like cap-inflation: the chokepoint prints the loud stderr WARN whenever its resolved POSTURE is WEAKER than the host-uniform authority; posture-divergence joins the `dev:preflight`/pre-push (WARN-only) sustained-pattern list AND the server-side `diverged` grading. And the tuning file — the enforcement AUTHORITY — gets integrity handling at least as strong as the holders file it governs: an unparseable/corrupt tuning file is **quarantined aside + WARN-ledgered + resolved to code-defaults** (never a silent fallback; §6 row), so a deleted or mangled authority cannot silently revert the whole host to dry-run.

**Tuning-file mutations are structurally visible — the announce rule is a mechanism, not prose (round-5 adversarial MEDIUM).** Every acquire/skip ledger event carries a short **content hash of the resolved tuning file**; the first run that observes the hash CHANGED from the previous ledger entry emits a `tuning-changed` ledger event + the loud stderr WARN (naming which fields differ, old → new). So ANY edit — enforce flip, cap raise, ttlSignal arm, hand edit — is detected by the next run on the host, regardless of who wrote it or whether they self-reported in `by`/`flippedAt`. Where a server exists, `/guards` and the route surface an armed `ttlSignal` and any above-code-default cap as distinct rows (change-detection, not just current-state). Arming `ttlSignal` via the tuning file — the strictly more dangerous lever — is additionally ANNOUNCED to the operator the same way the enforce flip is (§4's flip announcement), with the hash mechanism as the structural backstop when the announce step is skipped.

**Rendezvous schema evolution — versioned + tolerant from day one (round-4 integration; supersedes any blanket freeze reading).** The holders file, each holder row, witness records, and the tuning file all carry a schema-version field (`v: 1`) from the first ship, and every reader/writer follows the tolerant rule: (a) unknown top-level and per-row FIELDS are PRESERVED VERBATIM on any rewrite and never treated as corruption; (b) a row whose `state` is unrecognized is EXCLUDED from the cap count (the §1.1 fail-open direction), never signaled, and preserved on rewrite — **but preservation is BOUNDED, and exclusion is LOUD (round-5 adversarial + codex):** each excluded row ledgers a `schema-unknown` WARN (visible on the route histogram and in the preflight checks — a mixed-version host that is silently over-admitting shows up in the soak data), unknown-state rows still COUNT toward §2.4's poison-sanity ceiling (they occupy file bytes, so they must trip the bloat guard), and a row is preserved verbatim only while its pid is ALIVE or it is younger than `HOST_TEST_TTL_MAX_MS` — a dead-pid row older than the pinned ceiling is droppable by ANY reader (WARN-ledgered, never signaled; nothing legitimate outlives pid-death plus the 4h ceiling by this design's own rules, so unbounded preservation buys no correctness); (c) unknown tuning-file fields are ignored without invalidating the known ones. This is what makes mixed-version hosts — a multi-worktree dev host is GUARANTEED to run mixed checkouts for weeks after any change — safe by construction instead of by luck: an older actor can neither strip a newer actor's fields (breaking e.g. a tombstone's fingerprint) nor false-saturate on a state it doesn't know. The §2.6 pre-authorized post-soak refinement (waiter tickets / priority admission) MUST be additive under this rule, or bump `v` with a coordinated same-host cutover. §5 carries the round-trip test (a future-versioned row with unknown fields/state survives a prune pass intact and does not count toward admission).

**Migration wiring (Migration Parity):** type in `src/core/types.ts` under `intelligence`; default in `src/config/ConfigDefaults.ts` (the codebase's only-add-missing mechanism, matching the `spawnCap` treatment — no hand-written `migrateConfig` block); CLAUDE.md awareness in BOTH `src/scaffold/templates.ts` (new agents) AND `PostUpdateMigrator.migrateClaudeMd` (existing agents), including the "a rejected push may be contention" + outer-timeout guidance (§2.6); CapabilityIndex entry (§2.7). **Audience split:** the chokepoint files (5 vitest configs, the globalSetup, the core, the new semaphore module) are instar-repo SOURCE — they reach the incident population (builder/worktree hosts) by `git` pull into a checkout, NOT via `PostUpdateMigrator`; the route + config knob + CLAUDE.md awareness ship to deployed agent servers via the update/migrate path.

**Coverage honesty during rollout:** a worktree on a branch that predates this change has NO globalSetup — its suites run unbounded and invisible to the holders file until the branch rebases onto a base containing the change. On a dev host with many live worktrees this is the common case for a while; merge-to-main + normal rebase cadence is the closure path. The soak review treats old-branch runs as UNCOVERED, never as evidence about the cap.

### 2.10 The wait must not read as a hang (closes the very cascade this spec targets)

An agent `/build` session blocked in globalSetup for the wait budget shows a tmux frame with a once-a-minute line and no spinner — a plausible false-positive for the `ActiveWorkSilenceSentinel` (30-min threshold) and load-stall watchers whose false kills MOTIVATED this spec. Before shipping, the wait's frame signature is validated against the live silence/load-stall predicates; the printed wait line carries an active-work indicator those predicates recognize, OR the waiting run registers as known-blocked so no watcher kills it. The builder picks whichever mechanism the validation proves out, PREFERRING the active-work indicator (zero new coupling); the known-blocked registration is the fallback. Otherwise the fix re-triggers the exact layer-3 kill it targets (adversarial F10).

### 2.11 Dry-run semantics (the ratified soak posture, made precise)

While the resolved posture is dry-run (§2.9), the chokepoint performs **FULL bookkeeping with ZERO enforcement side-effects**:

- Acquire/release runs for real: holders are written, counted, and released; dead-pid reclaim and quarantine run normally (pure bookkeeping, no signals). This is load-bearing for the soak — if dry-run skipped bookkeeping, the holders file would always be empty and every would-block verdict vacuous.
- A run that WOULD block instead logs `would-block` (with the live holder set) to the ledger and **admits**.
- The TTL reclaim frees the stale slot and logs `stale-holder-reclaimed` in BOTH postures — because the DEFAULT reclaim is capacity-only and never signals anyway (§2.4), the dry-run/enforcing distinction for the TTL path is now narrow: it governs only whether a `would-block` ADMITS vs BLOCKS. Process-signaling is doubly gated — it requires BOTH enforcing posture AND the separate `INSTAR_HOST_TEST_TTL_SIGNAL=1` arm — and that arm is never turned on during the soak, so the soak validates TTL classification (via `stale-holder-reclaimed` frequency and any `reclaim-mismatch`) with zero risk of a wrong kill.
- **The config-eval clamps are dry-run-honest too (round-5 lessons-aware MEDIUM):** in dry-run the targeted-class maxWorkers ≤ 4 clamp (§2.3) and the nested-excess clamp (§2.5) do NOT reshape the real pool — the run executes with its natural worker count, and the chokepoint ledgers `would-clamp` (targeted and nested-excess variants, carrying the natural resolved pool bound) instead. Lane classification still computes the WOULD-state (targeted argv + would-clamp) so `acquire` events carry meaningful run-class soak data (`wouldClamp: true`). "Log-only" therefore means genuinely zero behavior change on the incident population before the enforce flip; the clamp MACHINERY is validated pre-flip by the §5 test-time clamp validation plus the §4(g) would-clamp review. Nested sub-record bookkeeping (create/teardown/sweep) runs identically in both postures — it is bookkeeping, not enforcement.
- All ledger kinds flow in both postures, each stamped with the resolved posture AND resolved cap.

## 3. What this is NOT

- NOT a change to any test's content or vitest version (the §2.3 config-time maxWorkers clamp on targeted-lane runs is the sole pool-settings touch).
- NOT a guarantee that the bound holds through a persistent lock wedge: fail-open admits multiple roots while a lock stays wedged (§2.4), so the bound is temporarily LOST in the safe direction until the 10s age-reclaim fires. Witness records make those admissions auditable (visible in `admittedOpen`), not re-counted. This is the deliberate trade — a wedged lock must never wedge the dev loop.
- NOT an absolute "targeted runs cannot fork-storm": the per-run maxWorkers≤4 clamp bounds ONE run; the aggregate is bounded by the targeted LANE's atomic cap `INSTAR_HOST_TEST_TARGETED_MAX` (§2.3), so worst-case targeted-lane load is ~`TARGETED_MAX × 4` workers, not unlimited. The arithmetic (N slots × 4) is stated so the lane cap's value can be sized against core count.
- NOT a scheduler with priorities/preemption. Admission is barging (whoever wins the lock when a slot frees), bounded by the wait ceiling — **the spec makes NO FIFO fairness claim** (the holder-set primitive has no arrival ordering). Push-class starvation under barging is a named soak-review item with a pre-authorized refinement path (§2.6). A monotonic arrival ticket is a possible future refinement, explicitly out of scope.
- NOT cross-machine: holders file host-local by the `df -P` fail-closed contract. Multi-machine posture: **machine-local BY DESIGN** — a test run consumes THIS host's cores; the route needs no `?scope=pool` (nothing to aggregate). The flip file, df marker, ledger, and witness records are machine-local for the same reason. "Host-wide" means **across actors sharing one OS-user home** (`os.homedir()`); a multi-OS-user host (e.g. an operator's `git push` as `justin` vs an agent's `pnpm test` as `justin_instar_1`) is not bounded across users — stated, with single-user dev host as the covered case. The test cap and the LLM spawn cap are **additive, not jointly bounded** — and the combined worst case is stated as a BUDGET TABLE (round-4 codex) so the sum is a reviewed number, not an implication. Worst legal concurrent load at defaults on a 16-core host:

  | Term | Source | Worst case |
  |---|---|---|
  | Suite lane | cap 1 × full pool | ~16 workers |
  | Nested shelter under the suite slot | `NESTED_MAX` 2 × full pool + excess × 4 (§2.5; lane-scoped — FULL-pool shelter exists ONLY under the suite slot) | ~32 workers + 4/excess |
  | Targeted lane | `TARGETED_MAX` 6 × ≤4 workers | ~24 workers |
  | Nested shelter under targeted slots | `TARGETED_MAX` 6 × `NESTED_MAX` 2 × ≤4 workers + excess × 4 (§2.5 — separate roots holding no lane slot; each ≤4, but ADDITIVE — round-5 scalability + lessons-aware) | ~48 workers + 4/excess |
  | LLM spawn cap | 8 subprocesses (mostly IO-bound) | ~8 procs |

  Sum ≈ **120 worker threads (zero-excess) + excess tails at ≤4 each + 8 procs** — oversubscribed on 16 cores, deliberately: the design's goal is preventing the UNBOUNDED storm (29 roots ≈ ~300 workers + watchdog kill cascade), not fitting inside core count; the worst case above requires every lane and every nested budget simultaneously maxed (never yet observed even in the incident), is transient, self-limiting, fully ledger-attributed, and every term has a named trim lever (`INSTAR_HOST_TEST_MAX`, `NESTED_MAX`, `TARGETED_MAX`, the spawn cap) sized per-host via the tuning file. If the soak shows the corrected sum running hot, the pre-authorized trim is `nestedMax: 1` in the tuning file (halves both shelter terms, one write).
- NOT a gate on WHAT code may run: a deterministic resource bound (counting, no content judgment), same class as the shipped spawn cap — Signal-vs-Authority compliant (no brittle blocking authority over behavior).
- NOT zero-friction at cap=1: full suites serialize, and on a busy multi-agent host that is a real, accepted throughput cost (the operator directive priced it deliberately). The ledger's wait distributions make the cost measurable; `INSTAR_HOST_TEST_MAX=2` is the named relief valve.

## 6. Rollback & failure table

**Rollback:** `INSTAR_HOST_TEST_SEMAPHORE=off` (env, immediate, no release — the sole chokepoint switch; ledger-visible, never silent). Posture rollback from "enforcing wedged something": clear `enforcing` in the tuning file (or `INSTAR_HOST_TEST_ENFORCE=0` per-process) — back to dry-run, not off. The TTL signal arm is off by default and independently clearable (`INSTAR_HOST_TEST_TTL_SIGNAL` / tuning file). Full rollback = revert the PR. **On-disk artifacts, all inert and safe to delete after revert:** the holders file, the lock file (`host-test-runner-holders.lock`), the witness dir (`host-test-runner-witness/`), the df marker (`host-test-runner-dflocal.json`), the tuning file (`host-test-runner-tuning.json`), the event ledger (`host-test-runner-events.jsonl`), and up to 5 quarantined corrupt files.
