# ELI16 — Threadline Store-Split Fix (relay-spawn inbox split)

When two of my agents talk to each other, every incoming message is supposed to be written into two record books: a per-conversation logbook and a separate "inbox" file. A security check added a while back verifies every reply by looking ONLY in the inbox file — but each of the three delivery routes writes exactly ONE of the two books, and only one route writes the inbox. So an agent would try to reply to a message it genuinely received, pointing honestly at that exact message, and the server would refuse with "that message doesn't exist" — even though it's sitting right there in the logbook. In the worst case (observed live, on both agents) a session ends up with ZERO ways to reply at all and has to tear open a brand-new conversation just to get a message out. Two agents (Echo and sagemind) reproduced this against each other for two days and agreed on the diagnosis down to the exact lines of code.

## The fix, in plain words

Four pieces, one PR:

1. **Every route writes both books, through one shared funnel** — and every entry is filed under the conversation the security guard actually routed it to, not the one the sender merely claimed (that ordering flaw would otherwise let a hostile sender plant valid-looking entries in someone else's conversation).
2. **The reply check reads both books** — with an honest trust story: the inbox is cryptographically keyed, the logbook's chain is not, so going forward logbook entries get the same key, and only the frozen set of pre-fix entries is accepted on looser terms (an explicitly recorded, shrinking bridge). Refusals also become specific ("that id isn't in either book" vs "a book was unreadable") instead of one ambiguous error.
3. **Stop minting ghost conversations** — the guard was comparing a nickname against a fingerprint and concluding "stranger!", quarantining known peers' messages into fresh empty threads. Identity now resolves through a trust-ordered chain (mutually-verified pairings first, cryptographic handshakes second, the easily-poisoned local registry only as corroboration), anything ambiguous still isolates, and a signed-but-foreign sender can no longer walk into someone else's live conversation (a real hole the review closed). This piece must ship together with piece 2 or replies break in a new way — one PR enforces that.
4. **Stop duplicate re-sends** — retries now re-send the ORIGINAL message bytes under the ORIGINAL id (so the receiver recognizes them instead of answering twice), receipts are never retried, and a narrow, time-boxed body-fingerprint check catches old-style duplicates from not-yet-updated peers without ever eating legitimate short messages like "done".

## What the review changed

The nine-reviewer round (six internal + GPT-5.5 + Gemini externally + the constitution gate) found 24 material issues in the first draft. The big ones: the two books have genuinely different tamper-resistance (fixed with the keyed-going-forward + frozen-bridge design), entries were recorded before the security guard ran (fixed with post-guard filing), the duplicate-detector as first drafted could never catch what it aimed at and could swallow real messages (redesigned: scoped, time-boxed, length-floored, always audited), reply checks could have become slow whole-file scans on every send (now bounded O(1) lookups), and my "this only widens acceptance" rollout claim was simply false — one old workaround gets refused after the update, and the error text now tells the sender exactly how to adapt in one round trip.

## Decisions already made (frontloaded — nothing waits on you mid-build)

All review questions were resolved into seventeen recorded decisions in the spec, including: no retroactive backfill of old data (the union read covers history); one safe-direction emergency switch for the identity change (flipping it can only over-isolate, never over-trust); and shipping live with no dark flag because a correctness repair of an always-broken path helps nobody while dark.
