# Dashboard Throughput — plain-English overview

Instar’s dashboard already shows whether work is running, but it did not show how delivery changes over time. This feature adds a Throughput tab that answers that question from merged pull requests. It shows seven charts: merges, time from opening to merge, requested-change loops, reverts, commit iterations, changed lines, and one composite throughput index. The operator can switch between Codey, Echo, and the whole team and can look at 7, 14, or 30 days.

The numbers come from GitHub’s structured pull-request data. The server groups merged pull requests by Pacific day and author, computes exact medians after collecting the full window, and counts requested-change reviews, revert titles, commits, and lines changed. Unknown authors remain an explicit “other” group instead of being attributed to Codey or Echo. The composite uses the prototype’s documented weights: 40% volume, 20% speed, 25% quality, and 15% output. Those weights are returned with the response so consumers can see the formula that produced the result.

This is observation only. A high or low score cannot approve, block, prioritize, notify, merge, or otherwise act on work. GitHub is never mutated. If GitHub data cannot be read or parsed, the route reports that the series is unavailable instead of drawing a believable chart full of zeroes. Queries are bounded to the three supported windows, 400 merged pull requests, batches of 50 detail queries, and 30 reviews per pull request.

The dashboard uses the shared fluid panel contract rather than a narrow fixed-width content root. On a wide screen, metric cards use the available space and the composite index spans the full grid. At phone width, the cards become a single readable column and the author/window selectors remain available without horizontal scrolling. Completion evidence includes automated contract and dashboard-registry tests, TypeScript validation, repository invariants, and real browser captures at 2560 and 390 pixels.
